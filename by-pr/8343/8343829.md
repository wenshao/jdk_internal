## 背景

### 原始实现的问题

**JDK-8343829 之前**，十进制和十六进制解析有**重复代码**：

```java
// FloatingDecimal.java - 两个独立的解析方法

// 十进制解析
static FloatingDecimal parseDecimal(String s) {
    // ... 200+ 行代码
    // 解析十进制字符串
}

// 十六进制解析
static FloatingDecimal parseHexString(String s) {
    // ... 150+ 行代码
    // 解析十六进制字符串
    // 大部分逻辑与 parseDecimal 重复！
}
```

**问题**：
1. **代码重复** - 两个方法有大量相似逻辑
2. **维护困难** - 修复需要同时改两处
3. **不一致风险** - 可能产生不同的解析行为

---

## 核心变更

### 设计原则

**统一解析入口**：

```
之前:
parseDecimal(String s)  → 十进制解析逻辑
parseHexString(String s) → 十六进制解析逻辑

之后:
parseDecimal(String s, int isHex)  → 统一解析逻辑
```

### 代码变更

#### 1. 统一解析方法

**变更前**:
```java
// 十进制解析
static AsciiStringCodec parseDecimal(String s) {
    // 检查前缀
    boolean isNegative = false;
    int index = 0;
    char c = s.charAt(index);
    if (c == '-') {
        isNegative = true;
        index++;
    } else if (c == '+') {
        index++;
    }
    // ... 解析数字
}

// 十六进制解析 (重复逻辑!)
static AsciiStringCodec parseHexString(String s) {
    // 检查前缀
    boolean isNegative = false;
    int index = 2;  // 跳过 "0x"
    char c = s.charAt(1);
    if (c == '-') {
        isNegative = true;
        index = 3;
    } else if (c == '+') {
        index = 3;
    }
    // ... 解析数字 (几乎相同的逻辑)
}
```

**变更后**:
```java
// 统一解析方法
static AsciiStringCodec parseDecimal(String s, int isHex) {
    // isHex: 0 = 十进制, 1 = 十六进制
    boolean isNegative = false;
    int index = isHex != 0 ? 2 : 0;  // 十六进制跳过 "0x"

    char c = s.charAt(index);
    if (c == '-' || c == '+') {
        isNegative = (c == '-');
        index++;
    }

    // 统一的数字解析逻辑
    return isHex != 0
        ? parseHexDigits(s, index, isNegative)
        : parseDecDigits(s, index, isNegative);
}
```

#### 2. FDBigInteger 构造优化

**变更前**:
```java
// 使用 int[] 作为参数
FDBigInteger(int[] mantissa, int exponent)
```

**变更后**:
```java
// 使用 byte[] 作为参数 (更高效)
FDBigInteger(byte[] mantissa, int exponent)
```

### 变更统计

| 指标 | 变更前 | 变更后 | 差异 |
|------|--------|--------|------|
| **代码行数** | ~1200 | ~1090 | **-110** |
| **重复代码** | 高 | 低 | **减少** |
| **方法数量** | 2 | 1 | **合并** |

## 技术分析

### 统一的好处

#### 1. 代码一致性

```java
// 之前：两个方法可能有不同的边界处理
parseDecimal("  123  ")  // 可能跳过空格
parseHexString("  0x123  ")  // 可能不跳过空格

// 之后：统一的边界处理
parseDecimal("  123  ", 0)   // 统一行为
parseDecimal("  0x123  ", 1)  // 统一行为
```

#### 2. 维护简化

| 场景 | 之前 | 之后 |
|------|------|------|
| **修复 bug** | 需要改两处 | 只改一处 |
| **添加新功能** | 需要同步两处 | 只改一处 |
| **代码审查** | 需要检查两处 | 只检查一处 |

#### 3. 测试覆盖

```
之前:
- parseDecimal 测试
- parseHexString 测试
- 需要分别验证两个方法

之后:
- parseDecimal(isHex=0) 测试
- parseDecimal(isHex=1) 测试
- 测试更容易覆盖所有路径
```
