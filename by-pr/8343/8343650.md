## 背景

在 PR #19626 中，`StringLatin1` 和 `StringUTF16` 新增了优化的 `putCharsAt` 方法。这些方法具有以下特点：

1. **批量字符写入** - 一次性写入多个字符，减少循环开销
2. **边界检查消除** - 在方法内部统一处理边界检查
3. **JIT 友好** - 更容易内联和优化

`StringConcatHelper` 是 Java 字符串拼接的核心实现类（`+` 操作符的底层实现），其中的 `prepend` 方法用于将 boolean 值转换为 "true" 或 "false" 并添加到字符串中。

**改动位置**
```
src/java.base/share/classes/java/lang/StringConcatHelper.java
    └── prepend(long indexCoder, byte[] buf, boolean value, String prefix)
    └── prepend(int index, byte coder, byte[] buf, boolean value, String prefix)
```

## 问题分析

### 原始实现

```java
// 文件: java/lang/StringConcatHelper.java
// 行数: ~236-260

static long prepend(long indexCoder, byte[] buf, boolean value, String prefix) {
    int index = (int)indexCoder;
    byte coder = (byte)(indexCoder >> 32);
    if (coder == LATIN1) {
        if (value) {
            index -= 4;
            buf[index] = 't';     // 单独赋值 4 次
            buf[index + 1] = 'r';
            buf[index + 2] = 'u';
            buf[index + 3] = 'e';
        } else {
            index -= 5;
            buf[index] = 'f';     // 单独赋值 5 次
            buf[index + 1] = 'a';
            buf[index + 2] = 'l';
            buf[index + 3] = 's';
            buf[index + 4] = 'e';
        }
        index -= prefix.length();
        prefix.getBytes(buf, index, LATIN1);
    } else {
        // UTF16 分支类似，使用 StringUTF16.putChar
        if (value) {
            index -= 8;
            StringUTF16.putChar(buf, index, 't');      // 4 次 putChar 调用
            StringUTF16.putChar(buf, index + 1, 'r');
            StringUTF16.putChar(buf, index + 2, 'u');
            StringUTF16.putChar(buf, index + 3, 'e');
        } else {
            index -= 10;
            StringUTF16.putChar(buf, index, 'f');      // 5 次 putChar 调用
            StringUTF16.putChar(buf, index + 1, 'a');
            StringUTF16.putChar(buf, index + 2, 'l');
            StringUTF16.putChar(buf, index + 3, 's');
            StringUTF16.putChar(buf, index + 4, 'e');
        }
        index -= prefix.length();
        prefix.getBytes(buf, index, UTF16);
    }
    return ((long)coder << 32) | index;
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **多次数组写入** | "true" 需要 4 次写入，"false" 需要 5 次写入 | 每次写入都有边界检查 |
| **代码重复** | LATIN1 和 UTF16 分支逻辑相同，只是方法不同 | 维护成本高 |
| **UTF16 方法调用开销** | 每次 `putChar` 都是一次方法调用 | UTF16 分支需要 4-5 次方法调用 |

### 为什么 `putCharsAt` 更优？

```java
// StringLatin1.putCharsAt 实现
static void putCharsAt(byte[] value, int i, char c1, char c2, char c3, char c4) {
    value[i] = (byte)c1;
    value[i + 1] = (byte)c2;
    value[i + 2] = (byte)c3;
    value[i + 3] = (byte)c4;
}

// StringUTF16.putCharsAt 实现
static void putCharsAt(byte[] value, int i, char c1, char c2, char c3, char c4) {
    int end = i + 4;
    checkBoundsBeginEnd(i, end, value);  // 统一检查
    putChar(value, i, c1);
    putChar(value, i + 1, c2);
    putChar(value, i + 2, c3);
    putChar(value, i + 3, c4);
}
```

**优势**：
1. **统一边界检查** - `StringUTF16.putCharsAt` 只检查一次边界
2. **代码简洁** - 单行代码替代多行赋值
3. **内联优化** - 更容易被 JIT 编译器内联

## 解决方案

### 核心改动

```java
// 变更后: 使用 putCharsAt 方法

static long prepend(long indexCoder, byte[] buf, boolean value, String prefix) {
    int index = (int)indexCoder;
    byte coder = (byte)(indexCoder >> 32);
    if (coder == LATIN1) {
        if (value) {
            index -= 4;
            StringLatin1.putCharsAt(buf, index, 't', 'r', 'u', 'e');  // 单行调用
        } else {
            index -= 5;
            StringLatin1.putCharsAt(buf, index, 'f', 'a', 'l', 's', 'e');  // 单行调用
        }
        index -= prefix.length();
        prefix.getBytes(buf, index, LATIN1);
    } else {
        if (value) {
            index -= 8;
            StringUTF16.putCharsAt(buf, index, 't', 'r', 'u', 'e');  // 单行调用
        } else {
            index -= 10;
            StringUTF16.putCharsAt(buf, index, 'f', 'a', 'l', 's', 'e');  // 单行调用
        }
        index -= prefix.length();
        prefix.getBytes(buf, index, UTF16);
    }
    return ((long)coder << 32) | index;
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **代码行数减少** | 从 ~40 行减少到 ~20 行 |
| **统一边界检查** | UTF16 分支只需一次边界检查 |
| **更好的可读性** | 意图更清晰，一目了然 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 数组越界 | `index` 已通过 `ensureCapacity` 确保 |
| 方法签名变化 | `putCharsAt` 是内部 API，受控变更 |
| 行为差异 | `putCharsAt` 与原始赋值行为完全一致 |

## 设计决策

### Q1: 为什么不优化其他类似的字符串常量拼接？

> **Reviewer**: "null"、数字等字符串常量也可以用同样的方式优化吗？

**A**:
1. "null" 已经在其他地方使用 `putCharsAt`（见 `AbstractStringBuilder.append(null)`）
2. 数字拼接需要动态计算，不适合使用固定参数的 `putCharsAt`
3. 本次 PR 只优化 boolean 的 "true"/"false"，这是最常见的固定字符串常量

### Q2: 性能提升主要来自哪里？

> **Reviewer**: 是减少了代码行数，还是减少了运行时开销？

**A**: 主要是：
1. **UTF16 分支** - 从 4-5 次方法调用减少到 1 次
2. **边界检查** - UTF16 分支从 4-5 次检查减少到 1 次
3. **代码缓存** - 更少的字节码，更好的指令缓存局部性

### Q3: 这个改动与 PR #19626 的关系？

> **Reviewer**: 为什么不在 PR #19626 中一起优化？

**A**:
1. PR #19626 引入了 `putCharsAt` 方法，主要优化 `StringBuilder`
2. 本次 PR 是在 `putCharsAt` 稳定后，将其应用到其他地方
3. 遵循 "一次一个 PR" 原则，便于代码审查和测试
