## 背景

### 问题由来

这是 JDK-8343962 的**重做版本**（REDO）。原始 PR #21593 由于 Unsafe offset 溢出问题被回滚。

**为什么需要这个 PR？**
1. **代码重复**：StringLatin1 和 StringUTF16 都有 getChars 方法，逻辑相似
2. **BigDecimal 重复**：BigDecimal 内部也有类似的数字转字符逻辑
3. **性能不一致**：Latin1 分支比 UTF16 分支慢（UTF16 使用 putChar 无边界检查）
4. **内部 API 泄露**：需要通过 JavaLangAccess 暴露 getChars

### 重做的关键修复

原始 PR #21593 的关键问题：
```java
// 错误：可能发生整数溢出
int offset = charPos * 2;  // charPos 可能是负数
UNSAFE.putByte(buf, baseOffset + offset, value);
```

修复后：
```java
// 正确：使用 putCharUnaligned 处理
putCharUnaligned(buf, charPos, value);
```

**改动位置**
```
src/java.base/share/classes/
├── jdk/internal/util/
│   └── DecimalDigits.java    ← 新增 getCharsLatin1/getCharsUTF16
├── java/lang/
│   ├── StringLatin1.java     ← 调用 DecimalDigits.getCharsLatin1
│   ├── StringUTF16.java      ← 调用 DecimalDigits.getCharsUTF16
│   └── StringConcatHelper.java
└── java/math/
    └── BigDecimal.java       ← 调用 DecimalDigits，消除重复代码
```

## 问题分析

### 原始实现（重复代码）

```java
// StringLatin1.getChars
static int getChars(int i, int index, byte[] buf) {
    // ... 使用 DigitTens/DigitOnes 或 PACKED_DIGITS
    while (i <= -100) {
        q = i / 100;
        r = (q * 100) - i;
        i = q;
        buf[--charPos] = DigitOnes[r];
        buf[--charPos] = DigitTens[r];
    }
    // ...
}

// StringUTF16.getChars (类似逻辑)
static int getChars(int i, int index, byte[] buf) {
    // ... 几乎相同的逻辑，但使用 putChar
    while (i <= -100) {
        q = i / 100;
        r = (q * 100) - i;
        i = q;
        putChar(buf, --charPos, Integer.DigitOnes[r]);
        putChar(buf, --charPos, Integer.DigitTens[r]);
    }
    // ...
}

// BigDecimal 内部也有类似实现
private static int getChars(int i, int index, byte[] buf) {
    // 又是重复！
}
```

### 性能问题

| 问题 | Latin1 分支 | UTF16 分支 | 原因 |
|------|-------------|------------|------|
| **边界检查** | 每次数组访问 | 无（putChar 使用 Unsafe） | Latin1 使用普通数组访问 |
| **缓存局部性** | 较差 | 较差 | 每次只写一个字符 |
| **代码重复** | 是 | 是 | 维护成本高 |

### 原始 PR #21593 的 Bug

```java
// 问题代码（已回滚）
private static void getCharsLatin1(int i, int index, byte[] buf) {
    // ...
    while (i <= -100) {
        // ...
        int offset = (charPos -= 2);  // 可能是负数！
        // 当 charPos = -1 时，offset = -2
        // baseOffset + (-2) 可能溢出为正数
        UNSAFE.putByte(null, baseOffset + offset, (byte) v1);
    }
}
```

**问题场景**：
- 缓冲区起始位置不是 0
- `charPos` 变成负数
- `baseOffset + charPos` 可能溢出

## 解决方案

### 核心改动

#### 1. DecimalDigits 新增统一实现

```java
// 文件: jdk/internal/util/DecimalDigits.java

/**
 * 将整数转换为 Latin1 编码的字符。
 *
 * @param i     要转换的整数
 * @param index 下一个字符的索引（独占）
 * @param buf   目标缓冲区，Latin1 编码
 * @return 最高位数字或负号的索引
 * @apiNote 此方法假定所有必要的边界检查都由调用者完成。
 *          它使用 Unsafe 操作来消除边界检查，这是安全的，因为调用者保证缓冲区大小足够。
 */
static int getCharsLatin1(int i, int index, byte[] buf) {
    int q, r;
    int charPos = index;

    boolean negative = i < 0;
    if (!negative) {
        i = -i;
    }

    // 每次迭代生成 2 位数字
    while (i <= -100) {
        q = i / 100;
        r = (q * 100) - i;
        i = q;
        charPos -= 2;
        // 关键：使用 putCharUnaligned 避免溢出问题
        putCharUnaligned(buf, charPos, PACKED_DIGITS[r]);
    }

    // 处理剩余 1-2 位数字
    if (i < -9) {
        charPos -= 2;
        putCharUnaligned(buf, charPos, PACKED_DIGITS[-i]);
    } else {
        buf[--charPos] = (byte) ('0' - i);
    }

    if (negative) {
        buf[--charPos] = (byte) '-';
    }
    return charPos;
}

/**
 * 将整数转换为 UTF16 编码的字符。
 */
static int getCharsUTF16(int i, int index, byte[] buf) {
    // 类似逻辑，但写入 UTF16
    // ...
}
```

#### 2. 关键修复：putCharUnaligned

```java
// 使用 Unsafe.putCharUnaligned 而不是手动计算偏移
private static void putCharUnaligned(byte[] buf, int pos, int value) {
    // Unsafe.putCharUnaligned 会正确处理对齐和边界
    // 避免了手动偏移计算的溢出问题
    UNSAFE.putCharUnaligned(buf, (long) ARRAY_BASE_OFFSET + (pos << 1), (char) value, BIG_ENDIAN);
}
```

#### 3. StringLatin1 调用统一实现

```java
// 文件: java/lang/StringLatin1.java
static int getChars(int i, int index, byte[] buf) {
    // 直接委托给 DecimalDigits
    return DecimalDigits.getCharsLatin1(i, index, buf);
}
```

#### 4. BigDecimal 消除重复

```java
// 文件: java/math/BigDecimal.java (优化前)
private static int getChars(int i, int index, byte[] buf) {
    // 重复实现...
}

// 文件: java/math/BigDecimal.java (优化后)
private static int getChars(int i, int index, byte[] buf) {
    // 直接使用 DecimalDigits
    return DecimalDigits.getCharsLatin1(i, index, buf);
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **统一实现** | 消除 StringLatin1、StringUTF16、BigDecimal 的重复代码 |
| **Unsafe 操作** | 使用 putCharUnaligned 消除边界检查 |
| **PACKED_DIGITS 复用** | Latin1 和 UTF16 使用相同的查找表 |
| **UTF16 性能提升** | 现在也使用 Unsafe，与 Latin1 性能一致 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 内存越界 | 调用者确保缓冲区大小足够 |
| 偏移溢出 | 使用 putCharUnaligned 而不是手动计算偏移 |
| 端序问题 | 显式指定 BIG_ENDIAN |

## 设计决策

### Q1: 为什么原 PR #21593 被回滚？

> **Reviewer**: 为什么要重做这个 PR？

**A**: 原始 PR 有一个关键 Bug：
```java
// 原始代码
UNSAFE.putByte(null, baseOffset + charPos, value);

// 问题：当 charPos 是负数时，baseOffset + charPos 可能溢出
// 例如：baseOffset = 16, charPos = -2
// 16 + (-2) = 14 (正常)
// 但如果 baseOffset = Integer.MAX_VALUE, charPos = -2
// Integer.MAX_VALUE + (-2) 溢出为负数！
```

修复后使用 `putCharUnaligned`，它会正确处理偏移计算。

### Q2: 为什么不直接修复 StringLatin1，而是移到 DecimalDigits？

> **Reviewer**: 为什么不直接在 StringLatin1 中修复？

**A**:
1. **代码重复**：BigDecimal 也有相同逻辑
2. **职责分离**：DecimalDigits 专门负责数字转换
3. **避免内部 API 泄露**：不需要通过 JavaLangAccess 暴露

### Q3: putCharUnaligned 和 putChar 有什么区别？

> **Reviewer**: 为什么用 putCharUnaligned？

**A**:
- `putChar` 要求地址对齐（2 字节对齐）
- `putCharUnaligned` 允许非对齐访问
- byte[] 可能从任意地址开始，需要 unaligned

### Q4: 为什么 LATIN1 比 UTF16 慢？

> **Reviewer**: 优化前 LATIN1 为什么比 UTF16 慢？

**A**:
```java
// LATIN1: 每次访问都有边界检查
buf[--charPos] = value;  // JVM 插入边界检查

// UTF16: putChar 使用 Unsafe，无边界检查
putChar(buf, --charPos, value);  // Unsafe 操作
```

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **内存安全** | 🟡 中 | 使用 putCharUnaligned，调用者确保缓冲区大小 |
| **兼容性** | 🟢 无影响 | 纯内部实现 |
| **平台差异** | 🟡 中 | putCharUnaligned 是 HotSpot 特有 |
| **回归风险** | 🟡 中 | 修复了原 PR 的 Bug |

### 原始 PR 的 Bug 修复

**问题**：
```java
// 原始代码
UNSAFE.putByte(null, baseOffset + charPos, value);
```

**修复**：
```java
// 新代码
UNSAFE.putCharUnaligned(buf, ARRAY_BASE_OFFSET + (pos << 1), (char) value, BIG_ENDIAN);
```

1. 使用 `putCharUnaligned` 而不是 `putByte`
2. 使用 `ARRAY_BASE_OFFSET` 常量
