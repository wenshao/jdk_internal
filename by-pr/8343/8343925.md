## 背景

### 原始优化：JDK-8342650

**目标**: 将数字转字符的 `getChars` 方法集中到 `DecimalDigits` 类，实现代码复用和性能优化。

**改动**:
- 新增 `DecimalDigits.getCharsLatin1()` 和 `getCharsUTF16()` 方法
- 在 `Integer.toString()`, `Long.toString()`, `StringBuilder.append()` 等处使用新方法
- 使用 `ByteArrayLittleEndian` 优化 UTF16 编码

### 问题浮现

在 PR #21593 的测试中，`test/jdk/java/util/BitSet/HugeToString.java` 发生崩溃：

```
# A fatal error has been detected by the Java Runtime Environment:
#  SIGSEGV (0xb) at pc=0x00007f8c4e8b6b70, pid=1234, tid=1235
```

### 崩溃根因

经分析，这是 **Unsafe 地址溢出** 问题（与 JDK-8343984 相同）：

```java
// StringUTF16.getChars 中的问题代码
int packed = (int) StringLatin1.PACKED_DIGITS[r];
int inflated = ((packed & 0xFF00) << 8) | (packed & 0xFF);
charPos -= 2;
ByteArrayLittleEndian.setInt(buf, charPos << 1, inflated);
//                                   ^^^^^^^^^^^
//                        charPos 接近 Integer.MAX_VALUE 时溢出
```

## 问题分析

### 原始实现（JDK-8342650）

#### 1. DecimalDigits 类新增方法

```java
// 新增的优化方法
public class DecimalDigits {
    static int getCharsLatin1(int i, int index, byte[] buf) {
        // 使用 ByteArrayLittleEndian 优化
        // ...
    }

    static int getCharsUTF16(int i, int index, byte[] buf) {
        // 问题代码位置
        int packed = (int) StringLatin1.PACKED_DIGITS[r];
        int inflated = ((packed & 0xFF00) << 8) | (packed & 0xFF);
        charPos -= 2;
        ByteArrayLittleEndian.setInt(buf, charPos << 1, inflated);  // 溢出风险！
    }
}
```

#### 2. 调用链

```
BitSet.toString()
  └── StringBuilder.append(int)
      └── DecimalDigits.getCharsUTF16()  ← 崩溃点
          └── ByteArrayLittleEndian.setInt()
              └── Unsafe.putInt()
                  └── 地址计算溢出
```

### Bug 根因分析

| 问题 | 描述 |
|------|------|
| **直接原因** | `charPos << 1` 在 `charPos` 接近 `Integer.MAX_VALUE/2` 时溢出 |
| **深层原因** | `ByteArrayLittleEndian.setInt` 的 offset 参数是 `int`，无法处理大偏移 |
| **设计问题** | 优化代码未考虑超大数组场景 |
| **测试缺失** | 原始 PR 缺少超大数组测试 |

### 为什么 HugeToString 测试触发崩溃？

```java
// BitSet.HugeToString.java
BitSet bs = new BitSet();
bs.set(Integer.MAX_VALUE - 100);  // 设置高位
String s = bs.toString();  // 触发超大 StringBuilder

// StringBuilder 内部
int charPos = spaceNeeded;  // 接近 Integer.MAX_VALUE
ByteArrayLittleEndian.setInt(buf, charPos << 1, inflated);
//                                         ^^^
//                             charPos << 1 溢出 → 负数
//                             → 访问非法内存 → JVM 崩溃
```

## 解决方案

### 核心改动：完全回退

```java
// 修改后：恢复原有实现
public AbstractStringBuilder append(int i) {
    int spaceNeeded = count + DecimalDigits.stringSize(i);
    ensureCapacityInternal(spaceNeeded);
    if (isLatin1()) {
-       DecimalDigits.getCharsLatin1(i, spaceNeeded, value);
+       StringLatin1.getChars(i, spaceNeeded, value);
    } else {
-       DecimalDigits.getCharsUTF16(i, spaceNeeded, value);
+       StringUTF16.getChars(i, count, spaceNeeded, value);
    }
    this.count = spaceNeeded;
    return this;
}
```

### 回退范围

| 文件 | 回退内容 |
|------|----------|
| `AbstractStringBuilder.java` | 恢复 `StringLatin1.getChars` 调用 |

## 设计决策

### Q1: 为什么不完全删除 DecimalDigits 类？

> **Reviewer**: `DecimalDigits` 类还有其他用途，是否保留？

**A**: `DecimalDigits` 类中还有 `stringSize()` 和 `digitPair()` 等方法，仍在使用：

```java
// DecimalDigits 保留的方法
public static int stringSize(int x) { ... }  // 计算数字位数
public static int stringSize(long x) { ... }
public static short digitPair(int index) { ... }  // 数字对查找表
```

### Q2: 为什么不直接修复溢出问题？

> **Reviewer**: 可以像 JDK-8343984 那样直接修复溢出吗？

**A**: 更复杂，原因：

1. `ByteArrayLittleEndian.setInt` 的 offset 是 `int` 类型，无法改为 `long`
2. 需要重新设计优化方案
3. 回退更安全，后续可单独优化

**Reviewer (AlanBateman)**: Changes in this area need to be very carefully reviewed and tested. I think continue with the current plan to backout the original change and seek wider review and testing for the REDO.

### Q3: 后续如何重新优化？

> **Reviewer**: 重新优化时需要注意什么？

**A**:
1. 避免使用 `ByteArrayLittleEndian`，改用直接 `putChar` 调用
2. 确保 offset 计算使用 `long` 类型
3. 添加超大数组测试用例

**相关 PR**: [JDK-8343984](https://bugs.openjdk.org/browse/JDK-8343984) 修复了类似问题

## 教训总结

### 优化陷阱

1. **过早优化**: 在没有充分测试的情况下进行底层优化
2. **边界条件**: 忽略了超大数组的边界情况
3. **类型选择**: 使用 `int` 作为 offset 类型限制了可处理的数据大小

### 最佳实践

1. **测试先行**: 优化代码应包含边界条件测试
2. **渐进式**: 大型重构应分步骤进行
3. **快速回退**: 发现问题后及时回退比带病上线更好
