## 描述

This bug was found with the Template Framework VectorAPI Fuzzer: https://github.com/openjdk/jdk/pull/28873

For a good example with explanation see `test1` in the attached test.

-----------------------

Important reminder about masks:
- `BVectMask`: boolean mask 0/1, stored in a byte `0x00/0x01`.
- `NVectMask`: N-bit lanes, `0x0..0/0xF..F`, used on platforms like AVX2 that don't have special mask registers.
- `PVectMask`: platform specific mask registers, used on AVX512 (k-register) etc.

-----------------------

In `VectorLongToMaskNode::Ideal` we optimized this pattern:
```
VectorStoreMask(0x0..0/0xF..F to 0/1)
VectorMaskToLong
AndL(truncate)
VectorLongToMask -> 0/1
```
To this:
```
VectorMaskCast(0x0..0/0xF..F to 0/1)
```

However, as far as I understand, `VectorMaskCast` is not made to convert `NVectMask` `0x0..0/0xF..F` to a `BVectMask` boolean mask `0x00/0x01`. Rather, it produces `0x00/0xFF` (it sign-extends or truncates the mask). This is a type confusion, and leads to a wrong result because the mask has the wrong bit pattern. Later operations rely on the exact bit pattern.

If we want to convert from to boolean 0/1 mask, we have to use `VectorStoreMask` and `VectorLoadMask`.

In the pattern above, we should not have folded away the `VectorStoreMask`, which is done with this old line:
`src = src->in(1);`

Rather, we should keep the `VectorStoreMask` and only fold the pattern below, resulting in:
```
VectorStoreMask(L-mask to 0/1)
VectorMaskCast (0/1 to 0/1)
```

This keeps the types consistent.

We should also add asserts to `VectorMaskCast`, that check that we don't convert from full-bit masks to boolean masks.
