## 描述

(Found when working on Valhalla enablement for Zero)

See the bug for symptoms. Happens reliably under some compilation circumstances on my machine (when unrelated Hotspot patches are applied). When a broken Zero VM gets compiled this particular way, it always fails the same way.

The key issue is that `ffi_call` accepts the `void**` argument list, that is a list of pointers from where to pull the FFI call arguments, see `man ffi_call` for the example. Therefore, to pass `JNIEnv*` down to native call, we pull it into local, and get the address of that local to `ffi_call` argument list. *But*, our `env` local goes out of scope _before_ we get to the call, so FFI call can see the garbage there. We seem to get _lucky_ most of the time, but not all of the time.

So the fix is to do the same thing we already do for `void *mirror`. Deliberately keeping the patch small to simplify backports.

Additional testing:
 - [x] Linux x86_64 zero fastdebug, `make images` (used to fail reliably with additional patches, now passes)
 - [x] Linux x86_64 zero fastdebug, `make bootcycle-images` (still passes)
