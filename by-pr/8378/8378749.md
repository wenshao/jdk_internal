## 描述

**Background:**

With the (classic) CDS archive, pre-initialized heap objects are stored using the `KlassSubGraph` mechanism that divides up the heap objects into multiple groups. Some of these groups may be ignored under certain runtime conditions.

For example, if the JVM is running with a different set of modules than those used during the creation of the CDS archive, we ignore the groups of objects that are used to built the module graph.

This design is intended to use as much of the pre-initialized heap objects as possible. However, the downside is we have a complex and fragile system. Some of the assumptions are broken by the upcoming Automatic Heap Sizing (AHS) work.

**Fix summary -- all-or-nothing loading of heap objects:**

KlassSubGraphs a no longer used by the new AOT cache, and we don't intend to improve it further for the classic CDS archive. This PR make the loading of heap objects in the classic CDS archive all-or-nothing:

- All archived heap objects are disabled if (see `FileMapInfo::can_use_heap_region()`)
  - JVMTI class file load hook is enabled
  - The archived module graph cannot be used due to incompatible module settings

**Performance impact:**

The performance impact is minimal (single digit milliseconds) when archived heap objects are disabled. This affects only the classic CDS archive. There's no change when AOT caches are used.

I tested with 

```
perf stat -r 100 java --add-modules=jdk.compiler --version > /dev/null

BEFORE: 0.022718 +- 0.000202 seconds time elapsed  ( +-  0.89% )  -- some archived heap objects are used
AFTER:  0.023271 +- 0.000336 seconds time elapsed  ( +-  1.44% )  -- all archived heap objects are discarded
```
