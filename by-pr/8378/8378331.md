## 描述

Please review this change to how we handle weak references in G1.

**Summary**
When investigating how the StringTable cleaning is processed by the WeakProcessor, I found that the problem I was seeing was connected to humongous objects not being reclaimed properly.

I saw this when using JFR and the OopStorage for "Weak JFR Old Object Samples" was keeping the humongous objects alive. Looking closer at the IsAlive and KeepAlive closures, I realized that they do not handle humongous candidates properly to allow eager reclaim.

A very similar problem is also present for java.lang.ref.Reference processing, it is reported in: [JDK-8378336](https://bugs.openjdk.org/browse/JDK-8378336) and this PR will fix that issue as well.

**Testing** 
* Extended two tests to cover the problem seen, verified that it failed before this change but not after.
* Mach5 tier1-5
* Sanity performance testing for G1 showed no change (as expected)
