## 描述

The issue here is that we delete `AOTRuntimeConstant ` table and `AOTCodeCache` buffer after we dumped AOT archive. As result `AOTCodeCache::is_on_for_dump()` returns false after that.

The changes introduced by [JDK-8377554](https://bugs.openjdk.org/browse/JDK-8377554) use `AOTCodeCache::is_on_for_dump()` check which is true in assembly phase (`-XX:AOTMode=create`) regardless if we do AOT compilation or normal JIT compilation or stubs generation.

In failing case normal JIT compilation calculates mach nodes sizes when `AOTCodeCache::is_on_for_dump()` is `true`  so it use code [JDK-8377554](https://bugs.openjdk.org/browse/JDK-8377554) load runtime constants. But when that JIT started generate code `AOTCodeCache::is_on_for_dump()` become `false` and runtime constants are directly embedded into code  and we hit assert since pre-calculated sizes do not match.

Suggested fix is don't not explicitly delete AOT code tables and buffers during AOT assembly phase so that `AOTCodeCache::is_on_for_dump()` kept `true` and compiled code (which reference them) could be generated and used. Note, we already keep AOT code tables and buffers during whole production run.

During testing I also found that we set AOT code and profiling flags too early before some AOT cache flags are finalized. I moved setting to `CDSConfig::ergo_initialize()`
