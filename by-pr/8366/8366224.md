## 背景

两位数十进制格式化是 Java 中最常见的操作之一，特别是在日期时间格式化中：

- **月份 (01-12)**: `LocalDate::getMonthValue`
- **日期 (01-31)**: `LocalDate::getDayOfMonth`
- **小时 (00-23)**: `LocalTime::getHour`
- **分钟 (00-59)**: `LocalTime::getMinute`
- **秒数 (00-59)**: `LocalTime::getSecond`

**为什么重要？**
- 日期时间格式化是日志、JSON 序列化、数据导出中的热点路径
- `DateTimeFormatter` 类在企业应用中被广泛使用
- 之前的实现每次都需要条件判断和除法运算

## 解决方案

### 核心改动：DecimalDigits.appendPair 和 appendQuad

```java
// 文件: src/java.base/share/classes/jdk/internal/util/DecimalDigits.java

/**
 * 预计算的数字对数组 (0-99)
 * 每个 short 值按小端序打包两个 ASCII 字符:
 *      00 -> '0' | ('0' << 8) -> 0x3030
 *      01 -> '0' | ('1' << 8) -> 0x3130
 *      ...
 *      99 -> '9' | ('9' << 8) -> 0x3939
 */
@Stable
private static final short[] DIGITS;

static {
    short[] digits = new short[128];  // 128 = 2^7，配合 & 0x7f 消除边界检查
    for (int i = 0; i < 10; i++) {
        short hi = (short) (i + '0');
        for (int j = 0; j < 10; j++) {
            short lo = (short) ((j + '0') << 8);
            digits[i * 10 + j] = (short) (hi | lo);
        }
    }
    DIGITS = digits;
}

/**
 * 将 0-99 的整数格式化为两位数字符串追加到 StringBuilder
 *
 * @param buf 目标 StringBuilder
 * @param v 要格式化的值 (应在 0-99 范围内)
 */
public static void appendPair(StringBuilder buf, int v) {
    // & 0x7f 确保索引在 [0, 127] 范围内，允许 JIT 消除边界检查
    int packed = DIGITS[v & 0x7f];
    // 使用 uncheckedNewStringWithLatin1Bytes，JVM 会通过逃逸分析消除临时对象
    buf.append(
        JLA.uncheckedNewStringWithLatin1Bytes(
            new byte[] {(byte) packed, (byte) (packed >> 8)}));
}

/**
 * 将 0-9999 的整数格式化为四位数字符串追加到 StringBuilder
 *
 * @param buf 目标 StringBuilder
 * @param v 要格式化的值 (应在 0-9999 范围内)
 */
public static void appendQuad(StringBuilder buf, int v) {
    int packedHigh = DIGITS[(v / 100) & 0x7f];
    int packedLow  = DIGITS[(v % 100) & 0x7f];
    buf.append(
        JLA.uncheckedNewStringWithLatin1Bytes(
            new byte[] {(byte) packedHigh, (byte) (packedHigh >> 8),
                        (byte) packedLow,  (byte) (packedLow  >> 8)}));
}
```

### DateTimeHelper 重构

```java
// DateTimeHelper.java - 修改前
public static void formatTo(StringBuilder buf, LocalDate date) {
    int year  = date.getYear(),
        month = date.getMonthValue(),
        day   = date.getDayOfMonth();
    int absYear = Math.abs(year);
    if (absYear < 1000) {
        if (year < 0) {
            buf.append('-');
        }
        buf.repeat('0', absYear < 10 ? 3 : absYear < 100 ? 2 : 1);
        buf.append(absYear);
    } else {
        if (year > 9999) {
            buf.append('+');
        }
        buf.append(year);
    }
    buf.append(month < 10 ? "-0" : "-").append(month)
       .append(day < 10 ? "-0" : "-").append(day);
}

// DateTimeHelper.java - 修改后
public static void formatTo(StringBuilder buf, LocalDate date) {
    int year    = date.getYear(),
        absYear = Math.abs(year);
    if (absYear < 10000) {
        if (year < 0) {
            buf.append('-');
        }
        DecimalDigits.appendQuad(buf, absYear);  // 使用新方法
    } else {
        if (year > 9999) {
            buf.append('+');
        }
        buf.append(year);
    }
    buf.append('-');
    DecimalDigits.appendPair(buf, date.getMonthValue());  // 使用新方法
    buf.append('-');
    DecimalDigits.appendPair(buf, date.getDayOfMonth());   // 使用新方法
}
```

## 为什么这个优化有效

### 1. 查找表策略

```
对于 value = 47:

原始实现:
├── 检查: 47 < 10? 否 (分支预测失败惩罚)
├── 检查: 47 < 100? 是 (分支命中)
├── 计算: 47 / 10 = 4 (idiv 指令, ~10 周期)
├── 计算: 47 % 10 = 7 (irem 指令, ~10 周期)
├── 转换: '0' + 4 = '4'
└── 转换: '0' + 7 = '7'

优化后:
├── 计算: 47 & 0x7f = 47 (位运算, 1 周期)
├── 加载: DIGITS[47] = 0x3739 (L1 缓存命中, ~4 周期)
├── 提取: packed & 0xFF = '7'
└── 提取: packed >> 8 = '9'
```

### 2. 内存布局

```
DIGITS 数组布局 (short[], 128 元素 = 256 字节):
索引    值 (十六进制)    打包内容
0       0x3030          '0' | ('0' << 8)
1       0x3130          '0' | ('1' << 8)
...
47      0x3739          '7' | ('9' << 8)  ← 47 变成 "97"
...
99      0x3939          '9' | ('9' << 8)
```

**小端序解释**: 低位字节是十位数字，高位字节是个位数字

### 3. 逃逸分析优化

```java
// 这段代码看似创建了临时对象
buf.append(
    JLA.uncheckedNewStringWithLatin1Bytes(
        new byte[] {(byte) packed, (byte) (packed >> 8)}));

// 但 JIT 编译器会通过逃逸分析优化为:
// 1. 直接在 StringBuilder 内部缓冲区写入字节
// 2. 不分配任何临时对象
// 3. 等价于直接的字节数组复制
```

### 4. 边界检查消除

```java
// JIT 的优化分析:
// 1. 数组长度是 128 (2^7)
// 2. 索引是 v & 0x7f
// 3. (v & 0x7f) 永远在 [0, 127] 范围内
// 4. 结论: 可以安全移除边界检查！
```

## 设计决策

### Q1: 为什么用 short[] 而不是 byte[]？

> **Reviewer**: 为什么每个元素用 short (2 字节) 而不是 byte[] (1 字节)？

**A**: 使用 short[] 可以将两个 ASCII 字符打包到一个元素中：

| 方案 | 内存大小 | 访问次数 | 优势 |
|------|----------|----------|------|
| `byte[200]` | 200 bytes | 2 次加载 | 直观 |
| `short[128]` | 256 bytes | 1 次加载 | **减少内存访问** |

1 次加载 2 个字符比 2 次加载各 1 个字符更快，且 `& 0x7f` 需要数组大小为 2 的幂。

### Q2: 为什么是 128 而不是 100 或 256？

> **Reviewer**: 数组大小为什么是 128？

**A**: 权衡考虑：
- **100**: 不够，无法使用 `& 0x7f` 消除边界检查
- **128**: 最小的 2 的幂，能容纳 0-99，浪费最少 (28 个元素)
- **256**: 浪费更多 (156 个元素)，可能增加缓存未命中

### Q3: uncheckedNewStringWithLatin1Bytes 会创建临时对象吗？

> **Reviewer**: 每次调用都创建 byte[]，会不会有性能问题？

**A**: 不会。JIT 的逃逸分析会优化掉这些临时对象：

```java
// 源代码
new byte[] {a, b}

// JIT 优化后 (伪代码)
// 直接在目标缓冲区写入，无分配
targetBuffer[pos++] = a;
targetBuffer[pos++] = b;
```

这在 JMH 基准测试中已验证，GC 压力没有增加。

### Q4: 为什么同时引入 appendPair 和 appendQuad？

> **Reviewer**: appendQuad 的必要性是什么？

**A**: 四位数格式化在日期格式化中也很常见：
- 年份后缀: 0000-9999
- `YearMonth.toString()` 需要格式化年份

使用 appendQuad 可以避免调用两次 appendPair，减少方法调用开销。
