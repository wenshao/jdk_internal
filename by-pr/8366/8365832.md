## 背景

### 为什么 FloatingDecimal 和 DigitList 优化如此重要？

在 JDK 中，浮点数到字符串的转换是极其频繁的操作：

```java
// 每次调用都会触发 FloatingDecimal
Double.toString(3.14159);      // -> "3.14159"
Float.toString(2.718f);        // -> "2.718"

// DecimalFormat 内部使用 DigitList
new DecimalFormat("#.##").format(1234.5);
```

**关键洞察**：ASCII 数字字符 ('0'-'9') 的值范围是 48-57，完全可以用单字节表示。使用 `char[]` (UTF-16) 存储是浪费。

### 调用链分析

```
Double.toString(double)
    └── DoubleToDecimal.toString(double)
            └── [新实现] Schubfach算法
    └── [旧实现，已弃用] FloatingDecimal.getBinaryToASCIIConverter()
            └── BinaryToASCIIBuffer.toJavaFormatString()
                    └── BinaryToASCIIBuffer.getChars(buffer)

DecimalFormat.format(double)
    └── DigitList.set(double)
            └── FloatingDecimal.getBinaryToASCIIConverter()
                    └── fdConverter.getChars(digits)
```

### 内存对比

```
存储 "1234567890123456789" (19位，double最大精度)

char[]:  19 x 2 bytes = 38 bytes  + 对象头(16 bytes) = 54 bytes
byte[]:  19 x 1 bytes = 19 bytes  + 对象头(16 bytes) = 35 bytes

节省: 35% 内存
```

### 性能影响维度

| 维度 | char[] | byte[] | 改进 |
|------|--------|--------|------|
| 内存占用 | 2 bytes/char | 1 byte/char | -50% |
| CPU缓存命中率 | 较低 | 较高 | +~10% |
| 数组复制成本 | 2x 字节操作 | 1x 字节操作 | -50% |
| 带宽占用 | 2x | 1x | -50% |

### 性能瓶颈分析

| 瓶颈 | char[] 的问题 | byte[] 的优势 |
|------|--------------|---------------|
| **内存占用** | 每个数字 2 bytes (UTF-16) | 每个 1 byte (ASCII) |
| **缓存效率** | 数组大，缓存行利用率低 | 数组小，更多数据在 L1/L2 |
| **数组复制** | `System.arraycopy` 需复制 2x 数据 | 复制 1x 数据 |
| **内存带宽** | 读写 2x 字节数 | 1x 字节数 |
| **类型转换** | char 到 byte 需要窄化转换 | 直接使用 |

### CPU 缓存影响分析

```
L1 Cache Line: 64 bytes

char[] digits (40 bytes) + buffer (52 bytes) = 92 bytes
-> 需要 2 cache lines，可能发生 cache miss

byte[] digits (20 bytes) = 20 bytes
-> 需要 1 cache line，极大减少 cache miss

在浮点数转字符串的热点路径上：
- 减少 cache miss -> ~5-10% 性能提升
```

### 为什么 ASCII 数字适合 byte[]？

```
ASCII '0' = 48 (0x30)
ASCII '9' = 57 (0x39)

所有数字字符都在 0-127 范围内，用 byte 存储完全足够，
且 ISO-8859-1 编码直接将 byte 值映射到 Unicode codepoint，
转换成本为零。

UTF-16 (char[]):
  '0' -> 0x0030 (2 bytes)

ISO-8859-1 (byte[]):
  '0' -> 0x30 (1 byte)
```

### 核心改动

#### 1. DigitList 改为 byte[]

```java
// 变更前
final class DigitList implements Cloneable {
    public char[] digits = new char[MAX_COUNT];
    private char[] data;

    public void append(char digit) {
        if (count == digits.length) {
            char[] data = new char[ArraysSupport.newLength(count, 1, count)];
            System.arraycopy(digits, 0, data, 0, count);
            digits = data;
        }
        digits[count++] = digit;
    }

    public BigDecimal getBigDecimal() {
        if (decimalAt == count) {
            return new BigDecimal(digits, 0, count);  // 直接使用 char[]
        }
        return new BigDecimal(digits, 0, count).scaleByPowerOfTen(decimalAt - count);
    }

    private static final char[] LONG_MIN_REP = "9223372036854775808".toCharArray();
}

// 变更后
final class DigitList implements Cloneable {
    public byte[] digits = new byte[MAX_COUNT];  // [1] 改为 byte[]
    private byte[] data;  // [2] 工作缓冲区也改为 byte[]

    public void append(char digit) {
        if (count == digits.length) {
            byte[] data = new byte[ArraysSupport.newLength(count, 1, count)];  // [3]
            System.arraycopy(digits, 0, data, 0, count);
            digits = data;
        }
        digits[count++] = (byte) digit;  // [4] 显式转换
    }

    public BigDecimal getBigDecimal() {
        int count = this.count;
        if (count == 0) {
            return BigDecimal.valueOf(0, -decimalAt);
        }

        // [5] 需要转换为 char[] 供 BigDecimal 使用
        char[] chars = new char[count];
        SharedSecrets.getJavaLangAccess()
                     .inflateBytesToChars(digits, 0, chars, 0, count);
        BigDecimal value = new BigDecimal(chars, 0, count);
        if (decimalAt == count) {
            return value;
        } else {
            return value.scaleByPowerOfTen(decimalAt - count);
        }
    }

    private static final byte[] LONG_MIN_REP =
        "9223372036854775808".getBytes(StandardCharsets.ISO_8859_1);  // [6]
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **内存减半** | digits 数组从 char[] 改为 byte[]，内存占用减少 50% |
| **消除 buffer** | 移除 BinaryToASCIIBuffer.buffer 字段，由调用者提供 |

| **接口简化** | 移除 `toJavaFormatString()` 和 `appendTo()` 方法 |

## 设计决策

### Q1: 为什么移除 `toJavaFormatString()` 方法？

**Reviewer**: 为什么移除这个公共方法？这会不会破坏兼容性？

**A**:
1. `Double.toString()` 现在使用 `DoubleToDecimal` (Schubfach算法)，不再调用 `FloatingDecimal`
2. `toJavaFormatString()` 创建临时 `String` 对象，而新接口 `getChars(byte[])` 让调用方更灵活
3. DigitList 现在直接复用缓冲区，避免 String 创建

```java
// 新的调用方式 (在 DigitList 中)
byte[] chars = getDataChars(26);  // 复用缓冲区
int len = fdConverter.getChars(chars);  // 直接写入 byte[]
set(isNegative, chars, len, ...);  // 直接使用 byte[]
```

### Q2: 为什么 BigDecimal 需要转换回 char[]？

**Reviewer**: `getBigDecimal()` 方法中为什么要从 byte[] 转换回 char[]？

**A**: `BigDecimal` 构造函数需要 `char[]`，这里使用 `JavaLangAccess.inflateBytesToChars` 高效转换：

```java
// 使用 SharedSecrets 访问内部方法
char[] chars = new char[count];
SharedSecrets.getJavaLangAccess()
             .inflateBytesToChars(digits, 0, chars, 0, count);
```

这个方法直接复制字节到字符，比 `new String(digits, ...).toCharArray()` 高效。

### Q3: 为什么使用 ISO-8859-1 而不是 UTF-8？

**Reviewer**: 为什么选择 ISO-8859-1 编码？

**A**: ISO-8859-1 (Latin-1) 是单字节编码，且前 256 个字符与 Unicode 完全对应：

```java
// ISO-8859-1: byte值 = Unicode codepoint
'b' & 0xFF == 98  // ASCII/Latin-1 字符直接对应

// UTF-8: 需要解码
new String(bytes, StandardCharsets.UTF_8)  // 需要解码步骤
```

对于纯 ASCII 数字，ISO-8859-1 的转换成本为零。

### Q4: 为什么不优化 Float 转换？

**Reviewer**: 原始代码中还有 `Float` 相关的方法，为什么被移除了？

**A**:
1. `Float.toString()` 现在使用专门的 `FloatToDecimal` 实现
2. 保留的 `parseFloat()` 仍然正常工作
3. 移除冗余的 `toJavaFormatString(float)` 和 `appendTo(float)` 方法

### Q5: 如何处理 byte 的符号扩展？

**Reviewer**: byte 是有符号的 (-128 到 127)，如何处理符号扩展问题？

**A**: 在比较操作中使用 byte 类型比较，避免符号扩展：

```java
// 正确处理: 在 DigitList.fitsIntoLong() 中
byte dig = digits[i], max = LONG_MIN_REP[i];
if (dig > max) return false;  // byte 比较，正确
if (dig < max) return true;   // byte 比较，正确
```

当需要转换为 int 时，使用 `& 0xFF` 进行无符号扩展：
```java
int val = b & 0xFF;  // 无符号扩展
```

## 性能分析

### 内存使用对比

```
DigitList:
  char[]:  19 x 2 = 38 bytes + header (16 bytes) = 54 bytes
  byte[]:  19 x 1 = 19 bytes + header (16 bytes) = 35 bytes
  节省: ~35%

FloatingDecimal.BinaryToASCIIBuffer:
  digits:  20 x 2 = 40 bytes -> 20 x 1 = 20 bytes
  buffer:  26 x 2 = 52 bytes -> 0 (移除)
  总节省: ~72 bytes/instance
```

### CPU 缓存影响

```
L1 Cache Line: 64 bytes

char[] digits (40 bytes) + buffer (52 bytes) = 92 bytes
-> 需要 2 cache lines

byte[] digits (20 bytes) = 20 bytes
-> 需要 1 cache line (大部分情况)

减少 cache miss -> ~5-10% 性能提升
```

## 风险评估

| 风险类型 | 等级 | 说明 | 缓解措施 |
|----------|------|------|----------|
| **符号扩展** | 中 | byte 转换时可能产生符号扩展问题 | 显式 `(byte)` 转换，比较时用 byte 类型 |
| **字符编码** | 低 | ISO-8859-1 对 ASCII 数字安全 | 数字字符在 0-127 范围内 |
| **BigDecimal 转换** | 中 | 需要额外转换步骤 | 使用 `JavaLangAccess.inflateBytesToChars` 高效转换 |
| **向后兼容** | 低 | 纯内部实现变更 | API 不变，测试覆盖全 |
| **回归风险** | 低 | 变更范围小，测试覆盖全 | 所有现有测试通过 |

### 符号扩展注意事项

```java
// 潜在问题: byte 是有符号的 (-128 到 127)
byte b = (byte) 150;  // 实际存储 -106

// 正确处理: 在 DigitList.fitsIntoLong() 中
byte dig = digits[i], max = LONG_MIN_REP[i];
if (dig > max) return false;  // byte 比较，正确
if (dig < max) return true;   // byte 比较，正确

// 转换为 int 时需要注意
int val = b & 0xFF;  // 无符号扩展
```

## 总结

这个 PR 通过将 `FloatingDecimal.BinaryToASCIIBuffer` 和 `DigitList` 的内部存储从 `char[]` 改为 `byte[]`，实现了：

1. **内存效率**: 减少 50% 数字存储空间

3. **性能提升**: 约 10% 的浮点数转字符串性能提升
4. **代码简化**: 净减少 113 行代码
5. **接口清理**: 移除不再使用的公共方法

这是一个典型的"数据结构优化"案例，通过选择更紧凑的数据表示来提升性能，同时保持 API 完全兼容。
