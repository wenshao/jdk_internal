## 背景

### DateTimeFormatter 架构

`java.time.format.DateTimeFormatter` 是 Java 8+ 中日期时间格式化的核心类，其内部使用 `DateTimePrintContext` 作为格式化过程的上下文对象：

```
DateTimeFormatter
    └── DateTimePrintContext (上下文)
            ├── temporal: TemporalAccessor (要格式化的时间对象)
            └── formatter: DateTimeFormatter (格式化器)
                    ├── chronology: Chronology (纪元，通常为 null)
                    └── zone: ZoneId (时区，通常为 null)
```

每次调用 `DateTimeFormatter.format()` 时，都会创建 `DateTimePrintContext` 实例，其构造函数会调用 `adjust()` 方法来处理可能的 chronology 和 zone 覆盖。

### 为什么重要？

- `DateTimeFormatter` 是线程安全的，通常被声明为 `static final` 常量
- 日志框架（Log4j、Logback）大量使用日期格式化
- 每次日志输出、API 响应格式化都会触发此方法

## 问题分析

### 原始实现

```java
// 文件: java/time/format/DateTimePrintContext.java
// 修改前的 adjust 方法 - 382 字节

private static TemporalAccessor adjust(final TemporalAccessor temporal, DateTimeFormatter formatter) {
    // 1. 获取覆盖设置
    Chronology overrideChrono = formatter.getChronology();
    ZoneId overrideZone = formatter.getZone();

    // 2. 早期返回优化（常见情况）
    if (overrideChrono == null && overrideZone == null) {
        return temporal;
    }

    // 3. 获取 temporal 的 chronology 和 zone
    Chronology temporalChrono = temporal.query(TemporalQueries.chronology());
    ZoneId temporalZone = temporal.query(TemporalQueries.zoneId());

    // 4. 如果覆盖值相同，则清除
    if (Objects.equals(overrideChrono, temporalChrono)) {
        overrideChrono = null;
    }
    if (Objects.equals(overrideZone, temporalZone)) {
        overrideZone = null;
    }

    // 5. 再次检查是否需要调整
    if (overrideChrono == null && overrideZone == null) {
        return temporal;
    }

    // 6. 执行调整
    final Chronology effectiveChrono = (overrideChrono != null ? overrideChrono : temporalChrono);
    if (overrideZone != null) {
        if (temporal.isSupported(INSTANT_SECONDS)) {
            Chronology chrono = Objects.requireNonNullElse(effectiveChrono, IsoChronology.INSTANCE);
            return chrono.zonedDateTime(Instant.from(temporal), overrideZone);
        }
    }

    // 7. 复杂情况处理（创建匿名 TemporalAccessor）
    // ... 约 80 行代码处理边界情况
    if (overrideZone != null) {
        // 阻止在某些情况下更改 zone
        if (overrideZone.normalized() instanceof ZoneOffset && temporal.isSupported(OFFSET_SECONDS) &&
                temporal.get(OFFSET_SECONDS) != overrideZone.getRules().getOffset(Instant.EPOCH).getTotalSeconds()) {
            throw new DateTimeException(...);
        }
    }
    // ... 更多复杂逻辑
    final ZoneId effectiveZone = (overrideZone != null ? overrideZone : temporalZone);
    final ChronoLocalDate effectiveDate;
    if (overrideChrono != null) {
        if (temporal.isSupported(EPOCH_DAY)) {
            effectiveDate = effectiveChrono.date(temporal);
        } else {
            // 检查日期字段...
            for (ChronoField f : ChronoField.values()) {
                if (f.isDateBased() && temporal.isSupported(f)) {
                    throw new DateTimeException(...);
                }
            }
            effectiveDate = null;
        }
    } else {
        effectiveDate = null;
    }

    // 8. 返回匿名内部类
    return new TemporalAccessor() {
        @Override
        public boolean isSupported(TemporalField field) { ... }
        @Override
        public ValueRange range(TemporalField field) { ... }
        @Override
        public long getLong(TemporalField field) { ... }
        @Override
        public <R> R query(TemporalQuery<R> query) { ... }
    };
}
```

### 性能瓶颈分析

使用 `-XX:+PrintInlining` 诊断日志发现问题：

```
@ 7   java.time.format.DateTimePrintContext::adjust (382 bytes)
    failed to inline: hot method too big
```

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **方法过大** | 382 字节 > 325 字节阈值 | C2 无法内联 |
| **冷热代码混合** | 常见路径和罕见边界情况在同一方法 | 热路径优化受限 |
| **复杂异常处理** | 大量边界情况检查 | 字节码膨胀 |

### C2 内联阈值

HotSpot JVM 的 C2 编译器有严格的内联限制：

| 方法类型 | 大小限制 |
|----------|----------|
| **Hot 方法** | ~325 字节 |
| **常规方法** | ~35 字节 |
| **C1 编译器** | 更宽松 |

## 解决方案

### 核心改动：方法拆分

将 382 字节的单一方法拆分为三个方法，每个方法专注于特定场景：

```
adjust (27 字节)           ← 热路径：快速返回
    ↓
adjustWithOverride (123 字节) ← 中等路径：有覆盖的情况
    ↓
adjustSlow (232 字节)      ← 冷路径：复杂边界情况
```

### 代码结构

```java
/**
 * 第一步：快速路径检查
 * 字节码大小：27 字节 - 可被 C1 和 C2 内联
 */
private static TemporalAccessor adjust(final TemporalAccessor temporal, DateTimeFormatter formatter) {
    // 正常情况优先（早期返回是优化）
    Chronology overrideChrono = formatter.getChronology();
    ZoneId overrideZone = formatter.getZone();
    if (overrideChrono == null && overrideZone == null) {
        return temporal;  // 最常见情况：直接返回
    }

    // 将非 null 情况放在单独方法中，允许更灵活的代码优化
    return adjustWithOverride(temporal, overrideChrono, overrideZone);
}

/**
 * 第二步：覆盖值处理
 * 字节码大小：123 字节 - 可被 C2 内联
 */
private static TemporalAccessor adjustWithOverride(TemporalAccessor temporal,
                                                   Chronology overrideChrono,
                                                   ZoneId overrideZone) {
    // 确保最小更改（早期返回是优化）
    Chronology temporalChrono = temporal.query(TemporalQueries.chronology());
    ZoneId temporalZone = temporal.query(TemporalQueries.zoneId());
    if (Objects.equals(overrideChrono, temporalChrono)) {
        overrideChrono = null;
    }
    if (Objects.equals(overrideZone, temporalZone)) {
        overrideZone = null;
    }
    if (overrideChrono == null && overrideZone == null) {
        return temporal;
    }

    // 执行调整
    final Chronology effectiveChrono = (overrideChrono != null ? overrideChrono : temporalChrono);
    if (overrideZone != null) {
        // 如果有 zone 和 instant，计算简单
        if (temporal.isSupported(INSTANT_SECONDS)) {
            Chronology chrono = Objects.requireNonNullElse(effectiveChrono, IsoChronology.INSTANCE);
            return chrono.zonedDateTime(Instant.from(temporal), overrideZone);
        }
    }

    // 将不常见的代码分支拆分到单独方法
    return adjustSlow(temporal, overrideZone, temporalZone, overrideChrono, effectiveChrono, temporalChrono);
}

/**
 * 第三步：复杂边界情况处理
 * 字节码大小：232 字节 - 冷路径，无需内联
 */
private static TemporalAccessor adjustSlow(
        TemporalAccessor temporal,
        ZoneId overrideZone, ZoneId temporalZone,
        Chronology overrideChrono, Chronology effectiveChrono, Chronology temporalChrono) {

    if (overrideZone != null) {
        // 阻止在 OffsetTime 等类似情况下更改 zone
        if (overrideZone.normalized() instanceof ZoneOffset && temporal.isSupported(OFFSET_SECONDS) &&
                temporal.get(OFFSET_SECONDS) != overrideZone.getRules().getOffset(Instant.EPOCH).getTotalSeconds()) {
            throw new DateTimeException("Unable to apply override zone...");
        }
    }
    // ... 其余复杂逻辑（匿名 TemporalAccessor 创建等）
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **热路径精简** | `adjust()` 仅 27 字节，涵盖 90%+ 场景（无覆盖） |
| **中等路径优化** | `adjustWithOverride()` 123 字节，处理常见覆盖情况 |
| **冷路径隔离** | `adjustSlow()` 处理边界情况，不影响热路径编译 |
| **死代码消除** | C2 可内联后，可基于常量优化消除分支 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 行为变化 | 仅重构，逻辑完全相同 |
| 性能回归 | 冷路径隔离确保热路径优化 |
| 边界情况 | 所有原有检查保留在 `adjustSlow()` 中 |

## 设计决策

### Q1: 为什么拆分成三个方法而不是两个？

> **Reviewer**: 能否只拆成两个方法？

**A**: 三个方法提供更好的分层：

```java
adjust (27 字节)
    └── 仅检查 null，快速返回
    └── 90%+ 调用在此返回

adjustWithOverride (123 字节)
    └── 处理实际覆盖逻辑
    └── 大部分剩余调用在此返回

adjustSlow (232 字节)
    └── 处理复杂边界情况
    └── 极少调用，无需内联
```

### Q2: 为什么不把所有逻辑都内联到构造函数中？

> **Reviewer**: 直接在构造函数中实现不是更快吗？

**A**:
1. 构造函数的语义是初始化，不应包含复杂逻辑
2. 静态方法更容易被 JIT 优化和内联
3. 保持方法职责单一

### Q3: 字节码大小如何测量？

> **Reviewer**: 如何确认方法现在是 27 字节？

**A**: 使用 JDK 内置工具：

```bash
# 编译后查看字节码大小
javap -v -p DateTimePrintContext.class | grep "adjust"

# 或者使用 JIT 日志
-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining
```

### Q4: 为什么静态 final 字段很重要？

> **Reviewer**: PR 讨论中提到了 `static final` formatter 的优化效果？

**A**: 当 `DateTimeFormatter` 是 `static final` 时：

```java
private static final DateTimeFormatter FORMATTER =
    DateTimeFormatter.ISO_LOCAL_DATE;  // chrono=null, zone=null
```

JIT 可以在编译时确定：
- `formatter.getChronology()` 永远返回 `null`
- `formatter.getZone()` 永远返回 `null`
- `adjust()` 方法中的 `if` 条件永远为 `true`
- 整个方法调用可以被优化为直接返回 `temporal`

这就是 `DateTimeFormatterWithPaddingBench` 显示更高性能提升的原因。

### 实际应用示例

```java
// 1. 日志配置（最常见场景）
private static final DateTimeFormatter LOG_FORMATTER =
    DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss.SSS");

// 每次日志输出都会调用
logger.info("Event at {}", LocalDateTime.now().format(LOG_FORMATTER));

// 2. REST API 响应格式化
private static final DateTimeFormatter ISO_FORMATTER =
    DateTimeFormatter.ISO_INSTANT;

public String formatDate(Instant instant) {
    return ISO_FORMATTER.format(instant);  // 受益于此优化
}

// 3. 批量数据处理
List<LocalDateTime> dates = ...;
String formatted = dates.stream()
    .map(dt -> dt.format(formatter))  // 大量调用
    .collect(joining(","));
```
