## 背景

### Float/Double.toString 的实现

Java 的浮点数转字符串通过 `FloatingDecimal` 类实现：

```java
// Float.java
public static String toString(float f) {
    FloatingDecimal.decimalTo(f);  // 使用 FloatingDecimal
}

// Double.java
public static String toString(double d) {
    FloatingDecimal.decimalTo(d);  // 使用 FloatingDecimal
}
```

### 历史遗留问题

早期 JDK 有两种浮点数转字符串的实现：
1. **FloatingDecimal**: 旧实现，使用 char[] 存储
2. **FloatToDecimal/DoubleToDecimal**: 新实现，性能更好

现在 `FloatToDecimal` 和 `DoubleToDecimal` 已经替代了旧实现，但 `FloatingDecimal` 仍然被一些地方使用（如 `BigDecimal`）。

**改动位置**
```
src/java.base/share/classes/java/
├── math/
│   └── FloatingDecimal.java  ← char[] -> byte[]
└── text/
    └── DigitList.java        ← char[] -> byte[]
```

## 问题分析

### 原始实现

```java
// 文件: java/math/FloatingDecimal.java (原始实现)

class FloatingDecimal {
    // 使用 char[] 存储数字
    private char[] digits;

    // 构造函数
    FloatingDecimal(double d) {
        // ... 处理逻辑
        digits = new char[n];
        // ... 填充 digits
    }
}

// 文件: java/text/DigitList.java (原始实现)

class DigitList {
    // 使用 char[] 存储数字
    private char[] digits = new char[21];

    // 使用场景
    void append(char digit) {
        digits[count++] = digit;
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **char[] 占用大** | 每个 char 2 bytes | 内存占用翻倍 |
| **类型转换** | char 到 byte 转换 | 额外的转换成本 |
| **未使用代码** | FloatToDecimal 后，部分代码不再使用 | 维护成本 |

### 为什么使用 byte[]？

| 方面 | char[] | byte[] |
|------|--------|--------|
| **元素大小** | 2 bytes | 1 byte |
| **适用场景** | Unicode 字符 | Latin1/数字 |
| **数字存储** | 浪费一半空间 | 恰好够用 |
| **缓存友好** | 较差 | 较好 |

## 解决方案

### 核心改动

#### FloatingDecimal.java

```java
// 文件: java/math/FloatingDecimal.java (优化后)

class FloatingDecimal {
    // 改为 byte[] 存储
    private byte[] digits;

    // 构造函数
    FloatingDecimal(double d) {
        // ... 处理逻辑
        digits = new byte[n];
        // ... 填充 digits (直接存储 byte)
    }

    // 移除未使用的方法
    // - BigDecimal 相关方法已移至其他地方
}
```

#### DigitList.java

```java
// 文件: java/text/DigitList.java (优化后)

class DigitList {
    // 改为 byte[] 存储
    private byte[] digits = new byte[21];

    // 使用场景
    void append(int digit) {
        digits[count++] = (byte) digit;
    }

    // 直接使用 byte[]，无需转换
    void getChars(byte[] dst, int length) {
        System.arraycopy(digits, 0, dst, 0, length);
    }
}
```

### 新增 BinaryToASCIIBuffer

```java
// 新增辅助类
class BinaryToASCIIBuffer {
    private final byte[] decimalDigits;

    /**
     * 获取字符数组。
     *
     * @apiNote 此方法返回内部 byte[]，调用者应确保不修改它。
     *          返回的数组是 Latin1 编码的。
     */
    byte[] getDecimalDigits() {
        return decimalDigits;
    }

    /**
     * 获取字符。
     *
     * @param index 字符索引
     * @return 指定索引处的字符（Latin1 编码）
     */
    char getChar(int index) {
        return (char) (decimalDigits[index] & 0xff);
    }

    /**
     * 将字符复制到目标数组。
     *
     * @param dst    目标数组
     * @param offset 目标偏移
     * @param length 复制长度
     */
    void getChars(byte[] dst, int offset, int length) {
        System.arraycopy(decimalDigits, 0, dst, offset, length);
    }
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **char[] -> byte[]** | 内存占用减半 |
| **消除类型转换** | 直接存储 byte，无需 char->byte 转换 |
| **移除未使用代码** | 清理 FloatToDecimal 后的冗余代码 |
| **添加文档** | 明确 byte[] 是 Latin1 编码 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 字符范围 | 数字 0-9 都在 Latin1 范围内 |
| 编码兼容 | 使用 Latin1，单字节映射 |
| 行为一致 | 纯内部实现，API 不变 |

## 设计决策

### Q1: 为什么 DigitList 也需要改？

> **Reviewer**: DigitList 不是用于数字格式化吗？

**A**: `DigitList` 内部只存储数字 0-9：
```java
// 只存储数字
digits[count++] = (char) ('0' + digit);
// 完全可以用 byte[] 存储
```

### Q2: BinaryToASCIIBuffer 是什么？

> **Reviewer**: 为什么要新增这个类？

**A**: 作为 `FloatingDecimal` 的辅助类：
```java
// 提供统一的接口
class BinaryToASCIIBuffer {
    byte[] getDecimalDigits();  // 获取内部 byte[]
    char getChar(int index);     // 获取单个字符
    void getChars(byte[]...);    // 复制字符
}
```

### Q3: 为什么移除某些方法？

> **Reviewer**: 移除的方法是什么？

**A**: 移除 `FloatToDecimal`/`DoubleToDecimal` 后不再使用的代码：
```java
// 移除的代码
- BigDecimal bigDecimalValue();  // 已移至 BigDecimal
- boolean isFinite();             // 不再需要
- // 其他未使用方法
```

### Q4: 性能提升为什么是 5-10%？

> **Reviewer**: 为什么提升比 Integer.toString 小？

**A**:
- 浮点数转换本身就比整数转换复杂
- 主要瓶颈在算法（除法、乘法），不在存储
- byte[] 主要帮助减少内存占用
