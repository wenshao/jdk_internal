## 背景

`java.util.UUID` 是 Java 中用于表示 128 位通用唯一标识符的核心类。其 `toString()` 方法是高频调用路径，在分布式系统中被大量使用。

**为什么重要？**
- UUID 是分布式系统的标准 ID 格式（RFC 4122 / RFC 9562）
- `UUID.toString()` 在日志、追踪、数据库主键场景中频繁调用
- 原始实现使用 `Long.formatUnsignedLong0()` 逐字节格式化，效率低

**PR #14578 与 PR #14577**
- **PR #14577**: `UUID#toString` 优化（合并前）
- **PR #14578**: `Long#fastUUID` 优化（本文档主题）
- 两个 PR 最终都指向同一个目标：优化 UUID 字符串转换

**改动位置**
```
src/java.base/share/classes/java/lang/Long.java
    └── fastUUID(long lsb, long msb)  <-- 删除此方法

src/java.base/share/classes/java/util/UUID.java
    └── toString()  <-- 完全重写

    └── digit(int...)  <-- 新增批量转换方法

    └── fastUUID(...)  <-- 删除接口方法
```

## 问题分析

### 原始实现

```java
// 文件: java/lang/Long.java (PR 前)
// 行数: ~450-480

static String fastUUID(long lsb, long msb) {
    if (COMPACT_STRINGS) {
        byte[] buf = new byte[36];
        // 逐段格式化，每次调用 formatUnsignedLong0
        formatUnsignedLong0(lsb,        4, buf, 24, 12);
        formatUnsignedLong0(lsb >>> 48, 4, buf, 19, 4);
        formatUnsignedLong0(msb,        4, buf, 14, 4);
        formatUnsignedLong0(msb >>> 16, 4, buf, 9,  4);
        formatUnsignedLong0(msb >>> 32, 4, buf, 0,  8);

        buf[23] = '-';
        buf[18] = '-';
        buf[13] = '-';
        buf[8]  = '-';

        return new String(buf, LATIN1);
    } else {
        // UTF16 分支，类似逻辑
        ...
    }
}

// 格式化辅助方法
private static void formatUnsignedLong0(long val, int shift, byte[] buf, int offset, int len) {
    int charPos = offset + len;
    int radix = 1 << shift;
    int mask = radix - 1;
    do {
        buf[--charPos] = Integer.digits[((int) val) & mask];  // 逐字节查表
        val >>>= shift;
    } while (charPos > offset);
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **循环格式化** | 每次调用 `formatUnsignedLong0` 都有循环开销 | 5 次调用 = 5 次循环 |
| **逐字节写入** | 每个字节单独查表、写入 | 32 个 hex 字符 = 32 次操作 |
| **方法调用深度** | `UUID.toString()` -> `jla.fastUUID()` -> `formatUnsignedLong0()` | 多层调用 |
| **分支预测** | COMPACT_STRINGS 条件分支 | CPU 分支预测失败 |
| **UTF16 分支** | 需要特殊处理双字节字符 | 额外代码路径 |

### 为什么 `formatUnsignedLong0` 这么"重"？

```java
// 每次格式化的简化流程
formatUnsignedLong0(long val, int shift, byte[] buf, int offset, int len) {
    int charPos = offset + len;
    do {
        // 1. 数组边界检查 (隐式)
        // 2. 查表操作 (Integer.digits[])
        // 3. 位运算
        // 4. 数组写入
        buf[--charPos] = Integer.digits[((int) val) & mask];
        val >>>= shift;
    } while (charPos > offset);  // 5. 循环条件检查
}
```

对于 `fastUUID`：
- UUID 格式固定：`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`
- 每段的长度是固定的
- 不需要循环，可以直接计算位置

## 解决方案

### 核心改动

PR 采用 **查找表 + 批量写入** 的策略：

```java
// 文件: java/util/HexDigits.java (新增)

/**
 * 预计算的查找表：每个 byte 对应其 hex 表示的 short 值
 * 例如: 0x00 -> '00' -> ('0' << 8) | '0' -> 12336
 *       0xFF -> 'ff' -> ('f' << 8) | 'f' -> 26214
 */
@Stable
private static final short[] DIGITS;

// 新增：批量转换 2 个 byte 为 int (4 个 hex 字符)
static int digit(int b0, int b1) {
    return (DIGITS[b0 & 0xff] << 16) | DIGITS[b1 & 0xff];
}

// 新增：批量转换 4 个 byte 为 long (8 个 hex 字符)
static long digit(int b0, int b1, int b2, int b3) {
    return (((long) DIGITS[b0 & 0xff]) << 48)
         | (((long) DIGITS[b1 & 0xff]) << 32)
         | (DIGITS[b2 & 0xff] << 16)
         | DIGITS[b3 & 0xff];
}
```

```java
// 文件: java/util/UUID.java (优化后的 toString)

@Override
public String toString() {
    long lsb = leastSigBits;
    long msb = mostSigBits;
    byte[] buf = new byte[36];

    // 批量写入 8 个 hex 字符 (使用 ByteArray.setLong)
    ByteArray.setLong(buf, 0,
        HexDigits.digit((int) (msb >> 56), (int) (msb >> 48),
                       (int) (msb >> 40), (int) (msb >> 32)));
    buf[8] = '-';

    // 批量写入 4 个 hex 字符 (使用 ByteArray.setInt)
    ByteArray.setInt(buf, 9,
        HexDigits.digit(((int) msb) >> 24, ((int) msb) >> 16));
    buf[13] = '-';

    ByteArray.setInt(buf, 14,
        HexDigits.digit(((int) msb) >> 8, (int) msb));
    buf[18] = '-';

    ByteArray.setInt(buf, 19,
        HexDigits.digit((int) (lsb >> 56), (int) (lsb >> 48)));
    buf[23] = '-';

    ByteArray.setLong(buf, 24,
        HexDigits.digit(((int) (lsb >> 40)), (int) (lsb >> 32),
                       ((int) lsb) >> 24, ((int) lsb) >> 16));

    ByteArray.setInt(buf, 32,
        HexDigits.digit(((int) lsb) >> 8, (int) lsb));

    return jla.newStringNoRepl(buf, StandardCharsets.ISO_8859_1);
}
```

### 关键优化点

| 优化点 | 说明 | 效果 |
|--------|------|------|
| **预计算查找表** | `HexDigits.DIGITS[]` 存储 byte -> hex short 映射 | O(1) 查表 |
| **批量合并** | `digit()` 方法将多个 byte 合并为 int/long | 减少操作数 |
| **VarHandle 写入** | `ByteArray.setLong/setInt` 使用 VarHandle | 直接内存写入 |
| **消除循环** | 从 5 次循环调用变为 6 次批量写入 | 消除分支 |
| **固定偏移** | 所有写入位置是编译时常量 | 更好的 JIT 优化 |

### 查找表原理

```java
// 初始化时预计算
static {
    short[] ds = new short[256];
    for (int i = 0; i < 256; i++) {
        // 高字节: hex 高位
        // 低字节: hex 低位
        ds[i] = (short) ((toHex(i >> 4) << 8) | toHex(i & 0xf));
    }
    DIGITS = ds;
}

// 使用示例：转换 byte 0xAB
short s = DIGITS[0xAB];  // 值为 ('a' << 8) | 'b' = 0x6162
// 高字节: 'a' (0x61)
// 低字节: 'b' (0x62)
```

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 内存越界 | 固定 36 字节缓冲区，写入位置计算精确 |
| 字符编码 | 使用 ISO-8859-1 (LATIN1)，与 hex 兼容 |
| 并发安全 | 方法无状态，使用局部变量 |
| 兼容性 | 输出格式完全符合 RFC 4122 |

## 设计决策

### Q1: 为什么删除 `Long.fastUUID()`？

> **Reviewer**: 为什么不保留 `Long.fastUUID()` 作为工具方法？

**A**:
1. **封装原则**: UUID 格式化逻辑应该属于 `UUID` 类，不是 `Long` 的职责
2. **性能考虑**: 删除 `JavaLangAccess` 间接调用，减少方法调用层次
3. **维护性**: 代码集中在一个地方，更容易维护

### Q2: 为什么使用 `short[]` 而不是 `byte[]`？

> **Reviewer**: 查找表为什么用 `short[]` 而不是 `byte[]`？

**A**:
```java
// byte[] 方案：每个 byte 对应 2 个 byte (2 个 hex 字符)
byte[] table = new byte[512];  // 256 * 2
// 需要两次数组访问

// short[] 方案：每个 byte 对应 1 个 short (2 个 hex 字符合并)
short[] table = new short[256];  // 节省一半内存
// 一次数组访问 + 位操作
```
- `short[]` 更紧凑，缓存友好
- 一次访问可以获取 2 个 hex 字符

### Q3: 为什么使用 `ByteArray.setLong` 而不是 `Unsafe`？

> **Reviewer**: 为什么不直接用 `Unsafe.copyMemory`？

**A**:
1. **安全性**: `ByteArray` 是高层封装，有边界检查
2. **可读性**: 代码意图更清晰
3. **性能**: VarHandle 最终会被 JIT 内联，与 Unsafe 性能相当
4. **兼容性**: 不依赖 HotSpot 特定实现

### Q4: 为什么不用 SIMD 指令？

> **Reviewer**: 现代 CPU 有 SIMD 指令（如 AVX），为什么不用？

**A**:
1. **Java 限制**: Java 没有直接的 SIMD 暴露（直到 Vector API）
2. **复杂度**: SIMD 代码复杂，难以维护
3. **收益有限**: UUID 只有 36 字节，SIMD 优势不明显
4. **未来可期**: Vector API (JEP 338) 可能提供更好的方案

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **功能正确性** | 低 | 输出格式完全一致，现有测试全部通过 |
| **内存使用** | 无影响 | 256 字节查找表，可忽略不计 |
| **兼容性** | 无影响 | 纯内部实现，API 不变 |
| **平台差异** | 低 | VarHandle 是标准 API |
| **回归风险** | 低 | 代码清晰，易于回滚 |

### 值得注意的细节

1. **字符编码**: 使用 `ISO_8859_1` (LATIN1) 与 hex 字符兼容
2. **VarHandle**: 依赖 JVM 对 `MethodHandles.byteArrayViewVarHandle` 的优化
3. **查找表初始化**: `@Stable` 注解确保 JVM 可以优化访问
