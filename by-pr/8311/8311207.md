## 背景

`UUID.toString()` 是一个**高频调用**的方法，在以下场景中被大量使用：

- **分布式追踪**: OpenTelemetry、Jaeger 等
- **数据库主键**: PostgreSQL UUID、MongoDB ObjectId
- **会话 ID**: Web 应用会话管理
- **日志关联**: 分布式日志追踪

原实现存在以下问题：

1. **字节序混乱**: 使用 `ByteArray` (Big-Endian) 而非 `ByteArrayLittleEndian`
2. **方法名不清晰**: `HexDigits.digit()` 名称不能体现字节序
3. **代码冗余**: 多次类似调用，缺乏复用

## 问题分析

### 原始实现

```java
// 文件: java/util/UUID.java
// 行数: ~470-500 (修复前)

public String toString() {
    long lsb = leastSigBits;
    long msb = mostSigBits;
    byte[] buf = new byte[36];

    // 1. 第一段: 8 个十六进制字符
    ByteArray.setLong(
        buf, 0,
        HexDigits.digit((int) (msb >> 56), (int) (msb >> 48),
                        (int) (msb >> 40), (int) (msb >> 32)));
    buf[8] = '-';

    // 2. 第二段: 4 个十六进制字符
    ByteArray.setInt(
        buf, 9,
        HexDigits.digit(((int) msb) >> 24, ((int) msb) >> 16));
    buf[13] = '-';

    // 3. 第三段: 4 个十六进制字符
    ByteArray.setInt(
        buf, 14,
        HexDigits.digit(((int) msb) >> 8, (int) msb));
    buf[18] = '-';

    // 4. 第四段: 4 个十六进制字符
    ByteArray.setInt(
        buf, 19,
        HexDigits.digit((int) (lsb >> 56), (int) (lsb >> 48)));
    buf[23] = '-';

    // 5. 第五段: 12 个十六进制字符
    ByteArray.setLong(
        buf, 24,
        HexDigits.digit(((int) (lsb >> 40)), (int) (lsb >> 32),
                        ((int) lsb) >> 24, ((int) lsb) >> 16));
    ByteArray.setInt(
        buf, 32,
        HexDigits.digit(((int) lsb) >> 8, (int) lsb));

    return jla.newStringNoRepl(buf, StandardCharsets.ISO_8859_1);
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **字节序转换** | `ByteArray` 是 Big-Endian，需要额外转换 | CPU 指令增加 |
| **方法调用开销** | 多次调用 `ByteArray.setLong/setInt` | 函数调用开销 |
| **HexDigits 字节序** | 原实现使用 Big-Endian，与内存布局不一致 | 额外的字节序反转 |

### 为什么 Big-Endian 更慢？

```java
// Big-Endian (ByteArray.setLong)
void setLong(byte[] buf, int off, long v) {
    buf[off + 7] = (byte) (v      );  // 最低字节在最高地址
    buf[off + 6] = (byte) (v >>  8);
    buf[off + 5] = (byte) (v >> 16);
    buf[off + 4] = (byte) (v >> 24);
    buf[off + 3] = (byte) (v >> 32);
    buf[off + 2] = (byte) (v >> 40);
    buf[off + 1] = (byte) (v >> 48);
    buf[off    ] = (byte) (v >> 56);  // 最高字节在最低地址
}

// Little-Endian (ByteArrayLittleEndian.setLong)
void setLong(byte[] buf, int off, long v) {
    // 直接使用 Unsafe.putLong，一次内存写入
    UNSAFE.putLong(buf, ARRAY_BYTE_BASE_OFFSET + off, v);
}
```

**Little-Endian 的优势**：
- 与 x86/x64 CPU 的原生字节序一致
- 可以使用单次内存写入指令
- 无需手动字节序反转

## 解决方案

### 核心改动

#### 1. 改用 `ByteArrayLittleEndian`

```java
// 修复后
- import jdk.internal.util.ByteArray;
+ import jdk.internal.util.ByteArrayLittleEndian;

public String toString() {
    long lsb = leastSigBits;
    long msb = mostSigBits;
    byte[] buf = new byte[36];

-   ByteArray.setLong(
+   ByteArrayLittleEndian.setLong(
        buf, 0,
-       HexDigits.digit((int) (msb >> 56), (int) (msb >> 48),
-                       (int) (msb >> 40), (int) (msb >> 32)));
+       HexDigits.packDigits((int) (msb >> 56), (int) (msb >> 48),
+                            (int) (msb >> 40), (int) (msb >> 32)));
    // ... 其他类似修改
}
```

#### 2. 重命名方法为 `packDigits()`

```java
// HexDigits.java

/**
 * Return a little-endian packed integer for the 4 ASCII bytes
 * for an input unsigned 2-byte integer.
 * {@code b0} is the most significant byte and {@code b1} is the least significant byte.
 */
- public static int digit(int b0, int b1) {
-     return (DIGITS[b0 & 0xff] << 16) | DIGITS[b1 & 0xff];
+ public static int packDigits(int b0, int b1) {
+     return DIGITS[b0 & 0xff] | (DIGITS[b1 & 0xff] << 16);
  }

/**
 * Return a little-endian packed long for the 8 ASCII bytes
 * for an input unsigned 4-byte integer.
 */
- public static long digit(int b0, int b1, int b2, int b3) {
-     return (((long) DIGITS[b0 & 0xff]) << 48)
-           | (((long) DIGITS[b1 & 0xff]) << 32)
-           | (DIGITS[b2 & 0xff] << 16)
-           | DIGITS[b3 & 0xff];
+ public static long packDigits(int b0, int b1, int b2, int b3) {
+     return DIGITS[b0 & 0xff]
+           | (DIGITS[b1 & 0xff] << 16)
+           | (((long) DIGITS[b2 & 0xff]) << 32)
+           | (((long) DIGITS[b3 & 0xff]) << 48);
  }
```

**关键变化**：
- 方法名从 `digit()` 改为 `packDigits()`，更清晰地表示"打包"操作
- 字节序从 Big-Endian 改为 Little-Endian
- 移位方向调整，匹配 Little-Endian 布局

#### 3. HexDigits 数组改为 Little-Endian

```java
// HexDigits.java

@Stable
private static final short[] DIGITS;

static {
    short[] digits = new short[16 * 16];
    for (int i = 0; i < 16; i++) {
-       short hi = (short) ((i < 10 ? i + '0' : i - 10 + 'a') << 8);
+       short lo = (short) (i < 10 ? i + '0' : i - 10 + 'a');
        for (int j = 0; j < 16; j++) {
-           short lo = (short) (j < 10 ? j + '0' : j - 10 + 'a');
+           short hi = (short) ((j < 10 ? j + '0' : j - 10 + 'a') << 8);
            digits[(i << 4) + j] = (short) (hi | lo);
        }
    }
    DIGITS = digits;
}
```

**Little-Endian 布局**：
- `DIGITS[0] = 0x3030` = `'0' | ('0' << 8)` = "00"
- `DIGITS[0x0a] = 0x6130` = `'0' | ('a' << 8)` = "a0"
- `DIGITS[0xff] = 0x6666` = `'f' | ('f' << 8)` = "ff"

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 字节序错误 | `ByteArrayLittleEndian` 与 Little-Endian `DIGITS` 匹配 |
| 输出变化 | 验证输出字符串完全相同 |
| 平台差异 | `ByteArrayLittleEndian` 在所有平台行为一致 |

## 设计决策

### Q1: 为什么不继续使用 `ByteArray`？

> **Reviewer**: `ByteArray` 已经存在，为什么引入 `ByteArrayLittleEndian`？

**A**:

1. **性能**: `ByteArrayLittleEndian` 可以使用 `Unsafe.putLong` 一次性写入 8 字节
2. **字节序**: x86/x64 是 Little-Endian，使用原生字节序更快
3. **一致性**: 与 `StringLatin1` 等其他类保持一致

### Q2: 为什么重命名 `digit()` 为 `packDigits()`？

> **Reviewer**: 方法名变化有什么意义？

**A**:

1. **清晰性**: `packDigits` 更准确地描述"打包"操作
2. **字节序**: 区分 Little-Endian 和 Big-Endian 版本
3. **未来扩展**: 为后续优化（如 `DecimalDigits.digitPair()`）铺路

### Q3: 为什么有些平台性能提升不明显？

> **Reviewer**: ARM 和某些 Intel 平台提升较小？

**A**:

| 平台 | 提升 | 原因 |
|------|------|------|
| Intel Sapphire Rapids | +33.8% | 支持更激进的内存合并优化 |
| AMD Genoa | +0.6% | 内存操作已经高度优化 |
| ARM Aliyun Yitian | +0.5% | ARM 内存模型不同 |
| Apple M1 Pro | +7.2% | Apple Silicon 优化良好 |

**结论**: 性能提升取决于 JIT 编译器的优化能力，但所有平台都有正向收益。
