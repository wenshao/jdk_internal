## 背景

`DateTimePrintContext` 是 Java 8 引入的 `java.time.format` 包中的核心类，用于在日期时间格式化过程中传递上下文信息。

**为什么重要？**
- `DateTimeFormatter` 是 JDK 中日期时间格式化的核心类
- 每次调用 `format()` 方法都会创建 `DateTimePrintContext` 实例
- 高频调用场景（如日志格式化）中，此对象创建极其频繁

**原始设计的缺陷**
```java
// 旧设计：可变状态
final class DateTimePrintContext {
    private int optional;  // 可变字段，用于嵌套可选格式化

    void startOptional() { optional++; }
    void endOptional() { optional--; }
}
```

这种可变设计有以下问题：
1. **阻止逃逸分析优化** - 可变对象无法被 JIT 完全优化到栈上
2. **线程安全性顾虑** - 虽然实际使用是单线程，但可变状态增加了认知负担
3. **代码复杂度** - 需要在 `try-finally` 块中管理状态

**改动位置**
```
src/java.base/share/classes/java/time/format/
    ├── DateTimePrintContext.java    ← 移除可变状态
    └── DateTimeFormatter.java       ← 修改接口签名，传递 optional 参数
```

## 问题分析

### 原始实现

```java
// 文件: java/time/format/DateTimePrintContext.java
// 旧版本（部分代码）

final class DateTimePrintContext {
    private final TemporalAccessor temporal;
    private final DateTimeFormatter formatter;
    private int optional = 0;  // 可变字段

    void startOptional() {
        optional++;
    }

    void endOptional() {
        optional--;
    }

    // 获取值时检查 optional 状态
    <R> R getValue(TemporalQuery<R> query) {
        R result = temporal.query(query);
        if (result == null && optional == 0) {  // 只有在非可选时才抛异常
            throw new DateTimeException("Unable to extract " + query);
        }
        return result;
    }
}
```

```java
// 文件: java/time/format/DateTimeFormatter.java
// 旧版本的调用方式

public CompositePrinterParser withOptional(boolean optional) {
    // ...
    @Override
    public boolean format(DateTimePrintContext context, StringBuilder buf) {
        int length = buf.length();
        if (optional) {
            context.startOptional();  // 修改可变状态
        }
        try {
            for (DateTimePrinterParser pp : printerParsers) {
                if (!pp.format(context, buf)) {
                    buf.setLength(length);
                    return true;
                }
            }
        } finally {
            if (optional) {
                context.endOptional();  // 恢复可变状态
            }
        }
        return true;
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **逃逸分析失败** | 可变字段导致对象无法被标为"不可逃逸" | 对象分配在堆而非栈 |
| **状态管理开销** | `startOptional()`/`endOptional()` 调用 | 方法调用 + 字段修改 |
| **JIT 优化受限** | 无法应用标量替换、锁消除等优化 | 性能上限受限 |

### 为什么不可变对象能优化？

HotSpot JIT 编译器的逃逸分析（Escape Analysis）可以优化不可变对象：
1. **栈上分配** - 对象不逃逸时，分配在栈上，方法结束时自动回收
2. **标量替换** - 将对象字段拆分为独立局部变量，消除对象创建
3. **锁消除** - 不可变对象无需同步

```java
// JIT 优化前
DateTimePrintContext ctx = new DateTimePrintContext(temporal, formatter);
ctx.startOptional();
// ... 使用 ctx
ctx.endOptional();

// JIT 优化后（标量替换）
TemporalAccessor _temporal = temporal;
DateTimeFormatter _formatter = formatter;
// 直接使用 _temporal 和 _formatter，无需对象创建
```

## 解决方案

### 核心改动

**1. 移除可变状态**

```java
// 新版本: DateTimePrintContext 完全不可变

final class DateTimePrintContext {
    private final TemporalAccessor temporal;
    private final DateTimeFormatter formatter;

    // 构造函数不接收 optional
    DateTimePrintContext(TemporalAccessor temporal, DateTimeFormatter formatter) {
        this.temporal = adjust(temporal, formatter);
        this.formatter = formatter;
    }

    // 移除 startOptional() 和 endOptional()

    // getValue 方法直接接收 optional 参数
    <R> R getValue(TemporalQuery<R> query, boolean optional) {
        R result = temporal.query(query);
        if (result == null && !optional) {
            throw new DateTimeException("Unable to extract " + query);
        }
        return result;
    }
}
```

**2. 修改接口签名**

```java
// DateTimePrinterParser 接口
interface DateTimePrinterParser {
-    boolean format(DateTimePrintContext context, StringBuilder buf);
+    boolean format(DateTimePrintContext context, StringBuilder buf, boolean optional);
}
```

**3. 更新所有实现类**

```java
// 新版本的调用方式
public boolean format(DateTimePrintContext context, StringBuilder buf, boolean optional) {
    int length = buf.length();
+    boolean effectiveOptional = optional | this.optional;  // 合并 optional 状态
    for (DateTimePrinterParser pp : printerParsers) {
-        if (!pp.format(context, buf)) {
+        if (!pp.format(context, buf, effectiveOptional)) {
            buf.setLength(length);
            return true;
        }
    }
    return true;
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除可变状态** | 移除 `optional` 字段，改为参数传递 |
| **简化代码结构** | 无需 `try-finally` 块管理状态 |
| **启用逃逸分析** | 不可变对象可被 JIT 优化 |
| **保持语义一致** | `optional | this.optional` 合并状态确保行为不变 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 行为变更 | `optional | this.optional` 确保嵌套可选格式化语义一致 |
| API 破坏 | `DateTimePrinterParser` 是内部接口，不暴露给用户 |
| 性能回归 | 参数传递开销远小于逃逸分析带来的收益 |

## 设计决策

### Q1: 为什么不用 `ThreadLocal` 保存 optional 状态？

> **Reviewer**: 使用 `ThreadLocal` 可以避免修改接口签名，为什么不用？

**A**:
1. `ThreadLocal` 有性能开销（查找、内存占用）
2. 增加了代码复杂度和认知负担
3. 不可变对象 + 参数传递是更清晰的设计

### Q2: 性能提升主要来自哪里？

> **Reviewer**: 是减少了状态管理，还是 JIT 优化效果？

**A**: 主要是 JIT 优化：
- 不可变对象可被逃逸分析优化
- 可进行标量替换，完全消除对象分配
- 参数传递的开销被 JIT 内联优化掉

### Q3: 为什么不直接内联 DateTimePrintContext？

> **Reviewer**: 既然这个类这么简单，为什么不直接内联到使用处？

**A**:
1. `DateTimePrintContext` 封装了复杂的 `adjust()` 逻辑
2. 代码复用 - 多个 `PrinterParser` 共享同一个上下文
3. 保持代码结构清晰，便于维护
