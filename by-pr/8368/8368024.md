## 背景

字符串拼接是 Java 中最常用的操作之一。从 JDK 9 开始，字符串拼接策略经历了重大演变。

**历史背景**
- **JDK 9**: 引入 `StringConcatFactory`，使用 `invokedynamic` 实现字符串拼接
- **JDK 20-21**: 引入多种字符串拼接策略
- **JDK 23+**: `InlineHiddenClassStrategy` 成为默认策略（[JDK-8336856](https://bugs.openjdk.org/browse/JDK-8336856)）

**问题来源**
`generateMHInlineCopy` 方法是早期字符串拼接优化的产物，用于生成内联的字符串拼接代码。随着 `InlineHiddenClassStrategy` 的成熟和普及，这个方法已经不再被使用。

**为什么重要？**
- 删除死代码减少维护负担
- 简化字符串拼接实现逻辑
- 降低新开发者理解成本

**改动位置**
```
src/java.base/share/classes/java/lang/
    ├── invoke/StringConcatFactory.java    ← 删除 generateMHInlineCopy
    └── StringConcatHelper.java           ← 删除辅助方法
```

4. **标记为 `noreg-cleanup`** - 无需新的回归测试

## 解决方案

### 核心改动

**删除的代码结构**：

```
删除内容统计：
├── 常量定义                    ~10 行
├── generateMHInlineCopy()     ~100 行
├── coder()                    ~5 行
├── checkOverflow()            ~10 行
├── mix() 系列方法 (5个重载)    ~50 行
├── prepend() 系列方法 (5个重载) ~200 行
├── newString()                ~20 行
├── newArrayWithSuffix()       ~20 行
├── newArray()                 ~15 行
├── initialCoder()             ~10 行
└── 其他辅助方法               ~300 行
───────────────────────────────────
总计                         ~800 行
```

**保留的代码**：

```java
// StringConcatHelper.java 中保留的核心方法

// 简单的字符串拼接（用于启动性能）
static String simpleConcat(Object first, Object second) {
    String s1 = stringOf(first);
    String s2 = stringOf(second);
    return s1 + s2;  // 使用简单的 + 操作符
}

// 数组分配（保留，被新策略使用）
static byte[] newArray(int length) {
    return (byte[]) UNSAFE.allocateUninitializedArray(byte.class, length);
}

// 带后缀的数组分配（更新签名）
@ForceInline
static byte[] newArrayWithSuffix(String suffix, int index, byte coder) {
    byte[] buf = newArray(index + suffix.length());
    if (coder == 0) {
        suffix.getBytes(buf, index, String.LATIN1);
    } else {
        suffix.getBytes(buf, index, String.UTF16);
    }
    return buf;
}
```

### 关键变更点

| 变更类型 | 说明 |
|----------|------|
| **删除废弃方法** | `generateMHInlineCopy()` 及所有辅助方法 |
| **更新方法签名** | `newArrayWithSuffix()` 参数从 `long indexCoder` 拆分为 `int index, byte coder` |
| **保留核心功能** | `simpleConcat()`, `newArray()` 等仍被使用的方法 |
| **清理常量** | 删除 `LATIN1`, `UTF16` 等私有常量 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 仍有隐藏引用 | 代码审查确认无活跃引用 |
| 行为变化 | 删除的是死代码，不影响运行时行为 |
| 测试覆盖 | 标记 `noreg-cleanup`，现有测试充分 |

## 设计决策

### Q1: 为什么不保留 `generateMHInlineCopy` 作为备用？

> **Reviewer**: 保留这个方法作为备用策略不好吗？

**A**:
1. **一年生产验证** - `InlineHiddenClassStrategy` 已充分验证
2. **维护成本** - 保留死代码增加维护负担
3. **代码清晰度** - 删除废弃代码让主线逻辑更清晰
4. **可以回退** - 如果需要，可以从 git 历史恢复

### Q2: `newArrayWithSuffix()` 为什么要修改签名？

> **Reviewer**: 为什么不直接删除这个方法？

**A**:
- `newArrayWithSuffix()` 仍被活跃代码使用
- 旧签名 `long indexCoder` 是为了兼容 `generateMHInlineCopy` 的设计
- 新签名 `int index, byte coder` 更清晰，更符合当前使用模式

### Q3: 删除这些代码会影响性能吗？

> **Reviewer**: 这些代码的存在会影响性能吗？

**A**:
- **不影响运行时性能** - JIT 编译器不会编译未使用的方法
- **减少类加载时间** - 更少的字节码需要验证和加载
- **减少元数据开销** - 更少的方法元数据占用内存
