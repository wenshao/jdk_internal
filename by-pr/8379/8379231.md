## 描述

For x86_64, `Assembler::mov64` always emits a 10-byte `movabs` regardless of the immediate value. For values that fit in 32 bits, we can use shorter encodings with `movl` and `movq`.

This matches what `MacroAssembler::movptr` already does. I only found `MacroAssembler::ic_call` that depends on the 10-byte encoding for patching and switched it to the other overload to keep `movabs`.

Measured ~460 bytes saved in renaissance dotty (single iteration), saw ~1K bytes saved during JDK build, by logging bytes saved (not in the PR).

I'm wondering if `Assembler` is the right level for this, since it's seems to be a thin mapping to machine instructions and instruction selection is normally `MacroAssembler`'s job. On the other hand, `mov64` is already a pseudo-instruction (not a real x86 mnemonic), so arguably it's fine to have it pick the best encoding.
