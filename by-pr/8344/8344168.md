# JDK-8344168: Change Unsafe base offset from int to long

> **Issue**: [JDK-8344168](https://bugs.openjdk.org/browse/JDK-8344168)
> **Author**: [Shaojin Wen](/by-contributor/profiles/shaojin-wen.md) (@wenshao)

## 背景

`Unsafe` 类是 JDK 中用于低级内存操作的核心类。`ARRAY_XXX_BASE_OFFSET` 常量表示各种数组类型的基础内存偏移量。

**问题起源**：
- JDK-8343925 和 JDK-8343984 修复了大量偏移量溢出问题
- 但根本原因是 `BASE_OFFSET` 是 `int` 类型
- 当 `BASE_OFFSET + index` 计算时，可能溢出

**为什么重要？**
- 影响所有数组访问的 Unsafe 操作
- 溢出可能导致内存访问错误
- 这是架构层面的修复

**改动位置**
```
src/java.base/share/classes/jdk/internal/misc/Unsafe.java
    └── ARRAY_BOOLEAN_BASE_OFFSET: int → long
    └── ARRAY_BYTE_BASE_OFFSET: int → long
    └── ARRAY_SHORT_BASE_OFFSET: int → long
    └── ARRAY_CHAR_BASE_OFFSET: int → long
    └── ARRAY_INT_BASE_OFFSET: int → long
    └── ARRAY_LONG_BASE_OFFSET: int → long
    └── ARRAY_FLOAT_BASE_OFFSET: int → long
    └── ARRAY_DOUBLE_BASE_OFFSET: int → long
    └── ARRAY_OBJECT_BASE_OFFSET: int → long
```

## 问题分析

### 原始实现（int 类型）

```java
// 文件: jdk/internal/misc/Unsafe.java
// 修改前: int 类型

public static final int ARRAY_BYTE_BASE_OFFSET = theUnsafe.arrayBaseOffset(byte[].class);
public static final int ARRAY_CHAR_BASE_OFFSET = theUnsafe.arrayBaseOffset(char[].class);
// ... 其他数组类型
```

### 溢出场景分析

```java
// 典型的 Unsafe 内存访问模式
byte[] array = new byte[largeSize];
int index = someIndex;

// 计算内存地址
long address = ARRAY_BYTE_BASE_OFFSET + index;  // 💥 可能溢出！

// 问题：ARRAY_BYTE_BASE_OFFSET 是 int
// 当 index 很大时，int + int 可能溢出
```

**具体溢出示例**：

```java
// 假设 ARRAY_BYTE_BASE_OFFSET = 16 (对象头大小)
int index = Integer.MAX_VALUE - 10;

// 计算
int offset = ARRAY_BYTE_BASE_OFFSET + index;
// = 16 + (2147483647 - 10)
// = 2147483653
// 但 int 最大值是 2147483647
// 结果：溢出变为负数！
```

### 真实世界的影响

| 问题 | 描述 | 严重程度 |
|------|------|----------|
| **内存损坏** | 溢出导致错误的内存地址 | 高 |
| **安全漏洞** | 可能被利用访问任意内存 | 高 |
| **数据损坏** | 读写错误位置 | 高 |
| **JVM 崩溃** | 访问非法内存地址 | 高 |

## 解决方案

### 核心改动：int → long

```java
// 修改后: long 类型

/** The value of {@code arrayBaseOffset(boolean[].class)} */
public static final long ARRAY_BOOLEAN_BASE_OFFSET
        = theUnsafe.arrayBaseOffset(boolean[].class);

/** The value of {@code arrayBaseOffset(byte[].class)} */
public static final long ARRAY_BYTE_BASE_OFFSET
        = theUnsafe.arrayBaseOffset(byte[].class);

/** The value of {@code arrayBaseOffset(short[].class)} */
public static final long ARRAY_SHORT_BASE_OFFSET
        = theUnsafe.arrayBaseOffset(short[].class);

// ... 所有数组类型都改为 long
```

### 受影响的常量

| 常量名 | 修改前 | 修改后 |
|--------|--------|--------|
| `ARRAY_BOOLEAN_BASE_OFFSET` | `int` | `long` |
| `ARRAY_BYTE_BASE_OFFSET` | `int` | `long` |
| `ARRAY_SHORT_BASE_OFFSET` | `int` | `long` |
| `ARRAY_CHAR_BASE_OFFSET` | `int` | `long` |
| `ARRAY_INT_BASE_OFFSET` | `int` | `long` |
| `ARRAY_LONG_BASE_OFFSET` | `int` | `long` |
| `ARRAY_FLOAT_BASE_OFFSET` | `int` | `long` |
| `ARRAY_DOUBLE_BASE_OFFSET` | `int` | `long` |
| `ARRAY_OBJECT_BASE_OFFSET` | `int` | `long` |

### 连带影响：53 个文件修改

由于这些常量被广泛使用，需要更新所有使用点：

```java
// 典型的修改模式

// 修改前
int offset = Unsafe.ARRAY_BYTE_BASE_OFFSET + index;
UNSAFE.putByte(array, offset, value);

// 修改后
long offset = Unsafe.ARRAY_BYTE_BASE_OFFSET + index;
UNSAFE.putByte(array, offset, value);
```

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| **类型转换** | `int` → `long` 是安全扩展 |
| **兼容性** | `long` 可以容纳所有 `int` 值 |
| **溢出预防** | `long + int` 不会溢出（在合理范围内） |

## 设计决策

### Q1: 为什么不改所有 offset 为 long？

> **Reviewer**: 有很多地方的 offset 还是 int，为什么不一起改？

**A**:

作者回复：

> "此 PR 只改 `BASE_OFFSET`，因为它们是常量。其他地方的 offset 是变量或参数，需要单独评估。"

分阶段策略：
1. **第一阶段**（此 PR）：改 `BASE_OFFSET` 常量
2. **后续阶段**：根据需要改其他 offset

### Q2: 为什么不用 @IntrinsicCandidate 处理溢出？

> **Reviewer**: 为什么不能在 intrinsic 中检测溢出？

**A**:

Intrinsic 无法完全解决溢出问题：

```java
// 即使有 intrinsic，计算仍然在 Java 层面发生
int offset = BASE_OFFSET + index;  // 这里已经溢出了
UNSAFE.putByte(array, offset);     // intrinsic 接收的是错误的值
```

根本解决方案是让 `offset` 类型为 `long`，而不是在 intrinsic 中检测。

### Q3: 这对 JVMCI 有什么影响？

> **Reviewer**: JVMCI (JVM Compiler Interface) 也需要更新吗？

**A**:

是的，后续 PR (JDK-8349743) 更新了 JVMCI：

```java
// JVMCI MetaAccessProvider 也使用 long
public interface MetaAccessProvider {
    // 修改前
    int arrayBaseOffset(Class<?> arrayClass);

    // 修改后
    long arrayBaseOffset(Class<?> arrayClass);
}
```

这确保了一致性。

### Q4: 为什么 sun.misc.Unsafe 也需要更新？

> **Reviewer**: `sun.misc.Unsafe` 是已废弃的类，为什么还要更新？

**A**:

虽然 `sun.misc.Unsafe` 已废弃，但：
1. **仍被广泛使用**：很多代码仍在使用
2. **一致性**：新旧 API 应该保持一致
3. **逐步迁移**：给用户时间迁移到 `jdk.internal.misc.Unsafe`

```java
// sun.misc.Unsafe（废弃但保留）
@Deprecated
public class Unsafe {
    public static final int ARRAY_BYTE_BASE_OFFSET = ...;  // 更新
}

// jdk.internal.misc.Unsafe（推荐）
public final class Unsafe {
    public static final long ARRAY_BYTE_BASE_OFFSET = ...;  // 主要更新
}
```

## 性能对比

### 对性能的影响

| 指标 | 影响 | 说明 |
|------|------|------|
| **计算性能** | 中性 | long 加法与 int 加法速度相同 |
| **内存使用** | 略增 | long 占用 8 字节 vs int 4 字节 |
| **代码大小** | 略增 | long 指令可能稍大 |
| **溢出安全性** | 大幅提升 | 消除溢出风险 |

### 为什么性能影响中性？

现代 x86-64 架构：
- `int` 加法：1 周期
- `long` 加法：1 周期
- 性能相同

### 溢出修复的收益

虽然微基准可能看不出差异，但**正确性收益**巨大：

```java
// 修复前：可能溢出
int offset = INT_BASE_OFFSET + largeIndex;  // 💥

// 修复后：安全
long offset = LONG_BASE_OFFSET + largeIndex;  // ✅
```

### 真实场景示例

```java
// 1. 大数组处理
byte[] largeArray = new byte[Integer.MAX_VALUE - 100];
// 修复前：BASE_OFFSET + (MAX_VALUE - 100) 可能溢出
// 修复后：安全计算

// 2. DirectByteBuffer
long address = UNSAFE.allocateMemory(size);
UNSAFE.putByte(null, address + BASE_OFFSET + largeIndex, value);
// 修复前：可能溢出
// 修复后：安全

// 3. 数组拷贝
UNSAFE.copyMemory(
    src, BASE_OFFSET + srcPos,
    dest, BASE_OFFSET + destPos,
    length
);
// 修复前：大位置可能溢出
// 修复后：安全
```

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **源码兼容性** | 🟡 中 | 需要重新编译使用这些常量的代码 |
| **二进制兼容性** | 🟢 低 | long 可以兼容 int 值 |
| **性能回归** | 🟢 无 | long 操作与 int 同速 |
| **测试覆盖** | 🟡 中 | 需要全面测试大数组场景 |

### 源码兼容性说明

```java
// 用户代码
int offset = Unsafe.ARRAY_BYTE_BASE_OFFSET;  // 编译错误！

// 需要修改为
long offset = Unsafe.ARRAY_BYTE_BASE_OFFSET;  // 正确
```

对于 JDK 内部代码，这是统一的修改。对于外部代码，需要重新编译。
