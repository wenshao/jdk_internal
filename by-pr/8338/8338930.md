## 背景

`StringConcatFactory` 是 Java 字符串拼接的底层实现，使用 `invokedynamic` 指令在运行时生成高效的拼接代码。

**现有策略**
- 对于小参数量（< 16）：强制内联
- 对于中等参数量：生成隐藏类并缓存实例
- 缓存使用 `SoftReference`，可以 GC 回收

**问题场景**
PR #20273 改进了字符串拼接性能，但当参数数量很大时（> 20）：
- 缓存命中率低（每个不同的参数类型组合都会生成新类）
- 缓存管理开销（SoftReference 检查、map 查找）
- 隐藏类实例化开销

**解决思路**
当参数数量非常大时，直接使用静态方法，将常量内联到字节码中，避免缓存和实例化。

## 问题分析

### 原始实现

```java
// 文件: java/lang/invoke/StringConcatFactory.java
// 行数: ~1254-1280

private static CallSite generate(Lookup lookup, MethodType concatArgs, String[] constants) {
    // 总是尝试从缓存获取
    var weakConstructorHandle = CACHE.get(concatArgs);
    if (weakConstructorHandle != null) {
        MethodHandlePair handlePair = weakConstructorHandle.get();
        if (handlePair != null) {
            try {
                var instance = handlePair.constructor.invokeBasic((Object)constants);
                return handlePair.concatenator.bindTo(instance);
            } catch (Throwable e) {
                throw new StringConcatException("Exception while utilizing the hidden class", e);
            }
        }
    }

    // 缓存未命中，生成新的隐藏类
    // ...
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **缓存查找** | ConcurrentHashMap.get() + SoftReference.get() | 每次调用都需要查找 |
| **缓存未命中** | 大参数量时类型组合太多，命中率低 | 频繁生成新类 |
| **实例化开销** | invokeBasic 创建隐藏类实例 | 增加启动时间 |
| **GC 压力** | SoftReference 在内存压力下被回收 | 重复生成类 |

### 为什么大参数量缓存效果差？

```
参数数量 = N
每个参数可能是: String, int, long, boolean, char, Object, float, double

类型组合数 ≈ 8^N

当 N = 20:
- 理论组合数 = 8^20 ≈ 1.15 × 10^18
- 实际应用中只会用到其中极小一部分
- 缓存命中率极低
```

## 解决方案

### 核心改动

```java
// 新增配置参数
private static final int CACHE_THRESHOLD;
static {
    String cacheThreshold = VM.getSavedProperty("java.lang.invoke.StringConcat.cacheThreshold");
    CACHE_THRESHOLD = cacheThreshold != null ? Integer.parseInt(cacheThreshold) : 256;
}

// 根据参数数量选择策略
private static CallSite generate(Lookup lookup, MethodType concatArgs, String[] constants) {
    boolean forceInline  = concatArgs.parameterCount() < FORCE_INLINE_THRESHOLD;
    boolean staticConcat = concatArgs.parameterCount() >= CACHE_THRESHOLD;  // 新增判断

    if (!staticConcat) {
        // 原有逻辑：尝试缓存
        var weakConstructorHandle = CACHE.get(concatArgs);
        // ...
    }

    // 生成代码时根据策略选择不同方式
    MethodTypeDesc prependArgs = prependArgs(concatArgs, staticConcat);
    // ...
}
```

### 静态方法 vs 实例方法

| 方面 | 实例方法（原策略） | 静态方法（新策略） |
|------|------------------|------------------|
| 方法签名 | `concat(Object... args)` | `static concat(Object... args)` |
| 常量处理 | 存储在实例字段 | 直接内联到字节码 |
| 缓存 | SoftReference 缓存 | 无缓存 |
| 实例化 | 需要 invokeBasic | 不需要 |

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **跳过缓存** | 参数 >= 256 时直接生成，不查缓存 |
| **静态方法** | 使用 `ACC_STATIC` 标志，无需实例 |
| **常量内联** | coder 和 length 在编译时计算并内联 |
| **新 JavaLangAccess** | 添加 `stringCoder(String)` 获取字符串 coder |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 字节码膨胀 | 常量内联增加少量字节，但可接受 |
| 方法签名变化 | 仅内部生成代码，不影响用户 |
| 性能回归 | 只影响大参数量场景，原有缓存效果本就差 |

## 设计决策

### Q1: 为什么阈值是 256？

> **Reviewer**: 为什么选择 256 作为缓存阈值？

**A**:
1. 经验值：256 个参数已经非常罕见
2. 缓存效率：此时缓存命中率几乎为 0
3. 可配置：通过 JVM 参数 `java.lang.invoke.StringConcat.cacheThreshold` 可调整

### Q2: 为什么不直接用静态方法？

> **Reviewer**: 为什么不所有场景都用静态方法？

**A**:
1. 小参数量时缓存效果很好
2. 实例方法可以共享常量数组，节省字节码
3. 静态方法需要为每个常量组合生成不同的类

### Q3: 静态方法如何访问常量？

> **Reviewer**: 没有实例字段，常量如何传递？

**A**: 通过 `ldc` 指令直接将常量加载到字节码中：

```java
// 实例方法方式:
aload(constantsSlot)
ldc(i)
aaload()  // 从数组获取常量

// 静态方法方式:
ldc(constants[i])  // 直接加载常量
```

## 性能对比

### 测试场景

```java
// 极端情况：300 个参数的字符串拼接
String result = a1 + a2 + a3 + ... + a300;
```

### 预期效果

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 首次调用时间 | 类生成 + 实例化 | 类生成（无实例化） | ~10-20% |
| 后续调用时间 | 缓存查找（可能未命中） | 直接调用静态方法 | ~5-15% |
| 内存占用 | SoftReference 开销 | 无缓存开销 | 更少 |
| GC 压力 | SoftReference 回收 | 无 GC 影响 | 更稳定 |

### 不同参数数量的行为

```
参数数量 vs 策略:
├── [0, 16):      强制内联 + 实例方法
├── [16, 256):    缓存 + 实例方法（可能命中缓存）
└── [256, ∞):     静态方法（无缓存）
```

## 实际影响场景

### 受益场景

| 场景 | 说明 | 预期效果 |
|------|------|----------|
| **批量 SQL 构建** | 生成包含大量列的 INSERT 语句 | 避免缓存未命中开销 |
| **日志聚合** | 一次性拼接大量日志字段 | 减少类生成时间 |
| **代码生成** | 模板引擎生成大块代码 | 静态方法直接调用 |

### 不受影响的场景

| 场景 | 说明 |
|------|------|
| **普通字符串拼接** | 绝大多数应用 < 256 个参数 |
| **小型应用** | 参数数量少，原有缓存策略有效 |
