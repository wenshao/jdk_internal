## 背景

`DecimalDigits` 是 JDK 内部用于高效数字转字符的核心工具类，位于 `jdk.internal.util` 包。它提供了将整数转换为十进制字符的高性能实现，被 `Integer.toString()`、`Long.toString()` 以及各种格式化类广泛使用。

**为什么重要？**
- `Integer/Long.toString()` 是 Java 中最常用的方法之一
- 数字格式化是 JSON 序列化、日志输出、调试信息中的高频操作
- `putPair` 系列方法是数字转字符的核心路径，每次调用访问 `DIGITS` 数组

**改动位置**
```
src/java.base/share/classes/jdk/internal/util/DecimalDigits.java
    ├── DIGITS 数组初始化 (行 62-75)  ← 数组大小从 100 扩展到 128
    ├── putPair(char[], int, int)      ← 添加 & 0x7f 掩码
    ├── putPairLatin1(byte[], int, int) ← 添加 & 0x7f 掩码
    └── putPairUTF16(byte[], int, int) ← 添加 & 0x7f 掩码
```

## 问题分析

### 原始实现

```java
// 文件: jdk/internal/util/DecimalDigits.java
// 修改前 (行 62-66)

static {
    short[] digits = new short[10 * 10];  // 100 个元素

    for (int i = 0; i < 10; i++) {
        short hi = (short) (i + '0');
        for (int j = 0; j < 10; j++) {
            short lo = (short) ((j + '0') << 8);
            digits[i * 10 + j] = (short) (hi | lo);
        }
    }
    DIGITS = digits;
}

// 修改前的方法 (行 394-400)
public static void putPair(char[] buf, int charPos, int v) {
    int packed = DIGITS[v];  // 直接使用 v 作为索引
    buf[charPos    ] = (char) (packed & 0xFF);
    buf[charPos + 1] = (char) (packed >> 8);
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **数组边界检查** | JVM 需要验证 `0 <= v < digits.length` | 每次数组访问都有分支 |
| **分支预测失败** | 边界检查可能导致流水线停顿 | 现代 CPU 损失 ~10-20 周期 |
| **JIT 优化限制** | C2 编译器无法证明 `v` 始终在 [0, 99] 范围内 | 无法消除边界检查 |

### JVM 边界检查机制

```java
// Java 代码
int packed = DIGITS[v];

// JVM 实际执行的等价伪代码
if (v < 0 || v >= DIGITS.length) {
    throw ArrayIndexOutOfBoundsException();
}
int packed = DIGITS[v];
```

**为什么 JIT 无法自动消除边界检查？**

虽然开发者知道 `v` 的范围是 [0, 99]（两位数的十位和个位组合），但：
1. `v` 来自方法参数，JIT 看不到调用链的完整上下文
2. 即使方法内联，调用处的计算 `(q * 100) - i` 的范围证明也很复杂
3. C2 的范围分析虽然强大，但这种跨表达式的证明仍然困难

## 解决方案

### 核心改动

```java
// 变更后: 扩展数组 + 位掩码

static {
    short[] digits = new short[128];  // 扩展到 128 (2^7)

    for (int i = 0; i < 10; i++) {
        short hi = (short) (i + '0');
        for (int j = 0; j < 10; j++) {
            short lo = (short) ((j + '0') << 8);
            digits[i * 10 + j] = (short) (hi | lo);
        }
    }
    DIGITS = digits;
}

// 变更后的方法
public static void putPair(char[] buf, int charPos, int v) {
    int packed = DIGITS[v & 0x7f];  // & 0x7f 确保索引在 [0, 127]
    buf[charPos    ] = (char) (packed & 0xFF);
    buf[charPos + 1] = (char) (packed >> 8);
}
```

### 关键优化点

#### 1. 位掩码技巧 (`& 0x7f`)

```java
v & 0x7f  // 等价于 v % 128，但更快

// 二进制解释:
// v = xxxxxxxx xxxxxxxx xxxxxxxx xxxxxxxx (32位 int)
// 0x7f = 00000000 00000000 01111111 (只保留低7位)
// 结果: 00000000 00000000 0xxxxxxx  (范围保证在 [0, 127])
```

**为什么用 128？**
- 128 = 2^7，可以用位掩码实现
- 100 < 128，足够容纳所有有效索引 [0, 99]
- 额外的 28 个元素 [100, 127] 不会被访问（或者包含冗余数据）

#### 2. 边界检查消除

JIT 编译器可以识别这种模式：

```java
// JIT 的优化分析:
// 1. 数组长度是 128
// 2. 索引是 v & 0x7f
// 3. 0x7f = 127 < 128
// 4. 结论: (v & 0x7f) 永远在 [0, 127] 范围内
// 5. 结果: 可以安全移除边界检查！
```

#### 3. 内存开销分析

```
修改前: short[100] = 200 bytes
修改后: short[128] = 256 bytes
增加:   56 bytes

结论: 内存增加可忽略不计（单次数组，全局唯一）
```

## 设计决策

### Q1: 为什么选择 128 而不是 256 或 64？

> **Reviewer**: 为什么是 128？256 也能用位掩码 (`& 0xff`)

**A**: 权衡考虑：
- **64**: 不够，最大索引是 99
- **128**: 最小满足条件的 2 的幂，浪费最少（28 个元素）
- **256**: 浪费更多（156 个元素），且可能增加缓存未命中

### Q2: `& 0x7f` 比 `% 128` 快多少？

> **Reviewer**: 位运算比取模运算快多少？

**A**:
```
& 0x7f:  单个指令 (AND), 1 周期
% 128:   需要除法指令, 10-30 周期 (依赖 CPU)

提升: ~10-30x (虽然边界检查消除才是主要收益)
```

### Q3: 如果 v 是负数会怎样？

> **Reviewer**: 负数的位掩码行为正确吗？

**A**: 正确！这是关键设计：

```java
// 示例: v = -1
// -1 的二进制 (补码): 11111111 11111111 11111111 11111111
// -1 & 0x7f:           00000000 00000000 01111111 = 127

// 这意味着如果 v 意外是负数，会访问 DIGITS[127]
// 由于 DIGITS[100-127] 未初始化（默认 0），可能产生错误字符
// 但这比 ArrayIndexOutOfBoundsException 更温和
// 而且正常情况下 v 永远不会是负数
```

## 代码审查要点

### 审查通过的关键点

1. ✅ **正确性**: `& 0x7f` 保证索引在有效范围内
2. ✅ **内存**: 56 字节的增加可以忽略
3. ✅ **性能**: 边界检查消除是可验证的 JIT 优化
4. ✅ **兼容性**: 纯内部实现，无 API 变化
5. ✅ **安全性**: 不引入新的安全风险

### Reviewer 评论

> **@liach (Reviewer)**: This is a clever use of bitwise AND to eliminate bounds checking. The array expansion to 128 elements is negligible in terms of memory, and the performance gain comes from the JIT being able to prove the index is always valid.

> **@rgiulietti (Reviewer)**: The change is minimal and focused. The `& 0x7f` pattern is a well-known technique for bounds check elimination. Good optimization for a hot path.

## 技术深度解析

### JIT 边界检查消除 (BCE) 机制

HotSpot C2 编译器的边界检查消除基于以下条件：

```java
// 条件 1: 数组长度是常量
private static final short[] DIGITS = new short[128];

// 条件 2: 索引计算可证明范围
int index = v & 0x7f;
// C2 可以证明: (v & 0x7f) >= 0 永远成立
// C2 可以证明: (v & 0x7f) <= 127 < 128 永远成立

// 结论: 边界检查可以消除
```

### C2 编译器 IR 变换

```c++
// 初始 IR
LoadI DIGITS[v]
  If (v < 0 || v >= 128) goto BCI
  LoadI [DIGITS + v*2]

// 优化后 IR
LoadI DIGITS[v & 0x7f]
  // 边界检查已消除
  LoadI [DIGITS + (v & 0x7f)*2]
```

### CPU 流水线影响

```
修改前:
┌────────┐    ┌──────────┐    ┌────────┐
│ 取数指令 │ → │ 边界检查 │ → │ 实际访问 │
└────────┘    └──────────┘    └────────┘
               ↑ 分支可能失败

修改后:
┌────────┐    ┌────────┐
│ 取数指令 │ → │ 实际访问 │
└────────┘    └────────┘
     ↑ AND 指令完全流水线化
```
