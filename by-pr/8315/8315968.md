## 背景

在 JDK 中，存在多个**数字到字符**的转换场景：

| 场景 | 位置 | 用途 |
|------|------|------|
| **十进制数字** | `StringLatin1.PACKED_DIGITS` | `Integer.toString`, `Long.toString` |
| **十进制数字** | `DecimalDigits.DIGITS` | Format Concat 优化 |
| **十六进制数字** | `HexDigits.DIGITS` | `UUID.toString`, `HexFormat` |
| **八进制数字** | `OctalDigits.DIGITS` | `Integer.toOctalString` |

**问题：这些数字表有重复！**

```java
// StringLatin1.java (原本)
@Stable
static final short[] PACKED_DIGITS = new short[] {
    0x3030, 0x3130, 0x3230, ... // 100 个元素
};

// DecimalDigits.java (原本)
@Stable
private static final short[] DIGITS;
static {
    short[] digits = new short[10 * 10];
    for (int i = 0; i < 10; i++) {
        short hi = (short) ((i + '0') << 8);
        for (int j = 0; j < 10; j++) {
            short lo = (short) (j + '0');
            digits[i * 10 + j] = (short) (hi | lo);
        }
    }
    DIGITS = digits;
}
```

**这两个表内容完全相同，但分散在不同位置！**

## 问题分析

### 代码重复问题

| 问题 | 影响 |
|------|------|
| **维护成本** | 两份代码需要同步维护 |
| **内存占用** | 两份数据占用双倍内存 |
| **优化困难** | 修改一处需要同时修改另一处 |

### 字节序问题

原 `StringLatin1.PACKED_DIGITS` 是 **大端序 (Big-Endian)**：

```java
// PACKED_DIGITS[i] = (hi << 8) | lo
// 例如: PACKED_DIGITS[12] = 0x3130 = '1' | ('0' << 8)
```

而 `DecimalDigits.DIGITS` 原本是 **小端序 (Little-Endian)**：

```java
// DIGITS[i] = (lo << 8) | hi
// 例如: DIGITS[12] = 0x3031 = '0' | ('1' << 8)
```

**字节序不一致会导致兼容性问题！**

---

## 解决方案

### 核心改动

#### 1. 移动类到 `jdk.internal.util` 包

```java
// 之前
package java.util;
final class DecimalDigits implements Digits { ... }

// 之后
package jdk.internal.util;
public final class DecimalDigits implements Digits { ... }
```

**为什么变成 `public`？**

- 需要被 `java.lang` 包中的类（如 `StringLatin1`）访问
- `jdk.internal.util` 本身就是内部 API，不影响外部

#### 2. 添加 `digitPair` 静态方法

```java
/**
 * For values from 0 to 99 return a short encoding a pair of
 * ASCII-encoded digit characters in little-endian
 * @param i value to convert
 * @return a short encoding a pair of ASCII-encoded digit characters
 */
public static short digitPair(int i) {
    return DIGITS[i];
}
```

这个方法使得其他类可以直接使用数字表，而不需要通过 `Digits` 接口。

#### 3. 统一为 Little-Endian

```java
// DecimalDigits.java (修改后)
@Stable
private static final short[] DIGITS = new short[] {
    0x3030, 0x3130, 0x3230, 0x3330, 0x3430, 0x3530, 0x3630, 0x3730, 0x3830, 0x3930,
    0x3031, 0x3131, 0x3231, 0x3331, 0x3431, 0x3531, 0x3631, 0x3731, 0x3831, 0x3931,
    // ... 共 100 个元素
};
```

**小端序格式：** `DIGITS[i] = lo | (hi << 8)`
- `DIGITS[0] = 0x3030` = `'0' | ('0' << 8)` = "00"
- `DIGITS[12] = 0x3031` = `'0' | ('1' << 8)` = "10"
- `DIGITS[99] = 0x3939` = `'9' | ('9' << 8)` = "99"

#### 4. 更新 `StringLatin1` 使用统一方法

```java
// StringLatin1.java (修改后)
static int getChars(int i, int index, byte[] buf) {
    // ...
    while (i2 <= -100) {
        q2 = i2 / 100;
        charPos -= 2;
-       ByteArrayLittleEndian.setShort(buf, charPos, PACKED_DIGITS[(q2 * 100) - i2]);
+       ByteArrayLittleEndian.setShort(buf, charPos, DecimalDigits.digitPair((q2 * 100) - i2));
        i2 = q2;
    }
    // ...
}

// 删除原来的 PACKED_DIGITS 数组
```

### 统一所有 Digits 类

```java
// 所有 Digits 类都变成 public，移到 jdk.internal.util
public sealed interface Digits permits DecimalDigits, HexDigits, OctalDigits {
    // ...
}

public final class DecimalDigits implements Digits {
    public static final Digits INSTANCE = new DecimalDigits();
    public static short digitPair(int i) { return DIGITS[i]; }
}

public final class HexDigits implements Digits {
    public static final Digits INSTANCE = new HexDigits();
    public static int digit(int b0, int b1) { ... }
    public static long digit(int b0, int b1, int b2, int b3) { ... }
}

public final class OctalDigits implements Digits {
    public static final Digits INSTANCE = new OctalDigits();
    // 修改为 little-endian
}
```

## 设计决策

### Q1: 为什么要从 `java.util` 移到 `jdk.internal.util`？

> **Reviewer**: 为什么不保持在 `java.util`？

**A**: 原因有三：

1. **内部 API**: 这些类不是公共 API，不应该在 `java.util` 中
2. **跨包访问**: `java.lang` 包中的类需要访问这些类
3. **命名规范**: `jdk.internal.util` 是 JDK 内部工具类的标准位置

### Q2: 为什么统一为 Little-Endian？

> **Reviewer**: 为什么不统一为 Big-Endian？

**A**: Little-Endian 更适合 `ByteArrayLittleEndian` 操作：

```java
// Little-Endian: 低字节在低地址
ByteArrayLittleEndian.setShort(buf, pos, 0x3031);
// buf[pos] = 0x30 ('0')
// buf[pos+1] = 0x31 ('1')
// 结果: "01" (正确)

// 如果是 Big-Endian: 0x3130
ByteArrayLittleEndian.setShort(buf, pos, 0x3130);
// buf[pos] = 0x30 ('0')
// buf[pos+1] = 0x31 ('1')
// 结果: "01" (巧合正确，但语义混乱)
```

### Q3: 为什么 `HexDigits` 的 `digit()` 方法不改为 `packDigits()`？

> **Reviewer**: 方法名为什么不统一？

**A**: `HexDigits` 的方法在后续 PR 中被重命名为 `packDigits()`，这个 PR 保持原有命名以减少变更范围。

## 性能对比

这是一个 **重构 PR**，主要目标是代码复用，非性能优化。

| 指标 | 修复前 | 修复后 | 变化 |
|------|--------|--------|------|
| **代码重复** | 2 份相同数据表 | 1 份统一数据表 | -50% |
| **内存占用** | 2 份数组 | 1 份数组 | -50% |
| **方法调用** | 直接数组访问 | 静态方法包装 | 0 (JIT 内联) |

**性能验证：**

由于 `digitPair()` 是简单的数组访问，JIT 编译器会完全内联：

```java
// 源代码
ByteArrayLittleEndian.setShort(buf, charPos, DecimalDigits.digitPair(v));

// JIT 内联后
ByteArrayLittleEndian.setShort(buf, charPos, DIGITS[v]);

// 进一步内联 DIGITS 访问
buf[charPos] = (byte) (DIGITS[v] & 0xFF);
buf[charPos + 1] = (byte) (DIGITS[v] >> 8);
```
