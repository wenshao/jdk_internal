## 背景

### 原始优化：JDK-8310929

**目标**: 优化 `StringUTF16.getChars()` 方法，使用 `ByteArrayLittleEndian` 提升小端序平台性能。

**改动**:
```java
// 优化前
putChar(buf, charPos, digit1);
putChar(buf, charPos + 1, digit2);

// 优化后（JDK-8310929）
int packed = PACKED_DIGITS[value];
int inflated = ((packed & 0xFF00) << 8) | (packed & 0xFF);
ByteArrayLittleEndian.setInt(buf, charPos << 1, inflated);
```

### 问题浮现

**AIX 平台测试失败**（Martin Doerr 反馈）:

```
We're getting test failures on AIX:
- compiler/intrinsics/Test8215792.java
- compiler/intrinsics/string/TestStringIntrinsics.java
- runtime/CompactStrings/TestMethodNames.java
- runtime/StringIntrinsic/StringIndexOfChar.java

Is there a problem with Big Endian?
```

### 根本原因

`ByteArrayLittleEndian` 类在小端序平台上高效，但在大端序平台上：

1. 字节序转换需要额外操作
2. `Integer.reverseBytes` 开销
3. 可能产生不正确的字节序结果

## 问题分析

### 字节序背景

| 平台类型 | 字节序 | 代表平台 |
|----------|--------|----------|
| **Little-endian** | 低字节在前 | x86/x64, ARM |
| **Big-endian** | 高字节在前 | s390x, AIX-ppc64, 传统 ppc64 |

### 原始实现（有问题）

```java
// JDK-8310929 的优化代码
static int getChars(int i, int index, byte[] buf) {
    // ...
    while (i <= -100) {
        q = i / 100;
        r = (q * 100) - i;
        i = q;

        int packed = (int) StringLatin1.PACKED_DIGITS[r];
        int inflated = ((packed & 0xFF00) << 8) | (packed & 0xFF);

        charPos -= 2;
        ByteArrayLittleEndian.setInt(buf, charPos << 1, inflated);
        //   ^^^^^^^^^^^^^^^^^^^
        //   问题：在大端序平台字节序不正确
    }
    // ...
}
```

### 为什么大端序出问题？

#### Little-endian（x86）内存布局

```
int inflated = 0x3132  // '1' (0x31) 和 '2' (0x32)
内存: 32 31 00 00  (低字节在前)

ByteArrayLittleEndian.setInt 写入后:
char[0] = 0x31  // '1'
char[1] = 0x32  // '2'  ✅ 正确
```

#### Big-endian（s390x）内存布局

```
int inflated = 0x3132
内存: 00 00 31 32  (高字节在前)

ByteArrayLittleEndian.setInt 执行 reverseBytes 后:
写入顺序可能不正确，导致:
char[0] = 0x32  // '2'
char[1] = 0x31  // '1'  ❌ 错误！
```

### 测试失败的具体场景

```java
// Integer.toString(-12345)
// 期望: "-12345"
// 实际: 字节序错误导致乱码

// StringUTF16.getChars 在大端序平台上
// 生成错误的字节序
```

## 解决方案

### 核心改动

移除 `ByteArrayLittleEndian` 依赖，恢复原有的 `putPair` 方法：

```java
// 修改后
static int getChars(int i, int index, byte[] buf) {
    // ...
    while (i <= -100) {
        q = i / 100;
        r = (q * 100) - i;
        i = q;

        charPos -= 2;
        putPair(buf, charPos, r);  // 使用平台无关的方法
    }
    // ...
}

// 新增辅助方法
private static void putPair(byte[] buf, int charPos, int v) {
    int packed = (int) StringLatin1.PACKED_DIGITS[v];
    putChar(buf, charPos, packed & 0xFF);
    putChar(buf, charPos + 1, packed >> 8);
}
```

### 关键修复点

| 修复点 | 说明 |
|--------|------|
| **移除 ByteArrayLittleEndian** | 避免字节序问题 |
| **恢复 putPair 方法** | 使用平台无关的 putChar |
| **保持 PACKED_DIGITS 优化** | 数字对查找表优化保留 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| **性能回归** | putChar 会被 JIT 内联优化 |
| **代码重复** | 提取为 putPair 私有方法 |
| **跨平台** | putChar 是平台无关的 API |

## 设计决策

### Q1: 为什么不条件化使用 ByteArrayLittleEndian？

> **Reviewer (cl4es)**: 可以在大端序平台使用 putPair，小端序平台使用 ByteArrayLittleEndian 吗？

**A**:
- 增加代码复杂度
- 需要运行时检测字节序
- 两个代码路径维护成本高
- 性能差异可能不明显（JIT 优化后）

```java
// 讨论的方案（未采用）
if (BIG_ENDIAN) {
    putPair(..);
} else {
    int inflated = ...;
    ByteArrayLittleEndian.setInt(buf, charPos << 1, inflated);
}
```

### Q2: 性能会受多大影响？

> **Reviewer (cl4es)**: This patch probably reverts performance numbers on little-endian.

**A**:
- `putChar` 是 `@IntrinsicCandidate`，会被 JIT 内联
- `putPair` 方法简单，易于内联
- 字节序转换的开销可能比额外的方法调用更大

### Q3: 为什么第一次提交没发现问题？

> **Reviewer**: 为什么 CI 没有捕获这个问题？

**A**:
- Oracle CI 和 GHA 不覆盖大端序平台
- s390x、AIX 等平台需要专用测试环境
- 依赖社区在大端序平台的测试反馈

## 性能对比

### Little-endian 平台

| 实现 | 说明 |
|------|------|
| **ByteArrayLittleEndian** | 单次 int 写入，依赖字节序转换 |
| **putPair (putChar x2)** | 两次 char 写入，JIT 内联后无额外开销 |

**结论**: 性能差异可忽略，JIT 会优化 `putPair`。

### Big-endian 平台

| 实现 | 说明 |
|------|------|
| **ByteArrayLittleEndian** | 需要 `Integer.reverseBytes`，开销大 |
| **putPair (putChar x2)** | 直接写入，无字节序转换 |

**结论**: `putPair` 在大端序平台更快。

## 教训总结

### 跨平台开发要点

1. **平台覆盖**: 不能只在 x86 平台测试
2. **字节序敏感**: 使用字节序相关的 API 要考虑大端序
3. **社区反馈**: 重视非主流平台的测试反馈

### 优化原则

1. **可移植性优先**: 性能优化不能牺牲跨平台兼容性
2. **简单方案**: `putChar` 虽然简单，但更可靠
3. **条件化复杂**: 平台特定的优化增加维护成本
