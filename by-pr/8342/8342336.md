## 背景

### ClassFile API

ClassFile API 是 JDK 22 引入的现代 API，用于读取、写入和转换 Java 类文件。它提供了结构化的方式来处理类文件格式，无需了解二进制格式细节。

**为什么需要清理 import？**
- ClassFile API 发展过程中 import 语句变得不一致
- 大量相关类的 import 导致代码冗长
- 统一的风格有助于代码维护

## 问题分析

### 原始实现问题

| 问题 | 说明 | 影响 |
|------|------|------|
| **无字母排序** | import 语句按任意顺序添加 | 难以查找 |
| **无 java/jdk 分隔** | `java.*` 和 `jdk.internal.*` 混在一起 | 风格混乱 |
| **过多的单独 import** | 文件中有 20+ 个来自同一包的 import | 代码冗长 |
| **静态 import 不一致** | 静态 import 放置位置不统一 | 风格混乱 |

### 具体示例

**问题 1: 37 个单独的 attribute import**

```java
// 优化前: Attribute.java
import java.lang.classfile.attribute.AnnotationDefaultAttribute;
import java.lang.classfile.attribute.BootstrapMethodsAttribute;
import java.lang.classfile.attribute.CharacterRangeTableAttribute;
import java.lang.classfile.attribute.CodeAttribute;
// ... 还有 33 个类似的 import
import java.lang.classfile.attribute.UnknownAttribute;

import jdk.internal.classfile.impl.BoundAttribute;
import jdk.internal.classfile.impl.UnboundAttribute;
```

**问题 2: 混合的 import 顺序**

```java
// 优化前
import java.util.Set;
import jdk.internal.classfile.impl.AccessFlagsImpl;
import java.lang.reflect.AccessFlag;  // 顺序混乱
import jdk.internal.javac.PreviewFeature;
import static java.util.Objects.requireNonNull;  // 混在普通 import 中
```

## 解决方案

### 核心改动

#### 1. 使用通配符 import

```java
// 优化后: Attribute.java
import java.lang.classfile.attribute.*;

import jdk.internal.classfile.impl.BoundAttribute;
import jdk.internal.classfile.impl.UnboundAttribute;
```

**好处**：
- 从 37 行减少到 1 行
- IDE 可以正确处理
- 新增 attribute 时无需更新每个文件

#### 2. 字母排序 + 分组

```java
// 优化后模板
// 1. java.* imports (字母序)
import java.lang.classfile.ClassModel;
import java.util.List;

// 2. jdk.* imports (字母序)
import jdk.internal.classfile.impl.ClassImpl;
import jdk.internal.javac.PreviewFeature;

// 3. static imports (字母序)
import static java.util.Objects.requireNonNull;
```

#### 3. 通配符使用规则

| 条件 | 使用通配符 |
|------|-----------|
| 同一包 import 数量 > 3 | 是 |
| 包是内聚的（相关类型） | 是 |
| 存在命名冲突 | 否 |

### 关键优化点

| 优化点 | 说明 | 效果 |
|--------|------|------|
| **通配符替换** | 3+ 个同类 import → 1 个通配符 | 减少 67% import 行数 |
| **字母排序** | 按字母顺序组织 import | 提高可读性 |
| **分组分隔** | java.* 和 jdk.* 之间空行 | 结构清晰 |
| **静态 import** | 单独分组在最后 | 风格一致 |

## 设计决策

### Q1: 为什么使用通配符 import？

> **Reviewer**: 通配符 import 不是通常被认为是不好的实践吗？

**A**: 对于 ClassFile API 这种内聚的包结构：
1. **语义清晰**：`attribute.*` 明确表示"所有属性类型"
2. **减少噪音**：让实际代码更突出
3. **易于维护**：新增类型时无需更新所有文件
4. **JDK 惯例**：内聚包可以使用通配符

### Q2: 为什么选择 3 作为阈值？

> **Reviewer**: 为什么是 3 个而不是 5 个或 10 个？

**A**:
1. **经验法则**：3 个 import 的行数已经超过通配符
2. **常见做法**：大多数 IDE 默认使用 3-5
3. **平衡点**：避免过度使用通配符

### Q3: 编译性能会受影响吗？

> **Reviewer**: 通配符 import 会影响编译速度吗？

**A**:
1. **现代编译器**：javac 已经优化了通配符处理
2. **影响极小**：只在增量编译时有轻微差异
3. **运行时无影响**：编译后的字节码完全相同
