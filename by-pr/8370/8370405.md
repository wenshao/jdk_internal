## 背景

### 问题发现

**Olivier Mattmann** 使用白盒 fuzzer 发现了这个 bug：

```
测试运行 → C2 编译 → 分配消除 → 去优化 → 崩溃！
```

### 根本原因

**问题序列**：

1. **Escape Analysis** 分析发现一个局部数组分配可以被消除
2. 所有存储都是 `StoreI` (int 存储)，看起来安全
3. 但有一个 `StoreI` 仍在循环中，暂时等待
4. **Loop Opts** 执行后，那个 `StoreI` 从循环中移出
5. **MergeStores** 运行，将两个 `StoreI` 合并为一个 `StoreL` (long 存储)
6. **分配消除** 执行，移除了数组分配
7. **去优化** 发生时，恢复的值类型错误 (int 变成了 long)

## 核心变更

### 问题演示

```java
// 原始代码
byte[] createArray() {
    int[] arr = new int[4];
    arr[0] = 0x1000000;  // StoreI
    arr[1] = 0x100;     // StoreI
    arr[2] = 0x10000;   // StoreI
    arr[3] = 0x10;      // StoreI
    return toByteArray(arr);
}
```

**C2 处理序列**：

```
1. Escape Analysis: 数组分配可消除
   SafePoint nodes: 4 x ConI

2. Loop Opts 后，某个 StoreI 仍在循环

3. MergeStores 合并:
   2 x StoreI → 1 x StoreL (错误!)

4. 分配消除执行，移除数组分配

5. SafePoint nodes: 3 x ConI, 1 x ConL (错误!)
```

### 去优化时的错误

```
原始: int[4] = {0x1000000, 0x100, 0x10000, 0x10}
错误: 恢复时变成了 long = 68719476737, ...

去优化重建数组:
- 期望: new int[4] = {0x1000000, 0x100, 0x10000, 0x10}
- 实际: new int[5] = {0x1, 0x10, 0x10000, 0x100, 0x1000000}
     (因为 ConL 被当作 int+long 对处理!)
```

### 解决方案

来自 **Quan Anh Mai** (Valhalla 中的类似修复)：

```cpp
// src/hotspot/share/opto/macro.cpp

// 在分配消除时，检查是否有类型不匹配的存储
bool ConnectionGraph::find_depending_fields() {
    for (auto* field : fields) {
        if (field->is_mismatched_store()) {
            return false;  // 放弃消除
        }
    }
    return true;
}
```

**修复**：如果发现类型不匹配的存储，**不要**进行分配消除。

## 技术分析

### MergeStores 的问题

```cpp
// 两个相邻的 int 存储
StoreI(conI_1, addr)    // 0x1000000
StoreI(conI_2, addr+4)  // 0x100

// MergeStores 尝试合并为 long 存储
StoreL(combine(conI_1, conI_2), addr)

// 问题: 类型从 int[] 变成了 long 存储到 int[]
```

### 类型编码

```cpp
// ObjectValue 中的类型编码
// [int=0, long=ConL] 表示 int 占位符 + long 值

// 正确: 4 个 int
{0x1000000, 0x100, 0x10000, 0x10}

// 错误: 1 个 long + 3 个 int (被解释为 5 个元素!)
{0x0, 0x10000000000010, 0x10000, 0x100, 0x1000000}
```

### 去优化序列

```
1. 编译时优化:
   - Escape Analysis: 消除数组分配
   - MergeStores: 合并存储
   - ObjectValue: 记录字段值

2. 去优化触发:
   - 解序列化 ObjectValue
   - 重建对象和字段
   - ❌ 类型错误: long 被当作 int 处理

3. 崩溃:
   - 数组长度错误
   - 字段值错误
```

## 经验教训

### 1. 逃逸分析的复杂性

**教训**: 多种优化交互时需要考虑全局影响

```
优化链:
Escape Analysis → MergeStores → Allocation Elimination

问题: 每个优化单独看是正确的，
      但组合后可能产生错误
```

### 2. 类型安全的重要性

**教训**: 类型系统需要在所有阶段保持一致

```cpp
// 存储: int → int → ❌ long
// 问题: 类型转换被忽略
```
