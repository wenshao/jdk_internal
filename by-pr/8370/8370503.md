## 背景

### 为什么重要

`Integer.toString()` 和 `Long.toString()` 是 Java 中最基础和最常用的方法之一：
- 每次日志记录 (`log.info("Value: " + value)`)
- JSON/XML 序列化（数字转字符串）
- HTTP 响应构建
- 调试输出

在 JDK 9 引入 Compact Strings (JEP 254) 后，Java 字符串内部默认使用 Latin1 编码（每个字符 1 字节）存储 ASCII/Latin1 字符，而不是 UTF16（每个字符 2 字节）。然而，为了向后兼容，JDK 仍保留了 `COMPACT_STRINGS = false` 的代码路径。

### 改动位置

```
src/java.base/share/classes/java/lang/
    ├── Integer.java
    │   ├── toString(int i)              ← 简化
    │   └── toUnsignedString0()          ← 简化
    │       └── formatUnsignedIntUTF16() ← 删除
    └── Long.java
        ├── toString(long i)             ← 简化
        └── toUnsignedString0()          ← 简化
            └── formatUnsignedLong0UTF16() ← 删除
```

## 问题分析

### 原始实现

在 JDK 26 之前，`Integer.toString()` 和 `Long.toString()` 需要同时处理两种编码模式：

```java
// 文件: src/java.base/share/classes/java/lang/Integer.java
// 行数: ~401-415 (变更前)

@IntrinsicCandidate
public static String toString(int i) {
    int size = DecimalDigits.stringSize(i);
    if (COMPACT_STRINGS) {
        byte[] buf = new byte[size];
        DecimalDigits.uncheckedGetCharsLatin1(i, size, buf);
        return new String(buf, LATIN1);
    } else {
        byte[] buf = new byte[size * 2];        // UTF16 需要 2 倍空间
        DecimalDigits.uncheckedGetCharsUTF16(i, size, buf);
        return new String(buf, UTF16);
    }
}
```

对于 `toUnsignedString0()` 方法（用于 `toHexString()`, `toOctalString()`, `toBinaryString()`）：

```java
// 文件: src/java.base/share/classes/java/lang/Integer.java
// 行数: ~363-377 (变更前)

private static String toUnsignedString0(int val, int shift) {
    int mag = Integer.SIZE - Integer.numberOfLeadingZeros(val);
    int chars = Math.max(((mag + (shift - 1)) / shift), 1);
    if (COMPACT_STRINGS) {
        byte[] buf = new byte[chars];
        formatUnsignedInt(val, shift, buf, chars);
        return new String(buf, LATIN1);
    } else {
        byte[] buf = new byte[chars * 2];           // UTF16 需要 2 倍空间
        formatUnsignedIntUTF16(val, shift, buf, chars);  // 单独的 UTF16 版本
        return new String(buf, UTF16);
    }
}

// UTF16 专用格式化方法 (被删除)
private static void formatUnsignedIntUTF16(int val, int shift, byte[] buf, int len) {
    int charPos = len;
    int radix = 1 << shift;
    int mask = radix - 1;
    do {
        StringUTF16.putChar(buf, --charPos, Integer.digits[val & mask]);
        val >>>= shift;
    } while (charPos > 0);
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **分支预测失败** | `if (COMPACT_STRINGS)` 分支在几乎 100% 情况下为 true | CPU 分支预测器几乎总是正确，但分支仍存在 |
| **代码膨胀** | 维护两套代码路径（Latin1 + UTF16） | 指令缓存压力，方法代码大小增加 |
| **C1 编译器内联限制** | 复杂的分支逻辑超过 C1 的内联阈值 (bytecode size > 35) | 阻止 C1 编译器内联，影响启动性能 |
| **维护负担** | 需要同时维护两个版本的格式化方法 | 代码复杂度高，容易引入 bug |

### 为什么 `COMPACT_STRINGS = false` 路径是"死代码"？

1. **默认启用**: 从 JDK 9 开始，`COMPACT_STRINGS` 默认为 `true`
2. **没有官方开关**: 没有标准的 JVM 参数禁用它
3. **实验性功能**: 禁用需要内部 API 或重新编译
4. **性能惩罚**: 禁用会使字符串操作性能显著下降

根据 PR 作者的统计，在实际生产环境中，几乎 100% 的 JVM 实例都使用默认的 `COMPACT_STRINGS = true` 配置。

## 解决方案

### 核心改动

使用 JDK 26 新引入的 `String.newStringWithLatin1Bytes()` 方法，直接创建 Latin1 编码的字符串，移除 `COMPACT_STRINGS` 条件分支和 UTF16 代码路径。

```java
// 文件: src/java.base/share/classes/java/lang/Integer.java
// 变更后

@IntrinsicCandidate
public static String toString(int i) {
    int size = DecimalDigits.stringSize(i);
    byte[] buf = new byte[size];
    DecimalDigits.uncheckedGetCharsLatin1(i, size, buf);
    return String.newStringWithLatin1Bytes(buf);  // 新 API
}

private static String toUnsignedString0(int val, int shift) {
    int mag = Integer.SIZE - Integer.numberOfLeadingZeros(val);
    int chars = Math.max(((mag + (shift - 1)) / shift), 1);
    byte[] buf = new byte[chars];
    formatUnsignedInt(val, shift, buf, chars);
    return String.newStringWithLatin1Bytes(buf);  // 新 API
}

// formatUnsignedIntUTF16() 方法被完全删除
```

### `String.newStringWithLatin1Bytes()` API

```java
// 文件: src/java.base/share/classes/java/lang/String.java
// 行数: 800-813

/**
 * {@return a new String created using the supplied latin1 bytes}
 * @param src a byte array with the bytes for a latin1 string
 */
static String newStringWithLatin1Bytes(byte[] src) {
    int len = src.length;
    if (len == 0) {
        return "";
    }

    if (COMPACT_STRINGS)
        return new String(src, LATIN1);
    return new String(StringLatin1.inflate(src, 0, src.length), UTF16);
}
```

这是一个**包私有**的内部 API，专门为 JDK 内部使用设计。

**重要细节**: 这个方法内部仍然处理了 `COMPACT_STRINGS` 检查，但这样做的好处是：
1. **集中管理**: COMPACT_STRINGS 逻辑在 String 类中统一处理
2. **调用方简化**: Integer/Long 等调用方不再需要关心编码问题
3. **零成本抽象**: 在 COMPACT_STRINGS=true 时，编译器可以完全优化

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除分支** | 移除 `if (COMPACT_STRINGS)` 分支，代码路径线性 |
| **代码减少** | 删除 77 行代码（-38 in Integer, -39 in Long） |
| **C1 内联** | 方法字节码大小 < 35，允许 C1 编译器内联 |
| **内存分配** | 只分配 `size` 字节，不再有 `size * 2` 的备用路径 |
| **方法删除** | `formatUnsignedIntUTF16()` 和 `formatUnsignedLong0UTF16()` 被完全删除 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 禁用 COMPACT_STRINGS 的用户 | 这是一个**极其罕见**的配置，没有官方支持，性能惩罚严重。即使禁用，`newStringWithLatin1Bytes` 内部仍会通过 `StringLatin1.inflate` 处理 |
| 非 Latin1 字符 | `toString()` 方法只输出 ASCII 数字（0-9 和负号），永远是 Latin1 |
| 向后兼容性 | API 行为完全相同，输出结果一致 |

## 设计决策

### Q1: 为什么不保留 UTF16 代码路径以备将来使用？

> **Reviewer Comment**: "Seems like `Integer.formatUnsignedIntUTF16()` is no longer needed." — @rgiulietti

**A**:
- `COMPACT_STRINGS` 从 JDK 9 (2017) 开始就是默认配置
- 没有官方的 JVM 参数禁用它
- 保留死代码增加维护负担
- 如果将来需要，可以从 Git 历史恢复

### Q2: 这个改动如何影响 C1 编译器的内联决策？

**A**:

C1 (Client Compiler) 是分层编译的第一层，专注于快速启动：
- **内联阈值**: 默认 bytecode size < 35 字节
- **变更前**: `toString(int)` 方法包含条件分支和两个代码路径 > 35 字节
- **变更后**: 单一代本路径 < 35 字节

```java
// 字节码大小对比
// 变更前 (伪字节码)
public static String toString(int i) {
    size = stringSize(i);           // ~5 bytes
    if (COMPACT_STRINGS) {          // ~3 bytes (分支)
        buf = new byte[size];       // ~8 bytes
        getCharsLatin1(...);        // ~10 bytes
        return new String(buf, L);  // ~8 bytes
    } else {                        // ~3 bytes (分支)
        buf = new byte[size*2];     // ~10 bytes
        getCharsUTF16(...);         // ~12 bytes
        return new String(buf, U);  // ~8 bytes
    }
}
// 总计: ~67 bytes (超过 C1 内联阈值)

// 变更后
public static String toString(int i) {
    size = stringSize(i);                       // ~5 bytes
    buf = new byte[size];                       // ~8 bytes
    getCharsLatin1(i, size, buf);               // ~10 bytes
    return String.newStringWithLatin1Bytes(buf); // ~8 bytes
}
// 总计: ~31 bytes (可以被 C1 内联)
```

### Q3: 为什么 `newStringWithLatin1Bytes()` 比直接 `new String(buf, LATIN1)` 好？

**A**:

语义上相同，但有以下好处：
1. **自文档化**: 方法名明确表达意图
2. **集中管理**: 如果将来 String 实现改变，只需修改一处
3. **性能标识**: 可以添加 `@IntrinsicCandidate` 或特殊优化
4. **代码审查**: 容易发现所有 Latin1 字符串创建点
5. **安全处理**: 内部处理了空数组和 COMPACT_STRINGS=false 的边缘情况

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **兼容性** | 🟢 无影响 | 纯内部实现，API 不变 |
| **正确性** | 🟢 低 | 数字转字符串永远是 Latin1。即使 COMPACT_STRINGS=false，newStringWithLatin1Bytes 内部会正确处理 |
| **性能回归** | 🟢 无风险 | 移除的代码路径几乎从不执行 |
| **平台差异** | 🟢 无影响 | 所有平台行为一致 |
| **维护负担** | 🟢 改善 | 减少代码量和复杂度 |

### 关于 COMPACT_STRINGS = false

这是一个**非标准配置**：
- 没有官方 JVM 参数支持
- 需要修改源代码重新编译或使用内部 API
- 性能惩罚严重（内存占用翻倍）
- 从 JDK 9 (2017) 到 JDK 26 (2025)，8 年间几乎无实际使用

如果用户确实需要禁用 Compact Strings，他们可以：
1. 继续使用 JDK 25 或更早版本
2. 自己维护补丁

**注意**: 即使 COMPACT_STRINGS=false，由于 `String.newStringWithLatin1Bytes` 内部会调用 `StringLatin1.inflate` 来处理，功能仍然正常工作，只是性能会退化（这正是预期的行为）。

- `Integer.java` - `toString()` 和 `toUnsignedString0()`

## 总结

这个 PR 是一个典型的"清理 + 优化"改动：

1. **清理**: 移除 77 行几乎从不执行的死代码（UTF16 路径）
2. **优化**: 简化代码路径，使方法能够被 C1 编译器内联
3. **简化**: 使用新的 `String.newStringWithLatin1Bytes()` API 提高代码可读性
4. **安全**: 移除的代码路径在现代 JVM 中几乎从不执行；即使 COMPACT_STRINGS=false，新实现仍能正确工作

这种改动体现了 JDK 开发的一个趋势：**渐进式清理遗留代码**，在保证兼容性的前提下，移除为极端向后兼容性而保留的复杂逻辑。
