## 背景

`StringUTF16` 是 Java 字符串 UTF-16 编码操作的核心内部类。在之前的 PR #16425 中，某些代码被重构，导致一个 import 语句不再被使用。

**为什么重要？**
- 未使用的 import 会造成代码混淆
- 现代 IDE 会将其标记为警告
- 代码清洁度是代码质量的重要指标

**改动位置**
```
src/java.base/share/classes/java/lang/StringUTF16.java
    └── 移除未使用的 import 语句
```

---

## 问题分析

### 原始实现

```java
// 文件: java/lang/StringUTF16.java
// 修改前: 包含未使用的 import

package java.lang;

import jdk.internal.access.JavaLangAccess;
import jdk.internal.misc.Unsafe;  // ← 未使用的 import
// ... 其他 imports
```

### 为什么 import 未被使用？

在 PR #16425 中，相关代码被重构，原本使用 `Unsafe` 的代码被替换为其他实现，但 import 语句没有被移除。

**未使用 import 的影响：**

| 影响类型 | 描述 | 严重程度 |
|----------|------|----------|
| **代码混淆** | 读者可能认为代码使用了 Unsafe | 低 |
| **IDE 警告** | 现代 IDE 会标记未使用的 import | 低 |
| **编译时间** | 增加微不足道的编译开销 | 极低 |
| **字节码** | 编译器自动忽略，不影响输出 | 无 |

---

## 解决方案

### 核心改动

```java
// 修改后: 移除未使用的 import

package java.lang;

import jdk.internal.access.JavaLangAccess;
// import jdk.internal.misc.Unsafe;  // ← 已移除
// ... 其他 imports
```

### 变更详情

| 文件 | 删除行数 | 新增行数 |
|------|----------|----------|
| `java/lang/StringUTF16.java` | 1 | 0 |

### 为什么这样清理是安全的？

1. **编译器验证**：如果 import 真的被使用，编译会失败
2. **测试覆盖**：所有 jtreg 测试通过
3. **代码审查**：由原作者提交，熟悉代码历史

---

## 设计决策

### Q1: 为什么这是一个单独的 PR？

> **Reviewer**: 为什么不将此清理与之前的 PR #16425 合并？

**A**:

PR #16425 是功能性变更，而这是一个清理任务。分开的原因：
- 原始 PR 可能已经合并
- 清理任务通常在主 PR 之后单独提交
- 保持 PR 的单一职责原则

### Q2: 未使用的 import 有什么危害？

> **Reviewer**: 编译器会忽略未使用的 import，为什么要移除？

**A**:

虽然编译器会忽略，但：
1. **代码可读性**：误导读者认为代码使用了 Unsafe
2. **IDE 集成**：产生警告，分散注意力
3. **代码规范**：大多数代码风格指南要求移除未使用的 import
4. **重构辅助**：当真正移除 Unsafe 使用时，未使用的 import 会混淆意图
