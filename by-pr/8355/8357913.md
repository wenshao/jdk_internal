## 背景

### @Stable 注解的作用

`@Stable` 是 HotSpot JVM 特有的注解，定义在 `jdk.internal.vm.annotation.Stable`。它告诉 JIT 编译器：

1. 字段引用本身不会改变（final 已保证）
2. 字段内容在构造后至多改变一次
3. 数组元素在初始化后不会改变

这允许 JIT 进行激进优化：
- **常量折叠**：将数组访问优化为常量
- **常量传播**：将数组值传播到使用点
- **消除边界检查**：基于已知的数组长度

### 为什么 BigInteger 和 BigDecimal 需要优化？

BigInteger 和 BigDecimal 是 Java 中处理大数的核心类，广泛用于：
- **加密算法**：RSA、DSA、ECDSA 等
- **金融计算**：精确的十进制计算
- **科学计算**：高精度数值计算

这些场景通常涉及大量重复计算，JIT 优化可以带来显著的性能提升。

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **数组访问开销** | JIT 无法确定数组内容是常量 | 每次访问都需要数组边界检查 |
| **无法常量传播** | JIT 无法将数组元素优化为常量 | 重复计算无法消除 |
| **缓存未命中** | 访问模式不适合 CPU 缓存 | 内存延迟 |

### 为什么 @Stable 有帮助？

```java
// 没有 @Stable
private static final long[] TABLE = {1, 10, 100};
// JIT 生成：每次访问 TABLE[i] 都需要边界检查

// 有 @Stable
@Stable
private static final long[] TABLE = {1, 10, 100};
// JIT 生成：TABLE[0] 直接编译为常量 1，TABLE[1] 直接编译为常量 10
```

## 解决方案

### 核心改动：添加 @Stable 注解

```java
// BigDecimal.java (修改后)
import jdk.internal.vm.annotation.Stable;

public class BigDecimal extends Number implements Comparable<BigDecimal> {
    // 添加 @Stable 注解
    @Stable
    private static final BigDecimal[] ZERO_THROUGH_TEN = {
        new BigDecimal(BigInteger.ZERO, 0, 0, 1),
        new BigDecimal(BigInteger.ONE, 1, 0, 1),
        // ...
    };

    @Stable
    private static final BigDecimal[] ZERO_SCALED_BY = {
        ZERO_THROUGH_TEN[0],
        new BigDecimal(BigInteger.ZERO, 0, 1, 1),
        // ...
    };

    @Stable
    private static final double[] DOUBLE_10_POW = {
        1.0e0, 1.0e1, 1.0e2, ...
    };

    @Stable
    private static final float[] FLOAT_10_POW = {
        1.0e0f, 1.0e1f, 1.0e2f, ...
    };

    @Stable
    private static final long[] LONG_TEN_POWERS_TABLE = {
        1, 10, 100, ...
    };

    @Stable
    private static final long[] THRESHOLDS_TABLE = {
        Long.MAX_VALUE,
        Long.MAX_VALUE/10L,
        ...
    };

    @Stable
    private static final BigInteger[] FIVE_TO_2_TO = new BigInteger[16 + 1];
    // ...

    @Stable
    private static final long[][] LONGLONG_TEN_POWERS_TABLE = {
        {0L, 0x8AC7230489E80000L},  // 10^19
        // ...
    };
}
```

```java
// BigInteger.java (修改后)
import jdk.internal.vm.annotation.Stable;

public class BigInteger extends Number implements Comparable<BigInteger> {
    // 添加 @Stable 注解
    @Stable
    private static final double[] logCache;
    // ...
}
```

| 数组 | 用途 | 元素数量 |

| 数组 | 用途 |
|------|------|
| logCache | 对数缓存 |

## 设计决策

### Q1: 为什么添加 @Stable 没有立即带来性能提升？

> **Reviewer**: 微基准测试显示没有性能提升，为什么还要添加？

**A**: 根据 PR 讨论中的回复：

```java
// 作者 @wenshao 的回复：
I ran `make test TEST="micro:java.math.BigDecimals` on a MacBook M1 Pro,
and there was no performance improvement with the Stable added by this PR.
```

**原因分析**：

1. **C2 优化器限制**：当前的 C2 编译器可能还没有利用 @Stable 注解进行数组常量传播
2. **未来优化潜力**：@Stable 注解为未来的 JIT 优化器提供了信息，可以解锁新的优化机会
3. **无副作用**：添加 @Stable 不会导致性能退化，只是为未来铺路

**Reviewer @liach 的评论**：
> "See #26341 for context; sometimes the constant values aren't immediately used by C2, but at least this will cause no regression and unlocks future gains."

### Q2: 为什么 BigInteger.mag 没有添加 @Stable？

> **Reviewer**: BigInteger.mag 数组也应该添加 @Stable 吗？

**A**: 最初尝试为 `BigInteger.mag` 添加 @Stable，但后来发现：

```java
// BigInteger.largePrime 方法会修改 mag 数组
BigInteger p = new BigInteger(1, mag);
p.mag[mag.length - 1] &= 0xFFFFFFFE;  // 修改 mag 数组内容！
```

这违反了 @Stable 的契约（数组元素应该只改变一次），因此 `BigInteger.mag` 不适合添加 @Stable。

### Q3: @Stable 的契约是什么？

> **Reviewer**: 添加 @Stable 需要满足什么条件？

**A**: @Stable 注解的契约：

```java
// @Stable 的语义
@Stable
final Type[] field;

// JIT 的理解：
// 1. field 引用本身不会改变（final 已保证）
// 2. field 数组长度不会改变
// 3. field 数组元素在初始化后不会改变
```

**使用条件**：
1. 字段必须是 final
2. 数组在静态初始化器或构造函数中完全初始化
3. 之后不再修改数组内容
4. 如果数组元素是对象引用，对象本身也不应被修改

## 性能对比

### 微基准测试结果

根据 PR 讨论，作者在 MacBook M1 Pro 上运行了微基准测试：

```bash
make test TEST="micro:java.math.BigDecimals"
```

**结果**：没有观察到明显的性能提升。

### 为什么没有立即的收益？

| 原因 | 说明 |
|------|------|
| **C2 优化器** | 当前 C2 可能尚未利用 @Stable 进行数组常量传播 |
| **访问模式** | 微基准的访问模式可能不适合触发优化 |
| **预热不足** | 可能需要更多的迭代才能触发 JIT 优化 |
| **未来潜力** | @Stable 为未来的 JIT 优化提供了信息 |

### 潜在的未来收益

```java
// 未来 JIT 优化器的可能优化
@Stable
private static final long[] TABLE = {1, 10, 100};

// 当前编译
long x = TABLE[i];  // 数组访问 + 边界检查

// 未来可能的编译
long x = (i == 0) ? 1 : (i == 1) ? 10 : 100;  // 完全展开为常量
```
