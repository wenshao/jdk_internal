## 概述

修复 `MergeLoadBench` JMH 基准测试在 JDK-8344168 合并后的失败问题。

**问题根因**: 在创建基准测试时的**复制粘贴错误**，误加了不必要的 `Unsafe.ARRAY_BYTE_BASE_OFFSET`。

**测试来源**: `MergeLoadBench` 由 [JDK-8343629](/by-pr/8343/8343629.md) 创建，用于测试 MergeLoad JIT 优化。

---

## 背景

### 触发 PR: JDK-8344168

[JDK-8344168](https://bugs.openjdk.org/browse/JDK-8344168): **Change Unsafe base offset from int to long**

这个 PR 将 `Unsafe.ARRAY_BYTE_BASE_OFFSET` 的类型从 `int` 改为 `long`，以支持大于 2GB 的数组。

### 失败原因

```java
// ❌ 错误：VarHandle.get 不需要 ARRAY_BYTE_BASE_OFFSET
char c = (char) CHAR_B.get(bytes4, Unsafe.ARRAY_BYTE_BASE_OFFSET + i * 2);
//                                            ^^^^^^^^^^^^^^^^^^^^^^^^
//                                            多余的偏移量！

// ✅ 正确：VarHandle.get 已经处理了数组基础偏移
char c = (char) CHAR_B.get(bytes4, i * 2);
```

**审查者评论** (liach):
> "Looks like a copy-paste error when the benchmark was created, but this wasn't statically detected until the VarHandle type checks failed due to the long offsets."

## 技术分析

### VarHandle vs Unsafe

| API | 偏移量处理 | 类型安全 |
|-----|-----------|---------|
| **Unsafe** | 需要手动计算 `base + offset` | ❌ 弱类型 |
| **VarHandle** | 自动处理数组基础偏移 | ✅ 强类型检查 |

### 为什么 VarHandle 不需要 ARRAY_BYTE_BASE_OFFSET?

```java
// Unsafe 方式 - 需要显式偏移
UNSAFE.getByte(bytes4, Unsafe.ARRAY_BYTE_BASE_OFFSET + i * 2);

// VarHandle 方式 - 自动处理
CHAR_B.get(bytes4, i * 2);  // VarHandle 内部已绑定 base offset
```

**VarHandle 内部实现**:
```java
// VarHandle 在创建时已经绑定了 base offset
static VarHandle makeVarHandle() {
    return MethodHandles.byteArrayViewVarHandle(char[].class,
                                                 ByteOrder.nativeOrder());
    // ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    // 这个方法内部已经处理了 ARRAY_BYTE_BASE_OFFSET
}
```

## 经验教训

### 1. VarHandle vs Unsafe 的使用

| 场景 | 推荐 API |
|------|----------|
| **新代码** | VarHandle (类型安全) |
| **底层操作** | Unsafe (需要时) |
| **JMH 基准** | 与生产代码一致 |

### 2. 复制粘贴错误防范

```java
// ❌ 不要直接复制 Unsafe 代码到 VarHandle 上下文
// Unsafe: Unsafe.ARRAY_BYTE_BASE_OFFSET + offset
// VarHandle: offset (不要加 base!)

// ✅ 创建专门的辅助方法
static char getCharViaVarHandle(byte[] bytes, int index) {
    return (char) CHAR_B.get(bytes, index * 2);
}
```

### 3. 类型检查的价值

JDK-8344168 的类型变更 (`int` → `long`) 揭示了这个隐藏的 bug：
- 之前 `int + int` 编译通过
- 之后 `long + long` 触发 VarHandle 类型检查失败
- **强类型检查在编译时发现了问题**
