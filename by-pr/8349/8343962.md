## 背景

### 问题起源

在 JDK 中，数字转字符的核心逻辑 `getChars` 方法在多个位置重复实现：

1. **StringLatin1.getChars(int/long)** - ~115 行代码
2. **StringUTF16.getChars(int/long)** - ~118 行代码
3. **BigDecimal.StringBuilderHelper.putIntCompact** - ~104 行代码

这些方法的核心算法完全相同，只是字符写入策略不同（byte数组 vs char数组）。

### 为什么重要？

- **高频路径**: `Integer.toString()` 和 `Long.toString()` 是 Java 中最常用的方法之一
- **字符串拼接**: 每次 `String +` 操作都会触发这些方法
- **格式化输出**: 日志、JSON序列化、调试输出都依赖这些方法
- **代码维护**: 三处重复代码增加维护成本和bug风险

### 改动位置

```
src/java.base/share/classes/
    ├── jdk/internal/util/DecimalDigits.java      ← 新增 getCharsLatin1/UTF16 方法
    ├── java/lang/StringLatin1.java              ← 删除 getChars 方法
    ├── java/lang/StringUTF16.java               ← 删除 getChars 方法
    ├── java/lang/Integer.java                   ← 更新调用点

    ├── java/math/BigDecimal.java                ← 删除 StringBuilderHelper
    └── jdk/internal/access/JavaLangAccess.java   ← 删除 getCharsLatin1/UTF16
```

## 问题分析

### 原始实现（代码重复）

**StringLatin1.getChars** (原始实现):
```java
// 文件: java/lang/StringLatin1.java
// 行数: ~90-200 (变更前已删除)

static int getChars(int i, int index, byte[] buf) {
    // Used by trusted callers. Assumes bounds checks done by caller.
    int q;
    int charPos = index;
    boolean negative = i < 0;
    if (!negative) {
        i = -i;
    }

    // Generate two digits per iteration
    while (i <= -100) {
        q = i / 100;
        charPos -= 2;
        writeDigitPair(buf, charPos, (q * 100) - i);
        i = q;
    }

    // ... rest of implementation
}

private static void writeDigitPair(byte[] buf, int charPos, int value) {
    short pair = DecimalDigits.digitPair(value);
    buf[charPos] = (byte)(pair);        // ← 数组访问，隐含边界检查
    buf[charPos + 1] = (byte)(pair >> 8); // ← 数组访问，隐含边界检查
}
```

**StringUTF16.getChars** (几乎相同的逻辑):
```java
// 文件: java/lang/StringUTF16.java
// 行数: ~1670-1770 (变更前已删除)

static int getChars(int i, int index, byte[] buf) {
    // 相同的算法，不同的字符写入策略
    // 使用 putChar(buf, charPos, c) 而不是 buf[charPos] = (byte)c
}
```

**BigDecimal.StringBuilderHelper** (第三次重复):
```java
// 文件: java/math/BigDecimal.java
// 行数: ~4170-4270 (变更前)

static class StringBuilderHelper {
    int putIntCompact(long intCompact) {
        // 又一次相同的算法实现
        // 使用 char[] 而不是 byte[]
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **数组边界检查** | `buf[index] = value` 每次都触发隐式边界检查 | 每次写入都有分支 |
| **分支预测失败** | 边界检查可能导致流水线停顿 | 现代 CPU 损失 ~10-20 周期 |
| **代码膨胀** | 相同逻辑在三个地方重复 | 指令缓存压力，维护困难 |
| **JIT优化限制** | 分散的实现难以统一优化 | 无法集中优化核心路径 |

### 原始PR (JDK-8342650) 回滚原因

PR #21593 被回滚是因为 **offset 计算溢出问题**:

```java
// 原始 PR #21593 中的有问题的代码
private static void putCharLatin1(byte[] buf, int charPos, int c) {
    // ❌ 当 charPos 接近 Integer.MAX_VALUE 时会溢出！
    UNSAFE.putByte(buf, ARRAY_BYTE_BASE_OFFSET + charPos, (byte) c);
}

// 修复后的代码 (本PR)
private static void uncheckedPutCharLatin1(byte[] buf, int charPos, int c) {
    // ✅ 先转换为 long 防止溢出
    UNSAFE.putByte(buf, ARRAY_BYTE_BASE_OFFSET + (long) charPos, (byte) c);
}
```

**溢出场景分析**:
```
假设 ARRAY_BYTE_BASE_OFFSET = 16 (典型值)
当 charPos = Integer.MAX_VALUE - 10:
    原始: 16 + (2147483647 - 10) = 2147483653 → 溢出为负数！
    修复: 16 + (long)(2147483647 - 10) = 2147483653L → 正确
```

## 解决方案

### 核心改动

#### 1. 统一的 DecimalDigits 实现

```java
// 文件: src/java.base/share/classes/jdk/internal/util/DecimalDigits.java
// 新增方法 (共 +279 行)

import static jdk.internal.misc.Unsafe.ARRAY_BYTE_BASE_OFFSET;

public final class DecimalDigits {
    private static final Unsafe UNSAFE = Unsafe.getUnsafe();

    // Latin1 变体 - 使用 Unsafe.putByte (无边界检查)
    public static int uncheckedGetCharsLatin1(int i, int index, byte[] buf) {
        int q;
        int charPos = index;

        boolean negative = i < 0;
        if (!negative) {
            i = -i;
        }

        // Generate two digits per iteration
        while (i <= -100) {
            q = i / 100;
            charPos -= 2;
            uncheckedPutPairLatin1(buf, charPos, (q * 100) - i);
            i = q;
        }

        // We know there are at most two digits left at this point.
        if (i <= -10) {
            charPos -= 2;
            uncheckedPutPairLatin1(buf, charPos, -i);
        } else {
            uncheckedPutCharLatin1(buf, --charPos, '0' - i);
        }

        if (negative) {
            uncheckedPutCharLatin1(buf, --charPos, '-');
        }
        return charPos;
    }

    // UTF-16 变体 - 使用 Unsafe.putCharUnaligned
    public static int uncheckedGetCharsUTF16(int i, int index, byte[] buf) {
        // 相同的算法，使用 uncheckedPutCharUTF16
        // ...
    }

    // ... long 变体 ...

    // Latin1 字符写入 - 无边界检查版本
    private static void uncheckedPutCharLatin1(byte[] buf, int charPos, int c) {
        assert charPos >= 0 && charPos < buf.length;
        // ✅ 修复: 先转为 long 防止溢出
        UNSAFE.putByte(buf, ARRAY_BYTE_BASE_OFFSET + (long) charPos, (byte) c);
    }

    // UTF-16 字符写入 - 无边界检查版本
    private static void uncheckedPutCharUTF16(byte[] buf, int charPos, int c) {
        assert charPos >= 0 && charPos < (buf.length >> 1);
        UNSAFE.putCharUnaligned(buf, ARRAY_BYTE_BASE_OFFSET + ((long) charPos << 1), (char) c);
    }

    public static void uncheckedPutPairLatin1(byte[] buf, int charPos, int v) {
        int packed = DIGITS[v & 0x7f];
        uncheckedPutCharLatin1(buf, charPos, packed & 0xFF);
        uncheckedPutCharLatin1(buf, charPos + 1, packed >> 8);
    }
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除边界检查** | `Unsafe.putByte` 不进行数组边界检查，由调用方保证安全 |
| **代码统一** | 单一实现，维护更容易，优化更集中 |
| **API简化** | 移除 JavaLangAccess 暴露的内部方法 |
| **溢出修复** | 使用 `(long) charPos` 防止 offset 计算溢出 |
| **Unsafe优化** | `putCharUnaligned` 比 `putChar` 更适合 UTF-16 场景 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 数组越界 | 调用方 (`Integer.toString`, `StringBuilder.append`) 已确保容量 |
| 溢出bug | 使用 `(long) charPos` 显式转换，防止整数溢出 |
| 类型错误 | 内部私有方法，调用链可控 |
| 内存错误 | `Unsafe` 操作只在受控环境中使用 |

## 设计决策

### Q1: 为什么用 `Unsafe.putByte` 而不是直接数组访问？

> **Reviewer**: 数组访问 `buf[index] = value` 不是更直观吗？

**A**: `Unsafe.putByte` 消除了隐式的边界检查：

```java
// 数组访问 - 每次都有边界检查
buf[charPos] = (byte) c;
// JVM 实际执行:
// if (charPos < 0 || charPos >= buf.length) throw ArrayIndexOutOfBoundsException
// buf[charPos] = c

// Unsafe.putByte - 无边界检查
UNSAFE.putByte(buf, ARRAY_BYTE_BASE_OFFSET + (long) charPos, (byte) c);
// JVM 直接写入内存，假设调用方已验证边界
```

在 `Integer.toString` 等热点路径中，这些微小的节省累积起来可以带来显著的性能提升。

### Q2: 为什么 Latin1 性能提升明显，但 UTF-16 略有下降？

> **Reviewer**: 基准测试显示 Latin1 提升 20%，但 UTF-16 下降 2-5%，为什么？

**A**:

```
Latin1 场景:
├── 原始: buf[charPos] = (byte)c      ← 有边界检查
├── 优化: Unsafe.putByte(...)         ← 无边界检查
└── 结果: 明显提升 (20%+)

UTF-16 场景:
├── 原始: StringUTF16.putChar(...)    ← 已经使用 Unsafe.putChar
├── 优化: Unsafe.putCharUnaligned(...) ← 类似的性能
└── 结果: 略有波动 (-2% ~ +2%)
```

UTF-16 的原始实现已经使用了 `Unsafe`，所以改进空间有限。微小的波动在测量误差范围内。

### Q3: 为什么删除 JavaLangAccess 的方法？

> **Reviewer**: `JavaLangAccess.getCharsLatin1/UTF16` 是给 `BigDecimal` 用的，为什么要删除？

**A**:

1. **不必要的暴露**: 这些方法只被 `BigDecimal` 使用，不需要通过 `JavaLangAccess` 暴露
2. **直接调用**: `BigDecimal` 可以直接调用 `DecimalDigits.getChars`
3. **API简化**: 减少内部 API 表面，降低维护成本

### Q4: `(long) charPos` 修复为什么能防止溢出？

> **Reviewer**: 整数溢出的具体场景是什么？

**A**:

```java
// 假设 ARRAY_BYTE_BASE_OFFSET = 16

// 场景: charPos = Integer.MAX_VALUE - 10 = 2147483637
// 原始代码:
int offset = ARRAY_BYTE_BASE_OFFSET + charPos;
// offset = 16 + 2147483637 = 2147483653
// 但是 int 最大值是 2147483647，所以 2147483653 溢出为 -2147483643！
// UNSAFE.putByte 会写入错误的地址 → 崩溃或内存损坏

// 修复后:
long offset = ARRAY_BYTE_BASE_OFFSET + (long) charPos;
// offset = 16L + 2147483637L = 2147483653L (正确)
// long 范围是 ±9.2×10^18，足够容纳这个值
```

### Q5: 为什么不一次性也优化 `Float` 和 `Double`？

> **Reviewer**: 数字转字符串的优化是否也应该应用到浮点数？

**A**:

1. **范围不同**: 本 PR 专注于十进制整数转换
2. **算法差异**: 浮点数转字符串需要更复杂的算法 (如 Dragonbox, Ryū)
3. **一次一个PR**: 遵循 OpenJDK 的"小步快跑"原则

### 溢出风险详细分析

```java
// 测试用例: 验证溢出修复
@Test
public void testOffsetOverflow() {
    // 创建接近 Integer.MAX_VALUE 的缓冲区
    byte[] buf = new byte[Integer.MAX_VALUE - 10];
    // 在接近末尾的位置写入
    int charPos = buf.length - 2;
    // 这会触发:
    // ARRAY_BYTE_BASE_OFFSET + charPos ≈ Integer.MAX_VALUE
    // 如果不转为 long，会溢出
    DecimalDigits.uncheckedPutCharLatin1(buf, charPos, 'A');
    // 验证写入成功
    assertEquals('A', buf[charPos]);
```

### 技术深度解析

#### Unsafe.putByte vs 数组访问

```java
// 数组访问的字节码
aload_1               // 加载 buf
iload_2               // 加载 charPos
bipush 65             // 加载 'A'
bastore               // 存储到数组 (隐含边界检查)

// Unsafe.putByte 的字节码
getstatic jdk/internal/misc/Unsafe.UNSAFE
aload_1               // 加载 buf
getstatic ARRAY_BYTE_BASE_OFFSET
iload_2               // 加载 charPos
i2l                   // 转为 long
ladd                  // 计算地址
bipush 65             // 加载 'A'
invokevirtual Unsafe.putByte  // 直接写入内存
```

#### 为什么 Latin1 提升更明显

```
Latin1 字符写入:
┌─────────────────────────────────────────────────────────┐
│ 原始实现: buf[charPos] = (byte) value                    │
│   ├── 字节码: bastore                                    │
│   ├── 边界检查: 每次访问都有                             │
│   └── CPU指令: BOUND + MOV                              │
│                                                          │
│ 优化后: Unsafe.putByte(buf, offset, value)              │
│   ├── 字节码: invokevirtual Unsafe.putByte              │
│   ├── 边界检查: 无 (调用方保证)                          │
│   └── CPU指令: MOV (直接内存写入)                        │
└─────────────────────────────────────────────────────────┘

UTF-16 字符写入:
┌─────────────────────────────────────────────────────────┐
│ 原始实现: StringUTF16.putChar(...)                       │
│   ├── 内部: Unsafe.putChar(...)                         │
│   ├── 边界检查: 已有 Unsafe 保护                         │
│   └── CPU指令: MOV (已优化)                             │
│                                                          │
│ 优化后: Unsafe.putCharUnaligned(...)                    │
│   ├── 边界检查: 无 (调用方保证)                          │
│   └── CPU指令: MOV (类似性能)                           │
└─────────────────────────────────────────────────────────┘
```

## 总结

这个 PR 是 JDK-8342650 的重做版本，主要贡献：

1. **代码重构**: 将三处重复的 `getChars` 实现统一到 `DecimalDigits`
2. **性能优化**: 使用 `Unsafe.putByte` 消除 Latin1 路径的数组边界检查
3. **Bug修复**: 使用 `(long) charPos` 防止 offset 计算溢出
4. **API简化**: 移除 `JavaLangAccess` 中不必要的方法暴露
5. **代码减少**: 净删除 35 行代码

**性能影响**: Latin1 编码场景提升 10-25%，UTF-16 场景基本持平。

**安全性**: 通过显式的 long 转换修复了原始 PR 的溢出问题，所有调用点已确保容量安全。
