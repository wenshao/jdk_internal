# JDK-8349400: Improve startup speed via eliminating nested classes

> **Issue**: [JDK-8349400](https://bugs.openjdk.org/browse/JDK-8349400)
> **Author**: Shaojin Wen (@wenshao)
> **Date**: 2025-05-29
> **Impact**: (-减少类加载数量, 优化元数据占用)

---

## 一眼看懂

| 维度 | 内容 |
|------|------|
| **解决了什么** | 消除 KnownOIDs 枚举中的 10 个匿名内部类，减少启动时的类加载开销 |

## 背景

### KnownOIDs 类的作用

`KnownOIDs` 是一个枚举类，用于将算法名称映射到 OID 字符串。它是 Java 加密架构 (JCA) 的核心组件，广泛用于证书解析、签名验证等安全操作。

```java
package sun.security.util;

public enum KnownOIDs {
    SHA_256("2.16.840.1.101.3.4.2.1", "SHA-256", "SHA256"),
    RSA("1.2.840.113549.1.1.1"),
    // ... 几百个 OID 定义
}
```

### 问题发现：Java Agent 场景

作者在 IntelliJ IDEA 中使用 Java Agent 调试时，发现以下现象：

```bash
java -verbose:class -javaagent:/path/to/idea_rt.jar Startup
```

输出显示：
```
[0.665s][info][class,load] sun.security.util.KnownOIDs
[0.666s][info][class,load] sun.security.util.KnownOIDs$1      <- 匿名类
[0.667s][info][class,load] sun.security.util.KnownOIDs$2      <- 匿名类
[0.667s][info][class,load] sun.security.util.KnownOIDs$3      <- 匿名类
[0.668s][info][class,load] sun.security.util.KnownOIDs$4      <- 匿名类
[0.668s][info][class,load] sun.security.util.KnownOIDs$5      <- 匿名类
[0.668s][info][class,load] sun.security.util.KnownOIDs$6      <- 匿名类

[0.669s][info][class,load] sun.security.util.KnownOIDs$10     <- 匿名类
```

**关键发现**：
- 默认 JVM 启动时不会加载这些匿名类（延迟加载）
- 但使用 Java Agent 时，KnownOIDs 类被触发初始化
- 初始化过程中，10 个匿名内部类被加载

## 问题分析

### 原始实现：匿名内部类

```java
// KnownOIDs.java (修改前)
public enum KnownOIDs {
    // ... 其他枚举常量

    KP_TimeStamping("1.3.6.1.5.5.7.3.8", "timeStamping") {
        @Override
        boolean registerNames() { return false; }
    },

    AD_TimeStamping("1.3.6.1.5.5.7.48.3", "timeStamping") {
        @Override
        boolean registerNames() { return false; }
    },

    PKCS1("1.2.840.113549.1.1", "RSA") {
        @Override
        boolean registerNames() { return false; }
    },

    X942_DH("1.2.840.10046.2.1", "DiffieHellman") {
        @Override
        boolean registerNames() { return false; }
    },

    OIW_DSA("1.3.14.3.2.12", "DSA") {
        @Override
        boolean registerNames() { return false; }
    },

    // ... 更多类似的匿名类

    // 默认实现
    boolean registerNames() {
        return true;
    }
}
```

### 为什么使用匿名内部类？

原始设计使用匿名内部类来覆盖 `registerNames()` 方法，用于控制某些 OID 是否注册其名称到查找表。

**设计意图**：
- 大多数 OID 需要注册名称（返回 true）
- 少数特殊 OID 不应注册名称（返回 false）
- 使用匿名类提供"常量特定行为"

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **类加载开销** | 每个匿名内部类是一个独立的类文件 | 10 个额外类需要加载 |
| **元空间占用** | 每个类占用元空间 | 元空间压力增加 |
| **验证开销** | JVM 需要验证每个类 | 字节码验证器工作增加 |
| **初始化链** | 父类初始化 -> 子类初始化 | 初始化时间延长 |

### 为什么默认启动不会加载？

```java
// KnownOIDs 的初始化
static {
    for (KnownOIDs o : KnownOIDs.values()) {
        register(o);  // 这里触发枚举常量的初始化
    }
}
```

在正常启动时：
- `KnownOIDs` 类被加载但可能不被立即初始化
- 枚举常量按需初始化（延迟加载）

使用 Java Agent 时：
- Agent 可能触发安全相关类的早期初始化
- 导致 KnownOIDs 被完全初始化
- 所有匿名内部类被加载

## 解决方案

### 核心改动：将方法覆盖转换为构造函数参数

```java
// KnownOIDs.java (修改后)
public enum KnownOIDs {
    // 直接使用构造函数参数控制 registerNames 行为
    KP_TimeStamping("1.3.6.1.5.5.7.3.8", "timeStamping", false),
    AD_TimeStamping("1.3.6.1.5.5.7.48.3", "timeStamping", false),
    PKCS1("1.2.840.113549.1.1", "RSA", false),
    X942_DH("1.2.840.10046.2.1", "DiffieHellman", false),
    OIW_DSA("1.3.14.3.2.12", "DSA", false),
    OIW_JDK_SHA1withDSA("1.3.14.3.2.13", "SHA1withDSA", false),
    OIW_SHA1withRSA_Odd("1.3.14.3.2.15", "SHA1withRSA", false),
    OIW_SHA1withDSA("1.3.14.3.2.27", "SHA1withDSA", false),
    OIW_SHA1withRSA("1.3.14.3.2.29", "SHA1withRSA", false),
    ITUX509_RSA("2.5.8.1.1", "RSA", false),

    // 其他枚举常量使用默认值 true

    // ... 枚举常量定义

    private final String stdName;
    private final String oid;
    private final String[] aliases;
    private final boolean registerNames;  // 新增字段

    // 原有的构造函数（registerNames 默认为 true）
    KnownOIDs(String oid) {
        this(oid, name(), new String[0], true);
    }

    KnownOIDs(String oid, String stdName) {
        this(oid, stdName, new String[0], true);
    }

    KnownOIDs(String oid, String stdName, String... aliases) {
        this(oid, stdName, aliases, true);
    }

    // 新增的构造函数（允许指定 registerNames）
    KnownOIDs(String oid, String stdName, boolean registerNames) {
        this(oid, stdName, new String[0], registerNames);
    }

    // 统一的主构造函数
    KnownOIDs(String oid, String stdName, String[] aliases, boolean registerNames) {
        this.oid = oid;
        this.stdName = stdName;
        this.aliases = aliases;
        this.registerNames = registerNames;
    }

    // 移除方法覆盖，改为直接使用字段
    // 之前：
    // boolean registerNames() { return true; }

    // 现在：
    // 无需方法，直接使用 registerNames 字段
}
```

### 关键变更点

#### 1. 添加 registerNames 字段

```java
private final boolean registerNames;  // 控制是否注册名称
```

#### 2. 添加新构造函数

```java
// 三参数构造函数：oid, stdName, registerNames
KnownOIDs(String oid, String stdName, boolean registerNames) {
    this(oid, stdName, new String[0], registerNames);
}
```

#### 3. 更新 register() 方法

```java
private static void register(KnownOIDs o) {
    KnownOIDs ov = name2enum.put(o.oid, o);
    if (ov != null) {
        throw new RuntimeException("ERROR: Duplicate " + o.oid +
                " between " + o + " and " + ov);
    } else if (debug != null) {
        debug.println(o.oid + " => " + o.name());
    }

    // 之前：if (o.registerNames())
    // 现在：if (o.registerNames)
    if (o.registerNames) {
        String stdNameUpper = o.stdName.toUpperCase(Locale.ENGLISH);
        if (Objects.nonNull(name2enum.put(stdNameUpper, o))) {
            throw new RuntimeException("ERROR: Duplicate " +
                    stdNameUpper + " exists already");
        }
        if (debug != null) {
            debug.println(stdNameUpper + " => " + o.name());
        }

        for (String a :  o.aliases) {
            String aliasUpper = a.toUpperCase(Locale.ENGLISH);
            if (Objects.nonNull(name2enum.put(aliasUpper, o))) {
                throw new RuntimeException("ERROR: Duplicate " +
                        aliasUpper + " exists already");
            }
            if (debug != null) {
                debug.println(aliasUpper + " => " + o.name());
            }
        }
    }
}
```

## 设计决策

### Q1: 为什么某些 OID 不应注册名称？

> **Reviewer**: `registerNames = false` 的语义是什么？

**A**: 这些 OID 有特殊的注册原因：

1. **命名冲突**：如 `KP_TimeStamping` 和 `AD_TimeStamping` 都使用 "timeStamping" 作为标准名称，如果都注册会产生冲突
2. **非直接使用**：如 `PKCS1` 是一个前缀 OID，JDK 内部使用但不希望用户直接通过名称查找
3. **历史遗留**：某些 OID 存在是为了向后兼容，但不应被新代码使用

### Q2: 为什么不保持延迟加载？

> **Reviewer**: 既然默认启动不会加载这些类，为什么还要优化？

**A**:

1. **Java Agent 场景很常见**：
   - IDE 调试（IntelliJ IDEA, Eclipse）
   - APM 工具（New Relic, DynaTrace）
   - 代码覆盖率工具（JaCoCo）
   - 性能分析工具（JProfiler, YourKit）

2. **安全模块的早期初始化**：
   - TLS 握手可能触发 KnownOIDs 初始化
   - 证书验证可能触发 KnownOIDs 初始化
   - 安全管理器可能触发 KnownOIDs 初始化

3. **优化成本很低**：
   - 代码变更简单（添加一个构造函数参数）
   - 无性能损失（反而略有改善）
   - 代码更清晰（消除了匿名类复杂性）

### Q3: 为什么不用枚举构造函数的默认方法？

> **Reviewer**: 可以在枚举构造函数中计算 `registerNames` 的值吗？

**A**: 可以，但不够清晰：

```java
// 可能的实现
KnownOIDs(String oid, String stdName, String... aliases) {
    this.oid = oid;
    this.stdName = stdName;
    this.aliases = aliases;
    // 基于硬编码的集合判断
    this.registerNames = !SKIP_SET.contains(stdName);
}
```

问题：
- 需要维护一个静态集合
- 不够显式（阅读代码时需要查找集合定义）
- 容易出错（添加新 OID 时可能忘记更新集合）

显式参数的方式更清晰、更安全。

## 性能对比

### 类加载数量

```
修改前：
KnownOIDs 类 + 10 个匿名内部类 = 11 个类

修改后：
KnownOIDs 类 = 1 个类

减少：10 个类 (90%+)
```

### 元空间占用

```
估算（每个匿名类约 1-2KB）：

修改前：11 x 2KB = 22KB
修改后：1 x 4KB = 4KB

减少：约 18KB
```

### 初始化时间

```
修改前：
1. 加载 KnownOIDs
2. 触发枚举常量初始化
3. 对每个匿名类：
   - 加载匿名类
   - 验证匿名类
   - 初始化匿名类
   - 调用 registerNames()

修改后：
1. 加载 KnownOIDs
2. 触发枚举常量初始化
3. 直接使用 registerNames 字段

减少：约 10 次类加载/验证/初始化循环
```
