## 背景

`HexFormat` 是 Java 17 引入的十六进制格式化工具类。`formatHex(byte[])` 方法将字节数组转换为十六进制字符串，这是哈希值、二进制数据编码的常见操作。

**为什么重要？**
- 哈希计算（MD5、SHA-256）后的输出编码
- 二进制协议的数据表示
- 调试和日志中的二进制数据显示
- 数据校验和计算

**改动位置**
```
src/java.base/share/classes/java/util/HexFormat.java
    └── formatOptDelimiter()  ← 核心优化方法
src/java.base/share/classes/jdk/internal/util/HexDigits.java
    └── digitPair(int, boolean)  ← 新增查找表方法
```

## 问题分析

### 原始实现

```java
// 文件: java/util/HexFormat.java
// 类: HexFormat

public String formatHex(byte[] bytes) {
    String s = formatOptDelimiter(bytes, 0, bytes.length);
    if (s != null) {
        return s;
    }
    // ... fallback ...
}

private String formatOptDelimiter(byte[] bytes, int fromIndex, int toIndex) {
    byte[] rep;
    if (!prefix.isEmpty() || !suffix.isEmpty()) {
        return null;
    }

    boolean ucase = digitCase == Case.UPPERCASE;
    int length = toIndex - fromIndex;
    if (delimiter.isEmpty()) {
        rep = new byte[checkMaxArraySize(length * 2L)];
        for (int i = 0; i < length; i++) {
            rep[i * 2] = (byte)toHighHexDigit(bytes[fromIndex + i]);
            rep[i * 2 + 1] = (byte)toLowHexDigit(bytes[fromIndex + i]);
        }
    }
    // ...
    return new String(rep, 0, rep.length, ISO_8859_1.INSTANCE);
}

private char toLowHexDigit(int value) {
    if (value < 10) {
        return (char)('0' + value);
    }
    if (digitCase == Case.LOWERCASE) {
        return (char)('a' - 10 + value);
    }
    return (char)('A' - 10 + value);
}

private char toHighHexDigit(int value) {
    // 类似逻辑
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **每次调用方法** | `toHighHexDigit`/`toLowHexDigit` 被调用 2n 次 | 方法调用开销 |
| **条件分支** | 每次都检查大小写 (`digitCase == Case.LOWERCASE`) | 分支预测失败 |
| **char 到 byte 转换** | `(byte)` 转换 | 虽小但累积 |
| **无缓存** | 每个字节都重新计算 | CPU 寄存器压力 |

### 执行流程

```
formatHex([0x12, 0x34, 0x56, ...])
  │
  └─ for each byte
      ├─ toHighHexDigit(byte)  ← 方法调用 + 条件判断
      ├─ toLowHexDigit(byte)   ← 方法调用 + 条件判断
      └─ 写入结果
```

### 为什么循环这么"重"？

对于 256 字节的输入：
- `toHighHexDigit` 调用：256 次
- `toLowHexDigit` 调用：256 次
- 条件分支：512 次 (每个方法内都有大小写判断)
- 字符到字节转换：512 次

## 解决方案

### 核心改动

#### 1. 在 `HexDigits` 中添加查找表方法

```java
// HexDigits.java
public final class HexDigits implements Digits {
    // 预计算的查找表：每个字节对应两个十六进制字符的 ASCII 码
    // 使用 short 存储，小端序：低位字节是第一个字符，高位字节是第二个字符
    private static final short[] DIGITS = {
        // 0x00-0x0F: '0'-'9', 'a'-'f' (小写)
        0x3030, 0x3130, 0x3230, 0x3330, 0x3430, 0x3530, 0x3630, 0x3730,
        0x3830, 0x3930, 0x4130, 0x4230, 0x4330, 0x4430, 0x4530, 0x4630,
        // ... 0x10-0xFF ...
        // 例如 0x12 -> '1' (0x31) 和 '2' (0x32) -> 0x3231 (little-endian)
    };

    /**
     * 返回 0-255 的值的十六进制 ASCII 字符对，编码为 short（小端序）。
     *
     * @param i 要转换的值 (0-255)
     * @param ucase true 大写，false 小写
     * @return 包含一对十六进制 ASCII 字符的 short
     */
    public static short digitPair(int i, boolean ucase) {
        /*
         * 位运算技巧：
         * 0b0100_0000_0100_0000 是选择字母的位掩码 (1 << 6)，
         * 大小写状态，右移 1 位后变成大小写之间的位偏移 (1 << 5)。
         *
         *  ([0-9] & 0b100_0000) >> 1 => 0
         *  ([a-f] & 0b100_0000) >> 1 => 32
         *
         *  [0-9] -  0 => [0-9]
         *  [a-f] - 32 => [A-F]
         */
        short v = DIGITS[i & 0xff];
        return ucase
                ? (short) (v - ((v & 0b0100_0000_0100_0000) >> 1))
                : v;
    }
}
```

#### 2. 在 `HexFormat` 中使用查找表

```java
// HexFormat.java
private String formatOptDelimiter(byte[] bytes, int fromIndex, int toIndex) {
    byte[] rep;
    if (!prefix.isEmpty() || !suffix.isEmpty()) {
        return null;
    }

    boolean ucase = digitCase == Case.UPPERCASE;
    int length = toIndex - fromIndex;
    if (delimiter.isEmpty()) {
        rep = new byte[checkMaxArraySize(length * 2L)];
        for (int i = 0; i < length; i++) {
            short pair = HexDigits.digitPair(bytes[fromIndex + i], ucase);
            int pos = i * 2;
            rep[pos] = (byte)pair;           // 低位字节
            rep[pos + 1] = (byte)(pair >>> 8); // 高位字节
        }
    } else if (delimiter.length() == 1 && (sep = delimiter.charAt(0)) < 256) {
        rep = new byte[checkMaxArraySize(length * 3L - 1L)];
        short pair = HexDigits.digitPair(bytes[fromIndex], ucase);
        rep[0] = (byte)pair;
        rep[1] = (byte)(pair >>> 8);
        for (int i = 1; i < length; i++) {
            int pos = i * 3;
            pair = HexDigits.digitPair(bytes[fromIndex + i], ucase);
            rep[pos - 1] = (byte) sep;
            rep[pos] = (byte)pair;
            rep[pos + 1] = (byte)(pair >>> 8);
        }
    }
    // ...
    return new String(rep, 0, rep.length, ISO_8859_1.INSTANCE);
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **查找表** | 预计算所有 256 个可能值的十六进制表示 |
| **位运算大小写** | 使用位操作而不是条件分支切换大小写 |
| **单次方法调用** | 每字节只调用一次 `digitPair` |
| **小端序存储** | 两个字符打包在一个 short 中 |

### 位运算技巧解析

```java
// 小写字母 'a' 的 ASCII: 0x61 = 0b0110_0001
// 大写字母 'A' 的 ASCII: 0x41 = 0b0100_0001
// 差异: 0x20 = 0b0010_0000 = (0b0100_0000 >> 1)

// 掩码 0b0100_0000_0100_0000 检测字母位
// 如果是字母，右移 1 位得到大小写差异
// 小写减去差异 = 大写
```

## 设计决策

### Q1: 为什么用 short 而不是 int？

> **Reviewer**: 为什么不直接用 int 存储两个字符？

**A**:
1. 两个 ASCII 字符正好是 16 bits，一个 short
2. 节省内存（查找表减半：256 shorts vs 256 ints）
3. CPU 处理 16 位值也很高效
4. 更好的缓存利用率

### Q2: 为什么用小端序？

> **Reviewer**: 为什么低位字节在前？

**A**:
1. Java 的内存模型天然小端序
2. `Unsafe.putShort` 等操作期望小端序
3. 与其他 JDK 代码保持一致

### Q3: 为什么 DIGITS 表从 256 减到 128？

> **Reviewer**: 代码审查中提到的减少是怎么回事？

**A**:
在 PR 后续迭代中讨论了这个问题：
1. 实际上只需要 128 个条目（0x00-0x7F）
2. 高位可以通过 `i & 0xff` 处理
3. 但最终版本保留了 256 以确保清晰性和正确性

### Q4: 为什么不移除 `toLowHexDigit` 和 `toHighHexDigit`？

> **Reviewer**: 既然有了查找表，为什么保留旧方法？

**A**:
1. 这些方法被其他地方使用（如 `formatHex(Appendable, byte[])`）
2. 移除会影响其他功能
3. 查找表优化主要针对批量操作

```java
@State(Scope.Benchmark)
@BenchmarkMode(Mode.Throughput)

public class HexFormatBenchmark {

    @Param({"16", "64", "256", "1024"})

    private HexFormat upperFormat;

    public String formatHexLower() {

        return upperFormat.formatHex(data);

}

├── 减少方法调用:      ~10-20% (2n -> n 次调用)
└── 更好的缓存局部性:  ~5-10% (紧凑的查找表)
```

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **行为变化** | 🟢 无风险 | 输出完全一致 |
| **兼容性** | 🟢 无影响 | 纯内部实现 |
| **性能回归** | 🟢 无风险 | 有性能测试验证 |
| **可维护性** | 🟡 中 | 位运算需要注释 |

### 内存占用

查找表内存开销：
- 256 shorts × 2 bytes = 512 bytes
- 在 `HexDigits` 类加载时分配一次
- 对于性能提升来说，内存开销可忽略
