# JDK-8316704: Regex-free parsing of Formatter and FormatProcessor specifiers

## 背景

`String.format()` 是 Java 中最常用的字符串格式化方法。原始实现使用正则表达式来解析格式说明符：

```java
// 原始实现
static final String FORMAT_SPECIFIER =
    "%(\\d+\\$)?([-#+ 0,(\\<]*)?(\\d+)?(\\.\\d+)?([tT])?([a-zA-Z%])";

static final Pattern FORMAT_SPECIFIER_PATTERN = Pattern.compile(FORMAT_SPECIFIER);
```

**问题**：
- 每次调用 `String.format()` 都会编译 Pattern（即使有缓存）
- 正则表达式匹配比手写解析器慢
- 对于简单的格式（如 `%d`, `%s`），正则表达式过于重量级

**为什么重要？**
- `String.format()` 在日志、模板、输出格式化中大量使用
- 这是 Claes Redestad 在 PR #2830 中开始的优化工作的延续
- 支持 Java 21+ 的字符串模板功能

**改动位置**
```
src/java.base/share/classes/java/util/Formatter.java
    └── FormatSpecifierParser (新类)
    └── parse(String) (重写)
src/java.base/share/classes/java/format/FormatProcessor.java
    └── 使用共享解析逻辑
```

## 问题分析

### 原始实现

```java
// Formatter.java
static List<FormatString> parse(String s) {
    ArrayList<FormatString> al = new ArrayList<>();
    int i = 0;
    int max = s.length();
    Matcher m = null;  // 正则匹配器
    while (i < max) {
        int n = s.indexOf('%', i);
        if (n < 0) {
            al.add(new FixedString(s, i, max));
            break;
        }
        if (i != n) {
            al.add(new FixedString(s, i, n));
        }
        i = n;
        if (s.charAt(i + 1) == '%') {
            al.add(new FormatSpecifier(c));
            i += 2;
            continue;
        }
        // 使用正则表达式匹配
        if (m == null) {
            m = FORMAT_SPECIFIER_PATTERN.matcher(s);
        }
        if (m.find(n) && m.start() == n) {
            al.add(new FormatSpecifier(s, m));
            i = m.end();
        } else {
            throw new UnknownFormatConversionException(String.valueOf(c));
        }
    }
    return al;
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **Pattern 编译** | 复杂的正则表达式需要编译 | 首次慢，后续有缓存开销 |
| **Matcher 创建** | 每次解析需要创建 Matcher | 对象分配 |
| **回溯** | 正则表达式可能需要回溯 | CPU 开销 |
| **分组提取** | 从 Matcher 中提取各个分组 | 多次方法调用 |

### 正则表达式复杂度

```
%(\\d+\\$)?([-#+ 0,(\\<]*)?(\\d+)?(\\.\\d+)?([tT])?([a-zA-Z%])
│        │                │        │         │        │
└─ 参数  └─ 标志          └─ 宽度   └─ 精度   └─ 日期   └─ 转换
```

6 个可选分组，大量的回溯可能性。

### 正则表达式的问题

```java
// 对于简单格式 "%d"
Pattern: "%(\\d+\\$)?([-#+ 0,(\\<]*)?(\\d+)?(\\.\\d+)?([tT])?([a-zA-Z%])"

// 匹配过程：
1. 尝试参数分组 (\d+\$)? - 失败，回溯
2. 尝试标志分组 ([-#+ 0,(\<]*)? - 失败，回溯
3. 尝试宽度分组 (\d+)? - 失败，回溯
4. 尝试精度分组 (\.\d+)? - 失败，回溯
5. 尝试日期分组 ([tT])? - 失败，回溯
6. 匹配转换 ([a-zA-Z%]) - 成功
```

即使是最简单的格式，也需要多次尝试。

## 解决方案

### 核心改动

#### 1. 手写解析器

```java
static final class FormatSpecifierParser {
    final ArrayList<FormatString> al;
    final String s;
    final int max;
    char first;
    int start;
    int off;
    char c;
    int argSize;
    int flagSize;
    int widthSize;

    FormatSpecifierParser(ArrayList<FormatString> al, char first,
                          int start, String s, int max) {
        this.al = al;
        this.first = first;
        this.c = first;
        this.start = start;
        this.off = start;
        this.s = s;
        this.max = max;
    }

    /**
     * 解析格式说明符：
     *   %[argument_index$][flags][width][.precision][t]conversion
     *
     * @return 格式说明符的长度。如果无效，返回 0。
     */
    int parse() {
        int precisionSize = 0;

        // (\d+\$)?
        parseArgument();

        // ([-#+ 0,(\<]*)?
        parseFlag();

        // (\d+)?
        parseWidth();

        if (c == '.') {
            // (\.\d+)?
            precisionSize = parsePrecision();
            if (precisionSize == -1) {
                return 0;
            }
        }

        // ([tT])?([a-zA-Z%])
        char t = '\0', conversion = '\0';
        if ((c == 't' || c == 'T') && off + 1 < max) {
            char c1 = s.charAt(off + 1);
            if (isConversion(c1)) {
                t = c;
                conversion = c1;
                off += 2;
            }
        } else if (isConversion(c)) {
            conversion = c;
            ++off;
        } else {
            return 0;
        }

        // 构造 FormatSpecifier
        if (argSize + flagSize + widthSize + precisionSize + t + conversion != 0) {
            if (al != null) {
                FormatSpecifier formatSpecifier =
                        new FormatSpecifier(s, start, argSize, flagSize,
                                           widthSize, precisionSize, t, conversion);
                al.add(formatSpecifier);
            }
            return off - start;
        }
        return 0;
    }

    private void parseArgument() {
        // (\d+\$)?
        int i = off;
        for (; i < max && isDigit(c = s.charAt(i)); ++i);
        if (i == off || c != '$') {
            c = first;
            return;
        }
        i++; // skip '$'
        if (i < max) {
            c = s.charAt(i);
        }
        argSize = i - off;
        off = i;
    }

    private void parseFlag() {
        // ([-#+ 0,(\<]*)?
        int i = off;
        for (; i < max && Flags.isFlag(c = s.charAt(i)); ++i);
        flagSize = i - off;
        off = i;
    }

    private void parseWidth() {
        // (\d+)?
        int i = off;
        for (; i < max && isDigit(c = s.charAt(i)); ++i);
        widthSize = i - off;
        off = i;
    }

    private int parsePrecision() {
        int i = ++off;
        for (; i < max && isDigit(c = s.charAt(i)); ++i);
        if (i != off) {
            int size = i - off + 1;
            off = i;
            return size;
        }
        return -1;
    }
}
```

#### 2. 简化的主循环

```java
static List<FormatString> parse(String s) {
    FormatSpecifierParser parser = null;
    ArrayList<FormatString> al = new ArrayList<>();
    int i = 0;
    int max = s.length();
    while (i < max) {
        int n = s.indexOf('%', i);
        if (n < 0) {
            al.add(new FixedString(s, i, max));
            break;
        }
        if (i != n) {
            al.add(new FixedString(s, i, n));
        }
        i = n;
        char c = s.charAt(i + 1);
        if (c == '%') {
            al.add(new FormatSpecifier(c));
            i += 2;
        } else {
            if (parser == null) {
                parser = new FormatSpecifierParser(al, c, i, s, max);
            } else {
                parser.reset(c, i);
            }
            int off = parser.parse();
            if (off > 0) {
                i += off;
            } else {
                throw new UnknownFormatConversionException(String.valueOf(c));
            }
        }
    }
    return al;
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除正则表达式** | 不再需要 Pattern 编译和 Matcher |
| **单次扫描** | 线性扫描字符串，无回溯 |
| **减少对象分配** | 只分配必要的对象 |
| **可预测的性能** | 不依赖正则引擎的优化 |

## 设计决策

### Q1: 为什么不保留正则表达式作为后备？

> **Reviewer**: 手写解析器可能有 bug，为什么保留正则作为后备？

**A**:
1. 两套解析器会带来维护负担
2. 行为必须完全一致，难以保证
3. 手写解析器已经过充分测试
4. 性能优势太大，不值得保留慢速路径

### Q2: 为什么使用 `reset()` 方法？

> **Reviewer**: 为什么不复用同一个 parser 实例？

**A**:
1. `reset()` 重置状态，避免创建新对象
2. `parse()` 可以被 `FormatProcessor` 单独调用（传 null al）
3. 这种设计支持更灵活的使用模式

### Q3: 为什么不在编译时解析？

> **Reviewer**: 对于常量格式字符串，为什么不在编译时解析？

**A**:
1. `String.format()` 通常使用动态格式字符串
2. 编译时优化需要 JDK 大幅改动
3. 当前的运行时优化已经足够好

### Q4: isConversion 为什么这么简单？

> **Reviewer**: `isConversion` 只检查字母范围，够吗？

**A**:
1. 格式说明符规范确实只使用字母
2. 后续的 `FormatSpecifier` 构造会验证有效性
3. 这是一个快速预检查，不是最终验证

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **行为变化** | 🟢 低 | 详尽的测试覆盖 |

### 边缘情况

解析器正确处理了以下边缘情况：
- 空格式字符串
- 只有 `%` 的情况

- 无效的转换字符
- 不完整的格式说明符
