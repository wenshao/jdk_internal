## 描述

This PR improves the implementaton of `Ideal` optimizations of shift nodes by ensuring that `Ideal` always returns the root of the subgraph when a modification was carried out.

### Context

A common pattern in the `Ideal` optimizations of shift nodes is to call `mask_and_replace_shift_amount` first. For a java `int`, for example, we would like to normalize `x << 33` into `x << 1`. The masked amount computed by `mask_and_replace_shift_amount` is then used for further optimizations. Because `mask_and_replace_shift_amount` directly modified the shift amount input, and we previously returned `nullptr` if no further optimization is carried out, we could end up in situations where we return `nullptr` even though `Ideal` modified the node, thus violating the invariant of `Ideal`.

In `PhaseIterativeGVN::transform_old`, we only add the users of a node to the worklist if the return of `Ideal` is not `nullptr`. Silent changes to shift nodes can then lead to missed optimization opportunities, as it was the case in the JBS report.

### Proposed fix
We need to track the changes carried out by `mask_and_replace_shift_amount` and return the subgraph root if the shift amount input was changed.

### Summary of the changes
This PR brings the following changes
- Modify `mask_and_replace_shift_amount` to not only compute the masked shift but also track changes to the shift input
- Ensure all call sites of `mask_and_replace_shift_amount` propagate the changes further
  - `LShiftNode::IdealIL`
  - `RShiftNode::IdealIL`
  - `URShiftINode::Ideal`
  - `URShiftLNode::Ideal`
- Add a regression test for the missed optimization opportunity
  - `TestMissingOptReplaceShiftAmount.java`
- Add a regression test to ensure we don't hit the recently added node hash check ([JDK-8375038](https://bugs.openjdk.org/browse/JDK-8375038))
  - `TestIdealReturnReplaceShiftAmount.java`
