## 描述

`LoadNode::make()` and `StoreNode::make()` are passed an address input
and an `adr_type`. `adr_type` should be consistent with
`gvn.type(adr)`. I added an assert in each and ran into
inconsistencies that the patch fixes. It should then be possible to
move forward with:

https://github.com/openjdk/jdk/pull/24258

that removes the `adr_type` parameter there and in the methods that
pass it around (only so `LoadNode::make()` and `StoreNode::make()` can
be called with an `adr_type`). That PR can't proceed because of the
inconsistencies that this patch fixes.

Other than some cases where a wrong `adr_type` is passed (hardcoding
the `adr_type` is error prone), there are 2 main issues:

- EA can move inputs to an `ArrayCopy` node to new new slices. It
  records that with fields in the `ArrayCopy` itself. The type that
  igvn records for the inputs can be different until it has time to
  fully run after EA.
  
- an ArrayCopy to a newly allocated object is performed on the raw
  memory slice but it uses the casted result for the allocation as
  input to the `Store` nodes that write to the object. To have a raw
  address, I changed that code to use the uncasted result of the
  allocation.
