| Issue | 标题 | 贡献 |

| [JDK-8339799](https://bugs.openjdk.org/browse/JDK-8339799) | Reduce work done in j.l.invoke bytecode generators | 本 PR |

### 字节码生成开销

Java 的 Lambda 表达式和 MethodHandle 在运行时需要动态生成字节码：

```
Lambda 表达式:
  () -> "hello"
    ↓
javac 生成 invokedynamic 指令
    ↓
LambdaMetafactory.bootstrap
    ↓
InnerClassLambdaMetafactory 生成类
    ↓ ClassFile API
写入字节码
```

**问题**: 这个生成过程存在**重复计算**和**冗余操作**。

## 核心变更

### 优化 1: 提前定义 ClassEntry，复用常量池

**变更前** (每次调用都重新创建 ClassDesc):
```java
// InnerClassLambdaMetafactory.java

private final ClassDesc lambdaClassDesc;  // ClassDesc

public InnerClassLambdaMetafactory(...) {
    // ...
    lambdaClassName = lambdaClassName(targetClass);
    lambdaClassDesc = ClassDesc.ofInternalName(lambdaClassName);
    // ...
}

// 使用时
ClassFile.of().build(lambdaClassDesc, clb -> {
    clb.withField(..., lambdaClassDesc, ...);
    cob.new_(lambdaClassDesc);
    cob.invokespecial(lambdaClassDesc, ...);
    cob.putstatic(lambdaClassDesc, ...);
});
```

**问题**: 每次使用 `lambdaClassDesc` 时，ClassFile API 都需要在常量池中查找或创建对应的 Entry。

**变更后** (提前创建 ClassEntry，直接复用):
```java
private final ConstantPoolBuilder pool = ConstantPoolBuilder.of();
private final ClassEntry lambdaClassEntry;  // ClassEntry

public InnerClassLambdaMetafactory(...) {
    // ...
    lambdaClassName = lambdaClassName(targetClass);
    lambdaClassEntry = pool.classEntry(
        ReferenceClassDescImpl.ofValidated(ConstantUtils.concat("L", lambdaClassName, ";"))
    );
    // ...
}

// 使用时 - 直接使用 ClassEntry
ClassFile.of().build(lambdaClassEntry, pool, clb -> {
    clb.withField(..., lambdaClassEntry, ...);
    cob.new_(lambdaClassEntry);
    cob.invokespecial(pool.methodRefEntry(lambdaClassEntry, ...));
    cob.putstatic(pool.fieldRefEntry(lambdaClassEntry, ...));
});
```

**效果**:
- ✅ 常量池条目只创建一次
- ✅ 避免重复查找
- ✅ 更高效的字节码生成

### 优化 2: 使用 invokeBasic() 替代 invokeExact()

**变更前**:
```java
if (factoryType.parameterCount() == 0) {
    // 非捕获 Lambda，预计算单例
    MethodHandle mh = caller.findConstructor(innerClass, constructorType);
    Object inst = mh.asType(methodType(Object.class)).invokeExact();
    return new ConstantCallSite(MethodHandles.constant(interfaceClass, inst));
}
```

**变更后**:
```java
if (factoryType.parameterCount() == 0) {
    // 非捕获 Lambda，预计算单例
    MethodHandle mh = caller.findConstructor(innerClass, constructorType);
    Object inst = mh.invokeBasic();  // 直接使用 invokeBasic
    return new ConstantCallSite(MethodHandles.constant(interfaceClass, inst));
}
```

**差异**:

| 方法 | 说明 | 开销 |
|------|------|------|
| `invokeExact()` | 类型精确，需要适配 | 需要类型转换 |
| `invokeBasic()` | 内部优化路径 | 无类型检查 |

**效果**: 零参数非捕获 Lambda 的实例化更快。

### 优化 3: 复用 argDescs

**变更前**:
```java
// 每次都重新创建 ClassDesc
for (int i = 0; i < argNames.length; i++) {
    Class<?> argType = factoryType.parameterType(i);
    // 这里会多次调用 classDesc()
    cob.putfield(lambdaClassDesc, argNames[i],
                 ClassDesc.ofDescriptor(argType.descriptorString()));
}
```

**变更后**:
```java
// 在构造函数中预先计算
argDescs = new ClassDesc[factoryType.parameterCount()];
for (int i = 0; i < argDescs.length; i++) {
    argDescs[i] = classDesc(factoryType.parameterType(i));
}

// 直接复用
for (int i = 0; i < argNames.length; i++) {
    cob.putfield(pool.fieldRefEntry(
        lambdaClassEntry,
        pool.nameAndTypeEntry(argNames[i], argDescs[i])
    ));
}
```

### 优化 4: 精简名称验证

**变更前**:
```java
private static String lambdaClassName(Class<?> targetClass) {
    String name = targetClass.getName();
    if (targetClass.isHidden())
        name = name.replace('/', '_');
    return name.replace('.', '/') + "$$Lambda";
}
```

**变更后**:
```java
private static String lambdaClassName(Class<?> targetClass) {
    String name = targetClass.getName();
    if (targetClass.isHidden())
        name = name.replace('/', '_');
    return name.replace('.', '/').concat("$$Lambda");  // 使用 concat
}
```

**改进**: `concat` 比 `+` 更明确地表示字符串连接。

### 优化 5: InvokerBytecodeGenerator 同步优化

`InvokerBytecodeGenerator` 采用了相同的优化模式：

**变更前**:
```java
private final ClassDesc classDesc;

public InvokerBytecodeGenerator(...) {
    // ...
    this.className = CLASS_PREFIX.concat(name);
    validateInternalClassName(name);
    this.classDesc = ReferenceClassDescImpl.ofValidated(
        concat("L", className, ";")
    );
    // ...
}
```

**变更后**:
```java
private final ConstantPoolBuilder pool = ConstantPoolBuilder.of();
private final ClassEntry classEntry;

public InvokerBytecodeGenerator(...) {
    // ...
    this.className = CLASS_PREFIX.concat(name);
    validateInternalClassName(name);
    this.classEntry = pool.classEntry(
        ReferenceClassDescImpl.ofValidated(concat("L", className, ";"))
    );
    // ...
}
```

## 技术分析

### ClassFile API 的 Entry 机制

ClassFile API 引入了 **Entry 类型系统**：

```
ClassDesc → 描述符
    ↓
ClassEntry → 常量池条目
    ↓
直接引用，无需重复查找
```

**使用 ClassEntry 的优势**:

| 方面 | ClassDesc | ClassEntry |
|------|-----------|------------|
| **常量池查找** | 每次查找 | 一次创建，永久复用 |
| **内存开销** | 临时对象 | 共享条目 |
| **生成速度** | 较慢 | 更快 |

### ConstantPoolBuilder 的作用

```java
ConstantPoolBuilder pool = ConstantPoolBuilder.of();

// 创建并缓存 Entry
ClassEntry entry = pool.classEntry(classDesc);

// 后续直接复用
pool.methodRefEntry(entry, nameAndType);
pool.fieldRefEntry(entry, nameAndType);
```

**工作机制**:
1. 第一次调用 `pool.classEntry()` 时创建 Entry
2. 后续调用返回缓存的 Entry
3. 避免重复创建和去重

## 经验教训

### 1. ClassFile API 最佳实践

**教训**: 使用 `ConstantPoolBuilder` 和 `Entry` 类型

```java
// ❌ 重复创建 ClassDesc
ClassFile.of().build(classDesc, cb -> {
    cb.withField(..., classDesc, ...);
});

// ✅ 使用 ConstantPoolBuilder
ConstantPoolBuilder pool = ConstantPoolBuilder.of();
ClassEntry entry = pool.classEntry(classDesc);
ClassFile.of().build(entry, pool, cb -> {
    cb.withField(..., entry, ...);
});
```

### 2. invokeBasic vs invokeExact

**教训**: 零参数场景使用 `invokeBasic()`

| 场景 | 推荐 |
|------|------|
| **零参数** | `invokeBasic()` |
| **有参数** | `invokeExact()` |

### 3. 预计算 > 重复计算

**教训**: 在构造函数中预计算常用值

```java
// ❌ 每次都计算
for (int i = 0; i < n; i++) {
    ClassDesc desc = ClassDesc.ofDescriptor(types[i].descriptorString());
    // ...
}

// ✅ 预先计算
ClassDesc[] descs = new ClassDesc[n];
for (int i = 0; i < n; i++) {
    descs[i] = ClassDesc.ofDescriptor(types[i].descriptorString());
}
for (int i = 0; i < n; i++) {
    // 使用 descs[i]
}
```

## 总结

JDK-8339799 是 ClassFile API 启动优化系列的重要部分，通过以下优化实现了 1.5-2.5% 的性能提升：

1. **ClassEntry 复用** - 避免常量池重复查找
2. **invokeBasic() 优化** - 零参数 Lambda 更快实例化
3. **预计算 argDescs** - 避免重复创建 ClassDesc
4. **ConstantPoolBuilder** - 统一常量池管理

这个优化展示了深入理解字节码生成机制和 ClassFile API 的重要性，以及微小改进累积带来的显著性能收益。
