# JDK-8339635: StringConcatFactory optimization for CompactStrings off

## 背景

`StringConcatFactory` 是 Java 字符串拼接的底层实现，负责在运行时生成高效的字符串拼接代码。它使用 `invokedynamic` 指令和字节码生成技术来优化字符串连接操作。

**CompactStrings 与 Coder**
- Java 9+ 引入了 CompactStrings，默认使用 LATIN1 (1 byte/char) 编码
- 当字符串包含非 LATIN1 字符时，自动切换到 UTF16 (2 bytes/char)
- 这种编码称为 "coder"，通过 `coder()` 方法获取

**问题场景**
当用户使用 `-XX:-CompactStrings` 关闭 CompactStrings 时：
- 所有字符串都使用 UTF16 编码
- 原有的 `coder` 方法仍然会被生成，但实际上完全没有必要
- 这增加了类加载时间和字节码生成开销

## 问题分析

### 原始实现

```java
// 文件: java/lang/invoke/StringConcatFactory.java
// 行数: ~1198-1210

/**
 * Construct the MethodType of the coder method. The first parameter is the initialized coder.
 * Only parameter types which can be UTF16 are added. Returns null if no such parameter exists.
 */
private static MethodTypeDesc coderArgsIfMaybeUTF16(MethodType concatArgs) {
    // 原始实现：始终检查参数类型
    int parameterCount = concatArgs.parameterCount();

    int maybeUTF16Count = 0;
    for (int i = 0; i < parameterCount; i++) {
        var cl = concatArgs.parameterType(i);
        if (canBeUTF16(cl)) {
            maybeUTF16Count++;
        }
    }

    if (maybeUTF16Count == 0) {
        return null;
    }
    // ... 生成 coder 方法
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **无用的方法生成** | CompactStrings 关闭时，coder 恒为 1，无需运行时计算 | 类加载时间增加 |
| **字节码膨胀** | 生成永远不会被调用的方法 | 元空间占用增加 |
| **启动延迟** | 首次字符串拼接需要生成隐藏类 | 应用启动时间变长 |

### 为什么 CompactStrings 关闭时 coder 恒定？

```java
// 当 CompactStrings = false 时：
String.COMPACT_STRINGS = false
String.LATIN1 = 0  // 不再使用
String.UTF16 = 1   // 所有字符串都使用 UTF16

// 因此：
// - 初始 coder 恒为 1 (UTF16)
// - 任何字符串的 coder 都是 1
// - coder 运算 (|) 结果始终是 1
// - 无需在运行时计算 coder
```

## 解决方案

### 核心改动

```java
// 变更后: 在 CompactStrings 关闭时跳过 coder 方法生成

private static MethodTypeDesc coderArgsIfMaybeUTF16(MethodType concatArgs) {
    // 新增：检查 CompactStrings 是否开启
    if (JLA.stringInitCoder() != 0) {
        return null;  // CompactStrings 关闭，跳过 coder 方法生成
    }

    // 原有逻辑...
    int parameterCount = concatArgs.parameterCount();
    // ...
}
```

### 新增 JavaLangAccess 方法

```java
// 文件: java/lang/System.java
// 新增内部访问方法

public byte stringInitCoder() {
    return String.COMPACT_STRINGS ? String.LATIN1 : String.UTF16;
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **早期返回** | CompactStrings 关闭时直接返回 null，避免后续计算 |
| **减少类生成** | 不生成 coder 方法，减少字节码生成开销 |
| **常量折叠** | 编译时已知 coder 值，可直接在字节码中使用常量 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 功能正确性 | CompactStrings 关闭时 coder 恒为 1，无需计算 |
| 配置兼容性 | 只影响 `-XX:-CompactStrings` 场景 |
| 边界情况 | 空参数列表的情况已被原有逻辑处理 |

## 设计决策

### Q1: 为什么不直接删除 coder 方法？

> **Reviewer**: 能否完全移除 coder 方法生成逻辑？

**A**: 不能。CompactStrings 是 JVM 运行时可配置的选项：
- 默认开启：需要运行时计算 coder
- 可选关闭：此时 coder 恒定，无需计算

因此需要动态检查，根据当前 JVM 配置决定是否生成。

### Q2: 为什么选择在 `coderArgsIfMaybeUTF16` 中检查？

> **Reviewer**: 为什么不在更早的阶段判断？

**A**:
1. 这是方法类型描述符构造的入口点
2. 返回 `null` 会自然地跳过后续的字节码生成
3. 最小化对现有代码路径的影响

### Q3: `stringInitCoder()` 的性能开销？

> **Reviewer**: 每次调用都会检查 `String.COMPACT_STRINGS`，会有性能损失吗？

**A**:
1. `String.COMPACT_STRINGS` 是 `final static boolean`，JIT 会内联
2. Hot 方法会被 JIT 编译为常量
3. 只在类生成时调用一次，不在热路径上
