> **Author**: Shaojin Wen (@wenshao)

## 背景

`HexFormat` 是 Java 17 引入的十六进制格式化工具类。原始实现使用了一个内部枚举 `Case` 来表示大小写：

```java
private enum Case {
    LOWERCASE,
    UPPERCASE
}
```

这个枚举虽然只有两个值，但它是一个独立的类，会在 JVM 启动时被加载。

**为什么重要？**
- 每个内部枚举类都需要单独加载
- 类加载涉及：验证、准备、解析、初始化等步骤
- 对于只有两个状态的场景，`boolean` 更轻量

**改动位置**
```
src/java.base/share/classes/java/util/HexFormat.java
    └── 移除内部枚举 Case，替换为 boolean ucase
```

## 问题分析

### 原始实现

```java
// 文件: java/util/HexFormat.java

public final class HexFormat {
    private static final HexFormat HEX_FORMAT =
        new HexFormat("", "", "", Case.LOWERCASE);

    private static final HexFormat HEX_UPPER_FORMAT =
        new HexFormat("", "", "", Case.UPPERCASE);

    private final String delimiter;
    private final String prefix;
    private final String suffix;
    private final Case digitCase;  // 枚举

    private enum Case {            // 内部枚举类
        LOWERCASE,
        UPPERCASE
    }

    public boolean isUpperCase() {
        return digitCase == Case.UPPERCASE;
    }

    // 枚举比较
    private String formatOptDelimiter(...) {
        boolean ucase = digitCase == Case.UPPERCASE;
        // ...
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **类加载开销** | 内部枚举需要作为独立类加载 | 启动时增加 ~0.1ms |
| **内存占用** | 枚举常量对象需要分配堆内存 | 每个常量 ~16 bytes |
| **比较开销** | 枚举比较需要引用相等性检查 | 比 boolean 慢 |
| **代码复杂度** | 枚举语法更复杂 | 字节码更多 |

### 验证类加载

运行一个空的 main 方法，使用 `-verbose:class` 可以看到：

```
[0.094s][info][class,load] java.util.HexFormat$Case source: jdk/modules/java.base
```

即使代码不使用 `HexFormat`，只要 `HexFormat` 被加载，`Case` 枚举类也会被加载。

### 类加载的开销

```
类加载过程:
1. 加载 (Loading)     - 找到 class 文件
2. 验证 (Verification) - 字节码验证
3. 准备 (Preparation) - 分配静态内存
4. 解析 (Resolution)  - 符号引用解析
5. 初始化 (Initialization) - 执行 <clinit>

对于枚举类，初始化还包括:
- 创建枚举实例
- 设置 values[] 数组
- 初始化 $VALUES 字段
```

## 解决方案

### 核心改动

```java
// 变更后: 使用 boolean

public final class HexFormat {
    private static final HexFormat HEX_FORMAT =
        new HexFormat("", "", "", false);  // false = lowercase

    private static final HexFormat HEX_UPPER_FORMAT =
        new HexFormat("", "", "", true);   // true = uppercase

    private final String delimiter;
    private final String prefix;
    private final String suffix;
    private final boolean ucase;  // 替换枚举

    // 移除内部枚举类
    // private enum Case { ... }  ← 删除

    public boolean isUpperCase() {
        return ucase;  // 直接返回
    }

    private String formatOptDelimiter(...) {
        boolean ucase = this.ucase;  // 直接使用
        // ...
    }

    char toLowHexDigit(int value) {
        if (value < 10) {
            return (char)('0' + value);
        }
        if (!ucase) {  // 简化条件
            return (char)('a' - 10 + value);
        }
        return (char)('A' - 10 + value);
    }
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除类加载** | 不再需要加载 `HexFormat$Case` 类 |
| **减少内存** | 不需要分配枚举常量对象 |
| **简化比较** | `boolean` 比较比引用比较更快 |
| **减少字节码** | 代码更紧凑 |

### 变更对比

| 方面 | 枚举版本 | Boolean 版本 |
|------|----------|-------------|
| 类数量 | +1 (内部枚举) | 0 |
| 实例字段 | 1 个引用 | 1 个 boolean |
| 比较 | `==` 引用比较 | `==` 原始比较 |
| 字节码 | getfield + if_acmpne | getfield + ifeq |

### 字节码对比

```java
// 枚举版本
getfield Field java/util/HexFormat.digitCase:Ljava/util/HexFormat$Case;
getstatic Field java/util/HexFormat$Case.UPPERCASE:Ljava/util/HexFormat$Case;

// Boolean 版本

```

## 设计决策

### Q1: 为什么不保留枚举以保证类型安全？

> **Reviewer**: 枚举提供了类型安全，`boolean` 可能会被误用。

**A**: 对于只有两个状态的场景：
1. `ucase` 是 private 字段，外部无法访问
2. 所有公共方法的行为完全一致
3. 内部实现细节不需要暴露类型安全性

### Q2: 这会影响 API 兼容性吗？

> **Reviewer**: `isUpperCase()` 的返回值会变化吗？

**A**: 不会。`isUpperCase()` 仍然返回 `boolean`，行为完全一致：
```java
// 之前
return digitCase == Case.UPPERCASE;  // boolean

// 之后
return ucase;  // boolean
```

### Q3: 为什么不直接用 `int` 常量？

> **Reviewer**: `public static final int LOWERCASE = 0;` 也可以？

**A**:
1. `int` 不如 `boolean` 语义清晰
2. `boolean` 只需 1 bit，`int` 需要 32 bits
3. JIT 对 `boolean` 的优化更好

### Q4: 序列化兼容性？

> **Reviewer**: `HexFormat` 是可序列化的，这会有影响吗？

**A**:
1. `digitCase` 字段从未被序列化
2. 它是 `private transient` 的
3. 序列化行为完全不变
