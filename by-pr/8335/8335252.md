## 背景

`Formatter.Conversion#isValid` 是一个用于验证格式化转换字符是否有效的方法。原始实现使用大型 switch 语句覆盖所有可能的转换字符。

**为什么重要？**
- `String.format()` 需要验证格式字符串
- `isValid` 是格式化路径上的热点方法
- 方法大小超过 325 字节时，JIT 拒绝内联

**改动位置**
```
src/java.base/share/classes/java/util/Formatter.java
    └── Formatter$Conversion.isValid()  ← 唯一改动的方法
```

## 问题分析

### 原始实现

```java
// 文件: java/util/Formatter.java
// 类: Formatter$Conversion

static boolean isValid(char c) {
    switch (c) {
        case 'b', 'B', 'h', 'H', 's', 'S', 'c', 'C', 'd', 'o',
             'x', 'X', 'e', 'E', 'f', 'g', 'G', 'a', 'A',
             't', 'T', 'n', '%', DECIMAL_FLOAT, HEXADECIMAL_FLOAT,
             HEXADECIMAL_FLOAT_UPPER, LINE_SEPARATOR, PERCENT_SIGN -> true;
        default -> false;
    };
}
```

### JIT 内联限制

HotSpot JIT 编译器有一个重要的限制：

```
-XX:FreqInlineSize=325  # 默认值
-XX:MaxInlineSize=35    # 小方法的默认值
```

当方法字节码超过 325 字节时，JIT 会拒绝内联：

```
@ 109   java.util.Formatter$Conversion::isValid (358 bytes)
        failed to inline: hot method too big
```

### 性能影响分析

| 影响点 | 原因 | 成本 |
|--------|------|------|
| **方法调用** | 无法内联，必须调用 | 每次调用 ~5-10ns |
| **分支预测** | 大型 switch 分支多 | 预测失败率高 |
| **代码缓存** | 大方法占用更多代码缓存 | 缓存压力 |

### 为什么 switch 这么"大"？

```java
// 每个常量 DECIMAL_FLOAT, HEXADECIMAL_FLOAT 等都是独立的 case
// 它们在编译时被展开为实际的字符值
// 这导致 switch 有很多分支，字节码很大

// switch 表达式的字节码大致结构：
tableswitch or lookupswitch
    // 每个 case 都是一个入口
    // 分支越多，字节码越大
```

## 解决方案

### 核心改动

```java
// 变更后: 简化 switch 表达式

static boolean isValid(char c) {
    return switch (c) {
        case 'b', 'B', 'h', 'H', 's', 'S', 'c', 'C', 'd', 'o',
             'x', 'X', 'e', 'E', 'f', 'g', 'G', 'a', 'A',
             't', 'T', LINE_SEPARATOR -> true;
        // 不要把 PERCENT_SIGN 放在 switch 里，
        // 那样会使方法大小超过 325，无法内联
        default -> c == PERCENT_SIGN;
    };
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **移除常量** | 将 `DECIMAL_FLOAT` 等常量从 switch 中移除 |
| **减少 case** | 减少分支数量，降低字节码大小 |
| **default 处理** | 将 `PERCENT_SIGN` 放在 default 中处理 |
| **可内联** | 方法大小降至 ~10 字节 |

### JIT 内联验证

优化后：

```
@ 109   java.util.Formatter$Conversion::isValid (10 bytes)   inline (hot)
  @ 4   jdk.internal.util.ImmutableBitSetPredicate$SmallImmutableBitSetPredicate::test (50 bytes)   inline (hot)
```

方法从 358 字节缩减到 10 字节，可以完全内联！

### 代码演进

PR 的实际实现经历了几次迭代：


## 设计决策

### Q1: 为什么只移除部分常量？

> **Reviewer**: 为什么不移除所有特殊常量？

**A**:
1. `DECIMAL_FLOAT`、`HEXADECIMAL_FLOAT` 等是内部常量
2. 它们实际上是字符常量的别名
3. `PERCENT_SIGN` = `'%'`，是最常用的转换字符

### Q2: default 分支的性能影响？

> **Reviewer**: `c == PERCENT_SIGN` 在 default 中会有额外检查吗？

**A**:
1. 大部分情况下，字符会在 switch 的 case 中匹配
2. default 只在非法字符或 `PERCENT_SIGN` 时执行
3. JIT 可以优化这种模式

### Q3: 为什么不用 `ImmutableBitSetPredicate`？

> **Reviewer**: 初稿使用了 `ImmutableBitSetPredicate`，为什么移除？

**A**:
1. `ImmutableBitSetPredicate` 虽然更灵活，但增加了复杂度
2. 简化后的 switch 已经足够小
3. 简单的代码更容易维护

### Q4: 为什么不使用 `@ForceInline`？

> **Reviewer**: 为什么不强制内联？

**A**:
1. `@ForceInline` 是更强制的手段，但有风险
2. 让方法自然地符合内联条件更安全
3. 避免过度依赖 HotSpot 特定注解
