## 背景

在 JDK 代码库中，计算整数/长整数转换为字符串后的长度（即数字的位数）是一个常见操作。这个操作在多个地方被重复实现：

1. `Integer.stringSize(int x)` - 在 `Integer` 类中
2. `Long.stringSize(long x)` - 在 `Long` 类中
3. `DateTimeFormatterBuilder` 中有两个私有副本

**为什么需要去重？**
- DRY 原则：相同逻辑不应该在多处维护
- 已有 `DecimalDigits` 类专门处理十进制数字相关操作
- 其他类（如 `StringConcatHelper`）需要通过 `JavaLangAccess` 间接调用

**改动位置**
```
src/java.base/share/classes/
    java/lang/Integer.java          (删除 stringSize 方法)
    java/lang/Long.java             (删除 stringSize 方法)
    java/lang/AbstractStringBuilder.java   (调用 DecimalDigits.stringSize)
    java/lang/StringConcatHelper.java      (调用 DecimalDigits.stringSize)
    java/time/format/DateTimeFormatterBuilder.java (删除私有副本)
    java/lang/System.java           (删除 JLA.stringSize 方法)
    jdk/internal/access/JavaLangAccess.java    (删除 stringSize 接口)
    jdk/internal/util/DecimalDigits.java      (新增 stringSize(long) 方法)
```

## 问题分析

### 原始实现 - 代码重复

**Integer.stringSize (被删除)**:
```java
// 文件: java/lang/Integer.java
static int stringSize(int x) {
    int d = 1;
    if (x >= 0) {
        d = 0;
        x = -x;
    }
    int p = -10;
    for (int i = 1; i < 10; i++) {
        if (x > p)
            return i + d;
        p = 10 * p;
    }
    return 10 + d;
}
```

**Long.stringSize (被删除)**:
```java
// 文件: java/lang/Long.java
static int stringSize(long x) {
    int d = 1;
    if (x >= 0) {
        d = 0;
        x = -x;
    }
    long p = -10;
    for (int i = 1; i < 19; i++) {
        if (x > p)
            return i + d;
        p = 10 * p;
    }
    return 19 + d;
}
```

**DateTimeFormatterBuilder 中的私有副本 (被删除)**:
```java
// 文件: java/time/format/DateTimeFormatterBuilder.java
// 有两个完全相同的私有 stringSize 方法实现
private static int stringSize(long x) {
    // ... 完全相同的代码
}
```

### 性能瓶颈分析

| 问题 | 影响 |
|------|------|
| **代码重复** | 相同逻辑在 4+ 处维护，容易产生不一致 |
| **间接调用** | `StringConcatHelper` 需要通过 `JavaLangAccess` 间接调用 |
| **维护成本** | 任何改进需要同步修改多处代码 |

## 解决方案

### 核心改动

**1. 在 DecimalDigits 中添加 long 版本**:
```java
// 文件: jdk/internal/util/DecimalDigits.java
// 新增方法
public static int stringSize(long x) {
    int d = 1;
    if (x >= 0) {
        d = 0;
        x = -x;
    }
    long p = -10;
    for (int i = 1; i < 19; i++) {
        if (x > p)
            return i + d;
        p = 10 * p;
    }
    return 19 + d;
}
```

**2. Integer 和 Long 委托给 DecimalDigits**:
```java
// 文件: java/lang/Integer.java
public static String toString(int i) {
    int size = DecimalDigits.stringSize(i);  // 替换 Integer.stringSize(i)
    // ...
}

static int stringSize(int x) {
    return DecimalDigits.stringSize(x);  // 保留为兼容方法，委托给 DecimalDigits
}
```

**3. 删除 DateTimeFormatterBuilder 中的私有副本**:
```java
// 文件: java/time/format/DateTimeFormatterBuilder.java
import jdk.internal.util.DecimalDigits;

// 删除两个私有 stringSize 方法
// 直接调用 DecimalDigits.stringSize(value)
```

**4. 删除 JavaLangAccess 中的 stringSize 方法**:
```java
// 文件: jdk/internal/access/JavaLangAccess.java
// 删除: int stringSize(long i);
```

**5. StringConcatHelper 直接调用 DecimalDigits**:
```java
// 文件: java/lang/StringConcatHelper.java
import jdk.internal.util.DecimalDigits;

static long mix(long lengthCoder, int value) {
    return checkOverflow(lengthCoder + DecimalDigits.stringSize(value));
}

static long mix(long lengthCoder, long value) {
    return checkOverflow(lengthCoder + DecimalDigits.stringSize(value));
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **单一真实来源** | 所有 `stringSize` 调用统一到 `DecimalDigits` |
| **减少间接调用** | `StringConcatHelper` 不再需要通过 `JLA` |
| **代码减少** | 删除 ~95 行重复代码 |

## 设计决策

### Q1: 为什么不直接删除 Integer.stringSize 和 Long.stringSize？

**A**: 这两个方法虽然现在是委托方法，但：
1. 可能被外部代码调用（尽管是包私有）
2. 保留可以保持二进制兼容性
3. JIT 编译器会内联这些委托调用，零性能损失

### Q2: 为什么不把 DecimalDigits.stringSize 设为 @IntrinsicCandidate？

**A**:
1. 这是一个小而简单的方法，JIT 已经能很好地内联优化
2. 循环展开后性能已经足够好
3. 对于偏向零的值（最常见情况），线性搜索实际上更快

### Q3: 为什么选择线性搜索而不是二分搜索？

**A**: 从代码注释可知：
> "values are biased heavily towards zero, and therefore linear search wins"

整数在实际应用中大部分是小值（0-100），线性搜索在这些情况下更快：
- 小值只需 1-2 次比较
- 二分搜索需要固定的 log2(10) ≈ 4 次比较
