## 背景

### 问题发现

当字符串拼接表达式的参数数量（arity）增加时，C2 JIT 编译器会消耗**指数级增长**的内存：

| 参数数量 | C2 内存占用 | 倍数 |
|----------|-------------|------|
| 13 | 6.3 MB | 基准 |
| 23 | 18 MB | 3x |
| 123 | **868 MB** | **138x** ⚠️ |
| 极端情况 | **~2 GB** | **318x** 💥 |

### 根本原因

```
字符串拼接: "a" + b + "c" + d + ... + "z" (123 个参数)
    ↓
StringConcatFactory.generateMHInlineCopy
    ↓ 构建复杂的 MethodHandle 组合树
    ↓
C2 JIT 尝试编译和优化这个巨大的 MH 树
    ↓
内存爆炸！💥
```

**MethodHandle 组合树复杂度**:
```
arity=2:  prepend(p1, p2)
arity=3:  prepend(prepend(p1, p2), p3)
arity=4:  prepend(prepend(prepend(p1, p2), p3), p4)
...
arity=n:  嵌套深度 n-1，节点数 ~O(n²)
```

当 C2 尝试内联和优化这个巨大的表达式时：
- **内联决策**: 每个节点都需要分析
- **类型推断**: 复杂的类型传播
- **逃逸分析**: 大量临时对象
- **图构建**: Ideal Graph 节点爆炸

---

## 解决方案

### 核心思路

引入**阈值机制**，当参数数量超过阈值时，回退到简单的字节码生成策略：

```
if (arity < HIGH_ARITY_THRESHOLD) {
    // 正常路径: MethodHandle 组合树
    return generateMHInlineCopy(...);
} else {
    // 高元数路径: 简单的 StringBuilder 字节码
    return SimpleStringBuilderStrategy.generate(...);
}
```

### 阈值选择

根据实验数据，选择 **20 作为默认阈值**：

| 阈值 | C2 内存 (23 args) | C2 内存 (13 args) | 类生成数量 |
|------|-------------------|-------------------|------------|
| 0 | 3.67 MB | 2.11 MB | 更多类 |
| 13 | 3.67 MB | 6.3 MB | 平衡 |
| 20 | 18 MB | 6.3 MB | 较少类 ⭐ |
| ∞ | 868 MB | 6.3 MB | 最少类 |

**选择 20 的原因**:
- 23 个参数的 18MB 内存开销是可接受的
- 保持大多数常见场景的 MethodHandle 优化
- 避免类生成数量过多

## 代码变更

### 1. 新增阈值配置

```java
// StringConcatFactory.java

private static final int HIGH_ARITY_THRESHOLD;

static {
    String highArity = VM.getSavedProperty("java.lang.invoke.StringConcat.highArityThreshold");
    HIGH_ARITY_THRESHOLD = highArity != null ? Integer.parseInt(highArity) : 20;
}
```

### 2. 分派逻辑

```java
// makeConcatWithConstants 方法

try {
    if (concatType.parameterCount() < HIGH_ARITY_THRESHOLD) {
        // 正常路径: MethodHandle 组合
        return new ConstantCallSite(
                generateMHInlineCopy(concatType, constantStrings)
                        .viewAsType(concatType, true));
    } else {
        // 高元数路径: 字节码生成
        return new ConstantCallSite(
                SimpleStringBuilderStrategy.generate(lookup, concatType, constantStrings));
    }
} catch (Error e) {
    throw e;
} catch (Throwable e) {
    throw new StringConcatException("Exception while spinning the class", e);
}
```

### 3. SimpleStringBuilderStrategy 实现

新增一个内部类，使用 ASM 直接生成 StringBuilder 链式调用的字节码：

```java
private static final class SimpleStringBuilderStrategy {
    static final int CLASSFILE_VERSION = 52; // JDK 8
    static final String METHOD_NAME = "concat";
    static final int ARGUMENT_SIZE_FACTOR = 4; // StringBuilder 初始容量估算

    static MethodHandle generate(Lookup lookup, MethodType args, String[] constants) throws Exception {
        String className = getClassName(lookup.lookupClass());
        ClassWriter cw = new ClassWriter(ClassWriter.COMPUTE_MAXS + ClassWriter.COMPUTE_FRAMES);

        // 生成类: package/HostClass$$StringConcat
        cw.visit(CLASSFILE_VERSION,
                ACC_SUPER + ACC_PUBLIC + ACC_FINAL + ACC_SYNTHETIC,
                className,
                null,
                "java/lang/Object",
                null
        );

        MethodVisitor mv = cw.visitMethod(
                ACC_PUBLIC + ACC_STATIC + ACC_FINAL,
                METHOD_NAME,
                args.toMethodDescriptorString(),
                null,
                null);

        mv.visitCode();

        // 创建 StringBuilder
        mv.visitTypeInsn(NEW, "java/lang/StringBuilder");
        mv.visitInsn(DUP);

        // 计算初始容量
        int len = 0;
        for (String constant : constants) {
            if (constant != null) {
                len += constant.length();
            }
        }
        len += args.parameterCount() * ARGUMENT_SIZE_FACTOR;
        iconst(mv, len);

        // 调用 StringBuilder(int capacity)
        mv.visitMethodInsn(INVOKESPECIAL, "java/lang/StringBuilder", "<init>", "(I)V", false);

        // 链式 append 调用
        int off = 0;
        for (int c = 0; c < args.parameterCount(); c++) {
            if (constants[c] != null) {
                mv.visitLdcInsn(constants[c]);
                sbAppend(mv, "(Ljava/lang/String;)Ljava/lang/StringBuilder;");
            }
            Class<?> cl = args.parameterType(c);
            mv.visitVarInsn(getLoadOpcode(cl), off);
            off += getParameterSize(cl);
            sbAppend(mv, getSBAppendDesc(cl));
        }
        if (constants[constants.length - 1] != null) {
            mv.visitLdcInsn(constants[constants.length - 1]);
            sbAppend(mv, "(Ljava/lang/String;)Ljava/lang/StringBuilder;");
        }

        // toString()
        mv.visitMethodInsn(INVOKEVIRTUAL, "java/lang/StringBuilder", "toString", "()Ljava/lang/String;", false);
        mv.visitInsn(ARETURN);

        mv.visitMaxs(-1, -1);
        mv.visitEnd();
        cw.visitEnd();

        byte[] classBytes = cw.toByteArray();

        // 定义隐藏类
        Lookup hiddenLookup = lookup.makeHiddenClassDefiner(className, classBytes, SET_OF_STRONG, DUMPER)
                                    .defineClassAsLookup(true);
        return hiddenLookup.findStatic(hiddenLookup.lookupClass(), METHOD_NAME, args);
    }
}
```

### 4. 生成的字节码示例

对于表达式 `"a" + b + "c" + d`：

```java
// 等效生成的代码
public static String concat(String b, String d) {
    return new StringBuilder(8)  // 2 常量 + 2 参数 * 4
        .append("a")
        .append(b)
        .append("c")
        .append(d)
        .toString();
}
```

## 性能对比

### C2 内存使用 (使用 -XX:CompileCommand=MemStat)

| 参数数量 | 原始策略 | 新策略 (阈值=20) | 改进 |
|----------|----------|------------------|------|
| **13** | 6.3 MB | 6.3 MB | 无变化 (低于阈值) |
| **23** | 18 MB | 3.67 MB | **5x** ⭐ |
| **123** | 868 MB | 4.75 MB | **183x** ⭐⭐⭐ |

### 类生成数量

高元数策略会为每个调用点生成一个隐藏类，这是权衡之一：

| 策略 | 类生成特点 | 适用场景 |
|------|------------|----------|
| **MethodHandle** | LambdaForm 共享，类少 | 低元数表达式 |
| **字节码生成** | 每个调用点一个类 | 高元数表达式 |

### 运行时性能

| 场景 | MethodHandle | 字节码生成 | 差异 |
|------|--------------|------------|------|
| 低元数 (< 20) | 更快 (可内联) | N/A | - |
| 高元数 (≥ 20) | 慢 (编译开销大) | 正常 | 字节码生成可能更快 |

## 设计决策

### Q1: 为什么选择 20 作为阈值？

> **Reviewer**: 阈值选择的标准是什么？

**A**: 基于实验数据：

1. **13 参数**: 6.3MB 内存开销 - 完全可接受
2. **23 参数**: 18MB 内存开销 - 边界可接受
3. **123 参数**: 868MB 内存开销 - 不可接受

选择 20 可以：
- 覆盖绝大多数实际场景（99%+ 的表达式 < 20 参数）
- 保持常见场景的 MethodHandle 优化优势
- 防止极端情况的内存爆炸

### Q2: 为什么不直接限制表达式大小？

> **Reviewer**: 为什么不直接拒绝编译大表达式？

**A**:
1. **合法用例**: 某些场景确实需要大量参数拼接
2. **渐进降级**: 回退到字节码仍然正确工作
3. **用户控制**: 系统属性允许用户自定义阈值

### Q3: 字节码策略的性能影响？

> **Reviewer**: 字节码生成会不会更慢？

**A**: 对于高元数表达式，字节码策略可能**更快**：

| 方面 | MethodHandle | 字节码生成 |
|------|--------------|------------|
| **首次调用** | 慢 (需要构建 MH 树) | 慢 (需要生成类) |
| **C2 编译** | 非常慢 (内存爆炸) | 正常 |
| **执行性能** | 可能略好 (更内联) | 正常 |

对于 123 个参数的表达式，字节码策略的总体性能可能更好，因为避免了 C2 编译开销。

### Q4: 为什么使用 ASM 而不是 ClassFile API？

> **Reviewer**: JDK-8331134 使用 ClassFile API，为什么这里用 ASM？

**A**: PR #18953 使用 ASM 是为了**便于向后移植**：

| 版本 | 实现方式 | 原因 |
|------|----------|------|
| **JDK 21+** | ClassFile API | 新 API，类型安全 |
| **JDK 17/21** | ASM | 兼容性 |

## 经验教训

### 1. MethodHandle 组合的复杂性

**教训**: MethodHandle 组合树的复杂度会随 arity 指数增长

```
arity=n 的 MH 树节点数 ≈ O(n²)
arity=123 → 节点数 ~15,000
```

### 2. C2 编译资源的边界

**教训**: 需要为 JIT 编译设置合理的资源边界

| 资源 | 建议限制 |
|------|----------|
| **内存** | < 100 MB/方法 |
| **时间** | < 10s/方法 |
| **IR 节点** | < 50,000 |

### 3. 渐进降级策略

**教训**: 对于极端情况，提供降级路径比硬限制更好

```
正常路径 → MethodHandle (优化)
    ↓ 阈值检测
降级路径 → 字节码生成 (正确性)
```

## 总结

JDK-8327247 是一个关键的 bug 修复，解决了复杂字符串拼接导致 C2 编译器内存爆炸的问题：

1. **问题**: 123 参数的字符串拼接消耗 868MB 内存
