## 背景

`UUID.toString()` 是 Java 中生成 UUID 字符串表示的核心方法，输出格式为：

```
123e4567-e89b-12d3-a456-426614174000
^^^^^^^ ^^^^ ^^^^ ^^^^ ^^^^^^^^^^^^
 8-4-4-4-12   标准格式
```

在 PR #16245 之前，UUID.toString 使用 `ByteArrayLittleEndian` 来处理字节序和十六进制转换。PR #16245 引入了 C2 编译器的优化，能够自动将多个小字节存储合并为更大的存储操作。

**改动位置**
```
src/java.base/share/classes/java/util/UUID.java
    └── toString()

src/java.base/share/classes/jdk/internal/util/HexDigits.java
    └── put4(byte[] buffer, int index, int value)
```

## 问题分析

### 原始实现

```java
// 文件: java/util/UUID.java
// 行数: ~467-495

@Override
public String toString() {
    long lsb = leastSigBits;
    long msb = mostSigBits;

    byte[] buf = new byte[36];

    // 使用 ByteArrayLittleEndian 和 HexDigits.packDigits
    ByteArrayLittleEndian.setLong(
        buf,
        0,
        HexDigits.packDigits((int) (msb >> 56), (int) (msb >> 48),
                            (int) (msb >> 40), (int) (msb >> 32)));
    buf[8] = '-';

    ByteArrayLittleEndian.setInt(
        buf,
        9,
        HexDigits.packDigits(((int) msb) >> 24, ((int) msb) >> 16));
    buf[13] = '-';

    ByteArrayLittleEndian.setInt(
        buf,
        14,
        HexDigits.packDigits(((int) msb) >> 8, (int) msb));
    buf[18] = '-';

    ByteArrayLittleEndian.setInt(
        buf,
        19,
        HexDigits.packDigits((int) (lsb >> 56), (int) (lsb >> 48)));
    buf[23] = '-';

    ByteArrayLittleEndian.setLong(
        buf,
        24,
        HexDigits.packDigits((int) (lsb >> 40), (int) (lsb >> 32),
                            ((int) lsb) >> 24, ((int) lsb) >> 16));
    ByteArrayLittleEndian.setInt(
        buf,
        32,
        HexDigits.packDigits(((int) lsb) >> 8, (int) lsb));

    try {
        return jla.newStringNoRepl(buf, StandardCharsets.ISO_8859_1);
    } catch (CharacterCodingException cce) {
        throw new AssertionError(cce);
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **复杂的 API 调用** | ByteArrayLittleEndian.setLong/setInt 是间接调用 | 方法调用开销 |
| **packDigits 中间步骤** | 需要先打包成 long/int，再写入 | 额外的计算和内存访问 |
| **代码可读性差** | 字节序处理和十六进制转换混在一起 | 维护成本高 |

### 为什么现在可以移除 ByteArrayLittleEndian？

PR #16245 引入的 C2 优化能够将：

```java
buf[0] = (byte) v1;
buf[1] = (byte) v2;
buf[2] = (byte) v3;
buf[3] = (byte) v4;
```

优化为单个 32 位存储操作：

```asm
; C2 生成的汇编代码
mov [buf + offset], eax   ; 一次性写入 4 字节
```

这意味着直接写入字节与使用 ByteArrayLittleEndian 的性能相同，但代码更简洁。

## 解决方案

### 核心改动

#### 1. 新增 HexDigits.put4 方法

```java
// 文件: jdk/internal/util/HexDigits.java

/**
 * 将 16 位整数转换为 4 个十六进制 ASCII 字节并写入 buffer
 * @param buffer 目标字节数组
 * @param index 写入位置
 * @param value 16 位整数（只使用低 16 位）
 */
public static void put4(byte[] buffer, int index, int value) {
    // 准备一个 int 值，使 C2 生成 4 字节写入而不是两个 2 字节写入
    int v = (DIGITS[value & 0xff] << 16) | DIGITS[(value >> 8) & 0xff];
    buffer[index]     = (byte)  v;
    buffer[index + 1] = (byte) (v >> 8);
    buffer[index + 2] = (byte) (v >> 16);
    buffer[index + 3] = (byte) (v >> 24);
}
```

**关键技巧**：将两个十六进制数字先打包成一个 int，然后分别写入 4 个字节。这使得 C2 能够优化为：

1. **低位 2 字节** - `v` 和 `v >> 8` 可能被合并为一次 16 位写入
2. **高位 2 字节** - `v >> 16` 和 `v >> 24` 可能被合并为一次 16 位写入
3. **或整个 int** - 可能被合并为一次 32 位写入

#### 2. 简化 UUID.toString

```java
// 变更后: 直接使用 put4

@Override
public String toString() {
    // 提取 long 的各个 int 部分
    int i0 = (int) (mostSigBits >> 32);
    int i1 = (int) mostSigBits;
    int i2 = (int) (leastSigBits >> 32);
    int i3 = (int) leastSigBits;

    byte[] buf = new byte[36];

    // 直接使用 put4 写入十六进制
    HexDigits.put4(buf, 0, i0 >> 16);          // 前 4 个十六进制数字
    HexDigits.put4(buf, 4, i0);                // 接下来的 4 个
    buf[8] = '-';

    HexDigits.put4(buf, 9, i1 >> 16);          // 时间高位和版本
    buf[13] = '-';

    HexDigits.put4(buf, 14, i1);               // 时间低位
    buf[18] = '-';

    HexDigits.put4(buf, 19, i2 >> 16);         // 时钟序列高位
    buf[23] = '-';

    HexDigits.put4(buf, 24, i2);               // 时钟序列低位
    HexDigits.put4(buf, 28, i3 >> 16);         // 节点 ID 前半部分
    HexDigits.put4(buf, 32, i3);               // 节点 ID 后半部分

    try {
        return jla.newStringNoRepl(buf, StandardCharsets.ISO_8859_1);
    } catch (CharacterCodingException cce) {
        throw new AssertionError(cce);
    }
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **移除中间层** | 不再通过 ByteArrayLittleEndian，直接写入字节 |
| **代码更清晰** | UUID 结构（8-4-4-4-12）在代码中清晰可见 |
| **利用 C2 优化** | 相邻的字节写入会被合并为更大的存储操作 |
| **减少方法调用** | 从 6 次 ByteArrayLittleEndian 调用减少到 0 次 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 字节序问题 | UUID 使用大端序，put4 正确处理了字节序 |
| 格式错误 | 输出格式与原实现完全一致 |
| 性能回归 | C2 优化确保性能不降低 |

## 设计决策

### Q1: put4 为什么不直接写入 4 个独立的字节？

> **Reviewer**: 为什么要先打包成 int？

**A**:
1. **C2 优化提示** - 将值准备为 int，让 C2 知道这是一个连续的 4 字节写入
2. **减少指令数** - `v << 16` 可以与 DIGITS 查找表合并
3. **更好的代码生成** - C2 可能将其优化为单个 32 位存储

```java
// 如果直接写 4 个独立字节
buffer[index]     = DIGITS[value & 0x0f];      // 需要额外的位操作
buffer[index + 1] = DIGITS[(value >> 4) & 0x0f];
buffer[index + 2] = DIGITS[(value >> 8) & 0x0f];
buffer[index + 3] = DIGITS[(value >> 12) & 0x0f];

// 使用 put4 的方式
int v = (DIGITS[value & 0xff] << 16) | DIGITS[(value >> 8) & 0xff];
// C2 可以看到这是一个连续的 4 字节操作
```

### Q2: 为什么保留 packDigits 方法？

> **Reviewer**: packDigits 还有其他用途吗？

**A**:
1. **向后兼容** - 可能有其他内部代码在使用
2. **不同用途** - packDigits 返回打包的值，put4 直接写入
3. **逐步迁移** - 可以在后续 PR 中逐步替换其他使用 packDigits 的地方

### Q3: i0/i1/i2/i3 的命名有什么意义？

> **Reviewer**: 为什么不直接使用 msb/lsb？

**A**:
1. **i0** - mostSigBits 的高 32 位
2. **i1** - mostSigBits 的低 32 位
3. **i2** - leastSigBits 的高 32 位
4. **i3** - leastSigBits 的低 32 位

这种命名反映了 UUID 的 128 位结构，与标准格式对应。
