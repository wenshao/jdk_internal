## 背景

`DirectClassBuilder::build` 是 ClassFile API 中生成类文件的核心方法。它在编译、代码生成、AOP 等场景中被大量使用。

**为什么重要？**
- 类文件生成是字节码操作的最终步骤
- `build` 方法的性能直接影响整体生成速度
- 原始方法过大（codeSize 361），不易被 JIT 内联

**改动位置**
```
src/java.base/share/classes/jdk/internal/classfile/impl/
    ├── DirectClassBuilder.java  ← 重构 build() 方法
    ├── DirectCodeBuilder.java   ← 提取 writeExceptionHandlers
    ├── BufWriterImpl.java       ← 添加 @ForceInline，新增 join() 方法
    └── Util.java                ← 添加 @ForceInline
```

## 问题分析

### 原始实现的主要问题

#### 1. 方法过大

```java
// DirectClassBuilder.build() - codeSize 361
public byte[] build() {
    // ... 200+ 行代码
    // 不易被 JIT 内联
}
```

#### 2. writeExceptionHandlers 过大

```java
// DirectCodeBuilder.writeExceptionHandlers - codeSize 183
private void writeExceptionHandlers(BufWriterImpl buf) {
    int pos = buf.size();
    int handlersSize = handlers.size();
    buf.writeU2(handlersSize);
    for (AbstractPseudoInstruction.ExceptionCatchImpl h : handlers) {
        // ... 异常处理逻辑
    }
}
```

#### 3. 缺少 @ForceInline

```java
// BufWriterImpl.writeIndex - codeSize 62
public void writeIndex(PoolEntry entry) {
    // 没有 @ForceInline，可能不被内联
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **方法过大** | 超过 HotSpot 内联阈值 | JIT 无法内联优化 |
| **重复计算** | 每次都调用 `size()` 方法 | 增加方法调用开销 |
| **缺少内联提示** | 没有 `@ForceInline` | JIT 保守优化 |
| **数组分配** | 每次都创建新数组连接 | 内存分配开销 |

## 解决方案

### 核心改动 1：重构 DirectClassBuilder.build()

```java
// 优化前
public byte[] build() {
    // 大量代码...
    // Join head and tail into an exact-size buffer
    byte[] result = new byte[head.size() + tail.size()];
    head.copyTo(result, 0);
    tail.copyTo(result, head.size());
    return result;
}

// 优化后
public byte[] build() {
    var constantPool = this.constantPool;  // ← 提取局部变量
    ClassEntry superclass = superclassEntry;
    // ... 重构代码

    // 使用新的 join 方法
    return BufWriterImpl.join(head, tail);  // ← 简化
}
```

### 核心改动 2：提取 writeExceptionHandlers

```java
// 优化前：一个大方法
private void writeExceptionHandlers(BufWriterImpl buf) {
    int pos = buf.size();
    int handlersSize = handlers.size();
    buf.writeU2(handlersSize);
    for (...) {
        // 所有逻辑
    }
}

// 优化后：拆分为两个方法
private void writeExceptionHandlers(BufWriterImpl buf) {
    int pos = buf.size();
    int handlersSize = handlers.size();
    buf.writeU2(handlersSize);
    if (handlersSize > 0) {
        writeExceptionHandlers(buf, pos);  // ← 提取
    }
}

private void writeExceptionHandlers(BufWriterImpl buf, int pos) {
    // 异常处理逻辑
}
```

### 核心改动 3：添加 @ForceInline

```java
// BufWriterImpl.java

@ForceInline  // ← 新增
public void writeIndex(PoolEntry entry) {
    int idx = AbstractPoolEntry.maybeClone(constantPool, entry).index();
    if (idx < 1 || idx > Character.MAX_VALUE)
        throw invalidIndex(idx, entry);  // ← 提取错误创建
    writeU2(idx);
}

@ForceInline  // ← 新增
public void writeU2(int x) {
    reserveSpace(2);
    // ...
}
```

### 核心改动 4：新增 join 方法

```java
// BufWriterImpl.java

/**
 * Join head and tail into an exact-size buffer
 */
static byte[] join(BufWriterImpl head, BufWriterImpl tail) {
    byte[] result = new byte[head.size() + tail.size()];
    head.copyTo(result, 0);
    tail.copyTo(result, head.size());
    return result;
}
```

### 核心改动 5：Util 方法添加 @ForceInline

```java
// Util.java

@ForceInline  // ← 新增
public static void writeAttributes(BufWriterImpl buf, List<? extends Attribute<?>> list) {
    int size = list.size();  // ← 缓存 size
    buf.writeU2(size);
    for (int i = 0; i < size; i++) {
        writeAttribute(buf, list.get(i));  // ← 使用索引
    }
}

@ForceInline  // ← 新增
static void writeList(BufWriterImpl buf, List<Writable> list) {
    int size = list.size();  // ← 缓存 size
    buf.writeU2(size);
    for (int i = 0; i < size; i++) {
        list.get(i).writeTo(buf);  // ← 使用索引
    }
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **方法拆分** | 将大方法拆分为小方法，便于内联 |
| **@ForceInline** | 提示 JIT 必须内联 |
| **局部变量缓存** | 减少重复的方法调用 |
| **提取工具方法** | join() 封装常用操作 |

## 设计决策

### Q1: 为什么要用 @ForceInline？

> **Reviewer**: 不是应该让 JIT 自己决定吗？

**A**:
1. **热点方法**: 这些方法是明确的热点路径
2. **编译器提示**: @ForceInline 是强烈的内联建议
3. **性能关键**: 这些方法在类文件生成时被频繁调用

### Q2: 为什么要提取 join 方法？

> **Reviewer**: 这不是简单的代码移动吗？

**A**:
1. **代码复用**: join 可能被其他地方使用
2. **清晰的语义**: join 操作有明确的含义
3. **便于优化**: 独立方法更容易被 JIT 优化

### Q3: 代码大小减少有什么好处？

> **Reviewer**: 361 -> 319 的减少有多大意义？

**A**:
1. **内联阈值**: HotSpot 默认内联阈值约 325 字节
2. **C1 编译**: Client Compiler 更容易内联小方法
3. **指令缓存**: 更小的代码占用更少的指令缓存

### Benchmark 方法

```java
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
public class DirectClassBuilderBenchmark {

    @Benchmark
    public byte[] buildClass() {
        return ClassFile.of().build(ClassDesc.of("Test"), cb -> {
            cb.withFlags(ClassFile.ACC_PUBLIC);
            cb.withMethod("test", MethodTypeDesc.of("(II)I"),
                ClassFile.ACC_PUBLIC | ClassFile.ACC_STATIC, mb -> {
                    var code = mb.codeBuilder();
                    code.iload(0).iload(1).iadd().ireturn();
                });
        });
    }
}
```
