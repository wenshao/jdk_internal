## 背景

`StackMapGenerator::processMethod` 是处理方法字节码、更新栈映射帧的核心方法。它在字节码遍历过程中被反复调用。

**为什么重要？**
- StackMap 生成是字节码验证的关键步骤
- `processMethod` 包含主要的字节码处理循环
- 方法的字节码大小影响 JIT 编译

**改动位置**
```
src/java.base/share/classes/jdk/internal/classfile/impl/StackMapGenerator.java
    └── processMethod()  ← 唯一改动的方法
```

## 问题分析

### 原始实现

```java
// 文件: jdk/internal/classfile/impl/StackMapGenerator.java

private void processMethod() {
    currentFrame.setLocalsFromArg(methodName, methodDesc, isStatic, thisType);
    currentFrame.stackSize = 0;
    currentFrame.flags = 0;
    int stackmapIndex = 0;
    boolean ncf = false;

    while (bcs.next()) {
        currentFrame.offset = bcs.bci();
        if (stackmapIndex < frames.size()) {
            int thisOffset = frames.get(stackmapIndex).offset;
            if (ncf && thisOffset > bcs.bci()) {
                throw generatorError("Expecting a stack map frame");
            }
            if (thisOffset == bcs.bci()) {
                if (!ncf) {
                    currentFrame.checkAssignableTo(frames.get(stackmapIndex));
                }
                Frame nextFrame = frames.get(stackmapIndex++);
                while (!nextFrame.dirty) {
                    if (stackmapIndex == frames.size()) return;
                    nextFrame = frames.get(stackmapIndex++);
                }
                bcs.reset(nextFrame.offset);
                bcs.next();
                currentFrame.copyFrom(nextFrame);
                nextFrame.dirty = false;
            } else if (thisOffset < bcs.bci()) {
                throw new ClassFormatError(String.format("Bad stack map offset %d", thisOffset));
            }
        } else if (ncf) {
            throw generatorError("Expecting a stack map frame");
        }

        // ... 字节码处理
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **重复字段访问** | `currentFrame`、`frames` 多次访问 | 增加字段读取开销 |
| **重复计算** | `bcs.bci()` 多次调用 | 重复的 getter 调用 |
| **字符串格式化** | `String.format("Bad stack map offset %d", ...)` | 异常路径的字符串构建 |
| **字节码大小** | 方法太大，不易内联 | codeSize 326 |

## 解决方案

### 核心改动

```java
// 优化后：提取局部变量，减少字段访问

private void processMethod() {
    var frames = this.frames;              // ← 提取到局部变量
    var currentFrame = this.currentFrame;  // ← 提取到局部变量

    currentFrame.setLocalsFromArg(methodName, methodDesc, isStatic, thisType);
    currentFrame.stackSize = 0;
    currentFrame.flags = 0;
    int stackmapIndex = 0;
    boolean ncf = false;

    while (bcs.next()) {
        int bci = bcs.bci();               // ← 缓存 bci
        currentFrame.offset = bci;

        if (stackmapIndex < frames.size()) {
            int thisOffset = frames.get(stackmapIndex).offset;
            if (ncf && thisOffset > bci) {
                throw generatorError("Expecting a stack map frame");
            }
            if (thisOffset == bci) {
                Frame nextFrame = frames.get(stackmapIndex++);
                if (!ncf) {
                    currentFrame.checkAssignableTo(nextFrame);
                }
                while (!nextFrame.dirty) {
                    if (stackmapIndex == frames.size()) return;
                    nextFrame = frames.get(stackmapIndex++);
                }
                bcs.reset(nextFrame.offset);
                bcs.next();
                currentFrame.copyFrom(nextFrame);
                nextFrame.dirty = false;
            } else if (thisOffset < bci) {
                throw generatorError("Bad stack map offset");  // ← 简化异常消息
            }
        } else if (ncf) {
            throw generatorError("Expecting a stack map frame");
        }

        // ... 字节码处理
    }
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **局部变量缓存** | 将 `frames`、`currentFrame` 提取到局部变量 |
| **减少 getter 调用** | 缓存 `bcs.bci()` 结果 |
| **简化异常消息** | 移除 `String.format`，使用简单字符串 |
| **代码大小优化** | codeSize 从 326 减少到 283 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 变量遮蔽 | 局部变量作用域明确 |
| 异常信息 | 简化后仍然包含足够信息 |
| 逻辑不变 | 只重构，不改变逻辑 |

## 设计决策

### Q1: 为什么要提取到局部变量？

> **Reviewer**: 这不是编译器会自动优化的吗？

**A**:
1. **字段访问 vs 局部变量**: 字段访问需要加载 `this`，局部变量直接访问
2. **JVM 限制**: JVM 规范对字段访问有特定限制，局部变量更灵活
3. **内联友好**: 局部变量更容易被 JIT 优化

### Q2: 为什么要简化异常消息？

> **Reviewer**: 移除偏移量信息不会影响调试吗？

**A**:
1. **异常路径**: 这是异常路径，不应该频繁触发
2. **上下文保留**: `generatorError` 会包含当前 BCI
3. **代码大小**: 简化字符串减少常量池压力

### Q3: 代码大小减少有什么好处？

> **Reviewer**: 326 到 283 的减少有多大意义？

**A**:
1. **内联阈值**: HotSpot 通常不内联超过 325 字节的方法
2. **编译优化**: 更小的方法更容易被 C1/C2 优化
3. **指令缓存**: 更小的代码占用更少的指令缓存

### Benchmark 方法

@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)

    @Benchmark
    public byte[] generateComplexMethod() {
        return ClassFile.of().build(ClassDesc.of("Test"), cb -> {
            cb.withMethod("test", MethodTypeDesc.of("(II)I"),
                ClassFile.ACC_PUBLIC | ClassFile.ACC_STATIC, mb -> {
                    var code = mb.codeBuilder();
                    // 生成包含多个分支的方法
                    for (int i = 0; i < 50; i++) {
                        var l = code.newLabel();
                        code.iload(0).ifne(l);
                        code.iconst_(i);
                        code.labelBinding(l);
                    }
                    code.ireturn();
                });
        });
    }
}
