## 背景

`DataInputStream.readUTF()` 是 Java 中读取 Modified UTF-8 编码字符串的核心方法，广泛应用于：

1. **Java 序列化** - ObjectOutputStream/ObjectInputStream 使用此格式
2. **RMI (远程方法调用)** - 参数和返回值的传输
3. **自定义协议** - 许多 Java 应用程序使用此格式进行数据交换

Modified UTF-8 是 Java 特有的编码格式，与标准 UTF-8 类似，但对 null 字符有特殊处理。

**改动位置**
```
src/java.base/share/classes/java/io/DataInputStream.java
    └── readUTF(DataInput in)
```

## 问题分析

### 原始实现

```java
// 文件: java/io/DataInputStream.java
// 行数: ~575-620

public static final String readUTF(DataInput in) throws IOException {
    int utflen = in.readUnsignedShort();
    byte[] bytearr;
    char[] chararr;
    if (in instanceof DataInputStream dis) {
        if (dis.bytearr.length < utflen) {
            dis.bytearr = new byte[utflen*2];
            dis.chararr = new char[utflen*2];
        }
        chararr = dis.chararr;
        bytearr = dis.bytearr;
    } else {
        bytearr = new byte[utflen];
        chararr = new char[utflen];
    }

    int c, char2, char3;
    int count = 0;
    int chararr_count=0;

    in.readFully(bytearr, 0, utflen);

    // 问题1: 逐字节检查 ASCII 前缀
    while (count < utflen) {
        c = (int) bytearr[count] & 0xff;
        if (c > 127) break;
        count++;
        chararr[chararr_count++]=(char)c;
    }

    // 问题2: 逐字符解码多字节序列
    while (count < utflen) {
        c = (int) bytearr[count] & 0xff;
        switch (c >> 4) {
            case 0: case 1: case 2: case 3: case 4: case 5: case 6: case 7:
                // 0xxxxxxx
                count++;
                chararr[chararr_count++]=(char)c;
                break;
            case 12: case 13:
                // 110xxxxx 10xxxxxx
                count += 2;
                if (count > utflen)
                    throw new UTFDataFormatException(...);
                c2 = (int) bytearr[count-1];
                if ((c2 & 0xC0) != 0x80)
                    throw new UTFDataFormatException(...);
                chararr[chararr_count++] = (char)(((c & 0x1F) << 6) | (c2 & 0x3F));
                break;
            case 14:
                // 1110xxxx 10xxxxxx 10xxxxxx
                count += 3;
                // ... 更多解码逻辑
                break;
            default:
                throw new UTFDataFormatException(...);
        }
    }
    return new String(chararr, 0, chararr_count);
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **逐字节检查** | 第一个 while 循环逐字节检查是否为 ASCII | CPU 分支预测失败 |
| **逐字节复制** | ASCII 字符逐个复制到 char 数组 | 内存带宽浪费 |
| **无法批量处理** | 即使大部分数据是 ASCII，也无法利用 SIMD | 现代 CPU 未充分利用 |
| **数组分配** | 非 DataInputStream 场景总是分配新数组 | GC 压力 |

### 为什么 `countPositives` 更快？

`JavaLangAccess.countPositives` 是一个内部 intrinsic 方法：

```java
// C2 会将其编译为 SIMD 指令 (如 SSE/AVX)
int countPositives(byte[] buf, int offset, int length);
```

**优势**：
1. **SIMD 加速** - 一次处理 16/32 字节
2. **分支预测友好** - 减少条件跳转
3. **内存访问优化** - 连续内存访问模式

## 解决方案

### 核心改动

```java
// 变更后: 使用 JLA.countPositives 和 JLA.inflateBytesToChars

public static final String readUTF(DataInput in) throws IOException {
    int utflen = in.readUnsignedShort();
    byte[] bytearr = null;

    // 优化1: 复用 DataInputStream 的缓存
    if (in instanceof DataInputStream dis) {
        if (dis.bytearr.length >= utflen) {
            bytearr = dis.bytearr;
        }
    }

    boolean trusted = false;
    if (bytearr == null) {
        bytearr = new byte[utflen];
        trusted = true;
    }

    in.readFully(bytearr, 0, utflen);

    // 优化2: 使用 countPositives 批量检查 ASCII
    int ascii = JLA.countPositives(bytearr, 0, utflen);
    if (ascii == utflen) {
        // 全是 ASCII，直接创建 String
        String str;
        if (trusted) {
            str = JLA.newStringNoRepl(bytearr, StandardCharsets.ISO_8859_1);
        } else {
            str = new String(bytearr, 0, utflen, StandardCharsets.ISO_8859_1);
        }
        return str;
    }

    // 缓存 bytearr 以便下次复用
    if (trusted && in instanceof DataInputStream dis) {
        dis.bytearr = bytearr;
        trusted = false;
    }

    char[] chararr;
    if (in instanceof DataInputStream dis) {
        if (dis.chararr.length < (utflen << 1)) {
            dis.chararr = new char[utflen << 1];
        }
        chararr = dis.chararr;
    } else {
        chararr = new char[utflen];
    }

    // 优化3: 使用 inflateBytesToChars 批量转换 ASCII 部分
    if (ascii != 0) {
        JLA.inflateBytesToChars(bytearr, 0, chararr, 0, ascii);
        count += ascii;
        chararr_count += ascii;
    }

    // 只处理非 ASCII 部分
    while (count < utflen) {
        // ... 原有的多字节解码逻辑
    }

    return new String(chararr, 0, chararr_count);
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **批量 ASCII 检测** | `countPositives` 使用 SIMD 加速 |
| **快速路径** | 全 ASCII 字符串直接返回，无需解码 |
| **批量字节转换** | `inflateBytesToChars` 批量转换 ASCII 部分 |
| **缓存复用优化** | 更智能的缓存复用策略 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| countPositives 行为变化 | 它是 intrinsic，行为由 JVM 保证 |
| 字符串编码错误 | 非 ASCII 部分仍使用原有逻辑 |
| 内存安全问题 | 边界检查由 intrinsic 方法处理 |

## 设计决策

### Q1: 为什么使用 `countPositives` 而不是 `countNonZeroAscii`？

> **Reviewer**: 两者有什么区别？

**A**:
- `countPositives` - 统计 0-127 范围内的字节（不包括 0）
- `countNonZeroAscii` - 统计 1-127 范围内的字节

在 Modified UTF-8 中，null 字符（0）会被编码为 2 字节序列，所以两者在这里是等价的。但 `countPositives` 是更通用的方法。

### Q2: 为什么在 `ObjectInputStream` 中先优化，现在才优化 `DataInputStream`？

> **Reviewer**: 为什么不同时优化？

**A**:
1. `ObjectInputStream` 的优化是在 PR #20866 中完成的
2. 本次 PR 将同样的优化应用到 `DataInputStream`
3. 两个类的代码结构略有不同，需要分别实现

### Q3: `trusted` 标志的作用是什么？

> **Reviewer**: 什么时候设置为 trusted？

**A**:
- `trusted = true` - 新分配的 bytearr，可以安全地缓存到 DataInputStream
- `trusted = false` - 使用的是 DataInputStream 的缓存，不能再次缓存

这个标志确保不会将共享的 bytearr 意外地存储到 DataInputStream 中。

@State(Scope.Thread)
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
public class ReadUTFBenchmark {

    private byte[] utfDataAscii;      // 纯 ASCII
    private byte[] utfDataMixed;      // 混合 ASCII 和 UTF-8
    private ByteArrayInputStream bais;
    private DataInputStream dis;

    @Setup
    public void setup() throws IOException {
        // 创建测试数据
        ByteArrayOutputStream baos = new ByteArrayOutputStream();
        DataOutputStream dos = new DataOutputStream(baos);
        dos.writeUTF("HelloWorld");  // ASCII
        utfDataAscii = baos.toByteArray();

        baos.reset();
        dos.writeUTF("Hello\u4e16\u754c");  // 混合
        utfDataMixed = baos.toByteArray();

    @Benchmark

        bais = new ByteArrayInputStream(utfDataAscii);
        dis = new DataInputStream(bais);
        return dis.readUTF();
    }

    @Benchmark
    public String readMixed() throws IOException {
        bais = new ByteArrayInputStream(utfDataMixed);
        dis = new DataInputStream(bais);
        return dis.readUTF();
    }
}

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **内存安全** | 🟢 低 | intrinsic 方法保证边界检查 |
| **兼容性** | 🟢 无影响 | 纯内部实现 |
| **回归风险** | 🟢 低 | 非 ASCII 部分保持原有逻辑 |
| **平台差异** | 🟡 中 | SIMD 依赖 CPU 支持 |

### 平台差异说明

`countPositives` 在不同平台上的实现：
- x86/x64: 使用 SSE/AVX 指令
- ARM: 使用 NEON 指令
- 其他: 回退到标量实现

性能提升在有 SIMD 的平台上更明显。
