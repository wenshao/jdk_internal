## 背景

`StackMapGenerator$Frame::checkAssignableTo` 是验证类型兼容性的核心方法。它在处理跳转目标时被调用，用于检查当前帧是否可以赋值给目标帧。

**为什么重要？**
- 类型验证是 JVM 安全的基础
- 每个跳转目标都需要调用此方法
- 数组复制操作在热路径上

**改动位置**
```
src/java.base/share/classes/jdk/internal/classfile/impl/StackMapGenerator.java
    └── Frame.checkAssignableTo(Frame target)  ← 唯一改动的方法
```

## 问题分析

### 原始实现

```java
// 文件: jdk/internal/classfile/impl/StackMapGenerator.java
// 内部类 Frame

void checkAssignableTo(Frame target) {
    if (target.flags == -1) {  // 目标帧未初始化
        target.locals = locals == null ? null : Arrays.copyOf(locals, localsSize);
        target.localsSize = localsSize;
        target.stack = stack == null ? null : Arrays.copyOf(stack, stackSize);
        target.stackSize = stackSize;
        target.flags = flags;
        target.dirty = true;
    } else {
        // ... 类型检查逻辑
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **不必要的栈复制** | 即使 `stackSize == 0` 也执行 `Array.copyOf` | 浪费内存分配 |
| **Array.copyOf 开销** | `Arrays.copyOf` 内部调用 `System.arraycopy` | 额外的数组创建和复制 |
| **压缩后扩展** | `Array.copyOf` 可能先压缩再扩展 | 冗余操作 |

**问题示例**：
```
场景：方法入口处（栈为空）跳转到另一个位置

原始实现：
1. locals 复制（必要）
2. stack 复制（stackSize = 0，但仍创建新数组）

优化后：
1. locals 复制（必要）
2. stack 跳过复制（stackSize = 0）
```

## 解决方案

### 核心改动

```java
// 优化后：条件复制 + clone

void checkAssignableTo(Frame target) {
    int localsSize = this.localsSize;  // ← 缓存到局部变量
    int stackSize = this.stackSize;    // ← 缓存到局部变量

    if (target.flags == -1) {
        // locals 总是需要复制
        target.locals = locals == null ? null : locals.clone();  // ← 用 clone
        target.localsSize = localsSize;

        // stack 只在非空时复制
        if (stackSize > 0) {  // ← 新增条件
            target.stack = stack.clone();  // ← 用 clone
            target.stackSize = stackSize;
        }
        target.flags = flags;
        target.dirty = true;
    } else {
        // ... 类型检查逻辑
    }
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **条件复制** | 只在 `stackSize > 0` 时复制栈 |
| **clone 替代 Array.copyOf** | `clone()` 直接复制，无需长度计算 |
| **局部变量缓存** | 减少字段访问 |

### clone vs Array.copyOf

```java
// Array.copyOf(int[], int)
public static int[] copyOf(int[] original, int newLength) {
    int[] copy = new int[newLength];  // ← 分配新数组
    System.arraycopy(original, 0, copy, 0, Math.min(original.length, newLength));
    return copy;
}

// Object.clone()
protected native Object clone();  // ← 原生方法，一次性完成复制
```

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| stackSize = 0 时未初始化 | 这是正确行为，空栈不需要数组 |
| clone() 返回 Object | Java 数组协变返回，编译器处理 |
| 浅拷贝问题 | Type 对象是不可变的，浅拷贝安全 |

## 设计决策

### Q1: 为什么用 clone 而不是 Array.copyOf？

> **Reviewer**: Array.copyOf 不是更安全吗？

**A**:
1. **性能**: `clone()` 是原生方法，一次调用完成
2. **准确长度**: `clone()` 复制整个数组，无需长度参数
3. **内存**: `Array.copyOf` 可能分配更大/更小的数组

### Q2: 为什么要检查 stackSize > 0？

> **Reviewer**: 空数组复制开销不大吧？

**A**:
1. **累积效应**: 每个方法的入口都会调用一次
2. **避免分配**: `new int[0]` 仍然分配对象
3. **逻辑清晰**: 空栈不需要复制

### Q3: 为什么不优化 locals？

> **Reviewer**: localsSize 也可能很小，为什么不一起优化？

**A**:
1. **locals 不为空**: 方法总是有参数/this
2. **收益递减**: locals 复制是必要的
3. **保持简单**: 只优化明显的热点
