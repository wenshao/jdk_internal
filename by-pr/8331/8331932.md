## 背景

### 问题起源：JDK-8309622

**PR #14404** 重构了 `BaseLocale` 的缓存机制，但引入了启动性能回归：

```java
// JDK-8309622 的变更 - 使用 lambda 表达式
private static final ReferencedKeySet<BaseLocale> CACHE =
        ReferencedKeySet.create(true, ConcurrentHashMap::new);  // lambda!

return CACHE.intern(new BaseLocale(language, script, region, variant),
    (b) -> new BaseLocale(  // lambda!
        LocaleUtils.toLowerString(b.language).intern(),
        LocaleUtils.toTitleString(b.script).intern(),
        LocaleUtils.toUpperString(b.region).intern(),
        b.variant.intern()));
```

**问题**：这些 lambda 表达式在启动时被触发，导致：

1. **Lambda 基础设施早期初始化**
   - `java.lang.invoke.LambdaMetafactory`
   - `InnerClassLambdaMetafactory`
   - 动态类生成

2. **类加载增加**
   - 取决于系统 Locale，不同路径加载的类数量不同
   - 影响测试和性能比较的一致性

3. **启动延迟**
   - 约 2-3ms 的额外启动时间

## 核心变更

### 设计思路

**将 lambda 表达式替换为匿名类**：

```java
// 变更前 - lambda 表达式
computeIfAbsent(key, Locale::createLocale)  // 方法引用
ConcurrentHashMap::new                       // 构造器引用
(b) -> new BaseLocale(...)                  // lambda
```

```java
// 变更后 - 匿名类实现 Function 接口
private static final Function<Object, Locale> LOCALE_CREATOR = new Function<>() {
    @Override
    public Locale apply(Object key) {
        if (key instanceof BaseLocale base) {
            return new Locale(base, null);
        }
        LocaleKey lk = (LocaleKey)key;
        return new Locale(lk.base, lk.exts);
    }
};
```

### 代码变更

#### 1. Locale.java - LOCALE_CREATOR 匿名类

**变更前**:
```java
private static Locale createLocale(Object key) {
    if (key instanceof BaseLocale base) {
        return new Locale(base, null);
    }
    LocaleKey lk = (LocaleKey)key;
    return new Locale(lk.base, lk.exts);
}

// 使用方法引用
return LOCALE_CACHE.computeIfAbsent(baseloc, Locale::createLocale);
```

**变更后**:
```java
private static final Function<Object, Locale> LOCALE_CREATOR = new Function<>() {
    @Override
    public Locale apply(Object key) {
        if (key instanceof BaseLocale base) {
            return new Locale(base, null);
        }
        LocaleKey lk = (LocaleKey)key;
        return new Locale(lk.base, lk.exts);
    }
};

// 使用匿名类实例
return LOCALE_CACHE.computeIfAbsent(baseloc, LOCALE_CREATOR);
```

#### 2. Locale.java - ConcurrentHashMap Supplier

**变更前**:
```java
private static final ReferencedKeyMap<Object, Locale> LOCALE_CACHE =
        ReferencedKeyMap.create(true, ConcurrentHashMap::new);  // 构造器引用
```

**变更后**:
```java
private static final ReferencedKeyMap<Object, Locale> LOCALE_CACHE =
        ReferencedKeyMap.create(true, ReferencedKeyMap.concurrentHashMapSupplier());
```

新增辅助方法：
```java
// ReferencedKeyMap.java
public static <K, V> Supplier<Map<ReferenceKey<K>, V>> concurrentHashMapSupplier() {
    return new Supplier<>() {
        @Override
        public Map<ReferenceKey<K>, V> get() {
            return new ConcurrentHashMap<>();
        }
    };
}
```

#### 3. BaseLocale.java - INTERNER 匿名类

**变更前**:
```java
private static final ReferencedKeySet<BaseLocale> CACHE =
        ReferencedKeySet.create(true, ConcurrentHashMap::new);  // lambda

return CACHE.intern(new BaseLocale(language, script, region, variant),
    (b) -> new BaseLocale(  // lambda
        LocaleUtils.toLowerString(b.language).intern(),
        LocaleUtils.toTitleString(b.script).intern(),
        LocaleUtils.toUpperString(b.region).intern(),
        b.variant.intern()));
```

**变更后**:
```java
private static final ReferencedKeySet<BaseLocale> CACHE =
        ReferencedKeySet.create(true, ReferencedKeySet.concurrentHashMapSupplier());

public static final UnaryOperator<BaseLocale> INTERNER = new UnaryOperator<>() {
    @Override
    public BaseLocale apply(BaseLocale b) {
        return new BaseLocale(
                LocaleUtils.toLowerString(b.language).intern(),
                LocaleUtils.toTitleString(b.script).intern(),
                LocaleUtils.toUpperString(b.region).intern(),
                b.variant.intern());
    }
};

return CACHE.intern(new BaseLocale(language, script, region, variant), INTERNER);
```

## 技术分析

### 为什么匿名类能避免 Lambda 初始化？

| 特性 | Lambda 表达式 | 匿名类 |
|------|--------------|--------|
| **编译时** | 转换为 `invokedynamic` | 生成独立 .class 文件 |
| **运行时** | `LambdaMetafactory` 动态生成类 | 直接加载类 |
| **首次调用** | 触发 bootstrap | 直接实例化 |
| **类生成** | 是 | 否（已编译） |

### Lambda 启动开销分解

```
Lambda 表达式首次调用:
1. invokedynamic 指令
2. 调用 LambdaMetafactory.metafactory
3. 生成动态类 (InnerClassLambdaMetafactory)
4. 链接动态类
5. 调用动态类实例

匿名类首次调用:
1. new 指令
2. 直接实例化
```

### 类加载影响

**变更前** (Lambda):
```
启动路径涉及 Locale:
  → 触发 BaseLocale 缓存初始化
    → 触发 lambda 表达式
      → 加载 LambdaMetafactory
        → 加载 InnerClassLambdaMetafactory
          → 生成动态类
```

**变更后** (匿名类):
```
启动路径涉及 Locale:
  → 触发 BaseLocale 缓存初始化
    → 实例化预定义的 INTERNER 匿名类
    → 无额外类加载
```

## 设计决策

### Q1: 为什么不直接缓存对象？

> **Reviewer**: 为什么需要这么复杂的缓存机制？

**A**: `BaseLocale` 和 `Locale` 需要规范化和缓存：

1. **规范化**: 语言/国家代码需要大小写转换
2. **缓存**: 避免重复创建相同的 Locale 对象
3. **弱引用**: 允许 GC 回收不常用的 Locale

### Q2: 为什么用匿名类而不是方法引用？

> **Reviewer**: 为什么不能用 `Locale::createLocale` 方法引用？

**A**: 方法引用也是 lambda 的一种形式，会触发 `LambdaMetafactory`。

```java
// 这仍然是 lambda，会触发 bootstrap
computeIfAbsent(key, Locale::createLocale)

// 必须用匿名类避免 bootstrap
computeIfAbsent(key, LOCALE_CREATOR)  // LOCALE_CREATOR 是匿名类
```

### Q3: 为什么会有后续修复 (JDK-8338897)？

> **Reviewer**: 不是已经修复了吗？

**A**: JDK-8331932 修复了主要问题，但仍有少量残留：

1. **部分缓存仍然早期初始化**
2. **可以进一步延迟初始化**

JDK-8338897 做了进一步的优化。

## 经验教训

### 1. 启动路径的 Lambda 需谨慎

**教训**: 启动时频繁调用的路径应避免使用 lambda 表达式

```java
// ❌ 避免在启动关键路径使用 lambda
private static final Supplier<Foo> FOO = () -> new Foo();

// ✅ 使用匿名类或静态工厂
private static final Supplier<Foo> FOO = new Supplier<>() {
    public Foo get() { return new Foo(); }
};
```

### 2. Lambda 有隐藏的初始化成本

**教训**: Lambda 的 `invokedynamic` 机制在首次调用时有额外开销

| 阶段 | Lambda | 匿名类 |
|------|--------|--------|
| **类加载** | 延迟 (首次调用) | 立即 (类加载时) |
| **类生成** | 动态生成 | 已编译 |
| **首次调用** | 慢 (bootstrap) | 快 (直接实例化) |

### 3. 不同 Locale 导致不同性能

**教训**: Locale 相关代码应确保所有代码路径性能一致

```java
// 问题: 不同 Locale 可能触发不同的代码路径
if (locale.equals("en")) {
    // 路径 A
} else {
    // 路径 B - 可能加载不同的类
}
```

## 总结

JDK-8331932 是一个重要的启动性能修复，解决了 JDK-8309622 引入的 2-3ms 启动回归：

1. **问题**: Lambda 表达式在启动时触发基础设施初始化
2. **解决方案**: 将 lambda 替换为匿名类
3. **效果**: 减少 2-4 个类加载，节省 2-3ms 启动时间
4. **权衡**: 代码稍显冗长，但避免启动开销

这个修复展示了在启动关键路径谨慎使用 lambda 的重要性，以及匿名类在延迟初始化方面的优势。
