## 背景

### JEP 280 String Concatenation 历史回顾

Java 9 引入了 JEP 280，使用 invokedynamic 实现字符串拼接。在此之前，该 PR 存在两种策略：

1. **MH InlineCopy 策略**：使用 MethodHandle 组合子构建表达式树
   - 优点：吞吐量好
   - 缺点：高参数数量时扩展性差，MethodHandle 树复杂

2. **SimpleStringBuilder 策略**：为每个调用点生成使用 StringBuilder 的类
   - 优点：扩展性好
   - 缺点：生成大量类，启动开销大

### 为什么需要新策略？

```
问题场景分析：
┌─────────────────────────────────────────────────────────┐
│ 传统 SimpleStringBuilder 策略                           │
├─────────────────────────────────────────────────────────┤
│ 每个调用点 → 生成一个独立的类                            │
│                                                          │
│ "Hello" + x + "World"  → StringConcat$1                 │
│ "Hello" + y + "World"  → StringConcat$2                 │
│ "Hello" + z + "World"  → StringConcat$3                 │
│                                                          │
│ 结果：1000个调用点 = 1000个类！启动时类加载开销巨大      │
└─────────────────────────────────────────────────────────┘

新 InlineHiddenClassStrategy 策略
┌─────────────────────────────────────────────────────────┐
│ 按形状生成类 (相同的参数类型共享一个类)                   │
│                                                          │
│ "Hello" + int + "World" → Concat1_Int                   │
│ "Hello" + long + "World" → Concat1_Long                 │
│                                                          │
│ 结果：1000个调用点 ≈ 20个类！大幅减少类加载              │
└─────────────────────────────────────────────────────────┘
```

## 问题分析

### 原始实现：SimpleStringBuilder 策略

```java
// StringConcatFactory.java (简化)
class SimpleStringBuilderStrategy {
    static MethodHandle generate(Lookup lookup, MethodType concatType, String[] constants) {
        // 为每个调用点生成一个类
        byte[] bytes = generateClass(concatType, constants);

        // 使用 lookup.defineHiddenClass 创建隐藏类
        Class<?> innerClass = lookup.defineHiddenClass(bytes, true).lookupClass();

        // 返回调用方法的 MethodHandle
        return MethodHandles.lookup().findStatic(innerClass, "concat", concatType);
    }

    // 生成的类类似：
    // class StringConcat$1 {
    //     static String concat(String arg0, int arg1) {
    //         StringBuilder sb = new StringBuilder();
    //         sb.append("prefix");
    //         sb.append(arg0);
    //         sb.append(arg1);
    //         sb.append("suffix");
    //         return sb.toString();
    //     }
    // }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **类加载开销** | 每个调用点生成独立类 | 启动时加载大量类 |
| **元数据占用** | 大量类占用元空间 | 内存压力大 |
| **JIT 编译开销** | 需要编译大量类 | 温热身时间长 |
| **方法调用** | 多次 StringBuilder.append 虚方法调用 | 吞吐量受限 |

### 为什么 MH InlineCopy 不够？

```java
// MH InlineCopy 构建的 MethodHandle 树
MethodHandle mh = MethodHandles.filterReturnValue(
    MethodHandles.filterReturnValue(
        MethodHandles.filterReturnValue(
            insertArguments(constantAppend, 0, constants[0]),

        appendArg1

// ... 更多嵌套
```

问题：
- 参数多时，MethodHandle 树深度呈线性增长
- JIT 优化复杂度高
- 内存占用大

### 1. StringConcatHelper 基础设施

#### StringConcatBase 抽象基类

```java
// StringConcatHelper.java
static abstract class StringConcatBase {
    @Stable
    final String[] constants;  // 常量字符串数组
    final int      length;     // 预计算的总长度
    final byte     coder;      // 预计算的 coder (LATIN1/UTF16)

    StringConcatBase(String[] constants) {
        int  length = 0;
        byte coder  = String.LATIN1;
        for (String c : constants) {
            length += c.length();
            coder  |= c.coder();
        }
        this.constants = constants;
        this.length    = length;
        this.coder     = coder;
    }
}
```

#### Concat1 单参数优化类

```java
static final class Concat1 extends StringConcatBase {
    Concat1(String[] constants) {
        super(constants);
    }

    @ForceInline
    String concat0(String value) {
        // 计算最终长度和 coder
        int length = stringSize(this.length, value);
        byte coder = (byte) (this.coder | value.coder());

        // 分配字节数组
        byte[] buf = newArray(length << coder);

        // 直接复制到目标位置
        String prefix = constants[0];
        prefix.getBytes(buf, 0, coder);
        value.getBytes(buf, prefix.length(), coder);
        constants[1].getBytes(buf, prefix.length() + value.length(), coder);

        return new String(buf, coder);
    }

    @ForceInline
    String concat(int value) {
        // 对于原始类型，使用 prepend 优化
        int length = stringSize(this.length, value);
        String suffix = constants[1];
        length -= suffix.length();
        byte[] buf = newArrayWithSuffix(suffix, length, coder);
        prepend(length, coder, buf, value, constants[0]);
        return new String(buf, coder);
    }

    // ... 其他重载: boolean, char, long, Object, float, double
}
```

### 2. InlineHiddenClassStrategy 策略

#### 策略选择流程

```java
// StringConcatFactory.java
public static CallSite makeConcatWithConstants(...) {
    // ...

    try {
        // 1. 首先尝试简单拼接 (针对简单情况)
        MethodHandle mh = makeSimpleConcat(concatType, constantStrings);

        // 2. 如果简单策略失败且参数数量少，尝试 MH InlineCopy
        if (mh == null && concatType.parameterCount() <= HIGH_ARITY_THRESHOLD) {
            mh = generateMHInlineCopy(concatType, constantStrings);
        }

        // 3. 如果都失败，使用新的隐藏类策略
        if (mh == null) {
            mh = InlineHiddenClassStrategy.generate(lookup, concatType, constantStrings);
        }

        mh = mh.viewAsType(concatType, true);
        return new ConstantCallSite(mh);
    }
}
```

#### 生成的类结构

对于 `"prefix" + int + long + boolean + String`，生成：

```java
class String$$StringConcat extends StringConcatHelper.StringConcatBase {
    // 继承字段: String[] constants, int length, byte coder

    String$$StringConcat(String[] constants) {
        super(constants);
    }

    String concat(int arg0, long arg1, boolean arg2, String arg3) {
        // 1. 转换非 String 参数
        String str3 = stringOf(arg3);

        // 2. 计算 coder 和 length
        int coder  = coder(this.coder, arg3);
        int len    = length(this.length, arg0, arg1, arg2, str3);

        // 3. 分配并填充缓冲区
        String[] constants = this.constants;
        byte[] buf = newArrayWithSuffix(constants[4], len, coder);
        prepend(len, coder, buf, constants, arg0, arg1, arg2, str3);

        return new String(buf, coder);
    }

    static int length(int length, int arg0, long arg1, boolean arg2, String arg3) {
        return stringSize(stringSize(stringSize(stringSize(
            length, arg0), arg1), arg2), arg3);
    }

    static byte coder(byte coder, String arg3) {
        return (byte)(coder | arg3.coder());
    }

    static void prepend(int length, byte coder, byte[] buf,
                        String[] constants, int arg0, long arg1,
                        boolean arg2, String arg3) {
        // 从后向前填充 (prepend)
        int pos = prepend(length, coder, buf, arg3, constants[3]);
        pos = prepend(pos, coder, buf, arg2, constants[2]);
        pos = prepend(pos, coder, buf, arg1, constants[1]);
        prepend(pos, coder, buf, arg0, constants[0]);
    }
}
```

### 3. 关键辅助方法

#### stringSize - 长度计算

```java
static int stringSize(int length, int value) {
    return checkOverflow(length + DecimalDigits.stringSize(value));
}

static int stringSize(int length, boolean value) {
    return checkOverflow(length + (value ? 4 : 5));  // "true" 或 "false"
}

static int stringSize(int length, String value) {
    return checkOverflow(length + value.length());
}

@ForceInline
static int checkOverflow(int value) {
    if (value >= 0) {
        return value;
    }
    throw new OutOfMemoryError("Overflow: String length out of range");
}
```

#### prepend - 反向填充

```java
static int prepend(int index, byte coder, byte[] buf, int value, String prefix) {
    if (coder == String.LATIN1) {
        // 先写 value
        index = StringLatin1.getChars(value, index, buf);
        // 再写 prefix
        index -= prefix.length();
        prefix.getBytes(buf, index, String.LATIN1);
    } else {
        // UTF16 路径
        index = StringUTF16.getChars(value, index, buf);
        index -= prefix.length();
        prefix.getBytes(buf, index, String.UTF16);
    }
    return index;
}
```

## 设计决策

### Q1: 为什么从后向前 (prepend) 而不是从前向后 (append)？

> **Reviewer**: 使用 prepend 看起来反直觉，为什么这样设计？

**A**: 从后向前填充有两个关键优势：

1. **减少计算**：对于原始类型，我们先计算其字符串长度，然后从最终位置开始填充，避免多次计算位置
2. **单次分配**：可以预先知道最终长度，一次性分配正确大小的数组

```
append 方式 (传统):
┌──────────────────────────────────────────┐
│ [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] │
│  ↑                                        │
│ 需要跟踪当前位置，每次更新               │
└──────────────────────────────────────────┘

prepend 方式 (新实现):
┌──────────────────────────────────────────┐
│ [p] [r] [i] [n] [t] [s] [u] [f] [i] [x] │
│                                  ↑        │
│ 从最终位置开始，每次减去已写入长度      │
└──────────────────────────────────────────┘
```

### Q2: 为什么需要 @Stable 注解？

> **Reviewer**: StringConcatBase.constants 上的 @Stable 有什么作用？

**A**: @Stable 告诉 JIT 编译器：

1. 数组引用本身不会改变（final 已保证）
2. 数组元素在构造后不会改变
3. JIT 可以激进优化，如常量折叠和常量传播

```java
@Stable
final String[] constants;  // JIT 知道 constants[0] 是常量
```

这允许 JIT 将 `constants[0].getBytes(buf, 0, coder)` 优化为直接操作。

### Q3: 为什么使用隐藏类而不是普通类？

> **Reviewer**: 隐藏类相比普通类有什么优势？

**A**:

| 特性 | 普通类 | 隐藏类 |
|------|--------|--------|
| 类加载器可见性 | 可见 | 不可见 |
| 反射访问 | 可访问 | 不可访问 |
| 卸载时机 | 类加载器卸载时 | 无引用时立即卸载 |
| 堆外存储 | 元空间 | 可使用堆外内存 |

隐藏类更适合动态生成的辅助类，因为：
- 不污染类命名空间
- 可以更快被 GC 回收
- 更好的内存管理

### Q4: 什么情况下会回退到 MH InlineCopy？

> **Reviewer**: HIGH_ARITY_THRESHOLD 控制的是什么？

**A**: 默认情况下 `HIGH_ARITY_THRESHOLD = 0`，意味着几乎总是使用隐藏类策略。

可以通过 `-Djava.lang.invoke.StringConcat.highArityThreshold=N` 调整：
- N > 0: 参数数量 <= N 时优先尝试 MH InlineCopy
- N = 0: 直接使用隐藏类策略

这个阈值主要是为了保留原有路径，以便在出现问题时可以回退。

## 性能对比

### 启动性能

启动性能的改善来自：

1. **减少类加载**：从每个调用点一个类 → 每种参数类型一个类
2. **按形状缓存**：相同参数类型的拼接共享同一个类
3. **隐藏类卸载**：不再使用的拼接类可以更快被 GC

### 温热身性能

温热身的改善来自：

1. **简单的字节码**：生成的类结构简单，JIT 容易优化
2. **@ForceInline**：小方法标记为强制内联
3. **静态调用**：对 StringConcatHelper 的调用是静态的，易于内联

### 吞吐量性能

吞吐量保持竞争力：

1. **单次分配**：只分配一次字节数组
2. **原始类型优化**：避免为 int/long/boolean 创建临时 String
3. **Coder 感知**：LATIN1 vs UTF16 高效处理
