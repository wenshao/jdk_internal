## 背景

`StringConcatHelper.simpleConcat` 是 Java 字符串拼接的核心方法，用于处理两个对象的字符串连接。

**调用路径**
```
代码: "hello" + "world"
  ↓
javac 编译: invokedynamic simpleConcat
  ↓
StringConcatHelper.simpleConcat(first, second)
  ↓
生成最终字符串
```

**原始实现问题**
- 使用复杂的 `mix` 和 `prepend` 机制
- `mix`：将长度和 coder 打包到一个 long 中
- `prepend`：反向填充字节数组
- 对于简单的两参数拼接过于复杂

## 问题分析

### 原始实现

```java
// 文件: java/lang/StringConcatHelper.java
// 行数: ~321-340 (原始版本)

@ForceInline
static String simpleConcat(Object first, Object second) {
    String s1 = stringOf(first);
    String s2 = stringOf(second);
    if (s1.isEmpty()) {
        return new String(s2);
    }
    if (s2.isEmpty()) {
        return new String(s1);
    }

    // 复杂的 mix/prepend 路径
    long indexCoder = mix(initialCoder(), s1);
    indexCoder = mix(indexCoder, s2);
    byte[] buf = newArray(indexCoder);
    indexCoder = prepend(indexCoder, buf, s2, s1);
    return newString(buf, indexCoder);
}

// mix: 将长度和 coder 打包到 long
static long mix(long lengthCoder, String s) {
    lengthCoder += s.length();
    return lengthCoder | s.coder();
}

// prepend: 反向填充（为多参数设计）
static long prepend(long indexCoder, byte[] buf, String value, String prefix) {
    // ... 复杂的反向填充逻辑
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **long 打包/解包** | mix 将长度和 coder 打包，prepend 需要解包 | 额外的位运算 |
| **反向填充** | prepend 设计为支持多参数反向遍历 | 两参数场景不需要 |
| **方法调用链** | simpleConcat → mix → mix → newArray → prepend | 调用深度 |
| **分支预测** | prepend 中有多个条件分支 | CPU 流水线停顿 |

### 为什么原始实现复杂？

```java
// 原始设计支持 N 个参数的通用拼接：
// arg1 + arg2 + arg3 + ... + argN

// 填充顺序（反向）：
// [ argN | ... | arg3 | arg2 | arg1 ]
//   ↑
//   从后向前填充

// 对于两参数：
// [ arg2 | arg1 ]
//   ↑
//   反向填充完全没必要
```

## 解决方案

### 核心改动

```java
// 变更后: 新增 doConcat 方法，使用直接正向复制

@ForceInline
static String simpleConcat(Object first, Object second) {
    String s1 = stringOf(first);
    String s2 = stringOf(second);
    if (s1.isEmpty()) {
        return new String(s2);
    }
    if (s2.isEmpty()) {
        return new String(s1);
    }
    return doConcat(s1, s2);  // 新方法
}

// 新的简化实现
@ForceInline
static String doConcat(String s1, String s2) {
    byte coder = (byte) (s1.coder() | s2.coder());
    int newLength = (s1.length() + s2.length()) << coder;
    byte[] buf = newArray(newLength);
    s1.getBytes(buf, 0, coder);       // 正向复制
    s2.getBytes(buf, s1.length(), coder);
    return new String(buf, coder);
}
```

### String.concat 也切换到新方法

```java
// 文件: java/lang/String.java
// 行数: ~2987-2995

public String concat(String str) {
    if (str.isEmpty()) {
        return this;
    }
-   return StringConcatHelper.simpleConcat(this, str);
+   return StringConcatHelper.doConcat(this, str);  // 直接调用简化版本
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除 long 打包** | 直接使用 byte 和 int，无需位运算 |
| **正向填充** | 按自然顺序复制，更符合 CPU 缓存预取 |
| **减少方法调用** | doConcat 直接到 getBytes，跳过中间层 |
| **JIT 友好** | 简单逻辑更容易内联和优化 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 边界检查 | getBytes 内部已处理 |
| coder 计算 | OR 运算正确合并两个 coder |
| 空字符串 | simpleConcat 入口已处理 |

## 设计决策

### Q1: 为什么保留 simpleConcat 入口？

> **Reviewer**: 为什么不直接修改 simpleConcat 的实现？

**A**:
1. `simpleConcat` 处理空字符串的特殊情况
2. `doConcat` 是两个非空字符串的纯拼接逻辑
3. 职责分离：入口处理边界，核心处理拼接

### Q2: 为什么 String.concat 也切换到 doConcat？

> **Reviewer**: String.concat 原本调用 simpleConcat，为什么改了？

**A**:
1. `String.concat` 的参数已经是非空 String
2. 跳过空字符串检查，直接进入核心逻辑
3. 减少一次方法调用

### Q3: getBytes 性能如何？

> **Reviewer**: getBytes 会不会有性能问题？

**A**:
1. `getBytes(byte[] buf, int offset, byte coder)` 是内部方法
2. 直接内存复制，使用 `System.arraycopy` 或 `Unsafe`
3. coder 决定复制策略（LATIN1 vs UTF16）
