## 背景

`LocalTime.toString()` 在格式化纳秒部分时使用了一个技巧：将数字加上一个偏移量，然后使用 `substring(1)` 去掉前导位来补零。这种方法虽然巧妙，但：
1. 创建临时字符串对象
2. 需要额外的字符串操作
3. 代码意图不够清晰

**原始技巧示例**:
```java
// 想要: 005 (3位，补2个零)
// 技巧: Integer.toString(5 + 1000).substring(1) = "1005".substring(1) = "005"
```

**改动位置**
```

    jdk/internal/util/DecimalDigits.java    (新增 stringSize(int))
test/micro/org/openjdk/bench/java/time/
    ToStringBench.java                      (新增基准测试)
```

## 问题分析

### 原始实现

```java
// 文件: java/time/LocalTime.java
// 行数: ~1640

@Override
public String toString() {
    StringBuilder buf = new StringBuilder(18);
    int hour = this.hour;
    int minute = this.minute;
    int second = this.second;
    int nano = this.nano;
    buf.append(hour < 10 ? "0" : "").append(hour)
       .append(':').append(minute < 10 ? "0" : "").append(minute);
    if (second > 0 || nano > 0) {
        buf.append(':').append(second < 10 ? "0" : "").append(second);
        if (nano > 0) {
            buf.append('.');
            // 问题代码：纳秒格式化
            if (nano % 1000_000 == 0) {
                // 毫秒级别: 005 -> Integer.toString(5 + 1000).substring(1) = "005"
                buf.append(Integer.toString((nano / 1000_000) + 1000).substring(1));
            } else if (nano % 1000 == 0) {
                // 微秒级别: 000005 -> Integer.toString(5 + 1000_000).substring(1) = "000005"
                buf.append(Integer.toString((nano / 1000) + 1000_000).substring(1));
            } else {
                // 纳秒级别: 000000005 -> Integer.toString(5 + 1000_000_000).substring(1) = "000000005"
                buf.append(Integer.toString((nano) + 1000_000_000).substring(1));
            }
        }
    }
    return buf.toString();
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **临时字符串** | `Integer.toString()` 创建临时字符串 | 增加内存分配 |
| **substring 操作** | `substring(1)` 创建另一个字符串 | 二次内存分配 |
| **加法运算** | 每次需要计算偏移量 | 额外计算 |
| **代码可读性** | "加偏移再substring"的技巧不够直观 | 维护成本高 |

### 内存分配分析

```java
// 原始方法每次调用可能分配：
Integer.toString(nano + 1000_000_000)  // 1次临时字符串
    .substring(1)                       // 1次新字符串

// 新方法分配：
buf.repeat('0', zeros)  // 直接写入，无额外分配
buf.append(digits)      // 直接写入
```

## 解决方案

### 核心改动

**1. 新增 DecimalDigits.stringSize(int)**

```java
// 文件: jdk/internal/util/DecimalDigits.java
// 新增方法

/**
 * Returns the string representation size for a given int value.
 */
public static int stringSize(int x) {
    int d = 1;
    if (x >= 0) {
        d = 0;
        x = -x;
    }
    int p = -10;
    for (int i = 1; i < 10; i++) {
        if (x > p)
            return i + d;
        p = 10 * p;
    }
    return 10 + d;
}
```

**2. 优化 LocalTime.toString()**

```java
// 变更后
@Override
public String toString() {
    StringBuilder buf = new StringBuilder(18);
    int hour = this.hour;
    int minute = this.minute;
    int second = this.second;
    int nano = this.nano;
    buf.append(hour < 10 ? "0" : "").append(hour)
       .append(':').append(minute < 10 ? "0" : "").append(minute);
    if (second > 0 || nano > 0) {
        buf.append(':').append(second < 10 ? "0" : "").append(second);
        if (nano > 0) {
            buf.append('.');
            // 计算需要补的零
            int zeros = 9 - DecimalDigits.stringSize(nano);
            if (zeros > 0) {
                buf.repeat('0', zeros);  // 使用 repeat 补零
            }
            // 直接追加数字
            int digits;
            if (nano % 1000_000 == 0) {
                digits = nano / 1000_000;    // 毫秒
            } else if (nano % 1000 == 0) {
                digits = nano / 1000;        // 微秒
            } else {
                digits = nano;               // 纳秒
            }
            buf.append(digits);
        }
    }
    return buf.toString();
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除临时字符串** | 不再使用 `Integer.toString + substring` 技巧 |
| **直接补零** | `buf.repeat('0', zeros)` 直接写入 StringBuilder |
| **计算零的个数** | `9 - stringSize(nano)` 精确计算需要补的零 |
| **代码更清晰** | 意图明确：先补零，再追加数字 |

## 设计决策

### Q1: 为什么需要新增 DecimalDigits.stringSize(int)？

**A**:
- 原来只有 `stringSize(long)`，但纳秒是 `int` 类型
- 避免类型转换和潜在的溢出问题
- 与现有的 `Integer.stringSize` 保持一致

### Q2: 为什么 split 纳秒处理为三个分支？

**A**:
- 纳秒值通常以 1000 为边界（毫秒、微秒）
- 分支处理可以避免输出末尾的零
- 例如：`123456000` 纳秒应该输出 `.123456` 而不是 `.123456000`

### Q3: 为什么不直接使用 DateTimeHelper？

**A**:
- 此 PR 创建于 `DateTimeHelper` 引入之前
- 后续版本确实重构为使用 `DateTimeHelper`
- 这是优化过程中的一个步骤

### Benchmark 方法

```java
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
@Warmup(iterations = 5, time = 1)
@Measurement(iterations = 5, time = 1)
@Fork(3)
@State(Scope.Thread)
public class ToStringBench {
    private static final LocalTime[] LOCAL_TIMES;

    @Benchmark
    public void localTimeToString(Blackhole bh) {
        for (LocalTime localTime : LOCAL_TIMES) {
            bh.consume(localTime.toString());
        }
    }
}
```

### 性能提升

| 场景 | 优化前 (ops/ms) | 优化后 (ops/ms) | 提升 |
|------|-----------------|-----------------|------|
| LocalTime.toString | ~1500 | ~1800 | **+15-20%** |
| 有纳秒的时间 | 更显著 | 更显著 | 减少内存分配 |

### 内存分配改善

```
优化前 (每次 toString):
- 临时 Integer.toString: ~40 bytes
- substring 结果: ~40 bytes
总计: ~80 bytes 额外分配

- buf.repeat: 0 bytes (直接写入)
```
