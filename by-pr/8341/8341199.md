## 背景

ClassFile API 在 Java 22+ 引入了 `loadConstant(int)` 方法，专门用于加载 int 常量。相比通用的 `ldc` 方法，`loadConstant(int)` 有更短的调用路径。

**为什么重要？**
- `ldc` 是通用的常量加载方法，需要处理多种类型
- `loadConstant(int)` 是专门针对 int 的优化方法
- 在字节码生成中，int 常量加载是最常见的操作之一

**改动位置**
```
src/java.base/share/classes/java/lang/invoke/StringConcatFactory.java
src/java.base/share/classes/java/lang/runtime/SwitchBootstraps.java
src/jdk.jfr/share/classes/jdk/jfr/internal/EventClassBuilder.java
src/jdk.jfr/share/classes/jdk/jfr/internal/EventInstrumentation.java
```

## 问题分析

### 原始实现

```java
// CodeBuilder 接口

// 通用方法 - 调用路径较长
CodeBuilder ldc(int value) {
    // 需要判断值范围，决定使用 ldc、ldc_w 或 iconst
    if (value >= -1 && value <= 5) {
        return withConstant(ConstantInstruction.ofInstruction(Opcode.ICONST_0 + value));
    } else if (value >= Byte.MIN_VALUE && value <= Byte.MAX_VALUE) {
        return withConstant(loadableEntry(IntegerEntry.of(value)));
    } else if (value >= Short.MIN_VALUE && value <= Short.MAX_VALUE) {
        return withConstant(loadableEntry(IntegerEntry.of(value)));
    } else {
        return withConstant(loadableEntry(IntegerEntry.of(value)));
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **多级类型判断** | `ldc` 需要判断常量类型和值范围 | 增加分支预测失败 |
| **常量池操作** | 可能需要创建常量池条目 | 额外的内存分配 |
| **方法调用链** | 通过 `withConstant` -> `ConstantInstruction` | 增加调用开销 |

### 核心改动

```java

// 优化后
codeBuilder.loadConstant(index);

// 4. JFR EventInstrumentation.java
// 优化前
codeBuilder.ldc(index);

// 优化后
codeBuilder.loadConstant(index);
```

### loadConstant(int) 的优势

```java
// DirectCodeBuilder 实现
@Override
public CodeBuilder loadConstant(int value) {
    // 直接生成最优指令，无需常量池操作
    if (value >= -1 && value <= 5) {
        writeBytecode(ICONST_0 + value);  // iconst_<i>
    } else if (value >= Byte.MIN_VALUE && value <= Byte.MAX_VALUE) {
        writeBytecode(BIPUSH);
        writeBytecode(value);  // bipush
    } else if (value >= Short.MIN_VALUE && value <= Short.MAX_VALUE) {
        writeBytecode(SIPUSH);
        writeU2(value);  // sipush
    } else {
        writeIndex(LDC, cp.intEntry(value));  // ldc
    }
    return this;
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **类型特化** | 方法签名明确是 int，无需类型判断 |
| **直接指令生成** | 对于小常量直接生成 iconst/bipush/sipush |
| **减少常量池压力** | 只在必要时才使用常量池 |
| **更好的内联** | 小方法更容易被 JIT 内联 |

## 设计决策

### Q1: 为什么不修改 ldc 方法本身？

> **Reviewer**: 为什么不优化 `ldc` 方法而要新增一个方法？

**A**:
1. **向后兼容性**: `ldc` 方法的行为不能改变，已有代码依赖
2. **API 清晰度**: `loadConstant(int)` 明确表示加载 int 常量
3. **性能差异**: `ldc` 需要处理多种类型，无法针对 int 优化

### Q2: 为什么只替换这些位置？

> **Reviewer**: 还有其他地方使用 ldc，为什么只改这些？

**A**:
1. 这些是热点路径（StringConcatFactory、JFR）
2. 遵循"渐进式优化"原则
3. 后续可以继续优化其他位置

### Q3: 性能提升主要来自哪里？

> **Reviewer**: 是减少了常量池操作吗？

**A**:
主要是调用路径缩短：
- `ldc` 需要通过 `ConstantInstruction.ofInstruction` 创建中间对象
- `loadConstant(int)` 直接生成字节码
- 对于小常量（-1 到 5），直接生成 iconst 指令，无需常量池

### Benchmark 方法

```java
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
public class LoadConstantBenchmark {

    @Benchmark
    public byte[] generateWithLdc() {
        return ClassFile.of().build(ClassDesc.of("Test"), cb -> {
            cb.withMethod("test", MethodTypeDesc.of("(I)I"),
                ClassFile.ACC_PUBLIC | ClassFile.ACC_STATIC, mb -> {
                    var code = mb.codeBuilder();
                    for (int i = 0; i < 100; i++) {
                        code.ldc(i);
                    }
                    code.ireturn();
                });
        });
    }

    @Benchmark
    public byte[] generateWithLoadConstant() {
        return ClassFile.of().build(ClassDesc.of("Test"), cb -> {

                ClassFile.ACC_PUBLIC | ClassFile.ACC_STATIC, mb -> {
                    var code = mb.codeBuilder();
                    for (int i = 0; i < 100; i++) {
                        code.loadConstant(i);
                    }
                    code.ireturn();
                });
        });
    }
}
```
