## 背景

### ClassFile API 使用规范

JDK 内部也在使用 ClassFile API 生成字节码（如 Lambda 实现、动态代理等）。这些代码应该遵循最佳实践，为外部用户提供示范。

**为什么重要？**
- JDK 内部代码是 ClassFile API 的参考实现
- 一致的编码风格提高可维护性
- 更简洁的代码更容易理解和审计

### 改动位置

```
src/java.base/share/classes/java/lang/invoke/
├── InnerClassLambdaMetafactory.java  ← Lambda 生成
├── AbstractValidatingLambdaMetafactory.java
└── StringConcatFactory.java          ← 字符串拼接
```

```java
builder.loadConstant(null);

// 优化后
builder.aconst_null();  // 更符合字节码语义
```

```java
// 问题 3: 没有利用流式 API

// 优化前

// 优化后
```

## 解决方案

### 核心改动

#### 1. 使用专用类型指令

```java
// 替换 loadLocal/storeLocal 为专用指令

// 优化前
builder.loadLocal(TypeKind.REFERENCE);
builder.storeLocal(TypeKind.REFERENCE);
builder.loadLocal(TypeKind.LONG);
builder.storeLocal(TypeKind.LONG);

// 优化后
builder.aload();           // 加载引用
builder.astore();          // 存储引用
builder.lload();           // 加载 long
builder.lstore();          // 存储 long
```

**好处**：
- 更简洁（7 字符 vs 17 字符）
- 直接对应字节码指令
- 编译期类型检查

#### 2. 使用 aconst_null

```java
// 优化前
builder.loadConstant(null);

// 优化后
builder.aconst_null();
```

**好处**：
- 更符合 JVM 字节码语义
- aconst_null 是单字节指令，更高效
- 意图更明确

#### 3. 流式编码风格

```java
// 优化前
ClassFile.build(constantPool, classBuilder -> {
    classBuilder.withMethod("toString", MTD_String, ACC_PUBLIC, methodBuilder -> {
        // 方法体
    });
    classBuilder.withMethod("hashCode", MTD_int, ACC_PUBLIC, methodBuilder -> {
        // 方法体
    });
});

// 优化后
ClassFile.build(constantPool, classBuilder -> {
    classBuilder
        .withMethod("toString", MTD_String, ACC_PUBLIC, mb -> {
            // 方法体
        })
        .withMethod("hashCode", MTD_int, ACC_PUBLIC, mb -> {
            // 方法体
        });
});
```

#### 4. 简化方法参数传递

```java
// 优化前
Util.fieldTypeSymbol(entry.nameAndType());

// 优化后
Util.fieldTypeSymbol(entry.type());  // 直接传递 type
```

### 完整对比示例

```java
// 优化前: InnerClassLambdaMetafactory 中的代码
var cb = mb.withCode(codeBuilder -> {
    codeBuilder.loadLocal(0, TypeKind.REFERENCE)
        .loadConstant(null)
        .loadLocal(1, TypeKind.REFERENCE)
        .invokevirtual(className, "equals", "(Ljava/lang/Object;)Z")
        .ifThen(ifBuilder -> {
            ifBuilder.loadConstant("arg$1")
                .areturn();
        });
    codeBuilder.loadLocal(1, TypeKind.REFERENCE)
        .areturn();
});

// 优化后
var cb = mb.withCode(codeBuilder -> {
    codeBuilder.aload(0)
        .aconst_null()
        .aload(1)
        .invokevirtual(className, "equals", "(Ljava/lang/Object;)Z")
        .ifThen(ifBuilder -> {
            ifBuilder.ldc("arg$1")
                .areturn();
        });
    codeBuilder.aload(1)
        .areturn();
});
```

## 设计决策

### Q1: 为什么 aconst_null 比 loadConstant(null) 好？

> **Reviewer**: 两者功能相同，为什么要改？

**A**:
1. **语义清晰**：aconst_null 直接对应 JVM aconst_null 指令
2. **性能**：aconst_null 是单字节指令 (0x01)，更高效
3. **一致性**：与字节码规范保持一致

### Q2: 专用指令是否降低类型安全性？

> **Reviewer**: aload() 不如 loadLocal(REFERENCE) 明确类型？

**A**:
1. **编译期检查**：ClassFile API 在编译期验证类型匹配
2. **上下文清晰**：从变量声明可以看出类型
3. **标准做法**：这是字节码操作的标准实践

### Q3: 流式风格是否影响可读性？

> **Reviewer**: 嵌套的 lambda 是否难以理解？

**A**:
1. **常见模式**：函数式编程的标准风格
2. **IDE 支持**：现代 IDE 可以很好地格式化
3. **减少重复**：避免重复的 builder 名称

### 最佳实践推广

这些改动建立了以下最佳实践：

1. **使用专用指令**：aload/lload/astore/lstore 等
2. **使用 aconst_null**：而不是 loadConstant(null)
3. **流式风格**：充分利用 builder 模式
4. **简化参数**：直接传递所需参数，避免中间对象
