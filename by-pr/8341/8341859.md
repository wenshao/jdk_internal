## 背景

### 性能基准测试的重要性

ClassFile API 的性能基准测试用于：
- 验证字节码生成的性能
- 检测性能回归
- 指导优化方向

**为什么稳定性很重要？**
- 不稳定的基准测试会掩盖真实的性能变化
- 噪声过大时难以检测小幅改进
- CI/CD 中的性能检查需要可靠的基准

## 问题分析

### 原始实现

```java
// 文件: ClassFileBenchmark.java (优化前)

@Benchmark
public void writeClass(Blackhole bh) {
    for (int i = 0; i < METHODS_COUNT; i++) {
        String methodName = "method" + i;  // 每次迭代都创建新字符串
        // ... 使用 methodName 生成字节码 ...
    }
}

@Benchmark
public void writeManyMethods(Blackhole bh) {
    for (int i = 0; i < LARGE_COUNT; i++) {
        String methodName = "method" + i;  // 大量字符串拼接
        // ...
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 问题 | 影响 |
|--------|------|------|
| **StringBuilder 分配** | 每次拼接创建新的 StringBuilder | 内存分配压力 |
| **字符串创建** | "method" + i 每次生成新字符串 | GC 压力增加 |
| **测试噪声** | 字符串构造时间计入测试时间 | 结果不稳定 |

### 为什么影响测试结果？

```
实际字节码生成时间: 1000ns
字符串构造时间:     50ns (5%)
-------------------------
测试测量时间:       1050ns

问题: 字符串构造时间波动大 (±20ns)
结果: 测试结果波动 ±2%
```

## 解决方案

### 核心改动

```java
// 文件: ClassFileBenchmark.java (优化后)

public class ClassFileBenchmark {
    private static final int METHODS_COUNT = 100;
    private static final String[] METHOD_NAMES;  // 缓存的方法名

    static {
        // 预先计算并缓存所有方法名
        METHOD_NAMES = new String[METHODS_COUNT];
        for (int i = 0; i < METHODS_COUNT; i++) {
            METHOD_NAMES[i] = "method" + i;
        }
    }

    @Benchmark
    public void writeClass(Blackhole bh) {
        for (int i = 0; i < METHODS_COUNT; i++) {
            String methodName = METHOD_NAMES[i];  // 直接使用缓存
            // ... 使用 methodName 生成字节码 ...
        }
    }
}
```

### 关键优化点

| 优化点 | 说明 | 效果 |
|--------|------|------|
| **静态初始化** | 在类加载时一次性生成所有方法名 | 基准测试期间零分配 |
| **数组查找** | `METHOD_NAMES[i]` 替代字符串拼接 | O(1) 时间复杂度 |
| **常量池复用** | 相同字符串使用常量池中的同一实例 | 减少内存占用 |

## 设计决策

### Q1: 为什么不使用 String.format()？

> **Reviewer**: `String.format("method%d", i)` 不是更简洁吗？

**A**:
1. **性能**：`String.format()` 比 `+` 拼接更慢
2. **开销**：需要解析格式字符串，创建临时对象
3. **缓存后等价**：既然我们缓存结果，原始方法的选择不影响运行时性能

### Q2: 为什么用数组而不是 Map？

> **Reviewer**: 用 `Map<Integer, String>` 会不会更灵活？

**A**:
1. **性能**：数组访问比 HashMap 快得多
2. **内存**：数组内存占用远小于 HashMap
3. **适用性**：索引是连续的整数，数组是最优选择

### Q3: 缓存大小如何确定？

> **Reviewer**: 如果测试参数变化怎么办？

**A**:
1. 当前实现根据实际测试需求确定大小
2. 如果测试参数变化，需要相应调整缓存大小
3. 这是合理的权衡，因为基准测试参数通常是固定的
