## 背景

### StackMapGenerator

`StackMapGenerator` 负责生成类的 StackMapTable 属性，这是 JVM 字节码验证的关键数据。`processFieldInstructions` 处理字段访问指令（getfield/putfield/getstatic/putstatic）。

**为什么重要？**
- StackMap 生成是字节码生成的关键步骤
- 代码大小影响 JIT 内联决策
- 字段访问指令在 Java 代码中非常频繁

## 问题分析

### 原始实现

```java
// 文件: StackMapGenerator.java (优化前)

private void processFieldInstructions(RawBytecodeHelper raw) {
    int bci = raw.curPc();
    int opcode = raw.curOpcode();

    switch (opcode) {
        case GETSTATIC:
            // 每次都访问 this.currentFrame
            currentFrame.popStack(TypeKind.REFERENCE);
            Symbol fieldSym = Util.fieldTypeSymbol(raw.curEntry().nameAndType());
            currentFrame.pushStack(fieldSig(fieldSym));
            break;

        case PUTSTATIC:
            currentFrame.popStack(fieldSig(Util.fieldTypeSymbol(raw.curEntry().nameAndType())));
            currentFrame.popStack(TypeKind.REFERENCE);
            break;

        case GETFIELD:
            currentFrame.popStack(TypeKind.REFERENCE);
            currentFrame.pushStack(fieldSig(Util.fieldTypeSymbol(raw.curEntry().nameAndType())));
            break;

        case PUTFIELD:
            Symbol fieldSym = Util.fieldTypeSymbol(raw.curEntry().nameAndType());
            currentFrame.popStack(fieldSig(fieldSym));
            currentFrame.popStack(TypeKind.REFERENCE);
            break;
    }
}
```

### 性能瓶颈分析

| 瓶颈点 | 问题 | 影响 |
|--------|------|------|
| **重复访问 currentFrame** | 多次 `this.currentFrame` | 字段访问开销 |
| **popStack 内部数组访问** | `popStack` 内部访问 stack 数组 | 间接访问 |
| **重复 nameAndType 获取** | 每次都调用 `raw.curEntry().nameAndType()` | 方法调用开销 |
| **Util.fieldTypeSymbol 复杂** | 需要从 nameAndType 解析 | 可以简化 |

## 解决方案

### 核心改动

```java
// 文件: StackMapGenerator.java (优化后)

private void processFieldInstructions(RawBytecodeHelper raw) {
    int bci = raw.curPc();
    int opcode = raw.curOpcode();

    // 优化 1: 使用局部变量避免重复访问 this.currentFrame
    Frame currentFrame = this.currentFrame;
    currentFrame.curPc = bci;

    switch (opcode) {
        case GETSTATIC:
            // 优化 2: 使用 decStack 直接减少栈深度，避免 popStack 的数组访问
            currentFrame.decStack();
            // 优化 3: 直接传递 type 而不是 nameAndType
            Symbol fieldSym = Util.fieldTypeSymbol(raw.curEntry().type());
            currentFrame.pushStack(fieldSig(fieldSym));
            break;

        case PUTSTATIC:
            // 优化 2: decStack 替代 popStack
            currentFrame.decStack();
            currentFrame.decStack();
            break;

        case GETFIELD:
            currentFrame.decStack();
            Symbol fieldSym = Util.fieldTypeSymbol(raw.curEntry().type());
            currentFrame.pushStack(fieldSig(fieldSym));
            break;

        case PUTFIELD:
            currentFrame.decStack();
            currentFrame.decStack();
            break;
    }
}
```

### 关键优化点

#### 1. 局部变量缓存 currentFrame

```java
// 优化前: 每次访问都是 this.currentFrame
currentFrame.popStack(...);
currentFrame.pushStack(...);

// 优化后: 局部变量
Frame currentFrame = this.currentFrame;
currentFrame.popStack(...);
```

**好处**：
- 减少 `getfield` 指令（字节码层面）
- 更好的寄存器分配
- JIT 编译后可能完全消除

#### 2. 使用 decStack 替代 popStack

```java
// popStack 的实现
Type popStack() {
    Type[] stack = this.stack;
    int sp = this.sp;
    Type result = stack[--sp];
    this.sp = sp;
    return result;
}

// decStack 的实现
void decStack() {
    this.sp--;
}
```

**好处**：
- 不需要读取栈顶值
- 不需要返回值
- 更少的数组访问

#### 3. 简化 Util.fieldTypeSymbol

```java
// 优化前
Symbol fieldTypeSymbol(NameAndTypeEntry nameAndType) {
    return nameAndType.type().symbol();  // 需要获取整个 nameAndType
}

// 优化后
Symbol fieldTypeSymbol(Utf8Entry type) {
    return type.symbol();  // 直接使用 type
}
```

## 设计决策

### Q1: 为什么不直接修改 sp 字段？

> **Reviewer**: 为什么要有 decStack 方法，不直接修改 sp？

**A**:
1. **封装性**：保持对栈操作的控制
2. **一致性**：所有栈操作都通过方法
3. **调试友好**：可以在方法中添加断言

### Q2: 局部变量是否会增加内存占用？

> **Reviewer**: Frame 对象的引用拷贝有开销吗？

**A**:
1. **只是引用**：不是拷贝整个对象
2. **JIT 优化**：JIT 会将其优化为寄存器分配
3. **零开销**：编译后等价于直接访问

### Q3: 为什么删除 fieldTypeSymbol(NameAndTypeEntry)？

> **Reviewer**: 保持两个重载不是更灵活吗？

**A**:
1. **单一职责**：方法应该做一件事
2. **避免混淆**：两个重载可能造成调用歧义
3. **调用者调整**：调用者直接传递 type 更清晰


**优化后**:
```

...
```
