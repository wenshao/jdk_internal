## 背景

### StackMapGenerator 方法调用处理

`processInvokeInstructions` 处理所有方法调用指令（invokevirtual/invokestatic/invokeinterface/invokespecial/invokedynamic）。这是 Java 字节码中最复杂的指令类型之一。

**为什么重要？**
- 方法调用是 Java 程序中最频繁的操作
- StackMap 生成必须正确处理参数和返回值
- 方法调用指令的栈操作复杂度高

## 问题分析

### 原始实现

```java
// 文件: StackMapGenerator.java (优化前)

private void processInvokeInstructions(RawBytecodeHelper raw) {
    int bci = raw.curPc();
    int opcode = raw.curOpcode();

    switch (opcode) {
        case INVOKEVIRTUAL:
        case INVOKESPECIAL:
        case INVOKEINTERFACE:
            // 多次访问 this.currentFrame
            Symbol methodSym = Util.methodTypeSymbol(raw.curEntry().nameAndType());
            for (int i = 0; i < methodSym.parameterSlotCount(); i++) {
                currentFrame.popStack();  // 每次都访问 currentFrame
            }
            if (!methodSym.isVoid()) {
                currentFrame.pushStack(methodSym.returnKind().asType());
            }
            break;

        case INVOKESTATIC:
            Symbol methodSym = Util.methodTypeSymbol(raw.curEntry().nameAndType());
            for (int i = 0; i < methodSym.parameterSlotCount(); i++) {
                currentFrame.popStack();
            }
            if (!methodSym.isVoid()) {
                currentFrame.pushStack(methodSym.returnKind().asType());
            }
            break;

        case INVOKEDYNAMIC:
            // ... 类似逻辑
            break;
    }
}

// 文件: Util.java (优化前)
Symbol methodTypeSymbol(NameAndTypeEntry nameAndType) {
    // 需要从 nameAndType 获取 type，再解析
    return nameAndType.type().symbol();
}
```

### 性能瓶颈分析

| 瓶颈点 | 问题 | 影响 |
|--------|------|------|
| **重复访问 currentFrame** | 多次 `this.currentFrame` | 字段访问开销 |
| **popStack 循环开销** | 每个参数都调用 popStack | 方法调用 + 数组访问 |
| **nameAndType 解析** | 需要从中提取 type | 间接访问 |
| **重复的 methodSym 计算** | 每个分支都计算 | 代码重复 |

## 解决方案

### 核心改动

#### 1. 使用局部变量缓存 currentFrame

```java
// 优化前
currentFrame.popStack();
currentFrame.popStack();
currentFrame.pushStack(...);

// 优化后
Frame currentFrame = this.currentFrame;
currentFrame.popStack();
currentFrame.popStack();
currentFrame.pushStack(...);
```

#### 2. 使用 decStack 替代循环中的 popStack

```java
// 优化前
for (int i = 0; i < paramCount; i++) {
    currentFrame.popStack();  // 每次都读取数组
}

// 优化后
currentFrame.sp -= paramCount;  // 直接调整 sp
```

#### 3. 删除 Util.methodTypeSymbol(NameAndTypeEntry)

```java
// 优化前: 需要中间步骤
Symbol methodSym = Util.methodTypeSymbol(raw.curEntry().nameAndType());

// 优化后: 直接传递 type
Symbol methodSym = Util.methodTypeSymbol(raw.curEntry().type());
```

#### 4. 简化方法符号获取

```java
// 优化后的完整实现
private void processInvokeInstructions(RawBytecodeHelper raw) {
    int bci = raw.curPc();
    int opcode = raw.curOpcode();

    // 缓存 currentFrame
    Frame currentFrame = this.currentFrame;
    currentFrame.curPc = bci;

    switch (opcode) {
        case INVOKEVIRTUAL:
        case INVOKESPECIAL:
        case INVOKEINTERFACE: {
            // 直接使用 type，跳过 nameAndType
            Symbol methodSym = Util.methodTypeSymbol(raw.curEntry().type());
            int paramCount = methodSym.parameterSlotCount();
            // 直接减少 sp，避免循环
            currentFrame.sp -= paramCount;
            if (opcode != INVOKESTATIC) {
                currentFrame.sp--;  // 弹出 this
            }
            if (!methodSym.isVoid()) {
                currentFrame.pushStack(methodSym.returnKind().asType());
            }
            break;
        }

        case INVOKESTATIC: {
            Symbol methodSym = Util.methodTypeSymbol(raw.curEntry().type());
            int paramCount = methodSym.parameterSlotCount();
            currentFrame.sp -= paramCount;
            if (!methodSym.isVoid()) {
                currentFrame.pushStack(methodSym.returnKind().asType());
            }
            break;
        }

        case INVOKEDYNAMIC: {
            // ... 类似优化
            break;
        }
    }
}
```

### 关键优化点

| 优化点 | 说明 | 效果 |
|--------|------|------|
| **局部变量缓存** | 避免重复的 this.currentFrame | 减少字段访问 |
| **直接 sp 操作** | currentFrame.sp -= n 替代循环 | 消除循环开销 |
| **简化参数传递** | 直接传递 type 而非 nameAndType | 减少中间对象 |
| **删除冗余方法** | 删除 methodTypeSymbol(NameAndTypeEntry) | 代码简化 |

## 设计决策

### Q1: 直接操作 sp 字段安全吗？

> **Reviewer**: 为什么不用 decStack 方法？

**A**:
1. **批量操作**：一次减少多个，不是逐个减少
2. **性能考虑**：避免循环和方法调用
3. **类型安全**：调用指令的参数数量在编译期确定

### Q2: 为什么删除 methodTypeSymbol(NameAndTypeEntry)？

> **Reviewer**: 保留重载不是更灵活？

**A**:
1. **避免歧义**：只有一个清晰的入口
2. **强制优化**：调用者必须传递 type
3. **代码一致性**：与 fieldTypeSymbol 保持一致

### Q3: INVOKESTATIC 和其他指令的区别？

> **Reviewer**: 为什么 INVOKESTATIC 不弹出 this？

**A**:
1. **静态方法**：不接收 this 参数
2. **栈影响**：栈顶只有参数，没有对象引用
3. **JVM 规范**：符合 JVM 规范要求

### 字节码指令对比

**优化前 (循环)**:
```
// 处理 3 个参数的方法调用
iload_2              // load i
iconst_0
if_icmpge L1
aload_1              // load currentFrame
invokevirtual popStack
iinc 2, 1
goto L2
L1:
...

// 3 次方法调用 + 3 次数组访问
```

**优化后 (直接 sp 操作)**:
```
// 处理 3 个参数的方法调用

dup
getfield sp          // 读取 sp
iconst_3
isub                 // sp -= 3
putfield sp          // 写回 sp

// 1 次字段读取 + 1 次字段写入
```
