## 背景

### DecimalDigits 的特殊性

`DecimalDigits` 类包含使用 Unsafe 操作的方法：

```java
// DecimalDigits.java
static int getCharsLatin1(int i, int index, byte[] buf) {
    // 使用 Unsafe.putByte，无边界检查！
    // 调用者必须确保：
    // 1. buf 不为 null
    // 2. buf.length >= index
    // 3. i 是有效的 int 值
}
```

### 为什么需要文档？

| 问题 | 影响 |
|------|------|
| **缺少文档** | 调用者不知道前置条件 |
| **Unsafe 操作** | 错误使用可能导致内存损坏 |
| **维护困难** | 后续维护者不知道方法约束 |

**改动位置**

## 解决方案

### 核心改动

#### 1. 添加详细的 javadoc

```java
// 文件: DecimalDigits.java (优化后)

/**
 * 将整数转换为 Latin1 编码的字符。
 *
 * @param i     要转换的整数
 * @param index 下一个字符的索引（独占）
 * @param buf   目标缓冲区，Latin1 编码
 * @return 最高位数字或负号的索引
 * @throws NullPointerException 如果 buf 为 null（由 Unsafe.putByte 抛出）
 * @apiNote 此方法假定所有必要的边界检查都由调用者完成。
 *          它使用 Unsafe 操作来消除边界检查，这是安全的，因为调用者保证：
 *          <ul>
 *          <li>{@code buf} 不为 {@code null}</li>
 *          <li>{@code buf.length >= index}</li>
 *          <li>{@code index - 字符数 >= 0}</li>
 *          </ul>
 *
 * @warning 此方法使用 Unsafe 操作，错误使用可能导致内存损坏。
 *          确保满足所有前置条件后再调用。
 */
static int getCharsLatin1(int i, int index, byte[] buf) {
    // ... 实现
}
```

#### 2. 添加 assert 验证

```java
// 文件: DecimalDigits.java

static int getCharsLatin1(int i, int index, byte[] buf) {
    // 在调试模式下验证前置条件
    assert buf != null : "buf must not be null";
    assert index >= 0 : "index must be non-negative";
    assert index <= buf.length : "index must not exceed buf.length";

    // ... 实现
}
```

#### 3. 添加 @warning 标签

```java
/**
 * @warning 此方法使用 Unsafe 操作，错误使用可能导致内存损坏。
 *          确保满足所有前置条件后再调用。
 */
```

## 设计决策

### Q1: 为什么使用 @apiNote 而不是 @implSpec？

> **Reviewer**: 两个标签有什么区别？

**A**:
```java
// @apiNote: 面向 API 使用者的说明
@apiNote 此方法假定所有必要的边界检查都由调用者完成。

// @implSpec: 面向实现者的规范
@implSpec 此实现使用 Unsafe 操作来消除边界检查。
```

这里使用 `@apiNote` 更合适，因为主要面向调用者。

### Q2: 为什么需要 assert？

> **Reviewer**: javadoc 不够吗？

**A**:
```java
// javadoc: 文档说明
// assert: 运行时验证（仅在启用断言时）

// 启用断言后，错误使用会立即失败
java -ea com.example.MyClass
```

`assert` 提供了额外的安全网。

### Q3: 为什么不直接添加边界检查？

> **Reviewer**: 为什么不直接在方法内检查边界？

**A**:
```java
// 如果添加边界检查，就失去了性能优势
if (buf == null) throw NPE;       // 性能损失
if (index < 0 || index > buf.length) throw IOOBE;  // 性能损失

// Unsafe 操作的优势就是无边界检查
// 所以通过文档来约束调用者
```

这是性能和安全的权衡。

### Q4: @warning 是标准标签吗？

> **Reviewer**: 我没见过 @warning 标签？

**A**: `@warning` 不是标准 javadoc 标签，但是：
- 某些 IDE 会特别显示
- 比 @note 更醒目
- 适合用于安全警告

## 文档结构

### 标准文档模板

```java
/**
 * [方法简短描述]
 *
 * @param paramName [参数说明]
 * @return [返回值说明]
 * @throws [异常说明]
 * @apiNote [实现细节和前置条件]
 * @warning [使用警告]
 */
```

### 前置条件检查清单

调用这些方法时，必须确保：

| 条件 | 说明 | 验证方法 |
|------|------|----------|
| **buf != null** | 缓冲区不能为 null | Objects.requireNonNull |
| **buf.length >= index** | 缓冲区必须足够大 | 提前检查 |
| **index - 字符数 >= 0** | 不能写入负索引 | 提前计算 |

## 实际影响场景

### 正确使用

```java
// 正确：调用者确保所有条件
public void appendInt(int value, byte[] buf, int pos) {
    // 前置条件检查
    Objects.requireNonNull(buf);
    if (pos < 0 || pos + 11 > buf.length) {
        throw new IndexOutOfBoundsException();
    }

    // 安全调用
    DecimalDigits.getCharsLatin1(value, pos, buf);
}
```

### 错误使用

```java
// 错误：未检查边界
public void appendIntUnsafe(int value, byte[] buf, int pos) {
    // 未检查 pos 是否有效！
    DecimalDigits.getCharsLatin1(value, pos, buf);  // 可能越界
}
```
