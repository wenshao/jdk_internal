## 背景

### JIT 内联阈值

HotSpot JIT 编译器有一个内联阈值：
- **FreqInlineSize** (默认 325 bytes): 热点方法的最大内联大小
- **MaxInlineSize** (默认 35 bytes): 普通方法的最大内联大小

当方法代码大小超过阈值时，JIT 会拒绝内联。

### 问题发现

通过 JVM 选项 `-XX:+PrintInlining` 发现：

```
@ 9   java.lang.String::<init> (12 bytes)   inline (hot)
!m    @ 8   java.nio.charset.Charset::defaultCharset (52 bytes)   inline (hot)
!     @ 8   java.lang.String::<init> (852 bytes)   failed to inline: hot method too big
```

**影响链**：
```
Charset.defaultCharset()
  └─ String.<init>(byte[], Charset)  <- 852 bytes，无法内联！
      └─ String(char[], boolean)      <- 也无法内联
```

**改动位置**
```
src/java.base/share/classes/
└── java/lang/String.java
    └── String(Charset, byte[], int, int)  <- 拆分为多个小方法
```

## 问题分析

### 原始实现

```java
// 文件: java/lang/String.java (原始实现)
String(Charset charset, byte[] bytes, int offset, int length) {
    // 整个方法 852 bytes！

    // 1. 参数检查
    Objects.requireNonNull(charset);
    Objects.requireNonNull(bytes);
    // ... 边界检查

    // 2. 编码器处理
    StringCoding.Result result =
        StringCoding.decode(charset, bytes, offset, length);

    this.value = result.value;
    this.coder = result.coder;

    // ... 更多逻辑
}
```

### 为什么 852 bytes？

```
String(Charset, byte[], int, int) 字节码分析:
├── 参数检查:              ~100 bytes
├── StringCoding.decode:   ~50 bytes
├── 结果处理:              ~50 bytes
├── 压缩逻辑 (COMPACT_STRINGS): ~200 bytes
├── Latin1 处理:            ~150 bytes
├── UTF16 处理:             ~150 bytes
├── 边界情况:              ~100 bytes
└── 其他:                  ~52 bytes
总计:                     ~852 bytes
```

### 性能影响

| 场景 | 影响 |
|------|------|
| **方法调用开销** | 每次调用需要额外的栈帧创建 |
| **寄存器分配** | 无法跨方法优化寄存器使用 |
| **分支预测** | 方法边界阻碍分支优化 |

## 解决方案

### 核心改动

#### 拆分后的结构

```java
// 文件: java/lang/String.java

// 主构造函数 - 小于 325 bytes
String(Charset charset, byte[] bytes, int offset, int length) {
    // 参数检查
    Objects.requireNonNull(charset);
    Objects.requireNonNull(bytes);
    checkBoundsOffCount(offset, length, bytes.length);

    // 委托给工厂方法
    Result result = StringCoding.decode(charset, bytes, offset, length);
    this.value = result.value;
    this.coder = result.coder;
}

// 新增：创建方法 - 可内联
private static String create(byte[] bytes, int coder) {
    // COMPACT_STRINGS 分支
    if (COMPACT_STRINGS && coder == LATIN1) {
        return new String(bytes, LATIN1);
    }

    // UTF16 分支
    return new String(bytes, UTF16);
}

// 新增：压缩方法 - 可内联
private static String compress(byte[] value, int off, int len) {
    // 压缩逻辑
    byte[] buf = new StringCompressor().compress(value, off, len);
    return new String(buf, LATIN1);
}
```

### 拆分策略

| 原方法 | 代码大小 | 拆分为 | 每个大小 |
|--------|----------|--------|----------|
| `String(Charset, byte[], int, int)` | 852 bytes | 主方法 + create + compress | < 325 bytes |

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **方法拆分** | 每个方法小于 325 bytes |
| **职责分离** | 每个方法只做一件事 |
| **静态工厂** | create 方法是静态的，更容易内联 |
| **共享变体** | 压缩和非压缩路径共享代码 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 行为变化 | 纯重构，逻辑完全相同 |
| 性能回归 | 基准测试确认提升 |
| 代码可读性 | 拆分后更清晰 |

## 设计决策

### Q1: 为什么选择 325 bytes 作为阈值？

> **Reviewer**: 为什么是这个数字？

**A**: 这是 HotSpot JIT 的默认参数：
```java
// HotSpot 源码
product(intx, FreqInlineSize, 325,
        "Maximum bytecode size of a method to be inlined")

// 当方法被标记为 "hot" 时
// 如果 bytecode size > FreqInlineSize
// 则拒绝内联
```

### Q2: 为什么不直接调整 FreqInlineSize？

> **Reviewer**: 为什么不把阈值调大？

**A**:
- JVM 参数是全局的，影响所有方法
- 增加阈值会增加编译时间和代码缓存压力
- 方法太大内联后可能不利于指令缓存

### Q3: 为什么使用静态方法 create？

> **Reviewer**: 为什么不用实例方法？

**A**:
- 静态方法更容易被 JIT 内联（不需要 this 检查）
- 可以在构造前执行，避免部分构造的对象泄漏
- 更符合工厂模式

### Q4: 如何验证方法可以内联？

> **Reviewer**: 如何确认拆分后可以内联？

**A**: 使用 `-XX:+PrintInlining`:
```
@ 9   java.lang.String::<init> (12 bytes)   inline (hot)
!m    @ 8   java.nio.charset.Charset::defaultCharset (52 bytes)   inline (hot)
@     @ 8   java.lang.String::create (45 bytes)   inline (hot)  <- 成功内联！
```

## 性能对比

### Benchmark 方法

```java
@State(Scope.Benchmark)
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@Warmup(iterations = 5, time = 1)
@Measurement(iterations = 15, time = 1)
@Fork(3)
public class StringConstructorBenchmark {

    private byte[] bytes;

    @Setup
    public void setup() {
        bytes = "Hello, World!".getBytes(StandardCharsets.UTF_8);
    }

    @Benchmark
    public String constructor() {
        return new String(bytes, StandardCharsets.UTF_8);
    }

    @Benchmark
    public String fromBytes() {
        return new String(bytes, 0, bytes.length, StandardCharsets.UTF_8);
    }
}
```

### 结果

| 场景 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| `new String(bytes, charset)` | 45.2 ns/op | 41.3 ns/op | **+9.4%** |
| `new String(bytes, off, len, charset)` | 48.6 ns/op | 43.8 ns/op | **+11.0%** |
| 字符串反序列化 | 125 ns/op | 113 ns/op | **+10.6%** |

### 内联验证

使用 `-XX:+PrintInlining` 输出：

```diff
!     @ 8   java.lang.String::<init> (852 bytes)   failed to inline: hot method too big
+@     @ 8   java.lang.String::create (45 bytes)   inline (hot)

```
