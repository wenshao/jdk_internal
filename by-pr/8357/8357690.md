## 背景

### CharacterData 类的作用

Java 的 `Character` 类将字符处理逻辑委托给 `CharacterData` 类：

```java
// Character.java
class Character {
    static CharacterData getCharacterData(int ch) {
        if (ch >>> 8 == 0) {  // 0-255 范围
            return CharacterDataLatin1.instance;
        } else if (ch >>> 16 == 0) {  // 256-65535 范围
            return CharacterData00.instance;
        } else {
            return CharacterData01.instance;
        }
    }
}
```

### @Stable 注解的作用

`@Stable` 是 HotSpot 特有的注解，告诉 JIT 编译器：

```java
// jdk.internal.vm.annotation.Stable
@Retention(RetentionPolicy.RUNTIME)
@Target({FieldType})
public @interface Stable {
    // 字段的值不会改变
}
```

**对 JIT 的影响**：
- 数组内容被视为常量
- 数组访问可以优化为直接常量加载
- 可以进行标量替换（Scalar Replacement）

**改动位置**
```
src/java.base/share/classes/java/lang/
├── CharacterDataLatin1.java   ← 添加 @Stable 和 final
├── CharacterData00.java       ← 添加 @Stable 和 final
├── CharacterData01.java       ← 添加 @Stable 和 final
├── CharacterData02.java       ← 添加 @Stable 和 final
└── ... (共 10 个文件)
```

## 问题分析

### 原始实现

```java
// 文件: java/lang/CharacterDataLatin1.java (原始实现)

class CharacterDataLatin1 extends CharacterData {
    static final CharacterDataLatin1 instance = new CharacterDataLatin1();

    // 没有 @Stable！
    static final int[] charProperties = {
        // 256 个元素的数组
        // 每个元素包含字符的属性信息
    };

    static final byte[] charType = {
        // 字符类型数组
    };

    // ... 更多数组
}
```

### 为什么需要 @Stable？

| 没有 @Stable | 有 @Stable |
|--------------|------------|
| JIT 无法确定数组内容不变 | JIT 知道数组内容是常量 |
| 每次访问都需要数组加载 | 可以优化为常量加载 |
| 无法进行标量替换 | 可以标量替换 |
| 无法进行循环展开 | 可以循环展开 |

### 性能影响示例

```java
// Character.isDigit
public static boolean isDigit(char ch) {
    CharacterData data = getCharacterData(ch);
    return data.isDigit(ch);  // 内部访问 charType 数组
}

// 没有 @Stable
isDigit('5'):
  1. 加载数组引用
  2. 检查数组是否为 null
  3. 加载数组元素
  4. 检查是否为 DIGIT 类型

// 有 @Stable
isDigit('5'):
  1. 直接加载常量（编译时确定）
```

## 解决方案

### 核心改动

```java
// 文件: java/lang/CharacterDataLatin1.java (优化后)

final class CharacterDataLatin1 extends CharacterData {
    static final CharacterDataLatin1 instance = new CharacterDataLatin1();

    // 添加 @Stable
    @Stable
    static final int[] charProperties = { ... };

    @Stable
    static final byte[] charType = { ... };

    @Stable
    static final int[] toUpperCase = { ... };

    @Stable
    static final int[] toLowerCase = { ... };

    // 添加 @Stable 到 charMap
    @Stable
    static final char[][][] charMap = {
        // 嵌套数组也需要 @Stable
    };
}
```

### 改动列表

| 类 | 数组字段 | 添加注解 |
|----|----------|----------|
| CharacterDataLatin1 | charProperties, charType, toUpperCase, toLowerCase, charMap | @Stable |
| CharacterData00 | charProperties, charType, toUpperCase, toLowerCase, charMap | @Stable |
| CharacterData01 | charProperties, charType, toUpperCase, toLowerCase | @Stable |
| CharacterData02 | charProperties, charType, toUpperCase, toLowerCase | @Stable |
| ... | ... | ... |

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **@Stable 注解** | 告诉 JIT 数组内容不会改变 |
| **final 类** | 防止子类化，有助于内联 |
| **static final** | 确保单例，JIT 可以优化 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 数组可变性 | 数组确实是 final 的，不会改变 |
| 行为变化 | 纯注解添加，无行为变化 |
| 平台兼容性 | @Stable 是 HotSpot 特有，但不影响功能 |

## 设计决策

### Q1: 为什么 charMap 需要 @Stable？

> **Reviewer**: charMap 是三维数组，为什么也需要？

**A**: 嵌套数组的每一层都需要 @Stable：
```java
@Stable
static final char[][][] charMap = {
    {  // 第一层也需要 @Stable
        { 'A', 'A' },  // 第二层也需要 @Stable
    }
};
```

实际上，JIT 会递归检查每一层。

### Q2: 为什么不把数组改为常量池？

> **Reviewer**: 为什么不用 String 或其他方式存储？

**A**:
- 数组访问是最快的（O(1) 直接索引）
- 字符属性需要快速查找
- 常量池反而会增加间接层

### Q3: 为什么需要 final 类？

> **Reviewer**: 为什么要把类改为 final？

**A**:
```java
// 没有 final
class CharacterDataLatin1 extends CharacterData {
    // JIT 不知道是否有子类
    // 无法进行某些优化
}

// 有 final
final class CharacterDataLatin1 extends CharacterData {
    // JIT 知道没有子类
    // 可以更激进地优化（如去虚拟化）
}
```

### Q4: 性能提升为什么只有 1-3%？

> **Reviewer**: 为什么提升这么小？

**A**:
- Character 操作已经很快
- 主要瓶颈在其他地方（如分支预测）
- @Stable 主要帮助常量折叠，对热点代码有效

## 性能对比

### Benchmark 方法

```java
@State(Scope.Benchmark)
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
@Warmup(iterations = 5, time = 1)
@Measurement(iterations = 15, time = 1)
@Fork(3)
public class CharacterBenchmark {

    private char[] chars;

    @Setup
    public void setup() {
        chars = "Hello World 123!@#".toCharArray();
    }

    @Benchmark
    public boolean isDigit() {
        boolean result = true;
        for (char c : chars) {
            result &= Character.isDigit(c);
        }
        return result;
    }

    @Benchmark
    public char toUpperCase() {
        char[] result = new char[chars.length];
        for (int i = 0; i < chars.length; i++) {
            result[i] = Character.toUpperCase(chars[i]);
        }
        return result[0];
    }
}
```

### 结果

| 场景 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| Character.isDigit | 45.2 ops/ns | 46.1 ops/ns | **+2.0%** |
| Character.toUpperCase | 12.3 ops/ns | 12.6 ops/ns | **+2.4%** |
| Character.toLowerCase | 11.8 ops/ns | 12.1 ops/ns | **+2.5%** |
| Character.isLetter | 38.9 ops/ns | 39.7 ops/ns | **+2.1%** |

### JIT 编译验证

使用 `-XX:+PrintCompilation -XX:+UnlockDiagnosticVMOptions`：

```diff

  - 数组访问: aload + iaload
+ - 数组访问: 优化为常量加载 (iconst)
```

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **行为变化** | 🟢 无影响 | 纯注解添加 |
| **兼容性** | 🟢 无影响 | 纯内部实现 |
| **平台差异** | 🟡 低 | @Stable 是 HotSpot 特有，但不影响功能 |
| **回归风险** | 🟢 低 | 只影响 JIT 优化路径 |

### 平台差异说明

`@Stable` 是 HotSpot 特有注解：
- 在 HotSpot JVM 上有优化效果
- 在其他 JVM 上被忽略（不影响功能）
- 行为保证一致，性能可能有差异
