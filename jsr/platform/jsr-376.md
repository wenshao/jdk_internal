# JSR 376: Java Platform Module System (Project Jigsaw)

> **状态**: ✅ Final | **JDK**: 9 (2017-09) | **链接**: https://jcp.org/en/jsr/detail?id=376

---

## 概述

JSR 376 定义了 Java 平台模块系统 (JPMS)，也称为 **Project Jigsaw**，是 Java 历史上最雄心勃勃的项目之一。从 2005 年启动到 2017 年交付，历时 **12 年**。

## 核心概念

### 模块 (Module)

模块是包和资源的集合，具有明确的依赖声明。

```java
// module-info.java
module com.example.myapp {
    // 依赖
    requires java.sql;
    requires transitive java.logging;
    
    // 导出
    exports com.example.api;
    exports com.example.internal to com.example.tests;
    
    // 服务
    uses com.example.spi.Service;
    provides com.example.spi.Service with com.example.impl.ServiceImpl;
}
```

### 关键字

| 关键字 | 说明 |
|--------|------|
| `module` | 定义模块 |
| `requires` | 声明依赖 |
| `requires transitive` | 传递依赖 |
| `requires static` | 可选依赖 |
| `exports` | 导出包 |
| `opens` | 开放反射访问 |
| `uses` | 使用服务 |
| `provides...with` | 提供服务实现 |

## 模块类型

### 具名模块 (Named Module)

```java
// module-info.java
module my.module {
    requires other.module;
}
```

### 自动模块 (Automatic Module)

```java
// 无 module-info.java，放在 modulepath 上
// 模块名来自 JAR 文件名
```

### 未命名模块 (Unnamed Module)

```java
// 放在 classpath 上的 JAR
// 可访问所有模块导出的包
```

## JDK 模块化

### Java SE 模块

```bash
# 查看所有 JDK 模块
java --list-modules

# 关键模块
java.base          # 核心类库
java.sql           # JDBC
java.logging       # java.util.logging
java.desktop       # AWT, Swing
java.xml           # XML 处理
```

### 紧凑配置

```bash
# jlink 创建自定义运行时
jlink --add-modules java.base,java.sql \
      --output /path/to/custom-jre
```

## 兼容性影响

### 强封装

```java
// JDK 内部 API 不再可访问
// --add-opens 绕过 (不推荐)
java --add-opens java.base/java.lang=ALL-UNNAMED
```

### 迁移策略

1. **无 module-info.java**: 放在 classpath
2. **自动模块**: 放在 modulepath
3. **具名模块**: 添加 module-info.java

## 争议与社区反馈

### 主要争议

#### 1. ClassPath vs ModulePath 混淆

```
问题: 开发者不清楚 JAR 应该放在哪里
- classpath: 遗留模式，类似 Java 8
- modulepath: 新模块化模式

社区反应: 褒贬不一，逐渐采纳
```

#### 2. 自动模块命名

```
问题: 无 module-info 的 JAR 获得自动生成的名称
- 规则: 从 JAR 文件名派生
- 问题: 不确定，可能破坏构建

示例:
  my-library-1.0.jar → module "my.library"

争议: 脆弱、版本敏感的命名
```

#### 3. 强封装破坏库兼容性

```java
// 之前常见的模式被破坏:
Field field = SomeClass.class.getDeclaredField("privateField");
field.setAccessible(true); // 现在对 JDK 内部类失败

// 解决方案 (不推荐):
java --add-opens java.base/java.lang=ALL-UNNAMED
```

#### 4. 分裂包问题

```
问题: 同一包在多个模块中 = 错误
- 常见于 shaded 依赖
- 没有简单的迁移路径

错误示例:
  Module A exports com.example.foo
  Module B also exports com.example.foo
  → Error: split package
```

### 知名批评

| 批评者 | 观点 |
|--------|------|
| **Stephen Colebourne** (Joda-Time 作者) | "模块系统对大多数应用来说太复杂" |
| **Martijn Verburg** (London JUG) | 担心生态碎片化，库维护者迁移负担 |
| **Red Hat** | 担心对现有框架的影响，请求更多时间 |

### 采纳统计 (2023)

```
使用 module-info.java 的应用: ~15-20%
使用 classpath 模式的应用: ~80-85%
使用 jlink 的应用: <5%
```
