## 2. 概述

JSR 292 的核心目标是让 JVM 更好地支持动态语言（如 JRuby、Groovy、Nashorn JavaScript）。在此之前，JVM 只有四种方法调用指令（`invokevirtual`、`invokeinterface`、`invokestatic`、`invokespecial`），它们都要求编译期确定方法签名和目标类型。

| 组件 | 说明 |
|------|------|
| **invokedynamic** | 新的字节码指令，允许运行时决定调用目标 |
| **MethodHandle** | 轻量级、类型安全的方法指针，可被 JIT 内联优化 |
| **MethodType** | 不可变对象，描述方法的参数类型和返回类型 |
| **CallSite** | invokedynamic 的调用点，持有目标 MethodHandle |
| **Bootstrap Method** | 首次执行 invokedynamic 时调用，完成链接 |

虽然设计初衷是服务动态语言，但 JSR 292 最终最大的受益者是 Java 自身——Lambda 表达式（JSR 335）和字符串拼接优化（JEP 280）都依赖 invokedynamic。

## 3. invokedynamic 指令

### JVM 五种方法调用指令对比

| 指令 | 用途 | 编译期确定 |
|------|------|-----------|
| `invokevirtual` | 虚方法调用 | 方法签名 + 接收者类型 |
| `invokeinterface` | 接口方法调用 | 接口类型 + 方法签名 |
| `invokestatic` | 静态方法调用 | 类 + 方法签名 |
| `invokespecial` | 构造函数、super、private | 类 + 方法签名 |
| **`invokedynamic`** | **用户自定义链接** | **仅 bootstrap 方法** |

### 字节码结构

```
// invokedynamic 指令格式
invokedynamic #bsm_index, 0
  // #bsm_index 指向 BootstrapMethods 属性中的条目
  // 第二个操作数始终为 0 (保留)

// BootstrapMethods 属性
BootstrapMethods:
  0: #15 invokestatic BSMClass.bootstrap:(
      Ljava/lang/invoke/MethodHandles$Lookup;
      Ljava/lang/String;
      Ljava/lang/invoke/MethodType;
  )Ljava/lang/invoke/CallSite;
```

### 执行流程

```
第一次执行 invokedynamic:
  ┌──────────────┐    ┌──────────────────┐    ┌───────────┐
  │ invokedynamic │───→│ Bootstrap Method │───→│ CallSite  │
  └──────────────┘    └──────────────────┘    └─────┬─────┘
                                                    │
                                              MethodHandle
                                                    │
后续执行:                                           ▼
  ┌──────────────┐    ┌───────────┐    ┌────────────────┐
  │ invokedynamic │───→│ CallSite  │───→│ 目标方法 (直接) │
  └──────────────┘    └───────────┘    └────────────────┘
```

## 4. MethodHandle API

### 基本使用

```java
// MethodHandles.Lookup: 访问上下文 (类似反射中的 setAccessible)
MethodHandles.Lookup lookup = MethodHandles.lookup();

// 查找静态方法
MethodHandle mh = lookup.findStatic(
    Integer.class,                          // 声明类
    "parseInt",                             // 方法名
    MethodType.methodType(int.class, String.class)  // 方法签名
);
int result = (int) mh.invokeExact("42");   // 调用

// 查找虚方法
MethodHandle mh = lookup.findVirtual(
    String.class,
    "substring",
    MethodType.methodType(String.class, int.class)
);
String s = (String) mh.invokeExact("Hello", 1);  // "ello"

// 查找构造函数
MethodHandle mh = lookup.findConstructor(
    ArrayList.class,
    MethodType.methodType(void.class)
);
ArrayList<?> list = (ArrayList<?>) mh.invoke();
```

### MethodType

```java
// 描述方法签名: (参数类型...) -> 返回类型
MethodType mt = MethodType.methodType(int.class, String.class);
// 对应: int method(String)

MethodType mt2 = MethodType.methodType(void.class);
// 对应: void method()

MethodType mt3 = MethodType.methodType(String.class, int.class, int.class);
// 对应: String method(int, int)

// MethodType 是不可变的，修改返回新实例
MethodType mt4 = mt3.changeReturnType(int.class);
MethodType mt5 = mt3.appendParameterTypes(boolean.class);
```

### MethodHandle 组合

```java
// 参数绑定 (partial application)
MethodHandle println = lookup.findVirtual(
    PrintStream.class, "println",
    MethodType.methodType(void.class, String.class)
);
MethodHandle bound = println.bindTo(System.out);
bound.invokeExact("Hello");  // System.out.println("Hello")

// 参数适配
MethodHandle adapted = MethodHandles.dropArguments(mh, 0, String.class);
MethodHandle filtered = MethodHandles.filterArguments(mh, 0, transformer);
MethodHandle folded = MethodHandles.foldArguments(mh, combiner);

// 条件分支 (guardWithTest)
MethodHandle guarded = MethodHandles.guardWithTest(
    testHandle,     // boolean test(...)
    trueHandle,     // 测试通过时调用
    falseHandle     // 测试失败时调用
);
```

### MethodHandle vs Reflection

| 特性 | Reflection | MethodHandle |
|------|-----------|-------------|
| 类型安全 | 运行时检查 | 编译期 + 运行时 |
| 性能 | 较慢 | 可被 JIT 内联 |
| 访问控制 | `setAccessible` | Lookup 上下文 |
| 组合能力 | 无 | 丰富的组合 API |
| 适用场景 | 框架/工具 | JVM 底层/高性能场景 |

## 5. CallSite 机制

```java
// CallSite: invokedynamic 的调用目标容器

// ConstantCallSite: 目标不可变（最常用，JIT 可充分优化）
CallSite cs = new ConstantCallSite(targetHandle);

// MutableCallSite: 目标可变
MutableCallSite mcs = new MutableCallSite(targetHandle);
mcs.setTarget(newTarget);  // 更换目标
MutableCallSite.syncAll(new MutableCallSite[]{ mcs });  // 全局同步

// VolatileCallSite: 目标可变 + volatile 语义
VolatileCallSite vcs = new VolatileCallSite(targetHandle);
vcs.setTarget(newTarget);  // 立即对所有线程可见
```

## 6. Bootstrap 方法

### 标准签名

```java
// Bootstrap 方法由 JVM 在首次遇到 invokedynamic 时调用
public static CallSite bootstrap(
    MethodHandles.Lookup caller,   // 调用者的查找上下文
    String name,                   // invokedynamic 指定的名称
    MethodType type                // invokedynamic 指定的类型
) {
    // 查找并返回目标方法
    MethodHandle target = caller.findStatic(
        MyClass.class, name, type
    );
    return new ConstantCallSite(target);
}
```

### LambdaMetafactory (JDK 8 使用的 Bootstrap)

```java
// Lambda 表达式的 bootstrap 方法
// JVM 在遇到 lambda 的 invokedynamic 时调用此方法
public static CallSite metafactory(
    MethodHandles.Lookup caller,
    String invokedName,           // 函数式接口方法名 (如 "run")
    MethodType invokedType,       // CallSite 的类型
    MethodType samMethodType,     // 函数式接口方法签名
    MethodHandle implMethod,      // 实现方法
    MethodType instantiatedMethodType
) {
    // 运行时生成函数式接口的实现类
    return new ConstantCallSite(/* generated class */);
}
```

## 7. 实际应用场景

### 1. Lambda 表达式 (JEP 126, JDK 8)

```java
// Java 源码
Runnable r = () -> System.out.println("Hello");

// 编译后字节码
invokedynamic #0:run:()Ljava/lang/Runnable;
// Bootstrap: LambdaMetafactory.metafactory
```

Lambda 使用 invokedynamic 而非匿名内部类，好处是延迟到运行时决定实现策略，未来可以切换实现方式而无需重新编译。

### 2. 字符串拼接优化 (JEP 280, JDK 9)

```java
// Java 源码
String s = "Hello, " + name + "!";

// JDK 8: 编译为 StringBuilder 链
new StringBuilder().append("Hello, ").append(name).append("!").toString()

// JDK 9+: 编译为 invokedynamic
invokedynamic #0:makeConcatWithConstants:(Ljava/lang/String;)Ljava/lang/String;
// Bootstrap: StringConcatFactory.makeConcatWithConstants
// 运行时选择最优策略
```

### 3. 动态语言 (JRuby, Nashorn)

```java
// 动态语言中，obj.foo() 的调用目标在编译期未知
// invokedynamic 允许运行时查找并缓存目标方法

// JRuby 使用 invokedynamic 实现 Ruby 方法调度
// Nashorn 使用 invokedynamic 实现 JavaScript 属性访问
```

### 4. switch 模式匹配 (JDK 21)

```java
// Pattern matching for switch 也使用 invokedynamic
switch (obj) {
    case Integer i -> handleInt(i);
    case String s  -> handleString(s);
    default        -> handleOther(obj);
}
// Bootstrap: SwitchBootstraps.typeSwitch
```

## 9. 性能考量

### invokedynamic 的性能优势

```
首次调用:
  invokedynamic → Bootstrap Method → CallSite 链接 (一次性开销)

后续调用:
  invokedynamic → CallSite → MethodHandle → 目标方法
  (ConstantCallSite 可被 JIT 完全内联，性能等同于直接调用)
```

### MethodHandle vs 直接调用

| 场景 | 相对性能 |
|------|---------|
| 直接调用 | 1.0x (基准) |
| MethodHandle (ConstantCallSite) | ~1.0x (JIT 内联后) |
| MethodHandle (MutableCallSite) | ~1.1x |
| Reflection | ~5-10x 慢 |

### JIT 优化

JVM 对 `ConstantCallSite` 中的 `MethodHandle` 做特殊优化：
- 内联 (inlining): 将目标方法代码直接嵌入调用点
- 常量折叠 (constant folding): 如果参数已知则预计算
- 消除冗余检查: 类型检查可在链接时完成
