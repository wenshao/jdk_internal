## 2. 概述

JSR 133 彻底重新定义了 Java 内存模型（Java Memory Model, JMM），是 Java 并发编程的理论基石：

| 特性 | 说明 |
|------|------|
| **Happens-Before 偏序** | 定义多线程间操作的可见性规则 |
| **volatile 语义增强** | 从简单的读写原子性升级为完整的获取/释放语义 |
| **final 字段安全发布** | 正确构造的对象的 final 字段对所有线程可见 |
| **同步动作语义** | synchronized、Thread.start()、Thread.join() 的精确定义 |
| **双重检查锁定修复** | volatile + DCL 模式终于正确 |

## 3. 旧内存模型的问题

JDK 1.4 及之前的内存模型（JLS 第 17 章原始定义）存在严重缺陷：

### 问题 1：volatile 语义过弱

```java
// 旧 JMM 下 volatile 仅保证原子读写，不保证可见性顺序
volatile boolean ready;
int value;

// Thread 1
value = 42;
ready = true;

// Thread 2
if (ready) {
    // 旧 JMM: 可能看到 value == 0！
    // 新 JMM: 保证看到 value == 42 ✓
    System.out.println(value);
}
```

### 问题 2：final 字段不安全

```java
// 旧 JMM 下 final 字段可能被其他线程看到默认值
public class FinalFieldExample {
    final int x;
    public FinalFieldExample() {
        x = 42;
    }
}
// 旧 JMM: 另一个线程可能看到 x == 0！
```

### 问题 3：不可变对象可变

```java
// 旧 JMM 下 String 理论上不是线程安全的
// 因为 final char[] value 可能在构造完成前被观察到
String s = new String("hello");
// 另一个线程可能看到部分构造的 String
```

## 4. Happens-Before 关系

### 定义

如果操作 A **happens-before** 操作 B，则 A 的结果对 B 可见，且 A 的执行顺序排在 B 之前（从内存可见性角度）。

### Happens-Before 规则

```
┌──────────────────────────────────────────────────────────────────┐
│  Happens-Before 规则 (Happens-Before Rules)                       │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 程序次序规则 (Program Order Rule)                               │
│     同一线程中，前面的操作 happens-before 后面的操作                   │
│                                                                  │
│  2. 监视器锁规则 (Monitor Lock Rule)                                │
│     unlock happens-before 后续的 lock                              │
│                                                                  │
│  3. volatile 变量规则 (Volatile Variable Rule)                     │
│     volatile 写 happens-before 后续的 volatile 读                   │
│                                                                  │
│  4. 线程启动规则 (Thread Start Rule)                                │
│     Thread.start() happens-before 新线程的所有操作                   │
│                                                                  │
│  5. 线程终止规则 (Thread Termination Rule)                          │
│     线程的所有操作 happens-before Thread.join() 返回                 │
│                                                                  │
│  6. 中断规则 (Interruption Rule)                                    │
│     thread.interrupt() happens-before 被中断线程检测到中断           │
│                                                                  │
│  7. 终结器规则 (Finalizer Rule)                                     │
│     构造函数结束 happens-before finalizer 开始                      │
│                                                                  │
│  8. 传递性 (Transitivity)                                          │
│     A hb B, B hb C → A hb C                                      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 传递性示例

```java
// Thread 1
x = 42;                  // (1)
volatile_flag = true;     // (2)  (1) hb (2) 程序次序

// Thread 2
if (volatile_flag) {      // (3)  (2) hb (3) volatile 规则
    assert x == 42;       // (4)  (3) hb (4) 程序次序
}
// 传递性: (1) hb (2) hb (3) hb (4)，所以 x == 42 成立
```

## 5. volatile 语义

### 新 JMM 下的 volatile

```java
// volatile 读 = 普通读 + LoadLoad 屏障 + LoadStore 屏障
// volatile 写 = StoreStore 屏障 + 普通写 + StoreLoad 屏障

volatile boolean flag;
int a, b, c;

// Thread 1 (Writer)
a = 1;              // 普通写
b = 2;              // 普通写
c = 3;              // 普通写
flag = true;        // volatile 写: 刷新上面所有写入到主存

// Thread 2 (Reader)
if (flag) {         // volatile 读: 从主存加载
    // 保证看到 a==1, b==2, c==3
    // volatile 写之前的所有写入，对 volatile 读之后的所有读取可见
}
```

### volatile 的正确用途

```java
// 1. 状态标志 (Status Flag)
volatile boolean shutdownRequested;

// 2. 安全发布 (Safe Publication)
volatile Config config;
public void updateConfig(Config newConfig) {
    config = newConfig;  // 所有线程立即可见
}

// 3. 双重检查锁定 (DCL) - 必须用 volatile
private volatile Singleton instance;
```

## 6. final 字段语义

### 安全发布保证

```java
public class SafePublication {
    private final int x;
    private final List<String> items;

    public SafePublication() {
        x = 42;
        items = new ArrayList<>();
        items.add("hello");
    }
    // 构造函数完成后:
    // 1. x == 42 对所有线程可见
    // 2. items 引用及其内容对所有线程可见
    // 3. 无需同步！
}
```

### final 字段语义的关键约束

```java
// ⚠ this 引用不得在构造函数完成前逸出 (escape)
public class BrokenFinal {
    final int x;
    static BrokenFinal instance;

    public BrokenFinal() {
        instance = this;  // ❌ this 逸出! 其他线程可能看到 x == 0
        x = 42;
    }
}

// ✓ 正确做法：构造完成后再发布
public class SafeFinal {
    final int x;
    public SafeFinal() { x = 42; }
}
SafeFinal obj = new SafeFinal();  // 构造完成后再赋值给共享变量
```

## 7. 双重检查锁定修复

### 旧 JMM 下的 DCL 问题

```java
// ❌ JDK 1.4 及之前: 此代码不安全！
class Singleton {
    private static Singleton instance;
    public static Singleton getInstance() {
        if (instance == null) {                 // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {         // 第二次检查
                    instance = new Singleton(); // 可能被重排序！
                    // 实际执行可能是:
                    // 1. 分配内存
                    // 2. 将引用赋给 instance (≠ null)
                    // 3. 调用构造函数
                    // 另一线程在步骤 2 后看到 instance != null
                    // 但对象尚未完全构造!
                }
            }
        }
        return instance;
    }
}
```

### 新 JMM 下的正确 DCL

```java
// ✓ JSR 133 之后: volatile 修复 DCL
class Singleton {
    private static volatile Singleton instance;  // 关键: volatile
    public static Singleton getInstance() {
        Singleton local = instance;              // 局部变量优化
        if (local == null) {
            synchronized (Singleton.class) {
                local = instance;
                if (local == null) {
                    instance = local = new Singleton();
                }
            }
        }
        return local;
    }
}

// ✓ 更简洁: Holder 模式 (利用类加载机制)
class Singleton {
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

## 8. 内存屏障

### 四种内存屏障（Memory Barriers / Fences）

```
┌──────────────────────────────────────────────────────────────────┐
│ 内存屏障类型     │ 作用                                           │
├──────────────────────────────────────────────────────────────────┤
│ LoadLoad         │ 确保前面的读在后面的读之前完成                    │
│                  │ Load1 → LoadLoad → Load2                      │
├──────────────────────────────────────────────────────────────────┤
│ StoreStore       │ 确保前面的写在后面的写之前刷出                    │
│                  │ Store1 → StoreStore → Store2                   │
├──────────────────────────────────────────────────────────────────┤
│ LoadStore        │ 确保前面的读在后面的写之前完成                    │
│                  │ Load1 → LoadStore → Store2                     │
├──────────────────────────────────────────────────────────────────┤
│ StoreLoad        │ 确保前面的写对后面的读可见 (最重量级)              │
│                  │ Store1 → StoreLoad → Load2                     │
│                  │ 通常由 mfence / lock 前缀实现 (x86)             │
└──────────────────────────────────────────────────────────────────┘
```

### 不同处理器架构的屏障需求

| 操作 | x86/x64 | ARM/POWER |
|------|---------|-----------|
| LoadLoad | 不需要 (TSO 保证) | 需要 (dmb ld) |
| StoreStore | 不需要 (TSO 保证) | 需要 (dmb st) |
| LoadStore | 不需要 | 需要 |
| StoreLoad | 需要 (mfence) | 需要 (dmb ish) |

x86 是**强内存序（Total Store Order）**，大部分屏障是空操作（no-op），仅 StoreLoad 需要实际屏障。ARM/POWER 是弱内存序，所有屏障都需要显式指令。

## 9. 对 Java 并发编程的影响

### 奠基 java.util.concurrent

JSR 133 的内存模型定义使 Doug Lea 的 `java.util.concurrent`（JSR 166）成为可能：

| 组件 | JMM 依赖 |
|------|----------|
| **ReentrantLock** | volatile state 字段 + CAS 提供 happens-before |
| **ConcurrentHashMap** | volatile 读保证可见性 |
| **AtomicInteger** | volatile + CAS (compareAndSet) |
| **CountDownLatch** | volatile state 的 acquire/release 语义 |
| **ForkJoinPool** | 工作窃取依赖 volatile 队列指针 |

### 安全发布模式

```java
// 1. volatile 字段
volatile Object shared;

// 2. synchronized 块
synchronized(lock) { shared = new Object(); }

// 3. final 字段 (JSR 133 新增保证)
final Object shared = new Object();

// 4. Atomic 引用
AtomicReference<Object> shared = new AtomicReference<>();

// 5. static initializer (类加载保证)
static final Object shared = new Object();
```

## 10. 与其他 JSR/JEP 的关系

### 直接关联

| JSR/JEP | 说明 |
|---------|------|
| JSR 166 | java.util.concurrent，基于 JMM 构建 |
| JEP 188 | JDK 9 VarHandle，替代 sun.misc.Unsafe 的内存访问 |
| JEP 193 | Variable Handles，精细的内存序控制 |

### 后续演进

| 版本 | 特性 | 说明 |
|------|------|------|
| JDK 9 | VarHandle | `MethodHandles.lookup()` 精细内存序 |
| JDK 9 | `@Contended` → `@jdk.internal.vm.annotation.Contended` | 避免伪共享 |
| JDK 21 | Virtual Threads (JEP 444) | 不改变 JMM 语义，但改变调度模型 |
| JDK 21 | Scoped Values (Preview) | 替代 ThreadLocal 的线程安全共享 |

### VarHandle 内存序选项

```java
// JDK 9+ VarHandle 提供精细的内存序控制
VarHandle vh = MethodHandles.lookup()
    .findVarHandle(MyClass.class, "x", int.class);

vh.set(obj, 1);                    // Plain: 无内存序保证
vh.setOpaque(obj, 1);              // Opaque: 保证原子性
vh.setRelease(obj, 1);             // Release: StoreStore + LoadStore
vh.setVolatile(obj, 1);            // Volatile: 完整屏障
int v = (int) vh.getAcquire(obj);  // Acquire: LoadLoad + LoadStore
```
