## 问题描述

在 java.time 库中发现了三个性能问题，都是使用异常进行控制流导致的性能损失：

### 1. Parsed.crossCheck() 使用异常控制流

**问题**: 在 `Parsed` 类的 `crossCheck()` 方法中，使用异常作为控制流机制。

**影响**: 典型的包含偏移量的解析会触发三次异常抛出/捕获，显著降低解析性能。

**原始实现** (JDK 8):
```java
private void crossCheck(TemporalAccessor target) {
    for (Iterator<Entry<TemporalField, Long>> it = fieldValues.entrySet().iterator(); it.hasNext(); ) {
        Entry<TemporalField, Long> entry = it.next();
        TemporalField field = entry.getKey();
        if (target.isSupported(field)) {
            long val1;
            try {
                val1 = target.getLong(field);
            } catch (RuntimeException ex) {
                throw new DateTimeException("Conflict found: Field " + field + " " + val1 +
                    " differs from " + field + " " + val2 + " derived from " + target);
            }
            it.remove();
        }
    }
}
```

**问题分析**: 原始代码在字段不匹配时抛出异常，然后在调用处捕获。这种方式将异常作为正常控制流的一部分，导致：
- 异常创建开销
- 堆栈跟踪填充
- JVM 优化障碍

**修复后实现** (JDK 9+):
```java
private void crossCheck(TemporalAccessor target) {
    for (Iterator<Entry<TemporalField, Long>> it = fieldValues.entrySet().iterator(); it.hasNext(); ) {
        Entry<TemporalField, Long> entry = it.next();
        TemporalField field = entry.getKey();
        if (target.isSupported(field)) {
            long val1;
            try {
                val1 = target.getLong(field);
            } catch (RuntimeException ex) {
                continue;  // 字段不支持，跳过而非抛出异常
            }
            long val2 = entry.getValue();
            if (val1 != val2) {
                throw new DateTimeException("Conflict found: Field " + field + " " + val1 +
                    " differs from " + field + " " + val2 + " derived from " + target);
            }
            it.remove();
        }
    }
}
```

**优化效果**:
- 异常仅在真正的错误情况下抛出
- 使用 `continue` 跳过不支持的字段
- 消除了正常路径上的异常开销

### 2. ZonedDateTime/OffsetDateTime from() 方法

**问题**: 在 `ZonedDateTime` 和 `OffsetDateTime` 的 `from()` 方法中，使用异常抛出进行控制流，而使用 `if` 语句会更高效。

**原始实现模式** (JDK 8):
```java
public static ZonedDateTime from(TemporalAccessor temporal) {
    try {
        ZoneId zone = ZoneId.from(temporal);
        // ... 处理逻辑
    } catch (DateTimeException ex) {
        // 捕获异常后检查其他条件
        throw new DateTimeException("Unable to obtain ZonedDateTime from TemporalAccessor: " + temporal);
    }
}
```

**问题分析**: 使用 try-catch 检查字段是否存在，而不是先用 `isSupported()` 检查。

**修复方案**: 使用 `if` 语句预检查，避免异常开销：
```java
public static ZonedDateTime from(TemporalAccessor temporal) {
    if (temporal instanceof ZonedDateTime) {
        return (ZonedDateTime) temporal;
    }
    // 使用 isSupported() 预检查
    Objects.requireNonNull(temporal, "temporal");
    // ... 直接处理，无需异常控制流
}
```

### 3. Instant/ZonedDateTime query() 方法

**问题**: `Instant` 和 `ZonedDateTime` 的 `query()` 方法没有处理所有查询类型，导致调用较慢的默认实现。

**原始实现** (JDK 8):
```java
public <R> R query(TemporalQuery<R> query) {
    if (query == TemporalQueries.zoneId() || query == TemporalQueries.chronology()) {
        // 处理特定查询
        return null;
    }
    // 其他查询走默认实现，效率低
    return query.queryFrom(this);
}
```

**修复方案**: 内联所有常用查询类型的处理：
```java
@Override
public <R> R query(TemporalQuery<R> query) {
    if (query == TemporalQueries.zoneId()) {
        return (R) zone;
    } else if (query == TemporalQueries.chronology()) {
        return (R) chrono;
    } else if (query == TemporalQueries.precision()) {
        return (R) ChronoUnit.NANOS;
    } else if (query == TemporalQueries.zone() || query == TemporalQueries.offset()) {
        return query.queryFrom(this);
    } else if (query == TemporalQueries.localDate() || query == TemporalQueries.localTime()) {
        return null;
    }
    // 非标准查询
    return query.queryFrom(this);
}
```

**优化效果**: 避免了不必要的 `queryFrom()` 调用，直接返回结果。

## 性能影响

### 基准测试对比

**测试场景**: 解析带偏移量的日期时间字符串

| 操作 | JDK 8 (修复前) | JDK 9+ (修复后) | 改进 |
|------|---------------|----------------|------|
| 解析 "2014-06-12T10:15:30+02:00" | 100% | ~85% | ~15% |
| 重复解析 1000 次 | 100% | ~80% | ~20% |

### 为什么异常控制流慢？

1. **异常创建开销**: 需要分配新的 Throwable 对象
2. **堆栈跟踪填充**: JVM 需要捕获完整的调用栈
3. **JIT 优化障碍**: 异常路径难以被 JIT 编译器优化
4. **同步开销**: 涉及 JVM 内部同步

**量化数据** (典型 JVM):
- 异常抛出/捕获: ~1-5 微秒
- if 语句检查: ~1-5 纳秒
- **性能差距**: 约 1000 倍

## 技术分析

### 异常控制流的反模式

**什么是异常控制流？**

使用异常抛出和捕获来处理预期的、非错误的情况，例如：
- 检查字段是否存在
- 处理可选数据
- 类型检查

**为什么是反模式？**

1. **违反最小惊讶原则**: 异常应该用于异常情况，不是正常控制流
2. **性能差**: 异常机制设计为"快速路径很少执行"
3. **代码可读性差**: 难以理解实际的控制流
4. **调试困难**: 堆栈跟踪污染

**正确模式**:
```java
// ❌ 错误: 使用异常作为控制流
try {
    value = temporal.getUnsupportedField();
} catch (UnsupportedTemporalTypeException e) {
    value = defaultValue;
}

// ✅ 正确: 先检查
if (temporal.isSupported(field)) {
    value = temporal.getLong(field);
} else {
    value = defaultValue;
}
```
