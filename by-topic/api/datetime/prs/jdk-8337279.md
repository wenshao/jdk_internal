## 概述

**核心目标**: 通过共享 StringBuilder 优化 `DateTimeFormatterBuilder` 中 `InstantPrinterParser.formatTo()` 的性能，减少对象分配。

**关键变更**:
1. 创建 `jdk.internal.util.DateTimeHelper` 工具类
2. 提取 `LocalDate/LocalTime/LocalDateTime` 的 `formatTo()` 方法到 DateTimeHelper
3. 重构 `DateTimeFormatterBuilder.format()` 方法，分离 `currentEra` 和 `beforeCurrentEra` 子方法

---

## 技术背景

### 性能问题

原始 `DateTimeFormatterBuilder$InstantPrinterParser#formatTo` 方法存在以下问题：

1. **代码过大**: codeSize = 444 字节，超过 HotSpot 内联阈值 (325 字节)
2. **多次分配**: 多次创建 StringBuilder 和中间 String 对象
3. **重复逻辑**: 日期时间格式化逻辑与 LocalDateTime 重复

### 基准测试结果

| 场景 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| **instantToString** | 4.734 ops/ms | 5.875 ops/ms | **+24.10%** |

**测试环境**: MacBook M1 Pro

## 方案演进

### 方案 1: 使用 LocalDateTime.toString() (已废弃)

```java
// 初始方案 - 被拒绝
LocalDateTime ldt = LocalDateTime.ofEpochSecond(inSecs, fractionalDigits >= 0 ? 0 : inNano, ZoneOffset.UTC);
return ldt.toString();  // 直接复用 toString()
```

**jodastephen 的反对意见**:
> `fractionalDigits == -1` 用于输出尽可能多的小数位（0-9位），LocalDateTime 无法处理这种情况
>
> `fractionalDigits == -2` 用于输出 0、3、6 或 9 位小数
>
> 注释明确指出 `// use INSTANT_SECONDS, thus this code is not bound by Instant.MAX`，表示 instant 值可能超出 LocalDateTime 范围

### 方案 2: 使用 SharedSecrets/JavaTimeAccess (已废弃)

```java
// SharedSecrets 方案 - 被作者放弃
JavaTimeAccess jta = SharedSecrets.getJavaTimeAccess();
jta.formatToLocalDateTime(sb, ...);
```

**放弃原因**:
> SharedSecrets 是最后手段，不应仅为了性能优化而使用
>
> 代码头部明确警告：
> ```
> Usage of these APIs often means bad encapsulation designs,
> increased complexity and lack of sustainability.
> Use this only as a last resort!
> ```

### 方案 3: 使用 jdk.internal.util.DateTimeHelper (最终方案)

```java
// 最终方案 - 创建新的内部工具类
package jdk.internal.util;

public final class DateTimeHelper {
    private DateTimeHelper() {}  // 私有构造函数

    public static void LocalDate_formatTo(StringBuilder sb, int year, int month, int day) {
        // 从 LocalDate 提取的格式化逻辑
    }

    public static void LocalTime_formatTo(StringBuilder sb, int hour, int minute, int second, int nano) {
        // 从 LocalTime 提取的格式化逻辑
    }

    public static void LocalDateTime_formatTo(StringBuilder sb, int year, int month, int day,
                                              int hour, int minute, int second, int nano) {
        LocalDate_formatTo(sb, year, month, day);
        sb.append('T');
        LocalTime_formatTo(sb, hour, minute, second, nano);
    }
}
```

## 核心优化

### 1. 分离 currentEra/beforeCurrentEra 方法

```java
// 优化前 - 单个方法过大
private void formatTo(Builder buf, long instantSecs, int nanos) {
    // 444 字节的代码...
    if (instantSecs >= -62135596800L) {  // 0000-01-01
        // currentEra 逻辑
    } else {
        // beforeCurrentEra 逻辑
    }
}

// 优化后 - 分离为两个方法
private void formatTo(Builder buf, long instantSecs, int nanos) {
    if (instantSecs >= -62135596800L) {
        formatToCurrentEra(buf, instantSecs, nanos);  // ~240 字节
    } else {
        formatToBeforeCurrentEra(buf, instantSecs, nanos);
    }
}
```

**效果**: codeSize 从 444 减少到 240，低于 325 字节内联阈值

### 2. 重用 DateTimeHelper

```java
// DateTimeFormatterBuilder 中使用
private void formatToCurrentEra(Builder buf, long inSecs, int inNano) {
    // ...
    StringBuilder sb = buf.getCharSeq().getStringBuilder();

    // 直接调用 DateTimeHelper，避免创建中间对象
    DateTimeHelper.LocalDateTime_formatTo(sb, year, month, day, hour, minute, second, nano);

    // ... 处理时区
}
```

## 性能分析

### 为什么分离方法能提升性能？

wenshao 的基准测试显示了一个意外的结果：

```bash
# 仅分离 currentEra/beforeCurrentEra，不做其他修改
Benchmark                       Mode  Cnt  Score   Error   Units
ToStringBench.instantToString  thrpt   15  4.716 ± 0.569  ops/ms  # -0.69%

# 当前版本 (DateTimeHelper)
Benchmark                       Mode  Cnt  Score   Error   Units
ToStringBench.instantToString  thrpt   15  5.713 ± 0.474  ops/ms  # +20.29%
```

**分析**:
1. **内联阈值**: HotSpot 的默认内联阈值是 325 字节
2. **方法大小**: 分离后每个方法 < 325 字节，可以被 JIT 内联
3. **缓存友好**: 更小的方法更适合 CPU 指令缓存

### 内存分配减少

```
优化前:
- DateTimeFormatterBuilder.formatTo()
  - 创建 StringBuilder
  - LocalDateTime.toString() 创建 String
  - 多次字符串拼接
  - 总分配: 3-4 个临时对象

优化后:
- DateTimeHelper.LocalDateTime_formatTo()
  - 共享 StringBuilder
  - 直接写入字符
  - 总分配: 1 个最终 String
```

## DateTimeHelper API 设计

### 完整 API

```java
package jdk.internal.util;

/**
 * Internal utility class for date-time formatting.
 * This class provides shared formatting methods to avoid
 * redundant StringBuilder allocations.
 */
public final class DateTimeHelper {

    // 私有构造函数，防止实例化
    private DateTimeHelper() {}

    // LocalDate 格式化
    public static void LocalDate_formatTo(StringBuilder sb,
                                          int year, int month, int day);

    // LocalTime 格式化
    public static void LocalTime_formatTo(StringBuilder sb,
                                          int hour, int minute, int second, int nano);

    // LocalDateTime 格式化
    public static void LocalDateTime_formatTo(StringBuilder sb,
                                              int year, int month, int day,
                                              int hour, int minute, int second, int nano);
}
```

### 设计考虑

1. **包位置**: `jdk.internal.util` 而非 `jdk.internal.time`
   - j3graham 建议 `jdk.internal.time` 更合适
   - wenshao 回应：等到有两个或更多类时再创建新包

2. **方法命名**: `ClassName_formatTo` 模式
   - 清晰表明来源类
   - 避免静态导入污染

3. **参数传递**: 基本类型而非对象
   - 避免不必要的对象创建
   - 直接访问内部字段

## 经验总结

### 1. 迭代式设计

这个 PR 展示了如何在专家反馈下迭代改进：

```
简单方案 (LocalDateTime.toString)
    ↓ jodastephen 指出问题
中间方案 (SharedSecrets)
    ↓ 作者意识到不合适
最终方案 (DateTimeHelper)
    ↓ 专家批准
```

### 2. OpenJDK 审查流程

- **技术正确性**: jodastephen 确保逻辑正确性
- **代码风格**: RogerRiggs 要求不使用跨包静态导入
- **封装设计**: naotoj 要求 final 类 + 私有构造函数
- **测试完整性**: tier1-3 测试必须通过

### 3. 性能优化原则

1. **测量而非猜测**: 基准测试驱动
2. **理解 JIT**: 内联阈值的重要性
3. **减少分配**: 共享 StringBuilder 是有效模式
