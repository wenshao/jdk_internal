## 背景

Java 中有两种主要的日期格式化方式：

1. **`DateTimeFormatter`** (java.time 包)
2. **`String.format()` / `Formatter`** (传统 API)

对于 ISO 标准日期格式 (`%tF` = `%Y-%m-%d`)，这两种实现在处理**极端年份**时存在不一致：

```java
// DateTimeFormatter 输出（正确）
LocalDate.of(-12345, 10, 3).format(DateTimeFormatter.ISO_DATE)
// 结果: "-12345-10-03"

// Formatter 输出（修复前错误）
String.format("%tF", LocalDate.of(-12345, 10, 3))
// 修复前结果: "922-10-03" (错误！)
// 修复后结果: "-12345-10-03" (正确)
```

**为什么会出现这个问题？**

原 `Formatter` 实现对年份的格式化没有正确处理：
- 负年份（公元前）
- 超过 4 位的正年份（年份 > 9999）

## 问题分析

### 原始实现

```java
// 文件: java/util/Formatter.java
// 行数: ~4490-4500 (修复前)

case DateTime.ISO_STANDARD_DATE: { // 'F' (%Y-%m-%d)
    char sep = '-';
    print(fmt, sb, t, DateTime.YEAR_4, l).append(sep);  // 直接使用 YEAR_4
    print(fmt, sb, t, DateTime.MONTH, l).append(sep);
    print(fmt, sb, t, DateTime.DAY_OF_MONTH_0, l);
    break;
}
```

**问题点分析：**

| 问题 | 原因 | 影响 |
|------|------|------|
| **负年份处理错误** | `YEAR_4` 格式化不支持负号 | 年份 -12345 显示为 922 |
| **超长年份处理错误** | `YEAR_4` 固定 4 位宽度，不支持 `+` 前缀 | 年份 12345 显示为 2345 |
| **历法类型判断缺失** | 没有区分 `IsoChronology` 和其他历法 | 日本历法等非 ISO 历法格式错误 |

### 为什么 `YEAR_4` 不够用？

```java
// DateTime.YEAR_4 的简化逻辑
private void print4DigitYear(int year, StringBuilder sb) {
    // 只处理 4 位，不支持符号位
    sb.append(String.format("%04d", year));
}

// 对于 year = -12345:
String.format("%04d", -12345)  // 结果: "-12345" (5位以上，被截断)
```

## 解决方案

### 核心改动

```java
// 修复后的实现

case DateTime.ISO_STANDARD_DATE: { // 'F' (%Y-%m-%d)
    char sep = '-';

    // 1. 判断历法类型
    ChronoField yearField;
    if (t.query(TemporalQueries.chronology()) instanceof IsoChronology) {
        yearField = ChronoField.YEAR;           // ISO 历法使用 YEAR
    } else {
        yearField = ChronoField.YEAR_OF_ERA;    // 其他历法使用 YEAR_OF_ERA
    }

    // 2. 手动处理年份
    int year = t.get(yearField);
    if (year < 0) {
        sb.append(getMinusSign(l));  // 添加负号（支持本地化）
        year = -year;
    } else if (year > 9999) {
        sb.append('+');              // 超长年份添加正号
    }

    // 3. 格式化年份（至少 4 位，补零）
    sb.append(localizedMagnitude(fmt, null, year, Flags.ZERO_PAD, 4, l))
      .append(sep);

    // 4. 月和日保持不变
    print(fmt, sb, t, DateTime.MONTH, l).append(sep);
    print(fmt, sb, t, DateTime.DAY_OF_MONTH_0, l);
    break;
}
```

### 关键修复点

| 修复点 | 说明 |
|--------|------|
| **历法判断** | 使用 `TemporalQueries.chronology()` 区分 ISO 和非 ISO 历法 |
| **符号处理** | 负年份添加本地化负号，超长正年份添加 `+` |
| **本地化支持** | 使用 `getMinusSign(l)` 而非硬编码 `-` |
| **宽度控制** | 使用 `localizedMagnitude` 确保 4 位最小宽度 |

### 新增辅助方法

```java
// 使用本地化的负号，而非硬编码 '-'
private static char getMinusSign(Locale locale) {
    return locale == null ? '-' : getDecimalFormatSymbols(locale).getMinusSign();
}
```

**为什么需要本地化？**

某些地区的负号不是 `-`：
- 阿拉伯: `U+0661-` (阿拉伯语减号)
- 其他: 可能有特定本地化需求

## 设计决策

### Q1: 为什么不直接使用 `DateTimeFormatter.ISO_DATE`？

> **Reviewer**: 直接委托给 `DateTimeFormatter` 不是更简单吗？

**A**: 这样做会有问题：

1. **性能开销**: `DateTimeFormatter` 创建和解析有额外开销
2. **API 设计**: `Formatter` 需要控制整个格式化流程
3. **一致性**: 其他日期格式化方法也采用类似实现

### Q2: 为什么需要区分 `IsoChronology` 和其他历法？

> **Reviewer**: 历法判断有什么特殊意义？

**A**: 日本历法等非 ISO 历法没有"负年份"概念：

```java
// 日本历法: 明治、大正、昭和...没有"公元前"
JapaneseChronology.INSTANCE.date(1899, 1, 1)
// 格式化为 "Meiji 32-01-01"，而非 "1899-01-01"

// 使用 YEAR_OF_ERA 而非 YEAR，避免负数问题
```

### Q3: 为什么添加 `+` 号？

> **Reviewer**: ISO 8601 标准是否要求超长年份带 `+` 号？

**A**: ISO 8601:2019 (E) 标准 4.3.2 节规定：

> 扩展年份表示法（超过 4 位）应以正号 `+` 开头

```
+12345-10-03  (正确)
12345-10-03   (不符合标准)
```

这是一个 **Bug 修复**，不涉及性能优化。但由于代码复杂度增加，需要验证性能影响：

| 场景 | 修复前 | 修复后 | 差异 |

| 超长年份 | 不适用 | ~165 ns | 修复后才可用 |

## 实际影响场景

### 受益场景

| 场景 | 示例 | 影响 |
|------|------|------|
| **历史日期处理** | 考古学、历史学研究 | 修复前无法正确格式化公元前日期 |
| **天文学计算** | 远期/远昔日期 | 修复前超出 4 位年份会错误 |
| **国际化应用** | 多地区日期格式 | 支持本地化负号 |
| **日志系统** | 统一日期格式 | 确保 `DateTimeFormatter` 和 `Formatter` 一致 |

### 不受影响场景

- 年份在 0-9999 范围内的日期（绝大多数场景）
- 使用 `DateTimeFormatter` 的代码（本来就是正确的）
