## 背景

在 JDK 代码库中，计算整数/长整数转换为字符串后的长度（即数字的位数）是一个常见操作。这个操作在多个地方被重复实现：

1. `Integer.stringSize(int x)` - 在 `Integer` 类中
2. `Long.stringSize(long x)` - 在 `Long` 类中
3. **`DateTimeFormatterBuilder` 中有两个私有副本** ← 与 java.time 直接相关

**改动位置**
```
src/java.base/share/classes/
    ├── java/lang/Integer.java          (删除 stringSize 方法)
    ├── java/lang/Long.java             (删除 stringSize 方法)
    ├── java/time/format/DateTimeFormatterBuilder.java (删除私有副本) ← java.time
    ├── jdk/internal/util/DecimalDigits.java      (新增 stringSize(long) 方法)
    └── ... 其他文件
```

## 问题分析

### 原始实现 - DateTimeFormatterBuilder 中的重复代码

```java
// 文件: java/time/format/DateTimeFormatterBuilder.java
// 修改前: 有两个完全相同的私有 stringSize 方法实现

private static int stringSize(long x) {
    int d = 1;
    if (x >= 0) {
        d = 0;
        x = -x;
    }
    long p = -10;
    for (int i = 1; i < 19; i++) {
        if (x > p)
            return i + d;
        p = 10 * p;
    }
    return 19 + d;
}

// ... 在另一个内部类中还有完全相同的副本
```

### 问题

| 问题 | 影响 |
|------|------|
| **代码重复** | 相同逻辑在 4+ 处维护，容易产生不一致 |
| **维护成本** | 任何改进需要同步修改多处代码 |
| **内存占用** | 重复的字节码增加类文件大小 |

---

## 解决方案

### 核心改动

**1. 在 DecimalDigits 中添加 long 版本**:
```java
// 文件: jdk/internal/util/DecimalDigits.java
public static int stringSize(long x) {
    int d = 1;
    if (x >= 0) {
        d = 0;
        x = -x;
    }
    long p = -10;
    for (int i = 1; i < 19; i++) {
        if (x > p)
            return i + d;
        p = 10 * p;
    }
    return 19 + d;
}
```

**2. DateTimeFormatterBuilder 直接调用 DecimalDigits**:
```java
// 文件: java/time/format/DateTimeFormatterBuilder.java
import jdk.internal.util.DecimalDigits;

// 删除两个私有 stringSize 方法
// 直接调用 DecimalDigits.stringSize(value)
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **单一真实来源** | 所有 `stringSize` 调用统一到 `DecimalDigits` |
| **代码减少** | 删除 ~95 行重复代码 |
| **维护性** | 只需在一处维护 `stringSize` 逻辑 |

## 设计决策

### Q1: 为什么不直接删除 Integer.stringSize 和 Long.stringSize？

**A**: 这两个方法现在是委托方法，但：
1. 可能被外部代码调用（尽管是包私有）
2. JIT 编译器会内联这些委托调用，零性能损失

### Q2: 为什么选择线性搜索而不是二分搜索？

**A**: 从代码注释可知：
> "values are biased heavily towards zero, and therefore linear search wins"

整数在实际应用中大部分是小值（0-100），线性搜索在这些情况下更快。
