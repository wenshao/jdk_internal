## 概述

**核心优化**: 将 `ZoneOffset.MINUTES_15_CACHE` 的实现从 `ConcurrentHashMap<Integer, ZoneOffset>` 改为 `AtomicReferenceArray<ZoneOffset>`，避免 int → Integer 的装箱开销。

**命名变更**: 缓存从 `MINUTES_15_CACHE` 重命名为 `QUARTER_CACHE`（15分钟 = 1/4小时 = 1季度）。

---

## 技术背景

### ZoneOffset 缓存需求

ZoneOffset 需要缓存标准的时区偏移量：

```
时区偏移范围: -18:00 到 +18:00
间隔: 15 分钟

总数 = (18 + 18) * 4 + 1 = 145 个偏移量
```

### 原实现 (ConcurrentHashMap)

```java
// 优化前 - JDK 24
private static final ConcurrentHashMap<Integer, ZoneOffset> MINUTES_15_CACHE;

public static ZoneOffset ofTotalSeconds(int totalSeconds) {
    if (totalSeconds % 900 == 0) {  // 15 * 60 = 900秒
        Integer totalSecs = totalSeconds;  // 装箱!
        ZoneOffset result = MINUTES_15_CACHE.get(totalSecs);
        if (result == null) {
            result = new ZoneOffset(totalSeconds);
            MINUTES_15_CACHE.putIfAbsent(totalSecs, result);
        }
        return result;
    }
    return new ZoneOffset(totalSeconds);
}
```

**问题**:
1. **Integer 装箱**: 每次缓存访问都需要 `Integer.valueOf(totalSeconds)`
2. **对象分配**: 装箱产生大量短生命周期 Integer 对象
3. **HashMap 开销**: 需要计算哈希、处理冲突

---

## 优化方案

### 新实现 (AtomicReferenceArray)

```java
// 优化后 - JDK 25
private static final int QUARTER_HOUR = 15 * 60;
private static final int OFFSET_MILLISECONDS = 18 * 60 * 60;
// 缓存大小 256 (2^8)，支持 & 0xFF 优化
private static final int CACHE_SIZE = 256;
private static final int CACHE_INDEX_OFFSET = 128;

private static final AtomicReferenceArray<ZoneOffset> QUARTER_CACHE =
    new AtomicReferenceArray<>(CACHE_SIZE);

public static ZoneOffset ofTotalSeconds(int totalSeconds) {
    if (totalSeconds % QUARTER_HOUR == 0) {
        int index = (totalSeconds / QUARTER_HOUR) + CACHE_INDEX_OFFSET;
        ZoneOffset result = QUARTER_CACHE.get(index);
        if (result == null) {
            result = new ZoneOffset(totalSeconds);
            if (!QUARTER_CACHE.compareAndSet(index, null, result)) {
                result = QUARTER_CACHE.get(index);
            }
        }
        return result;
    }
    return new ZoneOffset(totalSeconds);
}
```

## 性能优势

### 1. 零装箱

| 操作 | 优化前 | 优化后 |
|------|--------|--------|
| **缓存 Key** | `Integer.valueOf(int)` | 直接使用 int |
| **对象分配** | 每次 get 分配 Integer | 零分配 |

### 2. C2 JIT 优化

**数组边界检查消除**:

```java
// 使用 & 0xFF 确保索引在 [0, 255] 范围内
int index = quarters + 128;
ZoneOffset result = QUARTER_CACHE.get(index & 0xFF);
```

**JIT 生成的机器码**:
```asm
; C2 编译后，消除边界检查
mov rax, [rdi + r8*8]  ; 直接数组访问
; 无需额外的范围检查指令
```

**liach 的解释**:
> We use a size of 256 so C2 can apply array bound check elimination as it knows the resulting value from `& 0xFF` is ensured to be in the range of `[0, 255]`.

### 3. 内存布局

| 数据结构 | 内存占用 | 缓存局部性 |
|----------|----------|-----------|
| **ConcurrentHashMap** | 动态增长，初始 ~16 桶 | 分散 |
| **AtomicReferenceArray** | 固定 256 * 8 = 2048 字节 | 连续 |

## PR 审查讨论

### naotoj 的建议

> **"Maybe you could write a test case for all the quarters that indeed return cached ZoneOffsets?"**

**响应**: 添加了测试用例

```java
@Test
public void test_quarter_cache() throws Exception {
    // 测试所有季度偏移量返回缓存的对象
    for (int hour = -18; hour <= 18; hour++) {
        for (int minute = 0; minute < 60; minute += 15) {
            ZoneOffset offset1 = ZoneOffset.ofHoursMinutes(hour, minute);
            ZoneOffset offset2 = ZoneOffset.ofHoursMinutes(hour, minute);
            assertSame(offset1, offset2);  // 确保是同一对象
        }
    }
}
```

### altrisi 的内存优化建议

> **"Why not have the key be quarters + 72 so the array can be smaller?"**

**wenshao 的回应**:
> **"The size of AtomicReferenceArray is 256, and using `& 0xff` can eliminate array bounds checking."**

**分析**:
- 最小数组大小: 145 (实际需要的偏移量)
- 选择 256 的原因:
  - 2 的幂次方，便于位运算
  - 支持 `& 0xFF` 优化
  - 额外内存开销: (256 - 145) * 8 = 888 字节 (可接受)

### naotoj 的边界修正

> **"The offset range is inclusive, so you may want to test +18 hour as well"**

**修复**: 测试从 `< 18` 改为 `<= 18`

```java
// 修复前
for (int hour = -18; hour < 18; hour++)

// 修复后
for (int hour = -18; hour <= 18; hour++)
```

## 技术细节

### 缓存索引计算

```java
// 偏移量范围: -18:00 到 +18:00
// 季度数: -72 到 +72
// 索引偏移: +128
// 索引范围: 56 到 200 (在 [0, 255] 内)

int totalSeconds = hour * 3600 + minute * 60;
int quarters = totalSeconds / 900;  // 15分钟 = 900秒
int index = quarters + 128;         // 偏移到正数范围

// 使用 & 0xFF 确保安全访问
ZoneOffset result = QUARTER_CACHE.get(index & 0xFF);
```

### CAS 竞争处理

```java
ZoneOffset result = QUARTER_CACHE.get(index);
if (result == null) {
    result = new ZoneOffset(totalSeconds);
    if (!QUARTER_CACHE.compareAndSet(index, null, result)) {
        // CAS 失败，其他线程已设置
        result = QUARTER_CACHE.get(index);
    }
}
```

**优势**:
- 无锁快速路径 (缓存命中时)
- CAS 保证只创建一个 ZoneOffset 实例

## 经验总结

### 性能优化模式

**从通用到专用**:
1. `ConcurrentHashMap` → 通用解决方案
2. `AtomicReferenceArray` → 针对小范围整数的专用方案

**优化权衡**:
- **灵活性**: CHM 支持任意 key，Array 只支持连续范围
- **性能**: Array 零装箱 + 边界检查消除
- **内存**: Array 固定开销 vs CHM 动态增长

### 设计考虑

1. **选择 256 而非 145**:
   - 2 的幂次方
   - 支持位运算优化
   - 额外内存开销可接受

2. **使用 & 0xFF**:
   - 启用 C2 边界检查消除
   - 避免额外的比较指令

3. **命名优化**:
   - `MINUTES_15_CACHE` → `QUARTER_CACHE`
   - 更清晰表达 15 分钟 = 1/4 小时
