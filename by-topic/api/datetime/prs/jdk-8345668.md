## 问题描述

### 性能回归

**根本原因**: JDK-8288723 ("Avoid redundant ConcurrentHashMap.get call in java.time") 的修改导致 `ZoneOffset.ofTotalSeconds()` 性能严重下降。

### 性能对比

| 场景 | JDK 17 | JDK 21 | 回归 |
|------|--------|--------|------|
| **单线程** | 2.46 ns/call | 8.04 ns/call | **+226%** (更慢) |
| **多线程** | 0.31 ns/call | 1.03 ns/call | **+232%** (更慢) |

**测试代码**:
```java
// 100,000,000 次迭代测试
for (int i = 0; i < 100_000_000; i++) {
    if (ZoneOffset.ofTotalSeconds(0) != zoneOffset) {
        throw new RuntimeException();
    }
}
```

## 技术背景

### JDK-8288723: 问题的源头

**标题**: "Avoid redundant ConcurrentHashMap.get call in java.time"

**原始意图**: 优化 ConcurrentHashMap 的使用，避免冗余的 `get()` 调用。

**变更内容**:
```java
// 优化前 - JDK 17
ZoneOffset result = CACHE.get(totalSecs);
if (result == null) {
    result = ZoneOffset.ofTotalSeconds(totalSeconds);
    CACHE.putIfAbsent(totalSecs, result);
}

// 优化后 - JDK 21 (引入回归)
result = CACHE.computeIfAbsent(totalSecs, k -> ZoneOffset.ofTotalSeconds(k));
```

**问题**: `computeIfAbsent()` 虽然避免了两次查找，但在高度竞争情况下引入了锁开销，导致性能反而下降。

---

## 修复方案

### 部分回滚

**解决方案**: 部分回滚 JDK-8288723 的修改，恢复原来的 `get() + putIfAbsent()` 模式。

```java
// 修复后 - JDK 25
public static ZoneOffset ofTotalSeconds(int totalSeconds) {
    if (totalSeconds % 15 == 0) {
        Integer totalSecs = totalSeconds;
        ZoneOffset result = CACHE.get(totalSecs);
        if (result == null) {
            result = ZoneOffset.ofTotalSeconds(totalSeconds);
            CACHE.putIfAbsent(totalSecs, result);
        }
        return result;
    } else {
        return new ZoneOffset(totalSeconds);
    }
}
```

### 性能恢复

修复后的基准测试：

| 操作 | 修复前 | 修复后 | 恢复 |
|------|--------|--------|------|
| 1,000 次调用 | 3,946 ns | 2,241 ns | **+43%** |

## 用户影响

### 受影响的应用

1. **高频日志框架**
   - Log4j2, Logback 使用 ZoneOffset 处理时区
   - 每条日志可能调用多次 `ZoneOffset.ofTotalSeconds()`

2. **JSON 序列化**
   - Jackson JavaTimeModule 序列化带时区的日期时间

3. **微服务架构**
   - 多线程环境下性能回归更明显

### 临时解决方案

**JDK 提供的变通方法**:
```java
public static final class RegressionWorkaround {
    public static final ConcurrentHashMap<Integer, ZoneOffset> SECONDS_CACHE = new ConcurrentHashMap<>();

    public static ZoneOffset zoneOffsetOfTotalSeconds(int totalSeconds) {
        if (totalSeconds % (15 * 60) == 0) {
            Integer totalSecs = totalSeconds;
            ZoneOffset result = SECONDS_CACHE.get(totalSecs);
            if (result == null) {
                result = ZoneOffset.ofTotalSeconds(totalSeconds);
                SECONDS_CACHE.putIfAbsent(totalSecs, result);
            }
            return result;
        } else {
            return ZoneOffset.ofTotalSeconds(totalSeconds);
        }
    }
}
```

## 技术教训

### 优化陷阱

1. **理论 vs 实际**
   - `computeIfAbsent()` 理论上更高效
   - 实际上在高竞争场景下锁开销更大

2. **基准测试的重要性**
   - JDK-8288723 缺少充分的基准测试
   - 导致回归在发布后才被发现

3. **缓存策略**
   - ConcurrentHashMap 在读多写少场景下最优
   - `computeIfAbsent` 适合写少、计算昂贵的场景
   - ZoneOffset 创建成本低，不适合 `computeIfAbsent`
