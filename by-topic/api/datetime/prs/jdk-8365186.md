## 背景

### DateTimeFormatter 架构

`java.time.format.DateTimeFormatter` 是日期时间格式化的核心类，其内部使用 `DateTimePrintContext` 作为格式化上下文：

```
DateTimeFormatter
    └── DateTimePrintContext (上下文)
            ├── temporal: TemporalAccessor
            └── formatter: DateTimeFormatter
                    ├── chronology: Chronology (通常为 null)
                    └── zone: ZoneId (通常为 null)
```

每次调用 `format()` 都会创建 `DateTimePrintContext`，其构造函数调用 `adjust()` 方法处理 chronology 和 zone 覆盖。

## 问题分析

### 原始实现 - 单一大方法

```java
// 文件: java/time/format/DateTimePrintContext.java
// 修改前: 382 字节 - 超过 C2 内联阈值

private static TemporalAccessor adjust(TemporalAccessor temporal, DateTimeFormatter formatter) {
    Chronology overrideChrono = formatter.getChronology();
    ZoneId overrideZone = formatter.getZone();

    // 快速路径（常见情况）
    if (overrideChrono == null && overrideZone == null) {
        return temporal;
    }

    // 获取 temporal 的 chronology 和 zone
    Chronology temporalChrono = temporal.query(TemporalQueries.chronology());
    ZoneId temporalZone = temporal.query(TemporalQueries.zoneId());

    // 清除相同的覆盖值
    if (Objects.equals(overrideChrono, temporalChrono)) {
        overrideChrono = null;
    }
    if (Objects.equals(overrideZone, temporalZone)) {
        overrideZone = null;
    }

    // 再次检查
    if (overrideChrono == null && overrideZone == null) {
        return temporal;
    }

    // 执行调整（约 80 行复杂逻辑）
    // ... 处理 INSTANT_SECONDS、创建匿名 TemporalAccessor 等
    return new TemporalAccessor() { /* ... */ };
}
```

### 性能瓶颈诊断

使用 `-XX:+PrintInlining` 发现问题：

```
@ 7   java.time.format.DateTimePrintContext::adjust (382 bytes)
    failed to inline: hot method too big
```

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **方法过大** | 382 字节 > 325 字节阈值 | C2 无法内联 |
| **冷热混合** | 常见路径和边界情况在同一方法 | 热路径优化受限 |

### C2 内联阈值

| 方法类型 | 大小限制 |
|----------|----------|
| **Hot 方法** | ~325 字节 |
| **常规方法** | ~35 字节 |

---

## 解决方案

### 核心改动：方法拆分

将 382 字节拆分为三个方法：

```
adjust (27 字节)           ← 热路径：快速返回
    ↓
adjustWithOverride (123 字节) ← 中等路径：有覆盖
    ↓
adjustSlow (232 字节)      ← 冷路径：复杂边界情况
```

### 代码结构

```java
/**
 * 快速路径 - 27 字节，可被 C2 内联
 */
private static TemporalAccessor adjust(TemporalAccessor temporal, DateTimeFormatter formatter) {
    Chronology overrideChrono = formatter.getChronology();
    ZoneId overrideZone = formatter.getZone();
    if (overrideChrono == null && overrideZone == null) {
        return temporal;  // 90%+ 调用在此返回
    }
    return adjustWithOverride(temporal, overrideChrono, overrideZone);
}

/**
 * 中等路径 - 123 字节，可被 C2 内联
 */
private static TemporalAccessor adjustWithOverride(TemporalAccessor temporal,
                                                   Chronology overrideChrono,
                                                   ZoneId overrideZone) {
    Chronology temporalChrono = temporal.query(TemporalQueries.chronology());
    ZoneId temporalZone = temporal.query(TemporalQueries.zoneId());
    if (Objects.equals(overrideChrono, temporalChrono)) {
        overrideChrono = null;
    }
    if (Objects.equals(overrideZone, temporalZone)) {
        overrideZone = null;
    }
    if (overrideChrono == null && overrideZone == null) {
        return temporal;
    }

    final Chronology effectiveChrono = (overrideChrono != null ? overrideChrono : temporalChrono);
    if (overrideZone != null) {
        if (temporal.isSupported(INSTANT_SECONDS)) {
            Chronology chrono = Objects.requireNonNullElse(effectiveChrono, IsoChronology.INSTANCE);
            return chrono.zonedDateTime(Instant.from(temporal), overrideZone);
        }
    }

    return adjustSlow(temporal, overrideZone, temporalZone, overrideChrono, effectiveChrono, temporalChrono);
}

/**
 * 冷路径 - 232 字节，无需内联
 */
private static TemporalAccessor adjustSlow(
        TemporalAccessor temporal,
        ZoneId overrideZone, ZoneId temporalZone,
        Chronology overrideChrono, Chronology effectiveChrono, Chronology temporalChrono) {
    // 复杂边界情况处理...
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **热路径精简** | `adjust()` 仅 27 字节，涵盖 90%+ 场景 |
| **中等路径** | `adjustWithOverride()` 123 字节，可被 C2 内联 |
| **冷路径隔离** | `adjustSlow()` 处理边界情况，不影响热路径 |
| **死代码消除** | C2 内联后可基于常量优化消除分支 |

## 设计决策

### Q1: 为什么拆分成三个方法而不是两个？

**A**: 三个方法提供更好的分层：

- `adjust` (27 字节): 仅检查 null，90%+ 调用在此返回
- `adjustWithOverride` (123 字节): 处理实际覆盖逻辑
- `adjustSlow` (232 字节): 处理复杂边界情况，极少调用

### Q2: 为什么静态 final 字段很重要？

**A**: 当 `DateTimeFormatter` 是 `static final` 时，JIT 可在编译时确定：
- `formatter.getChronology()` 永远返回 `null`
- `formatter.getZone()` 永远返回 `null`
- 整个方法调用可优化为直接返回 `temporal`
