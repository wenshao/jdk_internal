## 问题描述

### 性能问题

原始的 `LocalDateTime.toString()` 实现存在性能问题：

```java
// 优化前 - JDK 23
@Override
public String toString() {
    return date.toString() + 'T' + time.toString();
}
```

**问题分析**:
1. **中间字符串创建**: `date.toString()` 和 `time.toString()` 各创建一个 String
2. **拼接分配**: `+` 操作创建第三个 String
3. **总分配**: 3 个 String 对象（约 72 字节 + 字符数据）

### 性能基准测试

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| **吞吐量** | 4.896 ops/ms | 13.703 ops/ms | **+179.88%** |
| **分配** | 3 strings | 1 string | -66.67% |

---

## 解决方案

### 优化后的代码

```java
// 优化后 - JDK 24
@Override
public String toString() {
    StringBuilder sb = new StringBuilder(19);  // 预分配容量
    date.toString(sb);  // 直接写入 StringBuilder
    sb.append('T');
    time.toString(sb);  // 直接写入 StringBuilder
    return sb.toString();
}
```

**关键改进**:
1. **预分配容量**: `new StringBuilder(19)` 避免扩容
2. **零中间字符串**: 直接写入 StringBuilder
3. **单次分配**: 最终只创建 1 个 String

### 依赖变更

此优化需要在 `LocalDate` 和 `LocalTime` 中添加新的重载方法：

```java
// 新增方法到 LocalDate
void toString(StringBuilder sb) {
    int yearValue = year;
    int monthValue = month;
    int dayValue = day;

    sb.append(yearValue);
    sb.append('-');
    if (monthValue < 10) {
        sb.append('0');
    }
    sb.append(monthValue);
    sb.append('-');
    if (dayValue < 10) {
        sb.append('0');
    }
    sb.append(dayValue);
}

// 新增方法到 LocalTime
void toString(StringBuilder sb) {
    int hourValue = hour;
    int minuteValue = minute;
    int secondValue = second;
    int nanoValue = nano;

    sb.append(hourValue);
    sb.append(':');
    if (minuteValue < 10) {
        sb.append('0');
    }
    sb.append(minuteValue);
    sb.append(':');
    if (secondValue < 10) {
        sb.append('0');
    }
    sb.append(secondValue);
    if (nanoValue > 0) {
        sb.append('.');
        if (nanoValue % 1000_000 == 0) {
            sb.append((nanoValue / 1000_000) + "000");
        } else if (nanoValue % 1000 == 0) {
            sb.append(((nanoValue / 1000) + "00").substring(0, 6));
        } else {
            sb.append((nanoValue + "00000000").substring(0, 9));
        }
    }
}
```

## 技术细节

### 内存分配对比

#### 优化前
```
date.toString() → String "2024-03-20" (10 chars)
time.toString() → String "10:30:45"   (8 chars)
拼接结果     → String "2024-03-20T10:30:45" (19 chars)

总分配: 3 String 对象 + 37 字符数据
```

#### 优化后
```
StringBuilder(19) → 容量 19，内部 char[19]
最终 toString()   → String "2024-03-20T10:30:45" (19 chars)

总分配: 1 String 对象 + 19 字符数据
```

### 为什么预分配 19？

```java
// LocalDateTime ISO-8601 格式长度计算
// YYYY-MM-DDTHH:MM:SS = 19 字符
// 4 + 1 + 2 + 1 + 2 + 1 + 2 + 1 + 2 + 1 + 2 = 19
// 年 + - + 月 + - + 日 + T + 时 + : + 分 + : + 秒

// 注意：纳秒部分会触发扩容，但这是罕见情况
// 最大长度 = 19 + 1 (点) + 9 (纳秒) = 29
```

## 审核讨论

### jodastephen 的关键评论

> The only case where the year > 9999 is when using the `with()` method to override the year field, as that is outside the range of standard `LocalDate`.

**背景**: ISO-8601 格式对于年份 > 9999 需要特殊处理：
```java
// 年份 > 9999 时，ISO-8601 要求扩展格式
// +YYYY-MM-DD (如 +12024-03-20)
// 这会影响 StringBuilder 预分配容量
```

**影响**: 对于正常范围的年份（-9999 到 9999），优化完全有效。

### liach 的审核意见

> LGTM, looks like a straightforward performance win.

**认可要点**:
- 代码变更简单直接
- 没有行为变化
- 性能提升显著

## 影响分析

### 直接影响
- **所有日志框架**: 使用 LocalDateTime 格式化时间戳
- **JSON 序列化**: Jackson/Gson 序列化 LocalDateTime
- **数据库操作**: JDBC 参数绑定时的字符串转换

### 间接影响
- **GC 压力**: 减少短生命周期对象，降低 Young GC 频率
- **CPU 缓存**: 更好的局部性，更少的内存访问

## 经验总结

### 性能优化原则

1. **避免中间对象**: String 拼接是常见性能杀手
2. **预分配容量**: 了解目标字符串大小，避免扩容
3. **批量写入**: 使用 StringBuilder 的 append 重载

### 代码审查要点

- 关注频繁调用的简单方法（如 toString）
- 测量基准，不要猜测
- 考虑对象分配的完整生命周期
