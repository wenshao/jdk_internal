## 概述

**优化目标**: 将 `java.time` 类中的 month/day 字段从 `int`/`short` 改为 `byte`，减少内存占用。

**结果**: **优化被回退** - 原因是破坏了跨版本序列化兼容性。

**关键问题**: 虽然使用了自定义 Externalizable 编码，但 Java 序列化系统对 `Class` 对象的表示方式导致了不兼容。

---

## 技术背景

### 受影响的类

| 类 | 优化前 | 优化后 |
|----|--------|--------|
| `LocalDate` | `int month, int day` | `byte month, byte day` |
| `YearMonth` | `int month` | `byte month` |
| `MonthDay` | `int day` | `byte day` |
| `HijrahDate` | `int month, int day` | `byte month, byte day` |

### 优化动机

1. **内存减少**: byte (1 字节) vs int (4 字节)
2. **为 Value Classes 预热**: 使 YearMonth/MonthDay < 64 位
3. **JVM 布局优化**: 给 JIT 更多优化空间

### 原始推理

```java
// 这些类使用自定义 Externalizable 编码
// Externalizable 中已经将值存储为 byte
// 因此认为字段类型改变不会影响序列化

public class LocalDate implements Serializable {
    private static final class Ser implements Externalizable {
        // writeExternal 已经将值写成 byte
        public void writeExternal(DataOutput out) throws IOException {
            out.writeInt(date.getYear());
            out.writeByte(date.getMonthValue());  // 已经是 byte
            out.writeByte(date.getDayOfMonth());   // 已经是 byte
        }
    }
}
```

**错误假设**: 因为实例序列化使用 byte，所以字段类型改变不影响兼容性。

## 问题发现

### 序列化不兼容

**症状**: `InvalidClassException` 当在 JDK 21/24 上反序列化 JDK 25 序列化的 Class 对象时。

```java
// 在 JDK 25 上
ObjectOutputStream out = ...;
out.writeObject(LocalDate.class);  // 序列化 Class 对象

// 在 JDK 21 上
ObjectInputStream in = ...;
Class<?> clazz = (Class<?>) in.readObject();  // InvalidClassException!
```

### 关键发现

**问题根源**: Java 序列化系统中 `Class` 对象的表示方式。

```java
// ObjectStreamClass 表示 Class 的序列化形式
class ObjectStreamClass {
    String name;
    long serialVersionUID;
    // ... 对于每个字段:
    String fieldName;
    Class<?> fieldType;  // ← 字段类型必须精确匹配!
}
```

### 兼容性检查规则

反序列化 Class 对象时，检查以下条件：

| 条件 | 要求 |
|------|------|
| 类名 | 匹配 |
| serialVersionUID | 匹配 |
| Serializable/Externalizable 声明 | 一致 |
| **字段类型** | **必须完全匹配** (short != byte) |
| 字段顺序 | 必须一致 |

**关键点**: 即使 SVUID 匹配，字段类型也必须匹配！

## 序列化机制详解

### java.time 的序列化代理模式

```java
public class LocalDate implements Serializable {
    // 序列化代理
    private Object writeReplace() {
        return new Ser(LocalDate.YEAR_MONTH_DAY, this);
    }

    // 反序列化
    private Object readResolve() {
        return LocalDate.of(year, month, day);
    }

    // Ser 是 Externalizable
    private static final class Ser implements Externalizable {
        public void writeExternal(ObjectOutput out) throws IOException {
            out.writeByte(type);
            out.writeInt(year);
            out.writeByte(month);  // 写入 byte
            out.writeByte(day);    // 写入 byte
        }
    }
}
```

### 为什么实例序列化不受影响？

```java
// LocalDate 实例序列化
LocalDate date = LocalDate.now();
oos.writeObject(date);
// 实际写入: Ser(proxy) -> year(int) + month(byte) + day(byte)

// 字段类型改变不影响这个流程，因为:
// 1. writeReplace 替换实例为 Ser
// 2. Ser.writeExternal 使用 byte 写入
// 3. Class 对象不会被写入流 (只写入类型标识符)
```

### 为什么 Class 对象序列化受影响？

```java
// LocalDate.class 序列化
oos.writeObject(LocalDate.class);
// 实际写入: ObjectStreamClass 描述符
// - 类名: "java.time.LocalDate"
// - SVUID: 1 (显式声明的固定值)
// - 字段: month (类型: byte), day (类型: byte)
//   (在旧版本中: month (类型: int), day (类型: int))

// 反序列化时:
// 1. 找到本地类 LocalDate
// 2. 比较流中的字段描述与本地类的字段
// 3. 流中: byte, 本地: int → 不匹配!
// 4. 抛出 InvalidClassException
```

## 技术教训

### 1. 序列化兼容性规则

**错误认知**: "SVUID 匹配就足够了"

**正确理解**: SVUID 只是条件之一，字段类型也必须匹配。

```java
// 即使显式声明 SVUID，字段类型改变仍会破坏兼容性
private static final long serialVersionUID = 1L;

// 以下改变仍会导致不兼容:
// int field → byte field
// String field → Object field
```

### 2. Class 对象 vs 实例序列化

| 类型 | 序列化方式 | 字段类型影响 |
|------|-----------|-------------|
| **实例** | writeReplace/readResolve | 不影响 (使用 Externalizable) |
| **Class 对象** | ObjectStreamClass | **影响** (描述符包含字段类型) |

### 3. 测试覆盖不足

**缺失的测试**:
```java
// 应该有但当时没有的测试:
@Test
public void testClassSerializationCrossVersion() {
    // 序列化 Class 对象
    byte[] data = serialize(LocalDate.class);

    // 模拟旧版本反序列化
    // (需要实际的跨版本测试框架)
}
```

### 4. 审查盲点

**审查时的假设**:
> "这些类使用 Externalizable，字段类型改变不影响"

**应该检查的**:
> "Class 对象的序列化是否受影响？"

## 序列化兼容性策略

### OpenJDK 默认策略

```
"forwards and backwards in perpetuity"
(永久向前和向后兼容)
```

### 检测方法

#### 1. 编译时检查

```bash
# javac --release 选项检查跨版本兼容性
javac --release 21 MyClass.java
```

**局限性**: 只能检查 API，不能检查序列化格式。

#### 2. 运行时测试

**建议方案**:
```java
// 序列化每个公共/受保护的 Class 到文件
// 在旧版本上反序列化，检测差异

@Test
public void crossVersionSerializationTest() {
    Class<?>[] classes = {
        LocalDate.class,
        LocalDateTime.class,
        // ... 其他类
    };

    for (Class<?> clazz : classes) {
        byte[] serialized = serialize(clazz);
        // 在旧版本运行: 反序列化并验证
    }
}
```

#### 3. 归档参考文件


## 经验总结

### 1. 优化 vs 兼容性

| 优化类型 | 兼容性风险 | 建议 |
|----------|-----------|------|
| **内部实现** | 低 | 可以优化 |
| **字段类型** | **高** | 需要严格测试 |
| **公共 API** | 高 | 避免破坏性变更 |

### 2. 序列化设计模式

**Externalizable 的局限性**:
- 保护实例数据
- **不保护** Class 对象的元数据表示

### 3. 测试策略

**必需的测试**:
1. 实例序列化 (已覆盖)
2. **Class 对象序列化** (之前缺失)
3. 跨版本测试 (需要改进)
