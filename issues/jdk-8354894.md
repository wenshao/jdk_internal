## 1. 概述

在特定负载模式下，虚拟线程调度器（ForkJoinPool）出现载体线程（carrier thread）饥饿，导致大量虚拟线程无法被调度执行。该问题在高并发 I/O 密集型场景下尤为严重，表现为应用响应时间飙升甚至完全无响应。

---

## 2. 问题描述

### 症状

```
1. 应用在高并发下响应时间从毫秒级飙升到秒级
2. jstack 显示大量虚拟线程处于 WAITING 状态
3. ForkJoinPool carrier thread 全部被占用
4. 新创建的虚拟线程无法被调度
5. CPU 使用率可能很低（carrier 被阻塞而非忙碌）
```

### 复现场景

```java
// 高并发 I/O 场景：大量虚拟线程同时执行阻塞操作
ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();

// 模拟 10,000 个并发请求
List<Future<String>> futures = new ArrayList<>();
for (int i = 0; i < 10_000; i++) {
    futures.add(executor.submit(() -> {
        // 阻塞式 HTTP 调用
        HttpClient client = HttpClient.newHttpClient();
        HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create("https://api.example.com/data"))
            .build();
        return client.send(request, HttpResponse.BodyHandlers.ofString()).body();
    }));
}
```

### 影响范围

| 场景 | 影响 |
|------|------|
| 微服务网关 | 请求积压，级联超时 |
| 批量 API 调用 | 任务无法完成 |
| 数据库连接池 | 连接等待超时 |
| 高并发爬虫 | 抓取停滞 |

## 3. 根因分析

### 调度器架构

```
┌─────────────────────────────────────────────────────────┐
│                 Virtual Thread Scheduler                  │
│                                                          │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐                   │
│  │Carrier-0│ │Carrier-1│ │Carrier-2│  ... (CPU 核心数) │
│  └────┬────┘ └────┬────┘ └────┬────┘                    │
│       │            │            │                         │
│  ┌────▼────────────▼────────────▼──────────────────┐    │
│  │              ForkJoinPool Task Queue              │    │
│  │   VT-1  VT-2  VT-3  VT-4  ...  VT-10000         │    │
│  └──────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘

问题: 当 carrier thread 被阻塞操作（如 synchronized 块内的 I/O）
     钉住（pin）时，无法调度其他虚拟线程。
```

### 核心问题

**Carrier Thread Pinning（载体线程钉住）**：

当虚拟线程在执行 `synchronized` 方法/块、JNI 调用或 `Object.wait()` 时，无法从载体线程卸载（unmount）。如果大量虚拟线程同时进入这些"钉住"状态，有限的载体线程全部被占用，新虚拟线程无法被调度。

```java
// 问题代码模式：synchronized 导致 pinning
synchronized (lock) {
    // 在 synchronized 块内执行阻塞 I/O
    // → 虚拟线程无法从 carrier thread 卸载
    // → carrier thread 被占用
    socket.getInputStream().read(buffer);
}
```

**二次问题 — 通知机制争用**：

在高并发场景下，`java.util.concurrent.locks.ReentrantLock` 的 `Condition.signal()` 在特定竞争条件下可能丢失通知，导致虚拟线程永久等待。

---

## 4. 解决方案

### JDK 26 修复

JDK 26 包含多项针对虚拟线程调度器的改进：

| 修复 | 描述 |
|------|------|
| **调度器优化** | 改进 carrier thread 的分配策略，减少饥饿概率 |
| **Pinning 监控** | 新增 JFR 事件 `jdk.VirtualThreadPinned` 增强，记录钉住时长和调用栈 |
| **通知修复** | 修复 `Condition.signal()` 在竞争条件下的通知丢失 |

### 用户侧缓解方案

```java
// ✅ 方案 1：替换 synchronized 为 ReentrantLock（可卸载）
private final ReentrantLock lock = new ReentrantLock();

public void processData() {
    lock.lock();
    try {
        // 在 ReentrantLock 下，虚拟线程可以正常卸载
        blockingIO();
    } finally {
        lock.unlock();
    }
}

// ✅ 方案 2：限制并发虚拟线程数
Semaphore semaphore = new Semaphore(500); // 限制并发数
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<?>> futures = tasks.stream()
        .map(task -> executor.submit(() -> {
            semaphore.acquire();
            try {
                return task.call();
            } finally {
                semaphore.release();
            }
        }))
        .toList();
}

// ✅ 方案 3：增加 carrier thread 数量
// 系统属性（谨慎使用）
-Djdk.virtualThreadScheduler.parallelism=16  // 默认为 CPU 核心数
-Djdk.virtualThreadScheduler.maxPoolSize=256 // 默认为 256
```

## 5. 诊断方法

### JFR 监控 Pinning

```bash
# 启动时开启 pinning 事件（阈值 20ms）
java -XX:StartFlightRecording=+jdk.VirtualThreadPinned#threshold=20ms ...
```

### JCMD 诊断

```bash
# 查看虚拟线程状态
jcmd <pid> Thread.vthread_scheduler

# 输出示例:
# Carrier thread count: 8
# Number of mounted virtual threads: 8
# Number of queued virtual threads: 4231  ← 队列积压
```

### 线程转储分析

```bash
jcmd <pid> Thread.dump_to_file -format=text threads.txt

# 查找 pinning 的虚拟线程
grep -A5 "WAITING" threads.txt | grep -B2 "synchronized"
```
