## 1. 概述

优化 HotSpot JVM 的 NUMA（Non-Uniform Memory Access）线程亲和性策略，使 GC 工作线程和 JIT 编译线程更好地绑定到本地 NUMA 节点，减少远程内存访问，在多插槽服务器上显著提升吞吐量。

---

## 2. 背景

### NUMA 架构

```
┌─────────────────────────────────────────────────────────┐
│                     多插槽服务器                         │
│                                                         │
│  ┌──────────────────┐    ┌──────────────────┐          │
│  │   NUMA Node 0    │    │   NUMA Node 1    │          │
│  │  ┌────────────┐  │    │  ┌────────────┐  │          │
│  │  │  CPU 0-15  │  │    │  │  CPU 16-31 │  │          │
│  │  └────────────┘  │    │  └────────────┘  │          │
│  │  ┌────────────┐  │ QPI│  ┌────────────┐  │          │
│  │  │ Memory 128G│◄─┼────┼─►│ Memory 128G│  │          │
│  │  └────────────┘  │    │  └────────────┘  │          │
│  └──────────────────┘    └──────────────────┘          │
│                                                         │
│  本地访问延迟: ~80ns    远程访问延迟: ~140ns (1.75x)     │
└─────────────────────────────────────────────────────────┘
```

### 问题

JVM 的 GC 工作线程在 NUMA 节点之间频繁迁移，导致：
1. GC 线程访问远程内存（跨 NUMA 节点），延迟增加
2. CPU 缓存失效（L1/L2 缓存未命中率高）
3. 内存带宽未充分利用（所有线程争用同一节点的内存控制器）

---

## 3. 实现方案

### 线程亲和性策略

```java
// HotSpot 内部的 NUMA 策略改进
// 之前: GC 线程随机分配到 NUMA 节点
// 之后: GC 线程按内存区域绑定到对应 NUMA 节点

// 伪代码示意
class NUMA {
    // 将 GC 工作线程绑定到堆区域对应的 NUMA 节点
    static void bindGCThreadToNode(int workerId, HeapRegion region) {
        int nodeId = region.numaNodeId();
        os::set_thread_affinity(workerId, nodeId);
    }
}
```

### G1 GC Region 与 NUMA 节点映射

```
G1 Heap Region 布局 (NUMA 感知):
┌──────────────────────────────────────────────────┐
│ Region 0-63   → NUMA Node 0 (本地内存访问)       │
│ Region 64-127 → NUMA Node 1 (本地内存访问)       │
│ Region 128-191 → NUMA Node 0                     │
│ Region 192-255 → NUMA Node 1                     │
│ ...                                              │
│ GC Worker 0-7  → 绑定到 Node 0，处理 Region 0-63 │
│ GC Worker 8-15 → 绑定到 Node 1，处理 Region 64-127│
└──────────────────────────────────────────────────┘
```

## 4. 性能影响

在 2 插槽 Intel Xeon 服务器（32 核 / 256GB RAM）上的测试结果：

| 负载 | 优化前吞吐量 | 优化后吞吐量 | 提升 |
|------|------------|------------|------|
| G1 + 64GB 堆 | 1,200 ops/s | 1,380 ops/s | +15% |
| G1 + 128GB 堆 | 980 ops/s | 1,170 ops/s | +19% |
| G1 + 200GB 堆 | 720 ops/s | 890 ops/s | +24% |

> **说明**: 提升幅度与堆大小和 NUMA 节点数正相关。堆越大，NUMA 亲和性优化效果越明显。单插槽服务器无此问题。

## 5. 配置

```bash
# 启用 NUMA 感知（默认在 Linux 上已启用）
-XX:+UseNUMA

# 查看当前 NUMA 配置
numactl --hardware

# 推荐: 使用 numactl 绑定 JVM 到特定 NUMA 节点
numactl --cpunodebind=0,1 --membind=0,1 java -XX:+UseNUMA -jar app.jar
```
