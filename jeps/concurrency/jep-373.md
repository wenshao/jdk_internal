## 1. 概述

JEP 373 用更简单、更现代的实现替换了 `java.net.DatagramSocket` 和 `java.net.MulticastSocket` 的底层实现。新实现基于 NIO（`java.nio.channels`）基础设施构建，取代了 JDK 15.0 时代遗留的原生代码实现。这是继 JEP 353（JDK 13 中重新实现 Socket/ServerSocket）之后，对 `java.net` 遗留套接字 API 底层实现现代化工作的完成。

---

## 2. 背景

### 2.1 JEP 353 的先例

JDK 15 中的 JEP 353 成功地将 `java.net.Socket` 和 `java.net.ServerSocket` 的底层实现替换为基于 NIO 的版本。该工作验证了用 NIO 通道重新实现遗留套接字 API 的可行性，并为 JEP 373 提供了实施模板。

### 2.2 DatagramSocket 的历史

`DatagramSocket` 自 JDK 1.0 起存在，`MulticastSocket` 自 JDK 1.1 起存在。两者的底层实现（`PlainDatagramSocketImpl`）大量使用 C/C++ 原生代码，跨越多个平台的本地方法实现。这些代码在 20 多年间不断被修补，积累了大量的技术债务。

---

## 3. 旧实现的问题

旧实现（`PlainDatagramSocketImpl`）依赖于大量的平台相关 C/C++ 原生代码，在 Linux、macOS、Windows 上各有独立的本地方法实现。原生代码中混合了套接字选项处理、缓冲区管理和错误处理逻辑，难以理解和修改。经过 20 多年的补丁和功能添加，代码结构变得支离破碎，修复一个平台上的缺陷经常需要在其他平台上进行对应修改。

此外，旧实现中存在固有的线程安全缺陷。多个线程并发访问同一个 `DatagramSocket` 时可能触发竞态条件，特别是在关闭套接字与正在进行的 I/O 操作之间的交互上。原生代码层的锁策略不一致，使得并发场景下的行为难以预测。

| 问题 | 影响 |
|------|------|
| 大量平台相关原生代码 | 维护成本高，修改风险大 |
| 代码结构碎片化 | 新功能开发困难 |
| 线程安全缺陷 | 并发场景下行为不可预测 |
| 调试困难 | 原生层问题排查需跨越 Java/C 边界 |

---

## 4. 新实现方案

### 4.1 基于 NIO 的架构

新实现使用 `java.nio.channels.DatagramChannel` 作为底层传输机制。`DatagramSocket` 和 `MulticastSocket` 的公共 API 保持不变，但内部实现委托给 NIO 通道完成实际的网络 I/O 操作。

```
┌─────────────────────────────┐
│  DatagramSocket (公共 API)   │
│  MulticastSocket (公共 API)  │
├─────────────────────────────┤
│  新实现层 (Java 代码)         │
│  委托给 DatagramChannel      │
├─────────────────────────────┤
│  NIO 基础设施                │
│  (Selector, Channel 等)     │
└─────────────────────────────┘
```

### 4.2 实现优势

新实现完全用 Java 编写（委托给已有的 NIO 通道），消除了对独立原生代码的依赖：DatagramSocket 和 DatagramChannel 共享同一套底层实现，减少重复代码；修复 NIO 层的缺陷自动惠及遗留 API；NIO 通道的线程安全模型更加清晰一致；纯 Java 代码栈更容易使用标准工具进行调试和分析。

## 5. 回退机制

为了降低迁移风险，JDK 15 提供了系统属性来切换回旧的实现：

```bash
java -Djdk.net.usePlainDatagramSocketImpl ...
java -Djdk.net.usePlainDatagramSocketImpl=true ...
```

该属性允许在发现新实现存在兼容性问题时快速回退，无需更改应用代码。与 JEP 353 中 Socket/ServerSocket 的回退机制类似，该系统属性被设计为临时性措施，旧实现在后续 JDK 版本中会被移除。

---

## 6. 与虚拟线程的兼容性

JEP 373 的一个重要动机是为虚拟线程（Project Loom，后来的 JEP 425/444）做准备。旧的原生实现在执行阻塞 I/O 时会固定（pin）底层平台线程，无法与虚拟线程的调度模型配合。基于 NIO 的新实现能够在底层使用非阻塞 I/O，当在虚拟线程中执行 `DatagramSocket.receive()` 等阻塞操作时，虚拟线程可以被调度器卸载（unmount），释放平台线程去执行其他任务。这对于构建高并发 UDP 服务至关重要。

---

## 7. 对开发者的影响

### 7.1 API 兼容性

新实现保持了 `DatagramSocket` 和 `MulticastSocket` 的完整公共 API 不变。现有代码无需修改即可在 JDK 15 上运行。

### 7.2 行为差异与自定义实现

在极少数边缘情况下，新实现的行为可能与旧实现略有不同，特别是在错误处理和异常消息方面。如果应用程序依赖于旧实现的特定未文档化行为，可以使用 `jdk.net.usePlainDatagramSocketImpl` 进行回退。依赖 `DatagramSocketImplFactory` 或自定义 `DatagramSocketImpl` 子类的应用程序也需要注意内部委托机制的变化，应优先迁移到标准 API。
