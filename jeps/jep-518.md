## 1. 背景与动机

### 问题

传统 JFR 采样使用信号中断线程：
- 信号处理开销
- 安全点影响
- 采样偏差

### 解决方案

JEP 518 引入协作式采样：
- 线程主动检查采样点
- 减少信号开销
- 更准确的采样

---

## 2. 技术细节

### 采样机制

```
传统采样:
信号 → 中断线程 → 记录栈 → 恢复

协作式采样:
线程执行 → 检查采样点 → 记录栈 → 继续
```

### 实现位置

```cpp
// 在解释器和编译代码中插入采样点
// src/hotspot/cpu/*/interp_masm_*.cpp

// 解释器入口点检查
void check_sample_point() {
    if (should_sample()) {
        record_stack();
    }
}
```

---

## 3. 开发者影响

### 性能改进

```
采样开销:
传统:  3-5%
协作式: 1-2%

采样准确性:
传统:  有偏差（信号延迟）
协作式: 更准确
```

### 配置

```bash
# 默认启用协作式采样
java -XX:StartFlightRecording MyApp

# 禁用（使用传统采样）
java -XX:-JFRCooperativeSampling -XX:StartFlightRecording MyApp
```
