# JEP 500: Prepare to Make Final Mean Final

> **状态**: 🗑️ 移除 (JDK 25) | **JEP 链接**: https://openjdk.org/jeps/500  
> **状态**: 正式发布  
> **Commit**: `26460b6f12c`  
> **作者**: Alan Bateman  
> **Reviewer**: liach, vlivanov, dholmes, vyazici

---

## 1. 背景与动机

### 问题

Java 的 `final` 字段本应不可变，但存在多种绕过方式：

```java
public class Constants {
    public final int VALUE = 42;
}

// 通过反射修改
Field field = Constants.class.getDeclaredField("VALUE");
field.setAccessible(true);
field.set(constants, 100);  // 成功修改！

// 通过 JNI 修改
// JNI 可以直接修改内存
```

这导致：
- **安全风险**: 敏感数据可被篡改
- **优化障碍**: JVM 无法假设 final 字段不变
- **语义混乱**: final 不代表真正不可变

### 解决方案

JEP 500 分阶段限制 final 字段修改：
1. **JDK 26**: 默认禁止，提供迁移选项
2. **未来版本**: 完全禁止

## 2. 使用示例

### 新增命令行选项

```bash
# 默认行为：限制 final 字段修改
java MyApp

# 允许修改（迁移模式）
java --finalization=enabled MyApp

# 严格模式（未来默认）
java --finalization=disabled MyApp
```

### 受影响的操作

```java
// ❌ 默认禁止
Field field = MyClass.class.getDeclaredField("FINAL_VALUE");
field.setAccessible(true);
field.set(obj, newValue);  // 抛出 IllegalAccessException

// ❌ 默认禁止
// JNI SetObjectField / SetIntField 等
```

### JFR 事件

```java
// 当 final 字段被修改时，触发 JFR 事件
// jdk.FinalFieldMutation

// 查看 JFR 录制中的事件
jfr print --events jdk.FinalFieldMutation recording.jfr
```

## 3. 实现分析

### 新增 API

```java
// java.lang.reflect.Field
public class Field {
    // 新增方法：检查是否可以修改 final 字段
    public boolean canModifyFinalField();
    
    // 修改行为：set 方法检查 final 限制
    @Override
    public void set(Object obj, Object value) throws IllegalAccessException {
        // 检查 --finalization 选项
        if (isFinal() && !canModifyFinalField()) {
            throw new IllegalAccessException(
                "Cannot modify final field: " + getName());
        }
        // ... 原有逻辑
    }
}
```

### 模块系统变更

```java
// java.lang.Module
public class Module {
    // 新增：声明模块需要修改 final 字段的能力
    public Module addUses(Class<?> service);
    
    // 新增：检查模块是否有修改 final 字段的权限
    public boolean canModifyFinalFields();
}
```

### JVM 层面

```cpp
// src/hotspot/share/runtime/arguments.cpp
// 新增命令行选项
product(ccstr, FinalFieldModification, "disabled", \
        "Control final field modification: disabled, enabled")

// src/hotspot/share/prims/jni.cpp
// JNI 字段访问检查
if (field->is_final() && !FinalFieldModificationEnabled) {
    // 拒绝修改
}
```

## 4. 开发者影响

### 迁移指南

```java
// 旧代码：依赖修改 final 字段
public class LegacyCode {
    private final int config;
    
    void updateConfig(int newConfig) throws Exception {
        Field f = getClass().getDeclaredField("config");
        f.setAccessible(true);
        f.set(this, newConfig);  // JDK 26 默认失败
    }
}

// 迁移方案 1：移除 final
public class MigratedCode {
    private int config;  // 移除 final
    // ...
}

// 迁移方案 2：使用 AtomicReference
public class MigratedCode2 {
    private final AtomicReference<Integer> config = new AtomicReference<>();
    // ...
}

// 迁移方案 3：临时启用修改（不推荐）
// 启动时添加 --finalization=enabled
```

### 兼容性检查

```bash
# 检查代码是否依赖 final 字段修改
# 查找 Field.setAccessible + Field.set 组合

# 使用 jdeps 分析
jdeps --final-field-access myapp.jar

# 运行时检查
java --finalization=disabled MyApp
# 如果有修改尝试，会抛出异常
```

### 最佳实践

```java
// ✅ 推荐：真正不可变
public final class ImmutableConfig {
    private final int value;
    
    public ImmutableConfig(int value) {
        this.value = value;
    }
    
    public int getValue() { return value; }
    
    // 需要新值时创建新对象
    public ImmutableConfig withValue(int newValue) {
        return new ImmutableConfig(newValue);
    }
}

// ❌ 避免：依赖反射修改 final
// 这种代码在 JDK 26 会失败
```

## 5. 分阶段计划

```
JDK 26 (2026)
├── 默认：禁止 final 字段修改
├── 选项：--finalization=enabled 允许修改
└── JFR：记录修改尝试事件

JDK 27+ (未来)
├── --finalization=enabled 继续可用
└── 警告：将在未来版本移除

JDK 未来版本
└── 完全禁止 final 字段修改
```
