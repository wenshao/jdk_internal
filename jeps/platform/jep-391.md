## 1. 概述

JEP 391 为 macOS/AArch64 平台提供原生 JDK 移植，使 Java 能够在 Apple Silicon（M1 及后续芯片）上以原生 ARM64 模式运行，无需依赖 Rosetta 2 x86_64 转译层。该移植作为 JDK 17 的正式功能交付，标志着 Java 平台对 Apple 新一代硬件架构的全面支持。

该移植建立在已有的 Linux/AArch64 基础之上，但需要大量 macOS 平台特定的适配——涉及 Mach-O 对象格式、W^X 内存保护、信号处理和 JIT 后端等多个层面。

---

## 2. 背景

### 2.1 Apple 的架构迁移

2020 年 6 月，Apple 在 WWDC 上宣布将 Mac 产品线从 Intel x86_64 架构迁移至自研的 ARM 架构（Apple Silicon）。这是 Mac 历史上第三次重大架构迁移（68k → PowerPC → Intel → ARM），Apple 为此提供了 Rosetta 2 转译技术作为过渡方案。

2020 年 11 月，首款搭载 Apple M1 芯片的 Mac 正式发售（ARMv8.5-A / AArch64 架构）。Apple 提供了 Rosetta 2 转译层作为过渡方案，但该方式存在明显不足：

| 方面 | Rosetta 2 转译 | 原生 AArch64 |
|------|---------------|-------------|
| **启动性能** | 首次运行需转译开销 | 无转译开销 |
| **运行时性能** | 约 70-80% 的原生性能 | 100% 原生性能 |
| **JIT 编译** | JIT 生成 x86_64 再转译 | JIT 直接生成 ARM64 |
| **内存占用** | 转译缓存增加内存消耗 | 无额外内存开销 |
| **长期支持** | Apple 可能在未来版本中移除 | 平台原生支持 |

### 2.3 已有的 AArch64 基础

Linux/AArch64 移植早在 JDK 9 时期就已纳入主线（JEP 237），经过多年发展已经成熟稳定。这意味着 HotSpot 的 AArch64 JIT 后端（C1 和 C2 编译器）、解释器、GC 屏障等核心组件已经存在，JEP 391 的工作重点在于 macOS 平台特定的适配，而非重新实现 AArch64 架构支持。

## 3. 技术挑战

### 3.1 Mach-O 对象格式

macOS 使用 Mach-O 格式而非 Linux 上的 ELF 格式。JDK 的本地库加载、调试信息生成等模块都需要适配 Mach-O 的段布局、符号表格式和重定位类型。

### 3.2 W^X 内存保护策略

Apple Silicon 上的 macOS 强制执行 W^X（Write XOR Execute）策略——同一内存页不能同时具有写入和执行权限。这对 JIT 编译器构成直接挑战：Linux/AArch64 上 JIT 可使用 RWX 页面，而 macOS/AArch64 要求使用 `pthread_jit_write_protect_np()` API 切换模式，并通过 `MAP_JIT` 标志分配 JIT 内存。

### 3.3 内存模型差异

Apple M1 实现了 TSO（Total Store Order）内存模型，与 x86 一致但与标准 ARM 弱内存模型不同。尽管 TSO 意味着部分内存屏障可省略，但 JDK 保留了标准 ARM 内存屏障以确保在未来非 TSO ARM 设备上的正确性。

### 3.4 信号处理机制

macOS 使用 Mach 异常端口机制而非 Linux 的 POSIX 信号处理。HotSpot 依赖信号处理实现隐式空指针检查、安全点轮询和栈溢出检测等功能，需要在 macOS/AArch64 上正确适配。

---

## 4. 关键变更

### 4.1 JIT 编译器后端适配

C1 和 C2 编译器的 AArch64 后端需要针对 macOS 的 W^X 策略修改：使用 `MAP_JIT` 分配代码缓存，通过 `pthread_jit_write_protect_np()` 切换写入/执行模式，编译完成后刷新指令缓存（`sys_icache_invalidate()`），去优化和内联缓存更新等运行时补丁同样需遵循 W^X 协议。

### 4.2 信号处理层

适配 macOS 的 Mach 异常端口机制：注册异常处理端口将硬件异常转换为 JVM 事件，适配 `ucontext_t` 结构差异，确保安全点在 macOS 异常模型下正确触发。

### 4.3 本地调用约定与构建系统

macOS/AArch64 的 C 调用约定与 Linux/AArch64（AAPCS64）存在细微差异（栈对齐、可变参数传递等），JNI 调用和解释器入口点需要适配。构建系统方面，需识别新平台并配置代码签名——Apple 要求所有 AArch64 二进制文件必须签名。

## 5. 性能对比

原生 macOS/AArch64 JDK 相较于通过 Rosetta 2 运行 x86_64 JDK，在各项基准测试中均展现出显著的性能优势：

| 基准测试场景 | Rosetta 2 (x86_64) | 原生 AArch64 | 提升幅度 |
|------------|-------------------|-------------|---------|
| **JVM 启动时间** | 基准线 | 约快 40-50% | 显著提升 |
| **SPECjbb 吞吐量** | 基准线 | 约快 20-30% | 明显提升 |
| **编译构建（如 Maven）** | 基准线 | 约快 25-35% | 明显提升 |
| **GC 暂停时间** | 基准线 | 约短 15-25% | 可观提升 |

性能提升主要来源于消除 Rosetta 2 指令转译开销、JIT 直接生成 ARM64 原生指令以及充分利用 M1 硬件特性。

## 6. 社区协作

### 6.1 多方参与的移植工作

macOS/AArch64 移植是 OpenJDK 社区多方协作的成果。Oracle 主导了 JEP 391 的开发，但多个组织贡献了关键工作：

| 组织 | 贡献领域 |
|------|---------|
| **Oracle** | JEP 391 lead，HotSpot 核心适配 |
| **Microsoft** | Windows/AArch64 移植（JEP 388），共享 AArch64 基础设施改进 |
| **SAP** | 早期 macOS/AArch64 探索性移植工作 |
| **Azul Systems** | Zulu 发行版的 macOS/AArch64 早期支持 |
| **Red Hat** | AArch64 后端维护和优化 |

### 6.2 与 JEP 388 的协同

Microsoft 主导的 JEP 388（Windows/AArch64 Port，JDK 16）与 JEP 391 几乎同期推进，两者共享了大量 AArch64 平台层的改进和修复。在 JEP 391 正式立项前，社区成员已通过非官方渠道提供实验性构建，验证了移植的可行性。

---

## 7. 对开发者的影响

对于 Java 应用开发者而言，从 macOS/x86_64 迁移到 macOS/AArch64 几乎是透明的——相同的 class 文件和 JAR 包无需修改即可运行。需要注意的是：依赖 x86_64 本地库（JNI）的应用需要获取 AArch64 版本或构建 Universal Binary；某些 JVM 调优参数（如线程栈大小）可能需要针对新平台重新评估。
