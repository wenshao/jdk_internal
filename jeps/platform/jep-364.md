## 1. 概述

JEP 364 将 Z Garbage Collector（ZGC）扩展至 macOS 平台。ZGC 最初在 JDK 14 中作为实验性功能引入，但仅支持 Linux。将其移植到 macOS 使开发者能够在本地开发环境中使用与生产环境一致的 GC，消除了跨平台开发时的行为差异。该移植是 ZGC 在 JDK 15 中正式转为生产特性（JEP 377）的重要前提。

---

## 2. 背景

### 2.1 ZGC 的核心设计

ZGC 是一种可伸缩的低延迟垃圾收集器，设计目标为：GC 暂停时间不超过 10ms，暂停时间不随堆大小增长，支持 TB 级别的堆。其核心机制依赖于彩色指针（colored pointers）和内存多重映射（multi-mapping）。

### 2.2 Linux 限定的问题

ZGC 在 JDK 11-13 中仅支持 Linux，这意味着大量使用 macOS 进行日常开发的 Java 开发者无法在本地测试 ZGC 行为。开发与生产环境的 GC 不一致可能导致难以复现的问题。

---

## 3. 技术挑战

### 3.1 内存多重映射

ZGC 的核心机制要求将同一物理内存映射到多个虚拟地址空间。Linux 上使用 `mmap` 配合共享文件描述符实现。macOS 不支持相同的方式，需要寻找替代方案。

### 3.2 macOS 虚拟内存限制

macOS 的虚拟内存管理与 Linux 存在差异：`mmap` 对匿名内存的多重映射支持有限，大页（huge pages）的支持机制不同，地址空间布局和保留策略也有所区别。

---

## 4. 关键变更

### 4.1 基于 mach_vm_remap 的多重映射

macOS 移植的核心创新是使用 Mach 内核的 `mach_vm_remap` API 实现内存多重映射：

| 平台 | 多重映射实现 |
|------|------------|
| **Linux** | `mmap` + `memfd_create` 共享文件描述符 |
| **macOS** | `mach_vm_remap` 将同一内存区域映射到多个虚拟地址 |

`mach_vm_remap` 允许在不同虚拟地址创建对同一物理页面的映射，满足了 ZGC 彩色指针对多重映射的需求。

### 4.2 平台抽象层重构

为支持多平台，ZGC 的内存管理代码进行了重构，将平台特定逻辑隔离到独立的后端模块中。这种分层设计也为后续的 Windows 移植（JEP 365）奠定了基础。

### 4.3 信号处理适配

macOS 上的信号处理机制与 Linux 不同。ZGC 依赖 SEGV 信号处理实现加载屏障的慢路径，需要在 macOS 的 Mach 异常处理框架下正确工作。

---

## 5. 实现细节

### 5.1 代码结构

macOS 特定的 ZGC 实现代码位于 `src/hotspot/os/bsd/gc/z/` 目录下，主要包括：内存预留与提交的 macOS 实现、`mach_vm_remap` 多重映射封装以及 macOS 大页支持的适配层。

### 5.2 测试覆盖

macOS 移植复用了 ZGC 现有的测试套件，确保在 macOS 上的行为与 Linux 一致。CI 系统新增 macOS ZGC 测试目标。

---

## 6. 社区影响

macOS 上的 ZGC 支持使开发者能够在日常开发中使用 `-XX:+UseZGC` 参数，与生产环境保持一致。这对于调试和性能调优尤为重要，开发者无需部署到 Linux 环境即可验证 ZGC 相关行为。
