## 1. 概述

JEP 388 为 Windows/AArch64 平台提供原生 JDK 移植，使 Java 能够在基于 ARM64 处理器的 Windows 设备上原生运行。该移植由 Microsoft 主导贡献，是 Microsoft 加入 OpenJDK 社区后的重要成果之一。JEP 388 在 JDK 16 中交付，为 Surface Pro X 等 Windows on ARM 设备以及 ARM 服务器上的 Windows 部署提供了 Java 支持。

---

## 2. 背景

### 2.1 Windows on ARM 生态

Microsoft 自 2017 年开始推动 Windows on ARM 平台，推出了多款基于 Qualcomm Snapdragon 处理器的设备：

| 设备类别 | 代表产品 | 处理器 |
|---------|---------|--------|
| **笔记本/平板** | Surface Pro X | Microsoft SQ1/SQ2 (Qualcomm) |
| **笔记本** | Lenovo Flex 5G | Qualcomm Snapdragon 8cx |
| **服务器** | Ampere Altra 运行 Windows Server | Ampere Altra (ARM64) |

在 JEP 388 之前，Windows/AArch64 上的 Java 只能通过 x86 模拟层运行 x86_64 版本的 JDK，存在显著的性能损耗。

Microsoft 在 2019 年正式加入 OpenJDK 社区，JEP 388 是 Microsoft 对上游的首个重大平台贡献。

---

## 3. 技术挑战

### 3.1 Windows 平台层与 AArch64 的交叉

JDK 已分别支持 Windows/x86_64 和 Linux/AArch64，但 Windows/AArch64 是两者的交叉组合，需要将 Windows 平台层代码与 AArch64 架构后端正确结合。Windows 的异常处理（SEH）、线程模型和内存管理 API 需要与 AArch64 架构特性协调。

### 3.2 Windows SEH 与 AArch64

Windows 的结构化异常处理（Structured Exception Handling）在 AArch64 上的实现与 x86_64 不同。HotSpot 利用硬件异常实现隐式空指针检查等优化，需要在 Windows/AArch64 的 SEH 框架下正确适配。

### 3.3 调用约定差异

Windows/AArch64 的 C 调用约定与 Linux/AArch64（AAPCS64）在可变参数传递和栈帧布局方面存在差异，JNI 调用桥接和解释器入口点需要适配。

---

## 4. 关键变更

### 4.1 HotSpot 平台层

新增 `src/hotspot/os_cpu/windows_aarch64` 目录，包含 Windows/AArch64 特定的信号处理、上下文切换和栈遍历代码。将已有的 Windows 平台代码和 AArch64 架构代码在此层面结合。

### 4.2 JIT 编译器适配

C1 和 C2 编译器的 AArch64 后端在 Windows 上需要适配异常处理和代码生成的细微差异。Windows 的代码签名和 DEP（Data Execution Prevention）机制需要 JIT 生成的代码符合 Windows 安全要求。

### 4.3 构建系统与本地库

新增 `windows-aarch64` 构建配置，适配 MSVC 的 ARM64 交叉编译工具链。确保 JDK 依赖的本地库（如 zlib、libpng、FreeType）在 Windows/AArch64 上正确编译和链接。

---

## 5. 设备生态

JEP 388 支持 Surface Pro X 等 Windows on ARM 桌面设备上的 Java 开发与运行，ARM64 Windows Server 上的云工作负载，以及 Windows/AArch64 CI/CD 构建节点。
