## 1. 概述

JEP 386 将 JDK 移植到 Alpine Linux 及其他使用 musl libc 的 Linux 发行版上。musl 是一个轻量级的 C 标准库实现，与 glibc 相比体积更小、启动更快，是 Alpine Linux 的默认 C 库。该移植使 Java 应用能够运行在极小的容器镜像中，对云原生和微服务场景具有重大意义。

---

## 2. 背景

### 2.1 Alpine Linux 与容器化

Alpine Linux 是一个面向安全和轻量的 Linux 发行版，在容器生态中被广泛采用。其核心优势在于极小的镜像体积：

| 基础镜像 | 体积 |
|---------|------|
| **Alpine Linux** | 约 5 MB |
| **Debian slim** | 约 80 MB |
| **Ubuntu** | 约 75 MB |
| **CentOS** | 约 200 MB |

### 2.2 musl vs glibc

Alpine Linux 使用 musl libc 而非 glibc。musl 是 C 标准库的另一个实现，设计目标是正确性、简洁性和静态链接友好。JDK 历来仅支持 glibc，因为 HotSpot 依赖了部分 glibc 特有的行为和扩展。

---

## 3. 技术挑战

### 3.1 线程栈管理差异

musl 和 glibc 在线程栈管理上存在重要差异。glibc 使用 `pthread_getattr_np` 获取线程属性，musl 对该扩展的支持不完整。HotSpot 依赖精确的线程栈边界信息进行栈溢出检测和保护页管理，需要针对 musl 进行适配。

### 3.2 动态链接器与系统调用

musl 的动态链接器（`ld-musl`）与 glibc 的 `ld-linux` 在符号解析顺序、`dlopen` 行为和线程局部存储（TLS）实现上存在差异。此外，HotSpot 使用了部分 glibc 特有的扩展函数（如 `malloc_info`、`backtrace`），musl 不提供这些扩展，需要提供替代实现或条件化使用。

---

## 4. 关键变更

### 4.1 musl libc 检测与适配

构建系统新增 musl libc 检测逻辑，通过 `ldd --version` 输出或 `__MUSL__` 宏识别 musl 环境。HotSpot 中条件化使用 glibc 扩展，在 musl 环境下使用标准 POSIX 替代方案。

### 4.2 线程栈处理与构建系统

针对 musl 的线程栈管理差异，HotSpot 修改了栈边界检测逻辑，使用兼容 musl 的方式获取线程栈信息。构建系统新增 `linux-x64-musl` 配置目标，支持在 Alpine Linux 上进行完整的 JDK 构建，并允许在 glibc 系统上交叉编译 musl 版本。

---

## 5. 容器化影响

musl/Alpine 支持对 Java 容器化部署产生了重大影响：

| 方面 | glibc 基础镜像 | musl/Alpine 基础镜像 |
|------|--------------|-------------------|
| **基础镜像体积** | 75-200 MB | 5 MB |
| **JDK 镜像体积** | 约 300-400 MB | 约 100-150 MB |
| **jlink 定制运行时** | 约 100-150 MB | 约 40-60 MB |

对于运行大量 Java 微服务的组织，镜像体积的减少直接转化为存储成本降低和部署速度提升。
