## 1. 概述

JEP 362 在 JDK 14 中将 Solaris/SPARC、Solaris/x64 和 Linux/SPARC 移植标记为
`Deprecated for Removal`——正式宣布这些平台端口已过时，将在未来版本中被移除。

该 JEP 是一个预告性变更，旨在给予依赖这些平台的用户和贡献者充分的迁移时间。
实际的源代码移除在 JDK 15 的 [JEP 381](jep-381.md) 中完成。

### 1.1 废弃的端口

- **Solaris/SPARC** (`os-arch = solaris-sparcv9`)
- **Solaris/x64** (`os-arch = solaris-amd64`)
- **Linux/SPARC** (`os-arch = linux-sparcv9`)

---

## 2. 背景：Solaris 与 SPARC 的地位变迁

### 2.1 曾经的核心平台

Solaris 和 SPARC 曾是 Java 的原生家园。Sun Microsystems 在 SPARC/Solaris 上开发了
最初的 Java 实现，该平台长期作为 Java 参考实现的首选运行环境。在企业级市场中，
SPARC 服务器运行 Solaris 操作系统曾是金融、电信等行业的标准配置。

### 2.2 市场份额的持续下降

进入 2010 年代，x86/x64 架构在服务器市场的主导地位日益巩固。Linux 取代了商业
Unix 系统在数据中心的核心位置。Oracle 于 2017 年宣布 SPARC 处理器停止开发，
Solaris 的更新节奏也大幅放缓。到 JDK 14 发布时，Solaris/SPARC 上的 Java 用户
已极为稀少。

---

## 3. JDK 14 中的变更

### 3.1 构建系统警告

在 JDK 14 中，针对 Solaris 和 SPARC 端口执行 `configure` 时会输出废弃警告，
提示这些端口将在未来版本中被移除。构建仍然可以完成，但需要用户确认知晓废弃状态。

### 3.2 标记受影响代码

Solaris 和 SPARC 相关的平台特定代码（包括 HotSpot、类库和构建脚本中的条件分支）
被标记为待移除状态，便于后续清理。

---

## 4. 为何废弃

### 4.1 维护负担不可持续

维护 Solaris 和 SPARC 端口需要持续投入工程资源：平台特定的 JIT 编译器后端、
操作系统调用适配、测试基础设施等。这些资源本可以用于开发新的 JVM 特性和优化。

### 4.2 阻碍新特性开发

多项正在研发的 JVM 改进（如 Loom 虚拟线程、ZGC 增强、Foreign Function API）
需要大量底层平台适配工作。维护已近乎无人使用的端口显著拖慢了这些新特性的交付速度。

### 4.3 测试基础设施成本

为 Solaris/SPARC 维护 CI/CD 测试环境需要专用硬件和配置，而这些硬件的采购和
维护成本日益攀升，与其产生的价值不成比例。

### 4.4 用户已大规模迁移

绝大多数 Java 用户已迁移至 Linux/x64、macOS/AArch64 或 Windows/x64 平台。
Solaris/SPARC 上的活跃 Java 部署数量极低。

---

## 5. 影响范围

### 5.1 受影响的组件

| 组件 | 影响 |
|------|------|
| HotSpot SPARC 后端 | 标记废弃，待移除 |
| Solaris 特定的 os_solaris 代码 | 标记废弃，待移除 |
| 构建系统的 Solaris/SPARC 支持 | 输出废弃警告 |
| Solaris 特定的 JDK 类库代码 | 标记废弃，待移除 |

### 5.2 不受影响的组件

- Linux/x64、Linux/AArch64 端口正常维护
- macOS 和 Windows 端口正常维护
- 所有与平台无关的 Java API 不受影响

---

## 6. 后续移除

JEP 362 的废弃为 JDK 14 中的 [JEP 381](jep-381.md) 铺平了道路。JEP 381
完成了 Solaris 和 SPARC 相关源代码的实际删除，释放了大量工程资源用于
Linux、macOS 和 Windows 平台的优化与新特性开发。
