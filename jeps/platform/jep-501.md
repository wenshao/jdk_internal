## 1. 概述

JEP 501 在 JDK 24 中将 Linux 32-bit x86 移植（linux-x86）标记为废弃并计划在未来
版本中移除。这是继 JEP 449（JDK 21 废弃 Windows 32-bit x86）之后，Java 平台
退出 32 位 x86 架构的又一关键步骤。

### 1.1 核心变更

- **构建警告**：在 linux-x86 平台上配置构建时输出废弃警告
- **`--enable-deprecated-ports`**：继续构建需要显式传入此配置选项
- **信号明确**：正式告知发行版维护者和用户 32-bit x86 Linux 支持即将终止
- **功能不变**：JDK 24 中 linux-x86 构建仍然完全可用

---

## 2. 背景：32 位平台的退出进程

JDK 21（2023）通过 JEP 449 率先废弃 Windows 32-bit x86 移植，JDK 23 通过 JEP 479
完成移除。主流 Linux 发行版也已大幅缩减 32-bit x86 支持：

| 发行版 | 32-bit x86 状态 |
|--------|----------------|
| Ubuntu | 自 18.04（2018）起停止 32-bit 安装镜像 |
| Red Hat / CentOS | RHEL 7+（2014）起不再支持 32-bit x86 |
| Fedora | 自 Fedora 31（2019）起停止 32-bit x86 |
| Debian | 仍提供 i386 架构，但用户极少 |

---

## 3. JDK 24 中的变更

### 3.1 构建系统变更

```bash
# JDK 24 在 linux-x86 上配置构建时输出废弃警告
# 继续构建需显式确认：
bash configure --enable-deprecated-ports

# 不传入该选项时，configure 将报错并终止
```

### 3.2 受影响的平台标识

```
os.arch = x86        # 受影响（32-bit x86）
os.arch = amd64      # 不受影响（64-bit x86-64）
os.arch = aarch64    # 不受影响（64-bit ARM）
```

---

## 4. 为何废弃

32-bit x86 移植需要额外维护 JIT 编译器后端、GC 屏障代码、平台特定汇编例程等。
新特性在 32-bit 上存在天然限制——虚拟线程栈管理更复杂，FFM API 存在指针大小差异，
Valhalla 内存布局差异显著，向量 API 的 SIMD 支持有限。

Oracle 和 OpenJDK 社区调研显示 linux-x86（32-bit）活跃用户极少，不足以支撑持续
工程投资。移除后可将资源重新分配到 RISC-V 64-bit、AArch64 优化等方向。

---

## 5. 移除时间线

| 时间 | 版本 | 事件 |
|------|------|------|
| 2023 | JDK 21 | JEP 449：废弃 Windows 32-bit x86 移植 |
| 2024 | JDK 23 | JEP 479：移除 Windows 32-bit x86 移植 |
| **2025** | **JDK 24** | **JEP 501：废弃 Linux 32-bit x86 移植** |
| **2025** | **JDK 25** | **JEP 503：移除 Linux 32-bit x86 移植** |

从废弃到移除仅间隔一个版本，延续了 Windows 32-bit 退出的快速节奏。

---

## 6. 迁移建议

| 当前环境 | 迁移方案 |
|---------|---------|
| 32-bit Linux 服务器 | 升级至 64-bit Linux，使用 x86-64 JDK |
| 嵌入式 32-bit 设备 | 评估 ARM 64-bit（AArch64）替代方案 |
| 32-bit JNI 本地库 | 重新编译为 64-bit，或使用 FFM API 替代 |
| 旧版 32-bit 应用 | 在 64-bit JVM 上运行（字节码不区分位宽） |
