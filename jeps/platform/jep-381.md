## 1. 概述

JEP 381 在 JDK 15 中完成了 Solaris 和 SPARC 平台端口的实际移除。这是
JDK 14 中 [JEP 362](jep-362.md) 废弃操作的后续步骤，将 Solaris/SPARC、
Solaris/x64 和 Linux/SPARC 的所有平台特定源代码从 OpenJDK 代码库中删除。

移除这些端口后，OpenJDK 贡献者可以将精力集中在 Linux、macOS 和 Windows
三大主流平台的优化与新特性开发上。

### 1.1 移除的端口

- **Solaris/SPARC** (`os-arch = solaris-sparcv9`)
- **Solaris/x64** (`os-arch = solaris-amd64`)
- **Linux/SPARC** (`os-arch = linux-sparcv9`)

---

## 2. 背景：从废弃到移除

### 2.1 JEP 362 的铺垫

JDK 14（2020 年 3 月）通过 JEP 362 正式废弃了 Solaris 和 SPARC 端口，
给予社区一个版本周期（约 6 个月）的过渡期。在废弃期间，社区反馈中没有出现
强烈的反对意见或保留请求，证实了这些端口的用户基础已极度萎缩。

### 2.2 快速推进的决定

从废弃到移除仅间隔一个版本（JDK 14 → JDK 15），反映了社区对该决策的广泛
共识。维护这些端口的机会成本远大于继续保留的收益。

---

## 3. JDK 15 中的变更

### 3.1 源代码删除

JEP 381 删除了 OpenJDK 源码树中所有 Solaris 和 SPARC 特定的代码，包括
`src/hotspot/os/solaris/`、`src/hotspot/cpu/sparc/`、`src/java.base/solaris/`
等目录以及构建系统中的相关条件分支。

### 3.2 构建系统清理

`configure` 脚本不再识别 Solaris 或 SPARC 作为目标平台。尝试为这些平台
构建 JDK 将直接报错。同时，许多 `#ifdef` 条件编译分支得到简化。

---

## 4. 移除的理由

### 4.1 释放工程资源

Solaris 和 SPARC 端口的维护占用了宝贵的工程时间。移除后，这些资源被重新
分配到以下关键项目：

- **Project Loom**（虚拟线程）的平台适配
- **ZGC / Shenandoah** 垃圾收集器的性能优化
- **Foreign Function & Memory API** 的底层实现
- **Project Panama** 的原生互操作支持

### 4.2 降低代码复杂度

SPARC 是唯一的大端（big-endian）CPU 架构端口。移除后，许多字节序相关的
条件分支得以简化。CI/CD 流水线也不再需要维护 Solaris/SPARC 测试环境，
加速了构建和测试周期。

---

## 5. 对开发者的影响

### 5.1 普通 Java 开发者

对运行在 Linux、macOS 或 Windows 上的 Java 应用程序**无任何影响**。所有
Java API 保持不变，跨平台兼容性不受影响。

### 5.2 仍需 Solaris/SPARC 的用户

- JDK 14 是最后一个支持 Solaris/SPARC 构建的版本
- 建议迁移至 Linux/x64 或 Linux/AArch64 平台

### 5.3 JDK 贡献者

移除后，贡献者只需关注 Linux、macOS、Windows 三个操作系统及 x64、AArch64
两个 CPU 架构，显著降低了贡献门槛。
