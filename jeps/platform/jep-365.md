## 1. 概述

JEP 365 将 Z Garbage Collector（ZGC）扩展至 Windows 平台，与 JEP 364（ZGC on macOS）同期在 JDK 15 中交付。此移植完成了 ZGC 对三大主流操作系统（Linux、macOS、Windows）的覆盖，为 ZGC 在 JDK 15 中正式转为生产特性（JEP 377）扫清了跨平台障碍。

---

## 2. 背景

### 2.1 Windows 平台的重要性

Windows 是 Java 开发者使用最广泛的桌面操作系统之一。企业环境中 Windows Server 的部署也相当普遍。ZGC 仅支持 Linux 意味着大量 Windows 用户和企业无法受益于其低延迟特性。

### 2.2 Windows 内存管理的特殊性

Windows 的虚拟内存管理与 POSIX 系统（Linux/macOS）存在根本差异。Windows 使用页面文件（paging file）支撑虚拟内存，缺少 `mmap` 和文件描述符共享等 POSIX 内存映射机制，需要完全不同的实现策略。

---

## 3. 技术挑战

### 3.1 内存多重映射

ZGC 的彩色指针机制要求同一物理内存映射到多个虚拟地址。Windows 没有与 Linux `mmap` 或 macOS `mach_vm_remap` 等价的简单 API。需要利用 Windows 特有的内存管理 API 实现此功能。

### 3.2 虚拟地址空间管理

Windows 对虚拟地址空间的管理更为严格：内存预留（`VirtualAlloc` 的 `MEM_RESERVE`）与提交（`MEM_COMMIT`）是显式分离的操作，地址空间碎片化处理与 Linux 不同，大页支持需要特殊权限配置。

---

## 4. 关键变更

### 4.1 基于页面文件映射的多重映射

Windows 移植的核心方案是使用 Windows 页面文件支撑的内存区段（section）实现多重映射：

| 步骤 | API | 作用 |
|------|-----|------|
| 创建内存区段 | `CreateFileMapping(INVALID_HANDLE_VALUE, ...)` | 在页面文件中创建共享内存区段 |
| 映射视图 | `MapViewOfFileEx` | 将区段映射到指定虚拟地址 |
| 多重映射 | 对同一区段多次调用 `MapViewOfFileEx` | 实现同一物理内存的多个虚拟地址映射 |

通过 `CreateFileMapping` 创建页面文件支撑的 section，然后使用 `MapViewOfFileEx` 将同一 section 映射到多个虚拟地址，从而满足 ZGC 彩色指针的多重映射需求。

### 4.2 占位符预留机制

Windows 10 1803 及更高版本引入了 `MEM_RESERVE_PLACEHOLDER` 标志，允许预留地址空间后再替换为文件映射视图。ZGC 利用此机制管理虚拟地址空间布局。

### 4.3 平台抽象层

与 JEP 364 共享的平台抽象层重构使得 ZGC 的核心逻辑保持平台无关。Windows 特定实现封装在独立的后端模块中，包括内存预留/提交、多重映射和大页支持。

---

## 5. 实现细节

### 5.1 系统要求

Windows 上的 ZGC 要求 Windows 10 版本 1803 或更高版本（或 Windows Server 2019），因为多重映射实现依赖于该版本引入的 `VirtualAlloc2` 和占位符机制。

### 5.2 大页支持

Windows 上的大页支持需要通过"锁定内存页"（Lock Pages in Memory）权限进行配置，与 Linux 的 `hugetlbfs` 或 `transparent hugepages` 机制不同。ZGC 在 Windows 上支持大页，但需要管理员进行额外的系统配置。

---

## 6. 社区影响

JEP 365 与 JEP 364 共同实现了 ZGC 的全平台覆盖，消除了平台限制，使 ZGC 能够在 JDK 14 中正式成为生产级特性。Windows 开发者和企业用户可以直接受益于 ZGC 的亚毫秒级暂停时间特性。
