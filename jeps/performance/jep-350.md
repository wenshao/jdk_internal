## 1. Overview

JEP 350 引入 **Dynamic CDS Archives**，允许 JVM 在应用退出时自动生成 AppCDS archive。用户只需添加 `-XX:ArchiveClassesAtExit=<filename>` 选项，JVM 便会在正常退出时将运行期间加载的应用类和库类自动 dump 到 archive 文件中。后续启动时通过 `-XX:SharedArchiveFile=<filename>` 即可使用该 archive，获得与静态 AppCDS 相同的启动加速和内存共享效果。

这将 JEP 310 的三步手动操作简化为一步自动操作，大幅降低了 AppCDS 的使用门槛，是 CDS 技术从"专家级调优"走向"开箱即用"的关键一步。

---

## 2. Background

JEP 310（AppCDS，JDK 10）将 CDS 扩展到应用类，但使用流程需要三个独立步骤：

```bash
# Step 1: 记录被加载的类
java -Xshare:off -XX:DumpLoadedClassList=classes.lst -cp myapp.jar com.example.Main
# Step 2: 生成共享 archive
java -Xshare:dump -XX:SharedClassListFile=classes.lst \
     -XX:SharedArchiveFile=app-cds.jsa -cp myapp.jar
# Step 3: 使用 archive 启动
java -Xshare:on -XX:SharedArchiveFile=app-cds.jsa -cp myapp.jar com.example.Main
```

这一流程要求用户理解 CDS 工作原理、维护 class list 文件、确保每步 classpath 一致，过于繁琐。尽管 AppCDS 能带来 20-30% 的启动改善，三步流程严重阻碍了采用。JEP 341（Default CDS Archives，JDK 13）通过自动生成默认 archive 部分缓解了问题，但应用类 archive 仍需手动创建。JEP 350 正是为彻底解决这一易用性问题。

## 3. How Dynamic CDS Works

生成 archive 只需一个选项：

```bash
java -XX:ArchiveClassesAtExit=app-dynamic.jsa \
     -cp myapp.jar com.example.Main
```

JVM 在运行应用的同时追踪所有被加载的类。当应用正常退出时（`System.exit()`、`main` 返回、或所有非 daemon 线程结束），自动将 class metadata dump 到 archive 文件中。无需额外的 training run 或中间文件。使用 archive：

```bash
java -XX:SharedArchiveFile=app-dynamic.jsa \
     -cp myapp.jar com.example.Main
```

JVM 启动时自动加载 dynamic archive 及其底层的 default archive，将预处理的 class metadata 通过 `mmap` 映射到内存，跳过已 archive 的类的 loading 和 verification。

**与静态 AppCDS 对比**:

| 方面 | 静态 AppCDS（JEP 310） | Dynamic CDS（JEP 350） |
|------|------------------------|------------------------|
| **操作步骤** | 三步（class list → dump → use） | 一步生成 + 一步使用 |
| **中间文件** | 需要 class list 文件 | 无需中间文件 |
| **training run** | 需要独立的 training run | 应用正常运行即 training |
| **Archive 内容** | 由 class list 精确控制 | 自动包含所有已加载的类 |

## 4. Archive Layering

Dynamic CDS archive 采用**分层架构**，建立在 default archive 之上：

```
┌──────────────────────────────────┐
│  Dynamic Archive (app.jsa)       │  ← 应用类 + 第三方库类
├──────────────────────────────────┤
│  Default Archive (classes.jsa)   │  ← JDK 核心类 (JEP 341)
└──────────────────────────────────┘
```

Dynamic archive 自动将 JEP 341 的 default archive 作为 **base archive**，加载时先加载 base 再加载 dynamic 层。这意味着 dynamic archive 仅包含应用特有的类（体积更小、生成更快），用户也可通过 `-XX:SharedArchiveFile=base.jsa:dynamic.jsa` 显式指定 base archive。如果 default archive 不可用，JVM 自动退化为完整 archive。

## 5. Performance Characteristics

Dynamic CDS 的性能改善与静态 AppCDS **基本相同**，运行时工作方式一致：通过 `mmap` 映射预处理的 class metadata，跳过 loading 和 verification。

| 指标 | Default CDS | Dynamic CDS | 备注 |
|------|-------------|-------------|------|
| **启动时间改善** | 10-15% | 20-35% | 覆盖应用类后效果显著 |
| **Metaspace 减少** | ~10% | ~25% | 共享类的 metadata 不重复分配 |
| **多实例内存共享** | 共享 JDK 类 | 共享 JDK + 应用类 | 容器化场景尤为明显 |

> **注意**: 以上数据为概念性估算。Archive 生成发生在 JVM shutdown 阶段，对运行时性能无影响。

---

## 6. Simplifying AppCDS Adoption

Dynamic CDS 大幅降低了 AppCDS 的采用门槛：

- **开发场景**: 开发者直接添加 `-XX:ArchiveClassesAtExit` 即可享受启动加速，无需理解 CDS 原理
- **CI/CD 集成**: 在测试运行时添加 archive 生成选项，即可为部署准备好 archive，无需维护 class list
- **容器化部署**: Docker 构建时在 warmup 运行中生成 archive，entrypoint 使用 `-XX:SharedArchiveFile` 即可

## 7. Evolution to AOT Cache

JEP 350 是 CDS 技术从手动操作走向自动化的转折点：

```
AppCDS (JEP 310, JDK 10) — 手动三步操作
  └→ Dynamic CDS (JEP 350, JDK 13) — 自动生成 archive ← 当前 JEP
       └→ AOT Class Loading & Linking (JEP 483, JDK 24) — 预完成 class linking
            └→ AOT Cache (JEP 516, JDK 26) — 完整 AOT 缓存
```

JEP 350 确立的 **"运行时自动收集 + 退出时自动 dump"** 模式成为后续 AOT 优化的基本范式。JEP 483 扩展 archive 至 class linking，JEP 516 进一步缓存 compiled code、profiling data 和 heap objects，JEP 514 简化了命令行接口。

## 8. Limitations

**Archive 生成条件**:
- 仅在 JVM 正常退出时生成 archive；`kill -9` 或 crash 时不会生成
- Archive 内容取决于运行期间实际加载的类

**Classpath 一致性**:
- 生成 archive 时的 classpath 必须与使用时一致
- 应用更新后 archive 自动失效，JVM 会回退到正常启动

**Class Loader 限制**:
- 仅支持 built-in class loader（bootstrap、platform、application）加载的类
- 自定义 class loader 和动态生成的类（proxy、lambda 等）不在 archive 范围内

**共享内容**:
- 仅共享 class metadata，不包含 JIT compiled code 或 profiling data
- 这些限制在 JEP 483（AOT linking）和 JEP 516（AOT Cache）中被逐步突破
