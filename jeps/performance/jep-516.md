## 1. Overview

JEP 516 引入 **AOT Cache**（Ahead-of-Time Cache），是 JDK 26 中对 Java 启动性能的重大增强。该特性将 CDS (Class Data Sharing) 的概念大幅扩展，不仅缓存 class metadata，还缓存 compiled native code、heap objects 和 method profiling data，从而显著减少 JVM 启动时间和预热时间。

AOT Cache 是 Project Leyden 的核心交付成果，也是 JDK 25 中 [JEP 514](jep-514.md)（AOT Command-Line Ergonomics）和 [JEP 515](jep-515.md)（Ahead-of-Time Method Profiling）的自然延伸。在 JDK 26 中，AOT Cache 将这些技术统一到一个完整的 cache 文件中，提供开箱即用的启动性能优化。

## 2. Background

### 2.1 JDK Startup Performance Problem

Java 应用的启动性能一直是一个广为人知的痛点：

- **Class Loading**: JVM 启动时需要加载、验证、解析大量 class 文件
- **JIT Compilation**: 方法需要经过 interpreter 执行后才能被 JIT 编译为 native code
- **Heap Initialization**: 核心库的 static fields 和常量需要在运行时初始化
- **Profiling Overhead**: JIT 编译器需要收集运行时 profiling data 来做优化决策

这些因素导致 Java 应用的启动时间通常在秒级，且需要较长的预热期才能达到峰值性能。在 cloud-native 和 serverless 场景中，这种特性尤为不利。

### 2.2 CDS Evolution

CDS（Class Data Sharing）技术经历了漫长的演进：

| JEP | JDK 版本 | 名称 | 贡献 |
|-----|----------|------|------|
| — | JDK 5 | CDS (Initial) | 共享核心类的 metadata |
| JEP 310 | JDK 10 | Application CDS | 扩展到应用类 |
| JEP 341 | JDK 12 | Default CDS Archives | 默认生成 CDS archive |
| JEP 350 | JDK 13 | Dynamic CDS Archives | 运行时动态生成 archive |
| JEP 483 | JDK 24 | AOT Class Loading & Linking | 预先完成 class loading 和 linking |

CDS 的核心思路是将 class loading 的结果持久化到文件中，下次启动时直接 memory-map 这些数据，跳过重复的 loading 和 verification 工作。但传统 CDS 只存储 class metadata，不包含 compiled code 或 heap objects。

### 2.3 AOT Compilation History

Java 的 AOT 编译尝试经历了曲折的历程：

- **JEP 295**（JDK 9）: 引入 `jaotc` 工具进行 AOT compilation。基于 Graal 编译器，但由于维护成本高、性能收益有限，在 JDK 17 中被移除
- **GraalVM Native Image**: 提供完整的 AOT 编译方案，但存在 closed-world 限制，不支持完整的 Java 动态特性
- **Project Leyden**（2022 启动）: Oracle 主导的长期项目，目标是在保持 Java 完整动态特性的前提下，通过 "shifting and constraining" 策略渐进式地改善启动性能

JEP 516 代表了 Project Leyden 的一个重要里程碑——在不牺牲 Java 动态性的前提下，将启动性能提升到接近 native 应用的水平。

### 2.4 JEP 514/515 as Predecessors

JDK 26 中的两个 JEP 为 JEP 516 奠定了基础：

- **[JEP 514](jep-514.md): AOT Command-Line Ergonomics** — 简化了 AOT 相关的命令行参数，引入 `-XX:AOTCache` 等统一接口，使用户无需理解底层细节即可使用 AOT 功能
- **[JEP 515](jep-515.md): Ahead-of-Time Method Profiling** — 在 training run 中收集 method profiling data（调用计数、分支概率、类型反馈等），使 JIT 编译器在后续启动时能立即做出优化决策，无需再次预热

JEP 516 将 JEP 514 的 ergonomic interface 和 JEP 515 的 profiling data 与 compiled code cache 统一整合，形成一个完整的 AOT Cache 解决方案。

## 3. How AOT Cache Works

### 3.1 Training Run

AOT Cache 的核心工作流程分为两个阶段：

```
阶段 1: Training Run
┌─────────────────────────────────────────────┐
│  java -XX:AOTCacheOutput=app.aot MyApp      │
│                                             │
│  1. 正常执行应用                              │
│  2. 记录 loaded classes                      │
│  3. 收集 method profiling data               │
│  4. 记录 heap object graph                   │
│  5. 触发 AOT compilation                     │
│  6. 输出 AOT cache 文件                       │
└─────────────────────────────────────────────┘

阶段 2: Production Run
┌─────────────────────────────────────────────┐
│  java -XX:AOTCache=app.aot MyApp            │
│                                             │
│  1. Memory-map AOT cache 文件                │
│  2. 直接使用 pre-loaded classes              │
│  3. 直接执行 pre-compiled native code        │
│  4. 恢复 pre-initialized heap objects        │
│  5. 应用快速启动并立即达到高性能               │
└─────────────────────────────────────────────┘
```

Training run 应使用有代表性的工作负载，以确保缓存的内容覆盖实际生产场景中的热点路径。

### 3.2 Cache Contents

AOT Cache 文件包含四类核心数据：

| 数据类型 | 说明 | 来源 |
|----------|------|------|
| **Class Metadata** | 已加载、验证、链接的 class 数据 | 继承自 CDS / JEP 483 |
| **Compiled Native Code** | C1/C2 编译的 machine code | JEP 516 新增 |
| **Method Profiling Data** | 调用频率、分支概率、类型反馈 | 来自 JEP 515 |
| **Heap Objects** | 预初始化的 Java 对象（如 String 常量池） | JEP 516 增强 |

```
AOT Cache 文件结构（概念模型）:

┌──────────────────────────────┐
│         Header               │  版本信息、校验和、配置指纹
├──────────────────────────────┤
│    Class Metadata Region     │  pre-loaded & pre-linked classes
├──────────────────────────────┤
│    Compiled Code Region      │  native code (C1/C2 output)
├──────────────────────────────┤
│    Profiling Data Region     │  method profiles, type feedback
├──────────────────────────────┤
│    Heap Objects Region       │  pre-initialized Java objects
├──────────────────────────────┤
│    Relocation Info           │  地址重定位表
└──────────────────────────────┘
```

### 3.3 Single-Command Usage

JEP 514 引入的 ergonomic improvements 在 JEP 516 中得到充分利用：

```bash
# 最简单的使用方式：自动缓存
java -XX:AOTCache MyApp

# 指定缓存文件
java -XX:AOTCache=app.aot -cp myapp.jar com.example.Main

# 指定 training 输出
java -XX:AOTCacheOutput=app.aot -cp myapp.jar com.example.Main

# 使用已有缓存
java -XX:AOTCache=app.aot -cp myapp.jar com.example.Main
```

### 3.4 Automatic vs Manual Cache Creation

JEP 516 支持两种缓存创建模式：

**Automatic Mode**（自动模式）:
- JVM 检测到没有可用缓存时，自动进行 training run
- 缓存文件存储在默认位置
- 后续启动自动使用缓存
- 适合开发环境和简单部署场景

**Manual Mode**（手动模式）:
- 开发者显式控制 training run 和 cache 使用
- 可以精确控制 training workload
- 缓存文件可以与应用一起部署
- 适合生产环境和 CI/CD pipeline

```bash
# Manual: 分步创建和使用
# Step 1: Training run
java -XX:AOTCacheOutput=app.aot -cp myapp.jar com.example.Main

# Step 2: Production run
java -XX:AOTCache=app.aot -cp myapp.jar com.example.Main
```

## 4. Key Improvements over CDS

JEP 516 AOT Cache 相比传统 CDS 有质的飞跃：

| 特性 | 传统 CDS | JEP 483 (JDK 26) | JEP 516 AOT Cache |
|------|----------|-------------------|---------------------|
| Class metadata 缓存 | Yes | Yes | Yes |
| Class loading 加速 | Yes | Yes | Yes |
| Class linking 加速 | No | Yes | Yes |
| Compiled code 缓存 | No | No | **Yes** |
| Method profiling data | No | No | **Yes** |
| Pre-initialized heap | Limited | Limited | **Enhanced** |
| 命令行易用性 | 需多步操作 | 需多步操作 | **单命令** |
| Warm-up 加速 | No | No | **Yes** |

核心差异总结：

1. **Compiled Code Storage**: 传统 CDS 只缓存 class metadata，JIT 编译仍需在每次启动时重新进行。AOT Cache 直接存储 C1/C2 编译产生的 native code，启动后立即可用
2. **Method Profiling Data**（from JEP 515）: 缓存 training run 中收集的 profiling data，使 JIT 编译器能跳过预热阶段，直接进行高质量的 C2 优化编译
3. **Pre-initialized Heap Objects**: 将 static initializer 执行后的 heap state 持久化，避免每次启动重新初始化
4. **Command-Line Ergonomics**（from JEP 514）: 将多步操作简化为单个 `-XX:AOTCache` 参数

## 5. Performance Impact

### 5.1 Startup Time Reduction

AOT Cache 对启动时间的影响因应用规模和类型而异：

| 应用类型 | 无缓存启动 | CDS 启动 | AOT Cache 启动 | 改善幅度 |
|----------|-----------|----------|----------------|----------|
| Hello World | ~50ms | ~35ms | ~15ms | ~70% |
| Spring Boot (小型) | ~3s | ~2s | ~0.5-1s | 特定工作负载下 60-80% |
| 大型企业应用 | ~15s | ~10s | ~3-5s | 特定工作负载下 50-70% |

> **注意**: 以上数据为概念性估算，实际改善幅度取决于具体应用特征、JVM 配置和硬件环境。请参阅 JEP 516 官方提案中的 benchmark 数据。

### 5.2 Memory Sharing

AOT Cache 支持多进程共享：

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Process 1  │  │  Process 2  │  │  Process 3  │
│  (MyApp)    │  │  (MyApp)    │  │  (MyApp)    │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────┬───────┴────────┬───────┘
                │                │
         ┌──────┴──────┐  ┌─────┴───────┐
         │  Shared     │  │  Private    │
         │  AOT Cache  │  │  JIT Code   │
         │  (mmap)     │  │  (per-proc) │
         └─────────────┘  └─────────────┘
```

AOT Cache 文件通过 `mmap` 共享映射，多个 JVM 进程可以共享同一份物理内存中的 class metadata 和 compiled code，在 container 和微服务场景中显著减少总内存占用。

### 5.3 Warm-up Time Reduction

传统 JVM 的预热过程：

```
时间轴 →
┌───────────────────────────────────────────────┐
│ Interpreter │   C1 编译   │  C2 编译  │ 峰值  │  无 AOT Cache
│  (慢)       │  (中等)     │  (优化中) │ 性能  │
└───────────────────────────────────────────────┘
0s           5s           15s        30s

┌───────────────────────────────────────────────┐
│ AOT Code │  C2 重编译（少量） │   峰值性能      │  有 AOT Cache
│ (快)     │  (基于缓存 profile)│               │
└───────────────────────────────────────────────┘
0s        2s                  5s
```

得益于 JEP 515 的 method profiling data，AOT Cache 中的 compiled code 质量接近完全预热后的 C2 代码。JVM 启动后仅需对少量未覆盖的方法进行 JIT 编译，大幅缩短达到峰值性能的时间。

## 6. Architecture

### 6.1 AOT Cache File Format

AOT Cache 使用扩展的 CDS archive 格式（`.aot` 文件）：

- **Header**: 包含 JDK 版本、平台信息、配置指纹（classpath hash 等），用于验证 cache 与当前运行环境的兼容性
- **Class Regions**: 按 class loader 组织的 class metadata，支持 memory-mapped 直接使用
- **Code Region**: 编译后的 native code，按方法组织，包含 relocation information
- **Heap Region**: 序列化的 Java 对象图（archived heap），包含 String 常量、Class mirrors 等
- **Profile Region**: 方法级 profiling data（invocation counts、branch frequencies、receiver types）

### 6.2 Integration with C1/C2 JIT

AOT Cache 与 HotSpot 的分层编译体系深度集成：

```
传统分层编译:
  Level 0 (Interpreter) → Level 1-3 (C1) → Level 4 (C2)

AOT Cache 加速路径:
  AOT Code (≈Level 4) → C2 Recompilation (可选，针对去优化后的方法)
```

- **C1 编译结果缓存**: training run 中 C1 编译的 code 直接存入 cache，启动后无需重新编译
- **C2 编译结果缓存**: 基于 profiling data 的 C2 优化代码也可缓存，提供接近峰值的执行性能
- **Deoptimization 兼容**: 如果运行时行为与 training run 不一致（如新的 class loading），JVM 可以 deoptimize 并回退到 JIT 编译，保持正确性
- **Speculative optimizations**: 缓存的 profiling data 指导 C2 做出 inlining、loop unrolling 等优化决策

### 6.3 Compatibility

AOT Cache 与其他 JVM 特性的兼容性：

| 特性 | 兼容性 | 说明 |
|------|--------|------|
| **Dynamic Class Loading** | Yes | 动态加载的类仍通过 JIT 编译 |
| **Reflection** | Yes | 反射调用路径保持不变 |
| **Serialization** | Yes | 无影响 |
| **Java Agents** | 部分 | Agent 修改的 class 可能绕过 cache |
| **JVMTI** | 部分 | Debug 相关功能可能禁用部分 AOT 优化 |
| **G1 GC / ZGC** | Yes | 与所有生产级 GC 兼容 |
| **Virtual Threads** | Yes | 无影响 |
| **CRaC** | 互补 | AOT Cache 解决启动性能，CRaC 解决状态恢复 |

**平台支持**: AOT Cache 支持所有 HotSpot 支持的平台（Linux、macOS、Windows）和架构（x86_64、AArch64 等），但 cache 文件不跨平台——需在目标平台上生成。

## 8. Project Leyden Context

[Project Leyden](https://openjdk.org/projects/leyden/) 是 Oracle 于 2022 年发起的 OpenJDK 长期项目，目标是通过 "shifting and constraining" 策略改善 Java 的启动时间、预热时间和占用空间。

**核心理念**: 将运行时工作（class loading、JIT compilation、heap initialization）提前到构建时或首次运行时完成，并将结果缓存起来供后续使用。与 GraalVM Native Image 的 closed-world 方案不同，Leyden 保持 Java 的完整动态特性（dynamic class loading、reflection、serialization 等）。

**Leyden 交付时间线**:

| 时间 | 交付物 | 说明 |
|------|--------|------|
| 2024 | JEP 483 (JDK 24) | AOT Class Loading & Linking |
| 2025 | JEP 514 (JDK 25) | AOT Command-Line Ergonomics |
| 2025 | JEP 515 (JDK 25) | AOT Method Profiling |
| 2026 | **JEP 516 (JDK 26)** | **AOT Cache（集大成）** |

JEP 516 是 Leyden 项目迄今最重要的交付成果。它将前序 JEP 的各项技术整合为统一的 AOT Cache，为 Java 开发者提供了一个简单、高效、完全兼容的启动性能优化方案。未来 Leyden 可能继续在 AOT Cache 基础上扩展——例如更激进的 code 缓存策略、跨 JDK 版本的 cache 迁移、以及与 container image 构建工具的集成。
