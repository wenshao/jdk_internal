## 1. Overview

JEP 310 引入 **Application Class-Data Sharing (AppCDS)**，将 JDK 10 以来仅限于 bootstrap class loader 的 CDS（Class Data Sharing）技术扩展到 application classes。通过将应用类的 metadata 预处理并存储到共享 archive 文件中，AppCDS 使 JVM 在启动时可以直接 memory-map 这些数据，跳过重复的 class loading 和 verification 工作，从而显著减少启动时间和内存占用。

AppCDS 是 CDS 技术演进的关键一步，也是后续 JEP 341（Default CDS Archives）、JEP 350（Dynamic CDS Archives）乃至 JEP 516（AOT Cache）的基础。在 JDK 12 中，AppCDS 首次使这项优化技术惠及所有 Java 应用，而不仅仅是 JDK 自身的核心类。

## 2. Background

### 2.1 CDS History

CDS（Class Data Sharing）最初在 **JDK 5**（2004 年）中引入，作为 HotSpot JVM 的一项性能优化特性。其原始设计仅支持 **bootstrap class loader** 加载的核心 JDK 类（`java.lang.*`、`java.util.*` 等）。

CDS 的基本思路是将 class loading 的结果（parsing、verification）持久化到共享 archive 文件（`classes.jsa`）中，JVM 启动时通过 `mmap` 直接映射使用，多个进程可共享同一份物理内存。

但这种方式在多年间仅对 JDK 内部类有效。大量应用类和第三方库仍然需要在每次启动时经历完整的 loading 和 verification 流程。JEP 310 正是为了打破这一限制。

### 2.2 Startup Bottleneck

Java 应用启动时需要经历 class loading（查找并读取 `.class` 文件）、bytecode verification（类型安全性检查）、parsing（解析 constant pool 和 metadata）、linking（解析符号引用、分配 vtable）等步骤。对于大型应用（如 Spring Boot、Jakarta EE），启动时需加载数千个类，累积开销显著。在 JDK 10 之前，CDS 仅加速 JDK 核心类部分，应用类的加载开销无法被优化。

### 2.3 Class Loading Overhead

Class loading 的开销主要来自以下几个方面：

| 阶段 | 开销来源 |
|------|----------|
| **I/O 读取** | 从 JAR 文件中解压并读取 `.class` 文件 |
| **Parsing** | 解析 class 文件二进制格式，构建 InstanceKlass |
| **Verification** | 字节码类型检查和结构验证（计算密集型） |
| **Linking** | 解析 constant pool 中的符号引用 |
| **内存分配** | 每个 JVM 进程独立为每个类分配 Metaspace |

在没有 AppCDS 的情况下，每个 JVM 进程都要独立完成上述全部工作，既浪费 CPU 时间，也浪费内存。

## 3. How AppCDS Works

AppCDS 的使用分为三个步骤：

### 3.1 Step 1: Create Class List

首先，运行应用并记录所有被加载的类：

```bash
java -Xshare:off -XX:DumpLoadedClassList=classes.lst \
     -cp myapp.jar com.example.Main
```

该命令以正常方式运行应用，同时将所有被加载的类名记录到 `classes.lst` 文件中（纯文本，每行一个类名，使用 `/` 作为包分隔符）。应确保 training run 覆盖应用的典型启动路径，以便 class list 包含所有常用类。

### 3.2 Step 2: Create Shared Archive

使用 class list 生成共享 archive 文件：

```bash
java -Xshare:dump -XX:SharedClassListFile=classes.lst \
     -XX:SharedArchiveFile=app-cds.jsa \
     -cp myapp.jar
```

该命令读取 class list，对其中的每个类执行 loading、parsing 和 verification，然后将处理后的 class metadata 写入 archive 文件（`app-cds.jsa`）。

### 3.3 Step 3: Use Archive

启动应用时指定使用共享 archive：

```bash
java -Xshare:on -XX:SharedArchiveFile=app-cds.jsa \
     -cp myapp.jar com.example.Main
```

JVM 启动时通过 `mmap` 将 archive 文件映射到内存，对于 archive 中已有的类，直接使用预处理的 metadata，跳过 loading 和 verification 步骤。仅对 archive 中不存在的类执行完整 loading 流程。

> **注意**: `-Xshare:on` 表示强制使用共享 archive，如果 archive 不可用则启动失败。也可以使用 `-Xshare:auto`（默认行为），在 archive 不可用时回退到正常启动。

## 4. What Gets Shared

AppCDS archive 中存储的数据包括：

| 数据类型 | 说明 | 共享方式 |
|----------|------|----------|
| **Class Metadata** | InstanceKlass、ConstantPool 等 HotSpot 内部数据结构 | 通过 mmap 跨进程共享 |
| **Constant Pools** | 已解析的常量池条目（字符串引用、类引用、方法引用等） | 直接映射，无需重新解析 |
| **Bytecode Verification Results** | 验证通过的标记，跳过重复验证 | 作为 class metadata 的一部分存储 |
| **Method Metadata** | 方法签名、access flags、exception tables | 随 class metadata 一起共享 |
| **Interned Strings** | 共享的字符串常量 | 通过 shared string table 共享 |

Archive 文件由 Header（JDK 版本、校验和、classpath 指纹）、Read-Only Region（constant pools、bytecode，跨进程共享）、Read-Write Region（可变 metadata，copy-on-write）、String/Symbol Tables、Relocation Info 构成。Read-only 数据通过操作系统内存页共享机制在多个 JVM 进程间共享物理内存。

## 5. Performance Impact

### 5.1 Startup Time Improvement

AppCDS 的启动时间改善幅度因应用规模而异，典型改善在 **20-30%** 范围内：

| 场景 | 无 CDS | 仅 JDK CDS | AppCDS | 改善幅度（vs 无 CDS） |
|------|--------|------------|--------|----------------------|
| 小型应用（少量类） | ~200ms | ~150ms | ~140ms | ~30% |
| 中型应用（数百个类） | ~1.5s | ~1.2s | ~1.0s | ~33% |
| 大型应用（数千个类） | ~5s | ~4s | ~3.5s | ~30% |

> **注意**: 以上数据为概念性估算，实际改善幅度取决于具体应用特征和硬件环境。改善主要来自跳过 bytecode verification 和减少 class 文件 I/O（mmap 替代逐个从 JAR 中解压读取）。

### 5.2 Memory Sharing Across Processes

在同一台机器上运行多个相同应用实例时，AppCDS 的内存共享优势尤为明显。Archive 中 read-only 区域通过 `mmap` 在进程间共享物理内存页，Metaspace 占用减少（共享类不需要独立分配）。在容器化部署和微服务场景中，多实例运行时总内存节省可观。

## 6. Evolution Timeline

JEP 310 是 CDS 技术演进链中的关键节点。从 JDK 10 到 JDK 26，CDS 经历了从基础 class metadata 共享到完整 AOT Cache 的渐进式演进：

```
CDS (JDK 5) — 仅 bootstrap 类
  └→ AppCDS (JEP 310, JDK 10) — 扩展到应用类 ← 当前 JEP
       └→ Default CDS Archives (JEP 341, JDK 12) — JDK 安装时自动生成 archive
            └→ Dynamic CDS Archives (JEP 350, JDK 13) — 运行结束时自动生成 archive
                 └→ AOT Class Loading & Linking (JEP 483, JDK 24) — 预完成 linking
                      └→ AOT Cache (JEP 516, JDK 26) — 完整 AOT 缓存
```

| 阶段 | JEP | JDK 版本 | 核心贡献 |
|------|-----|----------|----------|
| 初始 CDS | — | JDK 5 | 共享 bootstrap 类的 metadata |
| **AppCDS** | **310** | **JDK 10** | **扩展 CDS 到应用类** |
| Default Archives | 341 | JDK 12 | JDK 安装时自动生成默认 archive |
| Dynamic Archives | 350 | JDK 13 | 应用退出时自动 dump archive，无需手动三步操作 |
| AOT Class Loading | 483 | JDK 24 | 在 archive 中预完成 class linking |
| AOT Cache | 516 | JDK 26 | 缓存 compiled code、profiling data、heap objects |

JEP 310 的核心意义在于首次证明 CDS 技术可以从 JDK 内部扩展到任意 Java 应用，为后续所有增强奠定了架构基础。

## 7. Limitations and Constraints

AppCDS 在 JDK 10 中存在以下限制和约束：

**操作复杂性**:
- 需要三步操作（创建 class list → dump archive → 使用 archive），流程繁琐
- Class list 需要通过实际运行应用来生成，需要可运行的环境
- 后续 JEP 350（Dynamic CDS）和 JEP 514（AOT Ergonomics）逐步解决了这些问题

**Classpath 一致性要求**:
- 创建 archive 时的 classpath 必须与使用时一致
- JAR 文件的路径、顺序、内容发生变化时，archive 可能失效
- 应用更新后通常需要重新生成 archive

**Class Loader 限制**:
- 仅支持 built-in class loader（bootstrap、platform/extension、application）加载的类
- 自定义 class loader 加载的类不能被共享
- 动态生成的类（如 proxy classes、lambda classes）不在 archive 范围内

**共享内容有限**:
- 仅共享 class metadata，不包含 compiled native code
- JIT 编译仍需在每次启动时重新执行
- 不缓存 heap objects 或 profiling data
- 这些限制在后续 JEP 483 和 JEP 516 中被逐步突破

**平台约束**:
- Archive 文件与特定 JDK 版本和平台绑定，不可跨平台使用
- JDK 升级后需要重新生成 archive
