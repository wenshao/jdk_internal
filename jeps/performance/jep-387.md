## 1. Overview

JEP 387 引入 **Elastic Metaspace**，对 HotSpot JVM 的 Metaspace 内存管理进行全面重构。核心目标是更高效地管理 class metadata 所使用的 off-heap 内存，并将未使用的 Metaspace 内存及时归还给操作系统。

在 JDK 16 之前，Metaspace 存在严重的内存浪费问题：一旦为 class metadata 提交（commit）了物理内存页，即使对应的 class loader 被卸载、metadata 不再需要，这些内存也几乎不会被归还给操作系统。Elastic Metaspace 通过引入 buddy allocation 方案和基于 granule 的内存组织方式，使 Metaspace 能够弹性伸缩——按需增长，及时收缩——从而显著降低 JVM 的整体内存占用。

## 2. Background

### 2.1 From PermGen to Metaspace

**JDK 8**（2014 年）移除了 Permanent Generation（PermGen），用 **Metaspace** 取而代之。PermGen 是 Java heap 中的固定大小区域，用于存储 class metadata、interned strings 和 static variables。其固定大小的设计经常导致 `java.lang.OutOfMemoryError: PermGen space`，尤其在频繁部署的应用服务器场景中。

Metaspace 将 class metadata 存储在 native memory（off-heap）中，默认不设上限（由可用系统内存决定），解决了 PermGen 大小固定的问题。但 Metaspace 的初始实现在内存管理效率方面并不理想，引入了新的问题。

### 2.2 Memory Fragmentation Problem

旧版 Metaspace 使用 free list 分配器管理内存块。当 class loader 被卸载后，其占用的小块内存被释放回 free list，但这些碎片化的小块往往难以被后续分配请求复用。随着 class loader 的反复创建和销毁（例如在 OSGi 容器、热部署场景中），Metaspace 内部的碎片化越来越严重，实际可用内存远小于已提交的内存总量。

### 2.3 Commit-but-Never-Return Issue

旧版 Metaspace 最严重的问题是 **一旦向操作系统提交（commit）了虚拟内存页，就几乎不会归还**。即使大量 class loader 被卸载、metadata 被释放，Metaspace 的 committed 内存量也只增不减。对于长时间运行的应用（特别是经历过类加载高峰的应用），这意味着操作系统层面的内存浪费持续累积。在容器化部署中，这一问题尤为突出，因为容器通常有严格的内存限制。

## 3. New Design

### 3.1 Buddy Allocation Scheme

Elastic Metaspace 采用 **buddy allocation**（伙伴分配）算法管理内存块。Buddy allocation 是操作系统内核中广泛使用的经典内存分配策略，其核心思想是：

- 内存块的大小始终是 2 的幂次方
- 分配时，如果没有合适大小的块，将更大的块对半拆分
- 释放时，检查相邻的"伙伴"块是否也空闲，如果是则合并成更大的块

这种方案天然地减少了外部碎片化——空闲块总是可以高效合并，避免了旧版 free list 中大量不可复用小碎片的问题。

### 3.2 Granule-Based Memory Organization

新设计将 Metaspace 内存组织为固定大小的 **granule**（粒度单元）。每个 granule 是一段对齐的连续内存区域（默认大小通常与操作系统页大小对齐，典型值为 64KB 或更大）。

Granule 是 commit/uncommit 操作的最小单位。Metaspace 不再以任意大小的内存块为单位与操作系统交互，而是以 granule 为单位批量提交或取消提交物理内存。这种对齐的设计确保了 uncommit 操作的高效性和可靠性。

### 3.3 Uncommit and Return to OS

当 class loader 被卸载后，其占用的 Metaspace 内存块通过 buddy allocation 合并。如果合并后某个 granule 中的所有内存块都已空闲，该 granule 就可以被 **uncommit**——即通过 `madvise(MADV_DONTNEED)`（Linux）或等价的操作系统调用将物理内存页归还给操作系统。

这是 Elastic Metaspace 的核心价值所在：Metaspace 不再是"只增不减"的，而是可以真正地将内存归还给操作系统，使其可被其他进程或应用使用。

### 3.4 Elastic Grow/Shrink Behavior

新的 Metaspace 表现出弹性行为：

- **增长阶段**: 当需要更多 class metadata 空间时，按 granule 为单位 commit 新的物理内存
- **收缩阶段**: 当 class loader 被卸载、metadata 被释放后，空闲 granule 被 uncommit，committed 内存量下降
- **稳态阶段**: 在类加载趋于稳定后，Metaspace 的 committed 大小接近实际使用量

这种弹性行为使 JVM 的内存占用更加忠实地反映应用的实际需求，而非历史峰值。

## 4. Key Improvements

Elastic Metaspace 带来以下关键改善：

| 改善领域 | 旧版表现 | Elastic Metaspace 表现 |
|----------|----------|----------------------|
| **内存占用** | 大量 class loader 场景下浪费严重 | 仅含少量类的 class loader 占用显著减少 |
| **Class 卸载后的内存回收** | committed 内存几乎不归还 OS | 空闲 granule 及时 uncommit 归还 OS |
| **内存碎片化** | free list 碎片化严重 | buddy allocation 高效合并，碎片化大幅减少 |
| **Class 卸载速度** | 释放操作相对较慢 | 批量释放和合并更高效 |
| **OS 内存利用率** | Metaspace 长期占据不再使用的内存 | 未使用内存及时归还，OS 可分配给其他进程 |
| **容器环境适配** | 内存膨胀可能导致触及容器 limit | 弹性收缩降低 OOM-killed 风险 |

对于使用大量小型 class loader 的场景（如动态语言运行时、OSGi、热部署），内存节省尤为显著。旧版 Metaspace 为每个 class loader 分配的最小 chunk 较大，即使 class loader 仅加载一两个类也会浪费大量空间。Elastic Metaspace 的 buddy allocation 可以分配更细粒度的块，显著减少这种浪费。

## 5. Architecture

Elastic Metaspace 的内存组织采用分层结构：

```
VirtualSpaceList
 └→ VirtualSpaceNode (reserved 虚拟地址空间区域)
      └→ Granule (commit/uncommit 的最小单位)
           └→ Chunk (buddy allocation 管理的内存块)
                └→ Block (分配给 class loader 的实际 metadata 存储单元)
```

各层级的职责如下：

| 层级 | 说明 |
|------|------|
| **VirtualSpaceList** | 管理所有 VirtualSpaceNode 的链表，是 Metaspace 的顶层入口 |
| **VirtualSpaceNode** | 一段预先 reserve 的虚拟地址空间，按需 commit 其中的 granule |
| **Granule** | 固定大小的对齐内存区域，是 commit/uncommit 操作的原子单位 |
| **Chunk** | 通过 buddy allocation 从 granule 中分配的内存块，大小为 2 的幂次方 |
| **Block** | chunk 内部进一步细分的小块，直接用于存储 class metadata |

当 class loader 请求 metadata 空间时，Metaspace 从 chunk 中分配 block；当 class loader 被卸载时，其所有 block 被释放，chunk 通过 buddy allocation 尝试与伙伴合并，如果整个 granule 变为空闲则可被 uncommit。

## 6. Old vs New Metaspace

| 特性 | 旧版 Metaspace (JDK 8–15) | Elastic Metaspace (JDK 16+) |
|------|---------------------------|----------------------------|
| **分配算法** | Free list | Buddy allocation |
| **内存组织** | 可变大小 chunk | 对齐 granule + 2 的幂次方 chunk |
| **碎片化控制** | 差，free list 碎片化严重 | 好，buddy allocation 高效合并 |
| **内存归还 OS** | 几乎不归还 | 空闲 granule 主动 uncommit |
| **最小分配粒度** | 较大（浪费明显） | 更细粒度（减少浪费） |
| **Class 卸载效率** | 一般 | 更高效（批量合并） |
| **弹性伸缩** | 仅增长 | 增长 + 收缩 |
| **调优参数** | 有限 | 新增 MetaspaceReclaimPolicy 等 |
| **容器友好性** | 差 | 好 |

## 7. Tuning

Elastic Metaspace 引入了新的调优参数：

**`-XX:MetaspaceReclaimPolicy=<policy>`**

控制 Metaspace 的内存回收策略：

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| **balanced**（默认） | 在内存回收和性能开销之间取平衡 | 大多数应用 |
| **aggressive** | 更积极地 uncommit 空闲内存 | 内存受限环境（容器） |
| **none** | 不主动 uncommit | 追求最低延迟、不关心内存占用的场景 |

其他相关参数：

- **`-XX:MaxMetaspaceSize=<size>`**: 设置 Metaspace 最大值上限（默认无限制）
- **`-XX:MetaspaceSize=<size>`**: 触发 GC 的初始 Metaspace 阈值
- **`-XX:MinMetaspaceFreeRatio`** / **`-XX:MaxMetaspaceFreeRatio`**: 控制 GC 后 Metaspace 扩展/收缩的比例

在容器化部署中，建议使用 `aggressive` 策略以最大化内存归还，降低因 Metaspace 膨胀导致容器 OOM 的风险。
