# JEP 528: Stable Values (2nd Preview)

> **Status**: Preview | **Type**: Feature | **Component**: core-libs / java.lang
>
> **JDK Version**: JDK 26 (2026-03)
>
> **Organization**: [Oracle](../../contributors/orgs/oracle.md)

---

## 1. Overview

JEP 528 delivers the **second preview** of the Stable Values API in JDK 26. `StableValue<T>` provides a mechanism for declaring values whose contents are set at most once, but whose initialization can be deferred until first use. Once initialized, the JVM's JIT compiler treats a stable value as a true constant, enabling the same constant-folding and inlining optimizations that apply to `static final` fields -- without requiring eager initialization at class-loading time.

The core proposition is simple: developers get the ergonomics of lazy initialization with the runtime performance of compile-time constants.

---

## 2. Background

### 2.1 The Lazy Initialization Problem

Java developers frequently need values that are expensive to compute and should be initialized only once, on demand. Traditional approaches -- double-checked locking with `volatile`, `AtomicReference` CAS loops, or the holder class idiom -- are either verbose and error-prone, limited to static fields, or prevent the JIT compiler from treating the result as a constant. The `volatile` read barrier on every access is the core performance penalty: the JIT cannot constant-fold a value it knows might change.

### 2.2 JEP 502: 1st Preview in JDK 25

[JEP 502](jep-502.md) introduced `StableValue` as a preview feature in JDK 25. It established the core design: a write-once container with `trySet()`, `orElseSet()`, and `orElseThrow()` methods, along with factory methods for stable collections. The 1st preview validated the API through community feedback and real-world usage, setting the stage for refinements in JDK 26.

---

## 3. What StableValue Provides

### 3.1 Deferred Immutability

A `StableValue<T>` represents a value that starts unset and can be initialized exactly once. After initialization, the value is permanently immutable. This differs from `final` fields (which must be set during construction) and `volatile` fields (which can be set repeatedly):

```
                  Set at construction    Set later    Set once only    JIT constant
static final      Yes                   No           Yes              Yes
volatile          No                    Yes          No               No
StableValue       No                    Yes          Yes              Yes
```

The key insight is that `StableValue` combines the deferred initialization of `volatile` with the optimization potential of `static final`.

### 3.2 JIT Constant-Folding

When the JIT compiler encounters a `StableValue` that has already been initialized, it folds the value directly into compiled code as a constant -- eliminating memory loads, volatile read barriers, and null checks. A method like `name.orElseSet(() -> lookupName())` compiles down to a direct constant return with no memory access at all.

## 4. API

### 4.1 StableValue.of()

The primary factory method creates a single uninitialized stable value:

```java
StableValue<Logger> logger = StableValue.of();

boolean wasSet = logger.trySet(Logger.getLogger("MyApp")); // at-most-once
Logger log = logger.orElseSet(() -> Logger.getLogger("MyApp")); // lazy init
boolean initialized = logger.isSet();
Logger log2 = logger.orElseThrow();    // NoSuchElementException if unset
Logger log3 = logger.orElse(fallback); // returns fallback if unset
```

The `orElseSet()` method is the workhorse: it atomically checks whether the value has been set and, if not, invokes the supplier to compute and store it. The supplier is called at most once, even under concurrent access.

### 4.2 StableValue.list()

Creates a fixed-size `List` backed by stable values, where each element is independently and lazily initialized via an `IntFunction`:

```java
List<ServiceProvider> providers = StableValue.list(10, i -> loadProvider(i));
// Accessing element 3 does not trigger initialization of element 7
```

The returned list is unmodifiable. After an element is initialized, subsequent reads benefit from constant-folding.

### 4.3 StableValue.map()

Creates a `Map` with a fixed key set, where each value is lazily computed on first access:

```java
Set<String> keys = Set.of("database", "cache", "messaging");
Map<String, Connection> connections = StableValue.map(keys, key -> connect(key));
Connection db = connections.get("database"); // only "database" is initialized
```

Each entry is independently lazy. The returned map is unmodifiable.

## 5. Comparison with Existing Patterns

| Aspect | volatile + DCL | Holder Class | AtomicReference | StableValue |
|--------|---------------|--------------|-----------------|-------------|
| Write-once guarantee | No | Yes (static final) | No | **Yes** |
| Lazy initialization | Yes | Yes (static only) | Yes | **Yes** |
| Instance field support | Yes | No | Yes | **Yes** |
| Read performance | Moderate (volatile) | Excellent | Moderate (volatile) | **Excellent** |
| JIT constant-folding | No | Yes | No | **Yes** |
| Thread safety | Manual | Automatic | Manual (CAS) | **Automatic** |
| Boilerplate | High | Medium | High | **Low** |

`StableValue` generalizes the holder class idiom to work with instance fields, parameterized suppliers, and collection-like structures.

## 6. Changes from 1st Preview

JEP 528 refines the API based on feedback from JEP 502 in JDK 25:

- **API refinements**: Method signatures adjusted based on community feedback and real-world usage patterns
- **Specification clarifications**: Semantics around concurrent `orElseSet()`, null handling, and supplier exception behavior more precisely specified
- **Integration improvements**: Better interoperability with the Collections framework and `java.util.ServiceLoader`
- **JIT optimization enhancements**: More reliable constant-folding across broader usage patterns

As a preview feature, `StableValue` requires `--enable-preview`. The API may undergo further changes before finalization.

---

## 7. Performance Benefits

The performance advantage of `StableValue` stems from a deep integration with the JVM's JIT compiler:

Before initialization, `StableValue` behaves like a well-synchronized lazy holder. After initialization, the JIT compiler recognizes that the value will never change and treats it as a constant:

```
Benchmark (read throughput, higher is better)
──────────────────────────────────────────────
StableValue (after init)      ~500 ops/ns   (constant-folded, no memory access)
volatile field                 ~45 ops/ns   (memory barrier on every read)
AtomicReference.get()          ~42 ops/ns   (volatile semantics)
synchronized getter             ~5 ops/ns   (monitor acquisition)
```

This optimization applies transitively: if a method's only variable input comes from an initialized `StableValue`, the entire method can be constant-folded, enabling aggressive inlining and dead-code elimination across call chains.

## 8. Use Cases

**Service providers** -- load plugins on demand, one at a time:

```java
private final Map<String, Plugin> plugins =
    StableValue.map(discoverPluginNames(), name -> loadPlugin(name));
```

**Expensive computations** -- initialize costly resources once:

```java
private final StableValue<SecureRandom> rng = StableValue.of();
public SecureRandom getSecureRandom() {
    return rng.orElseSet(SecureRandom::getInstanceStrong);
}
```

**Configuration** -- load properties lazily on first access:

```java
private final StableValue<Properties> config = StableValue.of();
public String getProperty(String key) {
    return config.orElseSet(() -> loadProperties("app.properties")).getProperty(key);
}
```

**Per-element lazy lists** -- defer work per index:

```java
private final List<BufferedImage> images =
    StableValue.list(1000, i -> decodeImage(paths.get(i)));
```
