## 1. Overview

JEP 341 改进了 JDK 的开箱即用启动性能：在 JDK 构建过程中自动生成一个默认的 CDS（Class Data Sharing）archive 文件（`classes.jsa`），并将 `-Xshare:auto` 设为默认行为。这意味着从 JDK 12 开始，**所有 Java 应用无需任何额外配置即可享受 CDS 带来的启动加速**，JVM 会自动尝试使用预构建的共享 archive。

在此之前（JDK 12/11），用户需要手动运行 `java -Xshare:dump` 来生成 CDS archive，或者 JDK 安装程序在安装后执行此步骤。JEP 341 消除了这一摩擦点，使 CDS 成为真正的零配置优化。

---

## 2. Background

### 2.1 Manual CDS Creation

JDK 10 引入的 CDS 技术，以及 JDK 10 通过 JEP 310 扩展的 AppCDS，都要求用户主动执行步骤才能获得优化效果。对于基础 CDS（仅 bootstrap 类），流程如下：

```bash
# 手动生成默认 CDS archive
java -Xshare:dump
```

该命令会读取 JDK 内置的 class list，将核心类的 metadata 处理后写入 `lib/server/classes.jsa` 文件。然而这个步骤并非 JDK 安装的标准部分——在 tar.gz 解压安装和精简 Docker 镜像等场景中，`classes.jsa` 根本不存在。

### 2.2 Low Adoption of CDS

由于需要手动操作，CDS 的实际采用率很低。大多数开发者从未接触过 `-Xshare` 参数，容器化部署中通常缺少预生成的 archive，且 JDK 12 及之前某些配置下 `-Xshare:off` 是默认值。一项已经成熟的性能优化技术因缺少默认启用机制而未能惠及多数用户，JEP 341 正是为了解决这一问题。

## 3. What Changed

### 3.1 Archive Generated at Build Time

JEP 341 的核心变化是将 CDS archive 的生成集成到 **JDK 构建流程** 中。在 JDK 12 的 build 过程中，构建系统会在编译完 JDK 后自动执行以下步骤：

1. 使用 JDK 内置的默认 class list（包含核心 bootstrap 类和 platform 类）
2. 运行 `java -Xshare:dump` 生成 `classes.jsa`
3. 将生成的 archive 文件打包到 JDK 发行版中

这确保了无论用户通过何种方式获取 JDK（安装程序、tar.gz、Docker 镜像），都会包含一个可用的 CDS archive。

### 3.2 -Xshare:auto as Default

与 archive 文件的预构建配合，JDK 12 将 `-Xshare:auto` 设为默认的 CDS 行为：

| 模式 | 行为 | JDK 12 角色 |
|------|------|-------------|
| `-Xshare:off` | 不使用 CDS archive | 需显式指定 |
| `-Xshare:on` | 强制使用 archive，不可用则启动失败 | 需显式指定 |
| **`-Xshare:auto`** | **尝试使用 archive，不可用则正常启动** | **JDK 12 起的默认值** |

`-Xshare:auto` 模式保证了向后兼容性——如果 archive 文件缺失、损坏或与当前 JVM 配置不兼容，JVM 会静默回退到正常启动流程，不会影响应用运行。

### 3.3 classes.jsa Location

预构建的 archive 文件存放在 `$JAVA_HOME/lib/server/classes.jsa`，包含 JDK 核心类 metadata，文件大小通常 10-20MB。JVM 启动时自动在此路径查找默认 archive。

## 4. Out-of-Box Startup Improvement

JEP 341 带来的开箱即用启动改善主要体现在 JDK 核心类的加载加速上：

| 指标 | 无 CDS | 有默认 CDS | 改善 |
|------|--------|-----------|------|
| JDK 核心类加载时间 | 基线 | 减少约 30-40% | 跳过 parsing/verification |
| `java -version` 执行时间 | ~100ms | ~60-70ms | ~30% |
| 小型应用启动 | ~200ms | ~150ms | ~25% |
| 内存占用（Metaspace） | 基线 | 减少约 10-15% | 共享 metadata |

这些改善虽然幅度不如完整的 AppCDS（包含应用类），但由于是零配置的，对整个 Java 生态的实际影响更大。每个 JDK 12+ 的 JVM 进程都自动获得了这些优化，无需任何用户干预。

> **注意**: 默认 CDS archive 仅包含 JDK 核心类。要将应用类也纳入共享范围，仍需使用 JEP 310（AppCDS）的三步流程，或使用 JEP 350（Dynamic CDS Archives）在运行时自动生成。

## 5. Relationship to CDS Evolution

JEP 341 在 CDS 技术演进链中扮演"启用默认优化"的角色。它本身不引入新的共享能力，而是确保已有的 CDS 优化被所有用户默认享用：

```
CDS (JDK 5) — 技术可用，但需手动启用
  └→ AppCDS (JEP 310, JDK 10) — 扩展到应用类，仍需手动三步操作
       └→ Default CDS Archives (JEP 341, JDK 12) — 默认启用核心类 CDS ← 当前 JEP
            └→ Dynamic CDS Archives (JEP 350, JDK 13) — 自动生成应用类 archive
                 └→ AOT Class Loading & Linking (JEP 483, JDK 24) — 预完成 linking
                      └→ AOT Cache (JEP 516, JDK 26) — 完整 AOT 缓存
```

JEP 341 与相关 JEP 的协作关系：

- **JEP 310（AppCDS）**: JEP 341 使用 JEP 310 引入的 AppCDS 基础设施，但默认 archive 仅包含 JDK 类。用户仍可通过 AppCDS 流程添加应用类。
- **JEP 350（Dynamic CDS Archives）**: 在 JDK 12 中进一步降低门槛，允许应用退出时自动 dump archive。JEP 341 的默认 archive 可作为 Dynamic CDS 的 base layer。
- **JEP 483（AOT Class Loading & Linking）**: 在 JDK 12 中扩展 archive 内容至预链接，继续受益于 JEP 341 建立的默认启用模式。
- **JEP 516（AOT Cache）**: 在 JDK 12 中将 CDS 演进为完整 AOT 缓存方案。JEP 341 开创的"构建时生成、运行时自动使用"模式成为其设计基础。

---

## 6. Limitations

JEP 341 的默认 CDS archive 存在以下局限：

- **仅包含 JDK 核心类**：默认 archive 不包含应用类和第三方库，启动改善幅度有限
- **平台绑定**：archive 文件与特定 JDK 版本、平台（OS + 架构）绑定，不可跨平台使用
- **不包含 compiled code**：仅共享 class metadata，JIT 编译仍需每次启动时执行
- **64-bit only**：默认 CDS archive 仅在 64-bit JVM 上生成和使用
- **特定 GC 兼容性**：某些 GC 配置可能影响 archive 中 compressed oops 相关数据的兼容性
