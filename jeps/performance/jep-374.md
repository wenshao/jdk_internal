## 1. Overview

JEP 374 将 **biased locking** 在 JDK 15 中默认禁用，并将其标记为 deprecated。Biased locking 是一项自 JDK 6 以来一直默认启用的 HotSpot 同步优化技术，旨在减少无竞争场景下的 locking 开销。然而，随着现代硬件的发展和 HotSpot 代码库的演进，biased locking 带来的收益已不足以抵消其复杂性和维护成本。

该 JEP 的目标是：
- 将 `-XX:+UseBiasedLocking` 的默认值从 `true` 改为 `false`
- 将所有 biased locking 相关的命令行选项标记为 deprecated
- 为后续完全移除 biased locking 代码做准备

## 2. Background

### 2.1 What Is Biased Locking

Biased locking 是 HotSpot JVM 中的一项锁优化技术，核心思想是：如果一个对象的锁在其生命周期中只被一个线程获取（即无竞争场景），那么可以将该锁 "偏向"（bias）该线程，使得后续同一线程的加锁操作几乎零开销。

在传统的 thin lock（轻量级锁）实现中，即使只有一个线程反复进入 `synchronized` 块，每次加锁仍需要执行一次 **atomic CAS**（Compare-And-Swap）操作来设置 lock record。Biased locking 通过在对象的 mark word 中记录偏向线程的 ID，使得后续同一线程的加锁只需一次普通的内存比较（而非原子操作），从而消除了 CAS 的开销。

### 2.2 Mark Word Structure

在 64 位 HotSpot JVM 中，每个 Java 对象的 object header 包含一个 **mark word**（8 字节），其结构随锁状态的不同而变化。在 biased locking 启用时，mark word 的低位 bit pattern 用于区分不同的锁状态：

```
[unused:25 | hashcode:31 | unused:1 | age:4 | biased:1 | lock:2]  → unlocked (01)
[thread:54 | epoch:2     | unused:1 | age:4 | biased:1 | lock:2]  → biased   (01)
[ptr to lock record:62                                  | lock:2]  → thin lock (00)
[ptr to heavyweight monitor:62                          | lock:2]  → fat lock  (10)
```

当 biased locking 启用时，新分配的对象默认处于 "可偏向" 状态（biased bit = 1，thread ID = 0）。第一个获取锁的线程通过 CAS 将自己的 thread ID 写入 mark word，此后该线程的重复加锁只需检查 thread ID 是否匹配。

### 2.3 Biased Locking in Practice

Biased locking 的设计源于早期 Java 应用的特征——许多同步操作实际上是无竞争的。典型场景包括 `Vector`、`Hashtable`、`StringBuffer` 等 legacy 同步集合类（每个方法都是 `synchronized` 但通常单线程使用），以及启动阶段在单线程中创建和同步访问的对象。在 2006 年引入时，CAS 操作在多核处理器上的延迟相对较高，消除 CAS 可以带来可测量的性能提升。

## 3. Why Disable Biased Locking

### 3.1 Modern Hardware Makes CAS Fast

Biased locking 的核心价值在于避免 atomic CAS 的开销。然而，现代处理器（2010 年后的 x86 和 ARM 架构）中无竞争 CAS 操作的延迟已从早期多核时代的 ~20-40 ns 降低至 ~5-15 ns。在无竞争场景下，thin lock 的 CAS 操作与 biased lock 的普通比较之间的性能差距已经微乎其微，biased locking 的优化收益不再显著。

### 3.2 Maintenance Cost in HotSpot

Biased locking 的实现渗透到 HotSpot 代码库的多个子系统中，带来了沉重的维护负担：

- **Interpreter**：进入和退出 `synchronized` 块的代码需要处理 biased lock 状态检查和偏向
- **C1/C2 JIT Compilers**：编译器生成的锁代码需要包含 biased locking 的 fast path 和 slow path
- **Runtime**：biased lock 的 revocation（撤销偏向）逻辑跨越多个 runtime 文件
- **GC**：垃圾回收器需要感知 biased locking 状态，在特定 GC 阶段可能需要批量撤销偏向（bulk revocation）
- **Deoptimization**：deopt 路径需要处理 biased lock 状态的转换
- **Object identity hash code**：调用 `hashCode()` 或 `System.identityHashCode()` 会强制撤销偏向，因为 biased mark word 中没有空间存储 hash code

这些交叉依赖使得 HotSpot 中许多看似无关的变更都需要考虑 biased locking 的影响，增加了开发和代码审查的复杂度。

### 3.3 Safepoint Operations for Revocation

当其他线程尝试获取已偏向的锁时，JVM 必须撤销偏向状态（revocation）。这通常需要 **stop-the-world safepoint**——暂停所有应用线程，遍历偏向线程的栈以查找 lock record，然后将 mark word 转换为 thin lock 或 unlocked 状态。即使是单个对象的 revocation 也可能触发全局 safepoint。

虽然 HotSpot 实现了 **bulk revocation**（当某个类的撤销次数超过阈值时批量撤销）来减少 safepoint 频率，但这进一步增加了实现复杂性。在延迟敏感的应用中，revocation 引起的意外 safepoint 暂停是不可接受的。

### 3.4 Incompatibility with Compact Object Headers

**Project Lilliput** 致力于压缩 Java 对象头的大小，从 128-bit（mark word + class pointer）减少到 64-bit。在压缩的 object header 中，mark word 的可用 bit 数大幅减少，已经没有足够的空间同时容纳 biased locking 所需的 thread ID（54 bits）和 epoch（2 bits）信息。

因此，biased locking 与 compact object headers（[JEP 519](../gc/jep-519.md)）在根本上不兼容。移除 biased locking 是实现 compact object headers 的前提条件之一。

## 4. Object Header Layout: Lock States

禁用 biased locking 后，锁升级路径从 `biased → thin → fat` 简化为 `thin → fat`，mark word 中的 biasable 状态不再使用，锁状态从四种减少为三种：

| 锁状态 | Mark Word 内容 | Lock Bits | 加锁操作 |
|--------|---------------|-----------|----------|
| **Unlocked** | hashcode + age | `01` | — |
| **Biased**（已移除） | thread ID + epoch + age | `01` | 普通比较（同一线程）/ CAS（首次偏向） |
| **Thin Lock** | 指向 lock record 的指针 | `00` | 一次 CAS |
| **Fat Lock** | 指向 ObjectMonitor 的指针 | `10` | 膨胀为 monitor，系统调用（mutex） |

---

## 5. Performance Impact

禁用 biased locking 后，对现代应用的性能影响通常很小。现代框架（Spring Boot、Quarkus 等）已经避免不必要的同步，因此无可测量的差异。大量使用 `Vector`、`Hashtable` 等 legacy 同步集合的应用可能出现轻微回退（1-3%），但延迟敏感应用反而会从中受益——消除了 revocation safepoint 带来的延迟尖刺。总体而言，biased locking 在现代工作负载中的收益已不足以证明其维护成本的合理性。

## 6. How to Re-enable Biased Locking

在 JDK 15 到 JDK 17 期间，如果应用确实从 biased locking 中获益，可以通过以下方式临时重新启用：

```bash
java -XX:+UseBiasedLocking -cp myapp.jar com.example.Main
```

使用时 JVM 会输出 deprecated 警告。相关调优参数（均已 deprecated）：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `-XX:+UseBiasedLocking` | 启用 biased locking | `false`（JDK 15+） |
| `-XX:BiasedLockingStartupDelay` | 启动后延迟启用的毫秒数 | 0 |
| `-XX:BiasedLockingBulkRebiasThreshold` | 触发 bulk rebias 的撤销次数阈值 | 20 |
| `-XX:BiasedLockingBulkRevokeThreshold` | 触发 bulk revocation 的撤销次数阈值 | 40 |

> **重要**: 这些选项仅为临时过渡方案。从 JDK 18 开始，biased locking 的实现代码逐步被移除，这些参数在后续版本中不再生效。

## 7. Timeline

Biased locking 从引入到移除经历了约 15 年的生命周期：

| 版本 | 年份 | 事件 |
|------|------|------|
| JDK 6 | 2006 | Biased locking 引入并默认启用 |
| JDK 6 ~ 14 | 2006-2020 | 默认启用状态，作为标准同步优化 |
| **JDK 15** | **2020** | **JEP 374: 默认禁用并 deprecated** |
| JDK 18 | 2022 | 开始从 HotSpot 代码库中移除 biased locking 实现 |
| JDK 24 | 2025 | JEP 519（Compact Object Headers）进入预览，利用释放的 header 空间 |
