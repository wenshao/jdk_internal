# JEP 363: Remove the Concurrent Mark Sweep (CMS) Garbage Collector

> **Status**: Final | **Type**: Feature | **Component**: HotSpot / GC
>
> **Lead**: Stefan Karlsson (Oracle)
>
> **JDK Version**: JDK 14 (2020-03)
>
> **Organization**: Oracle

---

## 概览

JEP 363 从 HotSpot 虚拟机中彻底移除了 Concurrent Mark Sweep（CMS）垃圾收集器的实现代码。CMS 早在 JDK 9 中通过 [JEP 291](jep-291.md) 被标记为废弃（deprecated），并在使用时发出警告。经过两个主要版本的过渡期，JDK 14 正式完成了 CMS 的移除。

---

## 背景与历史

### CMS 的起源

CMS 垃圾收集器最早在 **JDK 1.4.2**（2003 年）中引入，是 HotSpot 中第一个真正意义上的并发垃圾收集器。它的设计目标是减少 GC 暂停时间，以满足对延迟敏感的服务端应用的需求。

在当时，可选的垃圾收集器只有 Serial GC 和 Parallel GC，两者都需要完全的 Stop-The-World 暂停来完成垃圾回收。CMS 通过在应用线程运行期间并发地执行大部分标记工作，显著降低了暂停时间，成为许多 Java 服务端应用的首选 GC。

### CMS 的工作原理

CMS 的全称是 Concurrent Mark-Sweep，其回收过程分为以下阶段：

```
应用线程:  ====[STW]=============================[STW]=======
CMS 线程:       初始标记 → 并发标记 → 并发预清理 → 最终标记 → 并发清除

阶段说明:
1. 初始标记 (Initial Mark)     — STW，标记 GC Roots 直接引用的对象
2. 并发标记 (Concurrent Mark)   — 与应用并发，遍历对象图
3. 并发预清理 (Preclean)        — 与应用并发，处理脏卡
4. 最终标记 (Remark)           — STW，修正并发标记期间的变化
5. 并发清除 (Concurrent Sweep)  — 与应用并发，回收死亡对象
```

CMS 的关键特征：

- **低暂停时间**：只有初始标记和最终标记需要 STW 暂停，暂停时间通常较短
- **并发执行**：大部分 GC 工作与应用线程并发进行
- **不压缩（Non-compacting）**：CMS 不移动存活对象，因此不会整理堆内存
- **内存碎片化**：由于不压缩，长时间运行后可能产生内存碎片

## 废弃与移除的动机

### 1. 维护负担沉重

CMS 的代码实现复杂度极高，与 HotSpot 的其他子系统有深度耦合。维护 CMS 需要大量工程资源，这些资源本可以用于改进更现代的垃圾收集器。CMS 的非压缩式设计导致了许多特殊代码路径，增加了 HotSpot 整体的维护难度。

### 2. G1 GC 已成为默认收集器

自 JDK 9 起（通过 [JEP 248](../jep-248.md)），G1 GC 取代 Parallel GC 成为默认垃圾收集器。G1 的设计目标涵盖了 CMS 的优势（低暂停时间），同时克服了 CMS 的主要缺陷：

| 问题 | CMS | G1 |
|------|-----|-----|
| **内存碎片化** | 不压缩，碎片化严重 | 区域化整理，避免碎片 |
| **Full GC** | 回退到 Serial Old，暂停极长 | 有自己的 Full GC 实现 |
| **暂停时间可预测性** | 暂停时间不可控 | 可设置暂停时间目标 |
| **大堆支持** | 大堆下 Full GC 灾难性 | 设计上支持大堆 |

### 3. JEP 291 的废弃警告

JEP 291 在 JDK 9 中正式废弃了 CMS，使用 `-XX:+UseConcMarkSweepGC` 时会打印以下警告：

```
Java HotSpot(TM) 64-Bit Server VM warning: Option UseConcMarkSweepGC was
deprecated in version 9.0 and will likely be removed in a future release.
```

从 JDK 9 到 JDK 14 经过了五个版本（JDK 9、10、11、12、13），给予了社区充分的迁移时间。

### 4. 更先进的低延迟 GC 已出现

JDK 11 及之后引入了更先进的低延迟垃圾收集器，进一步降低了保留 CMS 的必要性：

- **ZGC**（[JEP 333](jep-333.md)，JDK 11 实验性引入）：亚毫秒级暂停，支持 TB 级堆
- **Shenandoah**（[JEP 189](jep-189.md)，JDK 12 实验性引入）：并发压缩，低暂停

## 移除的具体影响

### 代码层面

- 移除了 CMS 相关的所有 C++ 源代码（约数万行）
- 移除了 CMS 特有的 JVM 选项（如 `-XX:+UseConcMarkSweepGC`、`-XX:CMSInitiatingOccupancyFraction` 等）
- 简化了 HotSpot GC 子系统的代码结构

### 用户层面

在 JDK 14 中使用 `-XX:+UseConcMarkSweepGC` 会导致 JVM 启动失败：

```bash
$ java -XX:+UseConcMarkSweepGC MyApp
Error: Could not create the Java Virtual Machine.
Error: A fatal exception has occurred. Program will exit.
Unrecognized VM option 'UseConcMarkSweepGC'
```

## 迁移指南

### 推荐的替代收集器

| 场景 | 推荐 GC | 启用方式 |
|------|---------|---------|
| **通用服务端应用** | G1 GC（默认） | `-XX:+UseG1GC` 或不指定 |
| **超低延迟需求** | ZGC | `-XX:+UseZGC` |
| **超低延迟需求（替代）** | Shenandoah | `-XX:+UseShenandoahGC` |
| **吞吐量优先** | Parallel GC | `-XX:+UseParallelGC` |

### G1 GC 迁移建议

对于大多数 CMS 用户，直接切换到 G1 GC 是最简单的路径：

```bash
# CMS (已移除)
# java -XX:+UseConcMarkSweepGC -Xmx4g MyApp

# G1 (推荐替代)
java -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -Xmx4g MyApp
```

需要注意的 CMS 专属选项在 G1 中没有直接对应：

- `-XX:CMSInitiatingOccupancyFraction` → G1 使用 `-XX:InitiatingHeapOccupancyPercent`
- `-XX:+UseCMSInitiatingOccupancyOnly` → G1 中无需此选项
- `-XX:+CMSClassUnloadingEnabled` → G1 默认卸载类

## 时间线总结

| 时间 | 事件 |
|------|------|
| **JDK 1.4.2**（2003） | CMS 垃圾收集器引入 |
| **JDK 8**（2014） | CMS 仍被广泛使用 |
| **JDK 9**（2017） | G1 成为默认 GC；JEP 291 废弃 CMS |
| **JDK 11**（2018） | ZGC 实验性引入，提供更好的低延迟选择 |
| **JDK 14**（2020） | **JEP 363 正式移除 CMS** |
