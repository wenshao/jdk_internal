# JEP 404: Generational Shenandoah (Experimental)

> **Status**: Experimental | **Type**: Feature | **Component**: HotSpot / GC
>
> **Lead**: Roman Kennke (Datadog；当时 Red Hat 开始开发)
>
> **JDK Version**: JDK 26 (预计 2026-09)
>
> **Organization**: Red Hat

---

## 概览

为 Shenandoah 垃圾收集器增加实验性的分代收集模式（Generational Mode）。分代 Shenandoah 将堆划分为年轻代和老年代，利用弱分代假说（Weak Generational Hypothesis）减少并发标记开销，提升吞吐量，同时保持 Shenandoah 原有的低暂停时间特性。

---

## 背景

### Shenandoah 的演进

Shenandoah GC 由 Red Hat 主导开发，经历了多个关键阶段：

1. **[JEP 189](jep-189.md) (JDK 12, 2019-03)**：Shenandoah 作为实验性功能首次引入 OpenJDK
2. **[JEP 379](jep-379.md) (JDK 15, 2020-09)**：提升为生产就绪的标准功能，移除实验性标志
3. **JDK 15 → JDK 25**：持续优化性能，但始终采用非分代（non-generational）设计

### 非分代设计的局限

在非分代模式下，Shenandoah 将所有对象统一管理，不区分对象的存活时间：

```
非分代 Shenandoah:
┌──────────────────────────────────────────┐
│              统一堆空间                    │
│  短命对象 + 长寿对象 混合存放              │
│  每次 GC 都需要扫描整个堆                  │
│  并发标记遍历所有存活对象                   │
└──────────────────────────────────────────┘
```

这种设计带来几个问题：

- **标记开销大**：每次并发标记都需要遍历整个堆中的所有存活对象
- **CPU 占用高**：大量 CPU 时间花费在扫描本不需要回收的长寿对象上
- **吞吐量受限**：频繁的全堆标记降低了应用可用的 CPU 时间

## 为什么需要分代

### 弱分代假说

弱分代假说（Weak Generational Hypothesis）是垃圾收集领域最重要的经验观察之一：

> 大多数对象在创建后很快就变得不可达（"朝生夕灭"），只有少数对象会长期存活。

实际生产环境中的典型数据：

| 对象类型 | 占比 | 特征 |
|----------|------|------|
| 短命对象（年轻代） | 80-95% | 方法局部变量、临时对象、中间计算结果 |
| 长寿对象（老年代） | 5-20% | 缓存、连接池、全局配置、长期数据结构 |

### 分代带来的优势

1. **减少标记开销**：年轻代 GC 只需标记年轻代对象，无需遍历整个堆
2. **提升吞吐量**：更少的 GC 工作意味着更多 CPU 时间留给应用
3. **降低内存占用**：短命对象更快回收，降低堆内存水位
4. **老年代低频回收**：长寿对象所在的老年代只需偶尔回收

## 工作原理

### 堆空间划分

分代 Shenandoah 将堆划分为两个逻辑代：

```
分代 Shenandoah 堆布局:
┌────────────────────┬─────────────────────────┐
│     年轻代          │        老年代             │
│   (Young Gen)      │      (Old Gen)           │
│                    │                           │
│ - 新分配的对象      │ - 经过多次 GC 存活的对象    │
│ - 频繁回收          │ - 低频回收                 │
│ - 快速并发标记      │ - 偶尔并发标记             │
│ - 大部分对象短命    │ - 大部分对象长寿            │
└────────────────────┴─────────────────────────┘
```

### 并发年轻代 GC

年轻代回收是分代 Shenandoah 的核心优化点：

```
年轻代 GC 周期:
  STW            并发              STW           并发
+---------+  +--------------+  +---------+  +---------------+
| 初始标记 |->| 并发标记(仅   |->| 再标记  |->| 并发疏散       |
| (roots) |  |  年轻代对象)  |  |         |  | (移动存活对象)  |
+---------+  +--------------+  +---------+  +---------------+
 (~0.1ms)    (仅扫描年轻代)    (~0.1ms)     (并发复制到老年代)
```

由于只扫描年轻代的存活对象，标记阶段的工作量大幅减少，GC 周期更短、更高效。

### 记忆集（Remembered Sets）

当老年代对象引用年轻代对象时，年轻代 GC 需要知道这些跨代引用，否则可能错误回收仍被引用的对象。分代 Shenandoah 使用记忆集来追踪这些引用：

```
老年代                     年轻代
┌──────────┐              ┌──────────┐
│ Object A │──引用──────>  │ Object X │
│          │              │ (不能回收) │
└──────────┘              └──────────┘
     │
     └── 记录在记忆集中 (Remembered Set)

年轻代 GC 时：
  GC Roots + 记忆集中的跨代引用 → 标记年轻代存活对象
```

记忆集通过写屏障（write barrier）维护：每当应用线程更新老年代对象的引用字段时，写屏障会检查是否产生了新的跨代引用并记录下来。

### 老年代 GC

老年代回收频率低于年轻代，采用与原始 Shenandoah 类似的全并发方式：

- 并发标记老年代存活对象
- 并发疏散（evacuate）碎片化的老年代区域
- 可与年轻代 GC 交替执行，避免长时间的全堆回收

## 使用方式

### 启用分代模式（实验性）

```bash
# 需要解锁实验选项
-XX:+UnlockExperimentalVMOptions -XX:+UseShenandoahGC -XX:ShenandoahGCMode=generational
```

### 与非分代模式切换

```bash
# 非分代模式（默认，生产就绪）
-XX:+UseShenandoahGC

# 分代模式（实验性，需要解锁）
-XX:+UnlockExperimentalVMOptions -XX:+UseShenandoahGC -XX:ShenandoahGCMode=generational
```

### 推荐调优参数

```bash
# 基础分代配置
-XX:+UnlockExperimentalVMOptions -XX:+UseShenandoahGC \
  -XX:ShenandoahGCMode=generational -Xmx8g

# 指定并发 GC 线程数
-XX:+UnlockExperimentalVMOptions -XX:+UseShenandoahGC \
  -XX:ShenandoahGCMode=generational -XX:ConcGCThreads=4

# 大堆应用
-XX:+UnlockExperimentalVMOptions -XX:+UseShenandoahGC \
  -XX:ShenandoahGCMode=generational -Xmx64g
```

## 与 Generational ZGC 的对比

[JEP 439](jep-439.md) 在 JDK 21 为 ZGC 引入了分代模式。分代 Shenandoah 与分代 ZGC 在目标上相似，但实现路径不同：

| 特性 | 分代 Shenandoah (JEP 404) | 分代 ZGC (JEP 439) |
|------|--------------------------|---------------------|
| **开发者** | Red Hat | Oracle |
| **引入版本** | JDK 26（实验性） | JDK 21（实验性） |
| **核心指针技术** | Brooks 指针 | 着色指针（Colored Pointers） |
| **屏障类型** | 读屏障 + 写屏障（分代需要） | 读屏障 + 存储屏障（分代需要） |
| **跨代引用追踪** | 记忆集（Remembered Sets） | 记忆集（Remembered Sets） |
| **并发压缩** | 是 | 是 |
| **最大堆** | 无硬限制 | 16TB |
| **生产就绪** | 实验阶段 | JDK 23 成为默认 |

两者都遵循相同的分代假说原理，主要区别在于底层指针处理和屏障实现机制。

## 性能预期

### 相比非分代 Shenandoah

| 指标 | 非分代 Shenandoah | 分代 Shenandoah（预期） |
|------|-------------------|------------------------|
| GC 暂停时间 | <10ms | <10ms（保持） |
| 并发标记开销 | 高（全堆） | 低（仅年轻代为主） |
| 吞吐量 | 基准 | 提升 10-30% |
| 内存占用 | 基准 | 略降（短命对象更快回收） |
| CPU 开销 | 基准 | 降低（更少的标记工作） |

### 预期改善场景

- **对象分配密集型应用**：大量短命对象的应用受益最大
- **大堆应用**：堆越大，减少全堆标记带来的收益越显著
- **Web 服务**：请求处理中产生的临时对象可快速回收
- **数据处理**：流式处理中的中间对象更高效回收

### 注意事项

- 分代模式引入额外的写屏障开销用于维护记忆集
- 在跨代引用极多的工作负载中，记忆集维护成本可能抵消分代收益
- 作为实验性功能，可能存在稳定性和性能波动
