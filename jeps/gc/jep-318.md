## 概览

Epsilon 是一个"无操作"垃圾收集器：它负责内存分配，但从不回收任何内存。当堆内存耗尽时，JVM 直接抛出 `OutOfMemoryError` 并终止。Epsilon 的名称来源于数学中表示"极小量"的符号 ε，意指该 GC 做的工作量趋近于零。

---

## 背景与动机

在 JDK 11 之前，所有 JVM 垃圾收集器都会执行实际的垃圾回收工作。然而，在某些场景下，开发者需要一个"什么都不做"的 GC：

1. **性能测试基线**：要衡量 GC 对应用性能的影响，需要一个不含 GC 开销的基线。使用 Epsilon 运行测试，再与其他 GC 的结果对比，可以精确量化 GC 的影响。

2. **短生命周期应用**：某些批处理任务或容器化微服务在堆耗尽之前就已完成工作，GC 回收内存是不必要的开销。

3. **内存压力测试**：在已知内存预算下，验证应用不会超出内存限制。

4. **GC 接口测试**：作为最小化的 GC 实现，用于验证 JVM 的 GC 接口是否正确。

---

## 设计原理

### 核心机制

Epsilon 的设计极其简单：

```
分配请求 → 线性分配 → 成功/OOM

+--------------------------------------------------+
|                     堆 (Heap)                      |
| [已分配][已分配][已分配][  ...  ][   未使用空间   ] |
|                                  ↑                 |
|                             分配指针               |
+--------------------------------------------------+
```

- **线性分配**：通过简单的指针碰撞（bump pointer）分配内存，效率极高
- **无回收**：已分配的内存永远不会被释放或回收
- **无 GC 暂停**：不存在 STW（Stop-The-World）暂停
- **OOM 终止**：堆空间用尽时抛出 `OutOfMemoryError`

## 使用场景

### 1. 性能测试（隔离 GC 影响）

```bash
# 使用 Epsilon 建立无 GC 基线
java -XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC \
     -Xmx4g MyBenchmark

# 使用 G1 GC 运行相同测试
java -XX:+UseG1GC -Xmx4g MyBenchmark

# 对比结果即可量化 G1 GC 的开销
```

### 2. 内存压力测试

```bash
# 验证应用在 512MB 内存限制下能否正常完成
java -XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC \
     -Xmx512m -XX:+HeapDumpOnOutOfMemoryError \
     MyApplication
```

### 3. 短生命周期批处理

```bash
# 容器中运行的一次性批处理任务
java -XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC \
     -Xmx2g BatchProcessor input.csv output.csv
```

### 4. 超低延迟场景

在对延迟要求极端严格的场景中，Epsilon 消除了所有 GC 暂停——前提是应用的内存使用量在堆容量范围内。

### 5. VM 接口测试

作为 GC 接口的最小实现，验证 HotSpot 的 GC 抽象层是否正确工作。

## 使用方式

### 启用 Epsilon GC

```bash
# 需要先解锁实验性选项
java -XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC [其他选项] MyApp
```

### 常用配置

```bash
# 大堆 + 监控分配速率
-XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC -Xmx8g -Xlog:gc

# OOM 时生成堆转储
-XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC \
    -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof
```

## 实现细节

Epsilon 是 GC 接口的最小化实现：

| 组件 | Epsilon 实现 |
|------|-------------|
| **内存分配** | 线性指针碰撞（bump pointer） |
| **垃圾回收** | 不实现 |
| **标记（Mark）** | 不实现 |
| **整理（Compact）** | 不实现 |
| **并发处理** | 不需要 |
| **写屏障 / 读屏障** | 不需要 |
| **TLAB 支持** | 支持（用于线程本地分配） |

Epsilon 的实现代码量极小（约几百行 C++），体现了 JVM GC 接口的简洁性。

## GC 对比

| 特性 | **G1 GC** | **ZGC** | **Shenandoah** | **Epsilon** |
|------|-----------|---------|----------------|-------------|
| **暂停时间** | 毫秒级 | 亚毫秒级 | 亚毫秒级 | 无暂停 |
| **内存回收** | 是 | 是 | 是 | **否** |
| **吞吐量开销** | 低 | 中 | 中 | **无** |
| **适合长期运行** | 是 | 是 | 是 | **否** |
| **适合短任务** | 是 | 是 | 是 | **是** |
| **屏障开销** | 写屏障 | 读/写屏障 | 读/写屏障 | **无屏障** |
| **分配效率** | 中 | 中 | 中 | **最高** |
| **内存利用率** | 高 | 高 | 高 | 单调递减 |
| **生产环境推荐** | 是 | 是 | 是 | **否**（特殊场景除外） |

## 何时不应使用 Epsilon

Epsilon 是一个特殊用途的 GC，以下场景**不应**使用：

1. **长时间运行的服务**：堆内存最终会耗尽，导致 OOM 崩溃
2. **内存分配量不可预测的应用**：无法保证在堆耗尽前完成工作
3. **生产环境的通用服务**：除非有非常明确的理由和充分的内存预算
4. **需要 GC 回收内存的场景**：这是显而易见的——Epsilon 不会回收任何内存
5. **堆内存受限的环境**：小堆 + Epsilon = 快速 OOM

> **警告**：Epsilon 要求 `-XX:+UnlockExperimentalVMOptions`，这本身就是一种安全机制，
> 提醒开发者这不是一个通用的 GC 选择。
