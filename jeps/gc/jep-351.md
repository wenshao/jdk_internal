# JEP 351: ZGC: Uncommit Unused Memory

> **Status**: Permanent | **Type**: Feature | **Component**: HotSpot / GC
>
> **Lead**: Per Liden (Oracle)
>
> **JDK Version**: JDK 13 (2019-09)
>
> **Organization**: Oracle

---

## 概览

增强 ZGC，使其能够将未使用的堆内存归还给操作系统。在此之前，ZGC 一旦分配了物理内存，就不会将其释放回 OS，即使这些内存长时间未被使用。JEP 351 解决了这一问题，使 ZGC 能够主动监测堆的使用情况，并在内存空闲一段时间后将其取消提交（uncommit），归还给操作系统。

---

## 背景

### ZGC 初始状态

ZGC 在 JDK 11 中通过 [JEP 333](jep-333.md) 作为实验性功能引入。早期实现专注于核心的低延迟 GC 能力——着色指针、读屏障、并发标记和并发重定位——但**不具备将未使用内存归还操作系统的能力**。

这意味着 ZGC 的内存占用表现为"只增不减"：堆内存一旦被提交（commit），即使后续不再需要，也会一直被 JVM 持有。

### 为什么需要归还内存

在传统的裸机服务器环境中，这个限制的影响相对较小。但在以下场景中，这成为了一个严重的问题：

- **容器环境**：在 Kubernetes / Docker 容器中，每个容器有严格的内存配额（cgroup 限制）。如果 GC 不归还内存，容器即使在低负载期间也会占用峰值内存，导致资源浪费
- **云环境按需付费**：云实例的内存直接关联费用，无法释放的内存意味着不必要的开销
- **多租户部署**：多个 JVM 共享同一物理主机时，一个 JVM 长期持有不需要的内存会影响其他应用的可用资源
- **负载波动场景**：许多服务的内存需求随时间变化显著（如白天高峰、夜间低谷），GC 应能适应这种变化

值得注意的是，G1 GC 和 Shenandoah GC 在当时已经具备了将未使用内存归还操作系统的能力，ZGC 缺乏这一功能使其在容器和云环境中的吸引力受限。

## 工作原理

### 内存管理模型

ZGC 使用基于页面（ZPage）的内存管理。堆被划分为三种大小的页面：

| 页面类型 | 大小 | 用途 |
|----------|------|------|
| Small | 2MB | 存放小对象（≤256KB） |
| Medium | 32MB | 存放中等对象（256KB–4MB） |
| Large | 可变（2MB 的倍数） | 存放大对象（>4MB） |

当一个 ZPage 不再包含任何存活对象时，它就成为可回收的空闲页面。

### 取消提交流程

JEP 351 引入了一个后台线程，负责监控和执行内存取消提交：

```
1. GC 回收完成后，空闲的 ZPage 被放入页面缓存（ZPageCache）
2. 后台线程定期检查页面缓存中的空闲页面
3. 如果某个页面的空闲时间超过配置的延迟阈值：
   a. 将该页面从页面缓存中移除
   b. 调用 OS 接口取消提交物理内存（如 madvise(MADV_DONTNEED)）
   c. 虚拟地址空间保留不变，仅释放物理内存
4. 当应用再次需要更多内存时，重新提交（commit）物理内存
```

### 配置参数

核心配置参数是 `-XX:ZUncommitDelay`，控制页面空闲多久后才会被取消提交：

```bash
# 默认延迟：300 秒（5 分钟）
-XX:+UseZGC

# 自定义延迟：60 秒
-XX:+UseZGC -XX:ZUncommitDelay=60

# 禁用取消提交（保持旧行为）
-XX:+UseZGC -XX:-ZUncommit
```

延迟机制的设计是有意义的：避免在内存需求短暂下降后立刻释放内存，随后又因需求回升而重新分配。频繁的提交/取消提交操作本身有性能代价。

## 与 -Xms / -Xmx 的交互

### 基本规则

取消提交功能遵循一个关键约束：**堆大小不会缩减到 -Xms（初始堆大小）以下**。

```bash
# 场景 1：-Xms < -Xmx → 可以取消提交
# 堆可以在 512m 和 4g 之间动态伸缩
-XX:+UseZGC -Xms512m -Xmx4g

# 场景 2：-Xms = -Xmx → 取消提交被隐式禁用
# 堆大小固定，不会缩减
-XX:+UseZGC -Xms4g -Xmx4g

# 场景 3：未设置 -Xms → 默认行为
# JVM 自行决定初始堆大小，可正常取消提交
-XX:+UseZGC -Xmx4g
```

### 设计考量

当 `-Xms` 等于 `-Xmx` 时隐式禁用取消提交是合理的：用户明确设定固定堆大小，通常是出于性能预测的需要（如避免页面错误带来的延迟抖动），此时不应违背用户的意图。

## 容器与云环境的收益

### 内存效率提升

在容器化部署中，JEP 351 带来了实际的内存节省：

- **降低内存配额需求**：容器不再需要按峰值内存分配配额，可以设置更接近平均使用量的限制
- **提升集群密度**：更多容器可以部署在同一节点上，因为非峰值时段释放的内存可供其他容器使用
- **改善弹性伸缩**：与 Kubernetes 的 VPA（Vertical Pod Autoscaler）配合更有效

### 推荐容器配置

```bash
# 容器环境推荐配置
-XX:+UseZGC \
-Xmx$(( MEMORY_LIMIT * 75 / 100 ))m \
-XX:ZUncommitDelay=120
```

设置较短的 `ZUncommitDelay`（如 120 秒）可以使内存更快地归还，适合负载波动频繁的微服务。

## 实现细节

### 平台支持

取消提交的底层实现依赖操作系统的内存管理接口：

| 平台 | 系统调用 | 说明 |
|------|----------|------|
| Linux | `madvise(MADV_DONTNEED)` | 通知内核释放物理页帧 |
| macOS | `madvise(MADV_FREE)` | 标记页面为可回收（JDK 14+ ZGC on macOS） |
| Windows | `VirtualFree(MEM_DECOMMIT)` | 取消提交虚拟内存页（JDK 14+ ZGC on Windows） |

### 与透明大页的关系

当启用透明大页（Transparent Huge Pages, THP）时，取消提交和重新提交的粒度可能受到影响。操作系统可能需要拆分或重组大页，这会引入额外的延迟。在延迟极其敏感的场景中，需要权衡大页带来的 TLB 命中率提升与取消提交操作的额外开销。
