## 1. 背景与动机

### 问题

Shenandoah GC 是低延迟垃圾回收器，但之前版本存在效率问题：

```
传统 Shenandoah:
- 所有对象统一处理
- 年轻代对象（短命）和老年代对象（长寿）同等对待
- 大量 CPU 时间用于扫描短命对象
```

### 解决方案

JEP 521 引入 **分代 Shenandoah**：
- 分离年轻代和老年代
- 年轻代对象频繁回收
- 老年代对象较少回收
- 提升吞吐量，保持低延迟

## 2. 使用示例

### 启用分代模式

```bash
# 启用 Shenandoah 分代模式
java -XX:+UseShenandoahGC -XX:ShenandoahGCMode=generational MyApp

# 或使用简化选项
java -XX:+UseShenandoahGC -XX:+ShenandoahGenerational MyApp
```

### 配置选项

```bash
# 分代模式自动管理年轻代大小，无需额外 JVM 参数

# 混合回收阈值
-XX:ShenandoahMixedGCThreshold=50

# 并发线程数
-XX:ConcGCThreads=4
```

## 3. 实现分析

### 分代架构

```
┌─────────────────────────────────────────────────────┐
│                    Heap                              │
├─────────────────────────┬───────────────────────────┤
│     Young Generation    │      Old Generation       │
│  ┌─────┬─────┬─────┐   │   ┌─────┬─────┬─────┐    │
│  │ Eden│ Eden│ Eden│   │   │ Old │ Old │ Old │    │
│  └─────┴─────┴─────┘   │   └─────┴─────┴─────┘    │
│  ┌─────┬─────┐         │                           │
│  │Surv │Surv │         │                           │
│  └─────┴─────┘         │                           │
└─────────────────────────┴───────────────────────────┘
```

### 回收策略

```
Young GC:
- 仅回收年轻代
- 频率高，速度快
- 大多数对象在年轻代死亡

Mixed GC:
- 回收年轻代 + 部分老年代区域
- 老年代区域根据垃圾比例选择

Full GC:
- 回收整个堆
- 极少发生
```

## 4. 开发者影响

### 性能对比

| 场景 | 传统 Shenandoah | 分代 Shenandoah |
|------|-----------------|-----------------|
| 分配密集 | 基准 | +15-25% |
| 短命对象多 | 基准 | +20-40% |
| 长命对象多 | 基准 | +5-10% |
| 混合负载 | 基准 | +10-20% |

### GC 暂停时间

```
传统模式:
- Young GC: 5-10 ms
- Mixed GC: 10-20 ms

分代模式:
- Young GC: 2-5 ms (更快)
- Mixed GC: 5-15 ms
```

### 最佳实践

```bash
# 推荐配置
-XX:+UseShenandoahGC
-XX:ShenandoahGCMode=generational
# 分代模式自动管理年轻代大小，无需额外 JVM 参数

# 大堆场景
-Xmx32g

# 低延迟场景
-XX:MaxGCPauseMillis=10
```
