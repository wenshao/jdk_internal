# JEP 344: Abortable Mixed Collections for G1

> **Status**: Final | **Type**: Feature | **Component**: HotSpot / GC
>
> **Lead**: Thomas Schatzl (Oracle)
>
> **JDK Version**: JDK 12 (2019-03)
>
> **Organization**: [Oracle](../../contributors/orgs/oracle.md)

---

## 概览

JEP 344 使 G1 垃圾收集器的混合回收（Mixed Collection）变为可中止的。当 G1 检测到混合回收可能超过暂停时间目标（`-XX:MaxGCPauseMillis`）时，可以提前中止回收，仅处理已完成的部分，将剩余工作推迟到后续周期。这使 G1 更好地遵守暂停时间承诺，提供更可预测的延迟表现。

---

## 背景与动机

### G1 混合回收的问题

G1 垃圾收集器使用启发式算法来选择混合回收的回收集合（Collection Set）。它根据历史数据预测每个区域（Region）的回收耗时，然后选择回收效率最高的区域集合。然而，这种预测并不总是准确的：

1. **预测误差**：回收一个区域的实际时间可能与预测值偏差较大，尤其是对象图结构复杂时
2. **不可中止的承诺**：一旦 G1 确定了回收集合并开始回收，即使已经超过暂停时间目标，也必须完成所有区域的回收
3. **暂停时间超标**：当预测不准确时，实际暂停时间可能显著超过 `-XX:MaxGCPauseMillis` 的设定值

```
JDK 12 之前的混合回收流程：

选择回收集合 → 开始回收 → 必须完成所有区域 → 暂停结束
                              ↑
                         即使已超时也无法中止
                         导致暂停时间超标
```

### 暂停时间目标的重要性

`-XX:MaxGCPauseMillis`（默认 200ms）是 G1 最核心的调优参数。对于延迟敏感的应用——如在线交易系统、实时数据处理——GC 暂停时间的可预测性至关重要。混合回收频繁超过暂停时间目标会导致应用延迟尖刺（latency spike），影响用户体验和 SLA 达成。

---

## 设计原理

### 集合分割：必须部分与可选部分

JEP 344 的核心设计是将混合回收的回收集合分割为两个部分：

```
回收集合（Collection Set）
+-------------------------------------------+
|  必须部分（Mandatory）                      |
|  - 所有年轻代区域（Eden + Survivor）        |
|  - 少量高优先级老年代区域                    |
|  必须完成回收，不可中止                      |
+-------------------------------------------+
|  可选部分（Optional）                       |
|  - 额外的老年代区域                         |
|  - 按回收效率排序                           |
|  - 可在时间不足时中止                       |
+-------------------------------------------+
```

- **必须部分**：包含所有年轻代区域和一小部分确定能在时间预算内完成的老年代区域。这部分必须完成回收，以保证 GC 的正确性
- **可选部分**：包含额外的老年代区域，按照回收效率从高到低排序。G1 在完成必须部分后，逐步处理可选部分中的区域

### 中止机制

G1 在处理可选部分时，每完成一个区域（或一小批区域）后会检查已用时间：

```
混合回收流程：

1. 回收必须部分（所有年轻代 + 部分老年代）
   ↓
2. 检查剩余时间预算
   ↓
3. 有剩余时间？
   ├── 是 → 回收下一批可选区域 → 回到步骤 2
   └── 否 → 中止回收，记录未完成的区域
   ↓
4. 暂停结束，未回收的区域留待下次混合回收
```

可选部分按小批次（chunk）处理，每个批次完成后进行时间检查。这种设计在中止灵活性和回收效率之间取得平衡。

## 与暂停时间目标的交互

JEP 344 显著改善了 G1 对 `-XX:MaxGCPauseMillis` 的遵守能力：

| 场景 | JDK 12 之前 | JDK 12 之后 |
|------|------------|------------|
| **预测准确** | 按时完成 | 按时完成（行为不变） |
| **预测偏高** | 超时完成，无法中止 | 中止可选部分，按时结束 |
| **极端情况** | 暂停时间大幅超标 | 必须部分仍可能超时，但影响大幅缩小 |

需要注意的是，必须部分的回收是不可中止的。如果必须部分本身的预测不准确，暂停时间仍可能超过目标。但由于必须部分的规模被刻意控制得较小，超标幅度通常有限。

---

## 实现细节

### 必须部分的大小控制

G1 在选择必须部分时采用保守策略：所有年轻代区域（Eden 和 Survivor）始终包含在必须部分中，老年代区域仅在预测时间有足够余量时才加入，预测的总时间不应超过暂停时间目标的一定比例。

### 可选部分的排序与未完成区域处理

可选区域按"回收效率"（每单位时间能回收的空间量）排序，高效率区域优先回收，确保即使提前中止也能最大化回收收益。被中止而未回收的可选区域不会丢失——它们会被纳入下一次混合回收的候选集合，G1 通过多次混合回收周期逐步完成所有需要回收的老年代区域。

---

## 延迟可预测性的改善

JEP 344 对延迟敏感型应用带来了实质性改善：

1. **更少的暂停超标事件**：可选部分的中止机制有效避免了长时间暂停
2. **更平滑的延迟分布**：GC 暂停时间的 P99/P999 百分位数显著降低
3. **更可靠的 SLA 达成**：应用更容易满足延迟相关的服务等级协议
4. **降低调优难度**：用户无需过于保守地设置 `-XX:MaxGCPauseMillis`，G1 能自适应调整

对于已经使用 G1 的应用，升级到 JDK 12 后无需任何配置更改即可自动受益于此改进。
