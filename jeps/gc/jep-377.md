# JEP 377: ZGC: A Scalable Low-Latency Garbage Collector (Production)

> **Status**: Permanent | **Type**: Feature | **Component**: HotSpot / GC
>
> **Lead**: Per Liden (Oracle)
>
> **JDK Version**: JDK 15 (2020-09)
>
> **Organization**: Oracle

---

## 概览

将 ZGC（Z Garbage Collector）从实验性功能提升为生产就绪（production-ready）的标准功能。ZGC 是一款可扩展的低延迟垃圾收集器，设计目标是将 GC 暂停时间控制在亚毫秒级别，且暂停时间不随堆大小增长。

---

## 背景

ZGC 经历了多个版本的演进，最终在 JDK 15 达到生产就绪状态：

1. **JEP 333 (JDK 11, 2018-09)**：ZGC 作为实验性功能首次引入，仅支持 Linux/x86_64
2. **JEP 364 (JDK 14, 2020-03)**：将 ZGC 移植到 macOS 平台
3. **JEP 365 (JDK 14, 2020-03)**：将 ZGC 移植到 Windows 平台
4. **JDK 11 → JDK 15 (两年稳定期)**：经历约两年的生产环境验证，修复大量问题，优化性能

在此期间，ZGC 已收到来自社区和企业用户的广泛正面反馈，证明其适合在生产环境中使用。

---

## 目标

- 将 ZGC 从实验性功能提升为标准产品功能
- 移除 `-XX:+UnlockExperimentalVMOptions` 要求
- 使 ZGC 可直接通过 `-XX:+UseZGC` 启用
- 推荐用于需要超低延迟的生产环境

---

## ZGC 核心特性

### 设计目标

- **亚毫秒级暂停时间**：GC 暂停时间控制在 1ms 以下
- **暂停时间与堆大小无关**：无论堆是 256MB 还是 16TB，暂停时间保持一致
- **支持超大堆**：可处理从 8MB 到 16TB 的堆内存
- **对应用吞吐量的影响可控**：相比 G1，吞吐量下降通常在 5-15% 以内

### 着色指针（Colored Pointers）

ZGC 最核心的设计创新是着色指针——将 GC 元数据编码在对象指针的高位中：

```
 63    47 46 45 44 43 42    0
+------+--+--+--+--+---------+
| 0000 |  |  |  |  | address |
+------+--+--+--+--+---------+
         |  |  |  |
         |  |  |  +-- Remapped
         |  |  +---- Marked1
         |  +------ Marked0
         +-------- Finalizable
```

着色指针的优势：

- **快速判断对象状态**：无需访问对象头，仅通过指针即可判断对象是否被标记或重定位
- **支持并发重定位**：旧地址和新地址可同时有效
- **减少内存屏障开销**：仅需读屏障（load barrier），无需写屏障

### 读屏障（Load Barrier）

ZGC 仅使用一种屏障类型——读屏障。当应用线程从堆中加载对象引用时触发：

```
// 伪代码：ZGC 读屏障
Object load_reference(Object* field) {
    Object ref = *field;
    if (is_bad_color(ref)) {
        ref = slow_path(field, ref);  // 修正指针颜色
    }
    return ref;
}
```

读屏障确保应用线程总是看到正确（已重定位）的对象引用，使得 ZGC 可以在不暂停应用的情况下完成对象的并发移动。

## 架构

### GC 阶段

ZGC 的工作几乎全部并发完成，仅有极短的 STW（Stop-The-World）暂停：

```
         STW             并发              STW              并发
     +---------+   +---------------+   +---------+   +------------------+
     | 初始标记 |-->| 并发标记/重映射 |-->| 再标记  |-->| 并发预备重定位    |
     +---------+   +---------------+   +---------+   +------------------+
      (~0.1ms)      (与应用并发)        (~0.1ms)       (与应用并发)
                                                              |
                                                              v
                                                    +------------------+
                                                    | 并发重定位        |
                                                    +------------------+
                                                     (与应用并发)
```

### 关键阶段说明

| 阶段 | 类型 | 说明 |
|------|------|------|
| 初始标记（Init Mark） | STW | 标记 GC Roots，极短暂停（<1ms） |
| 并发标记（Concurrent Mark） | 并发 | 遍历对象图，标记存活对象 |
| 再标记（Remark） | STW | 处理标记阶段遗留的少量引用（<1ms） |
| 并发预备重定位（Concurrent Prepare Relocate） | 并发 | 选择需要压缩的内存区域（Relocation Set） |
| 并发重定位（Concurrent Relocate） | 并发 | 移动存活对象到新位置 |
| 并发重映射（Concurrent Remap） | 并发 | 更新指向旧地址的引用（与下一轮标记合并） |

## 使用方式

### JDK 15+ (标准功能)

```bash
# 直接启用 ZGC，无需解锁实验选项
-XX:+UseZGC

# JDK 11-14（实验阶段）需要额外参数
-XX:+UnlockExperimentalVMOptions -XX:+UseZGC
```

### 推荐配置

```bash
# 基础低延迟配置
-XX:+UseZGC -Xmx16g

# 大堆应用
-XX:+UseZGC -Xmx128g -XX:SoftMaxHeapSize=120g

# 调整并发 GC 线程数
-XX:+UseZGC -XX:ConcGCThreads=4

# 容器环境（配合内存限制）
-XX:+UseZGC -XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0
```

## 性能特性与权衡

### 暂停时间对比（JDK 15）

| 堆大小 | G1 GC | ZGC | 改进 |
|--------|-------|-----|------|
| 4GB | 15ms | <1ms | 15x+ |
| 16GB | 45ms | <1ms | 45x+ |
| 64GB | 150ms | <1ms | 150x+ |
| 256GB | 500ms+ | <1ms | 500x+ |

### 吞吐量对比

| 基准测试 | G1 GC (基准) | ZGC | 差异 |
|----------|-------------|-----|------|
| SPECjbb2015 | 100% | 85-95% | -5~15% |
| DaCapo | 100% | 90-95% | -5~10% |
| Renaissance | 100% | 88-95% | -5~12% |

### 延迟 vs 吞吐量权衡

- **ZGC 优势**：暂停时间极短且可预测，适合延迟敏感型应用
- **ZGC 代价**：读屏障带来额外 CPU 开销，吞吐量略低于 G1
- **适用原则**：如果应用对 P99 延迟有严格要求（<10ms），ZGC 是最佳选择；如果追求最大吞吐量，G1 或 Parallel GC 更合适

## ZGC vs G1 vs Shenandoah 对比

| 特性 | ZGC | G1 GC | Shenandoah |
|------|-----|-------|------------|
| **开发者** | Oracle | Oracle | Red Hat |
| **暂停时间目标** | <1ms | <200ms (可调) | <10ms |
| **暂停时间与堆大小关系** | 无关 | 有关 | 基本无关 |
| **最大堆** | 16TB | 无硬限制 | 无硬限制 |
| **吞吐量** | 85-95% | 95-100% (基准) | 90-95% |
| **核心屏障** | 读屏障 (Load Barrier) | 写屏障 (Write Barrier) | 读屏障 (Load Barrier) |
| **指针技术** | 着色指针 | 无 | Brooks 指针 |
| **并发压缩** | 是 | 否（增量分区回收） | 是 |
| **分代支持** | JDK 21+ (JEP 439) | 始终分代 | JDK 26 (JEP 521) |
| **默认 GC** | 否 | 是 (JDK 9+) | 否 |
| **生产就绪版本** | JDK 15 | JDK 7u4 | JDK 15 |

## 后续演进

ZGC 在成为生产功能后持续进化，最重要的方向是引入分代收集：

| JEP | JDK | 内容 | 意义 |
|-----|-----|------|------|
| [JEP 439](jep-439.md) | JDK 21 | 分代 ZGC（Generational ZGC） | 通过分代收集显著提升吞吐量和减少内存开销 |
| [JEP 474](jep-474.md) | JDK 23 | 分代模式成为默认 | `-XX:+UseZGC` 默认使用分代模式 |
| [JEP 490](jep-490.md) | JDK 24 | 移除非分代模式 | 简化代码库，非分代 ZGC 完成历史使命 |

分代 ZGC 通过将堆划分为年轻代和老年代，使得短命对象可以更快速地被回收，同时保持 ZGC 原有的低延迟特性。这一演进使 ZGC 在吞吐量方面也大幅接近 G1。

## 适用场景

### 推荐使用

- 低延迟要求的金融交易系统
- 实时数据流处理平台
- 大堆内存应用（>32GB）
- 对 GC 暂停时间敏感的微服务
- 需要可预测延迟的在线服务

### 不推荐

- CPU 密集型批处理任务（选择 Parallel GC）
- 小内存应用（<256MB，选择 Serial GC）
- 吞吐量绝对优先的离线计算（选择 G1 或 Parallel GC）
