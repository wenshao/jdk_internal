# JEP 346: Promptly Return Unused Committed Memory from G1

> **Status**: Final | **Type**: Feature | **Component**: HotSpot / GC
>
> **Lead**: [Thomas Schatzl](../../by-contributor/profiles/thomas-schatzl.md) (Oracle)
>
> **JDK Version**: JDK 12 (2019-03)
>
> **Organization**: [Oracle](../../contributors/orgs/oracle.md)

---

## 概览

JEP 346 增强了 G1 垃圾收集器，使其能够在应用空闲期间主动将未使用的 Java 堆内存归还给操作系统。在此之前，G1 一旦提交（commit）内存就几乎不会释放，即使堆中大部分区域已空闲。该特性通过周期性 GC 触发堆收缩和内存反提交（uncommit），显著降低了容器和云环境下的内存占用与计费成本。

---

## 背景与动机

### G1 的内存持有问题

在 JDK 12 之前，G1 收集器存在一个明显的不足：它几乎从不主动将已提交的堆内存归还给操作系统。即使应用负载大幅下降，堆中存在大量空闲区域，这些内存仍然被 JVM 进程持有：

```
应用高峰期:
  堆使用: ████████████████████ 8GB / 8GB committed

应用空闲期 (JDK 12 之前):
  堆使用: ████░░░░░░░░░░░░░░░░ 2GB used / 8GB committed  ← 6GB 白白浪费
  操作系统看到的 JVM RSS: ~8GB

应用空闲期 (JDK 12 之后, JEP 346):
  堆使用: ████░░░░             2GB used / 3GB committed  ← 主动归还 5GB
  操作系统看到的 JVM RSS: ~3GB
```

### 云与容器时代的新需求

在传统部署模式中，应用独占物理服务器，未使用的内存不会产生额外费用。然而云计算和容器化改变了这一格局：

- **容器环境**：容器内存限制是硬上限，JVM 持有过多内存会导致其他容器无法调度或 OOM Kill
- **云计费**：云实例按内存规格计费，不释放空闲内存意味着用户为未使用的资源付费
- **弹性伸缩**：自动伸缩依赖实际资源使用率，JVM 的"虚高"内存占用会干扰伸缩决策

## 工作原理

### 周期性 GC 触发机制

JEP 346 引入了一个后台周期性任务，在应用空闲时触发 GC 以回收和归还内存：

```
定时检查 → 距上次GC超过interval? → 系统负载低于阈值? → 触发并发GC → 堆收缩 + 反提交 → 空闲区域归还OS
```

### 堆收缩与内存反提交

当周期性 GC 完成后，G1 执行以下步骤来归还内存：

1. **识别空闲区域**：标记不包含存活对象且不属于任何分配上下文的堆区域
2. **堆收缩判定**：根据当前堆使用率和 `-Xms`，计算可以释放的区域数量
3. **反提交内存**：对空闲区域调用操作系统反提交操作（如 Linux 的 `madvise(MADV_DONTNEED)`），将物理页面归还给操作系统
4. **更新簿记**：更新 G1 的区域管理数据结构，将这些区域标记为未提交状态

堆不会收缩到 `-Xms` 以下，因此合理设置 `-Xms` 与 `-Xmx` 的差值是充分利用该特性的关键。

## 关键参数

### 核心标志

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:G1PeriodicGCInterval` | 0（禁用） | 周期性 GC 的触发间隔（毫秒）。设为非零值以启用该特性 |
| `-XX:G1PeriodicGCSystemLoadThreshold` | 0.0 | 系统平均负载阈值。仅当 1 分钟平均负载低于此值时才触发周期性 GC。设为 0.0 表示不检查负载 |
| `-XX:G1PeriodicGCInvokesConcurrent` | true | 为 true 时触发并发 GC；为 false 时触发 Full GC |

### 推荐配置

```bash
# 容器/云环境推荐配置
java -XX:+UseG1GC \
     -Xms256m -Xmx4g \
     -XX:G1PeriodicGCInterval=15000 \
     -XX:G1PeriodicGCSystemLoadThreshold=0.0 \
     -XX:MaxGCPauseMillis=200 \
     MyApplication
```

关键要点：`-Xms` 应设置得尽量小，以允许 G1 在空闲时将堆收缩到较低水位；`G1PeriodicGCInterval` 设为 0（默认值）时该特性完全禁用。

## 云与容器场景

### Kubernetes 部署

在 Kubernetes 环境中，JEP 346 配合合理的资源配置可以显著优化利用率：

- **requests** 可以设置得较低（对应 `-Xms`），提高集群调度密度
- 节点上的可用内存更真实地反映实际使用情况
- HPA 基于内存的伸缩策略更加准确

假设一个 Java 微服务集群有 20 个实例，每个 `-Xmx=8GB`，但 80% 的时间实际使用不超过 2GB。启用 JEP 346 后，空闲时段每个实例可释放约 6GB，整个集群在低峰期可节省约 120GB 内存资源。

## 局限性

1. **默认禁用**：`G1PeriodicGCInterval` 默认为 0，必须显式设置才能启用
2. **最小堆限制**：堆不会收缩到 `-Xms` 以下，若 `-Xms` 等于 `-Xmx` 则该特性无效
3. **GC 开销**：周期性 GC 本身有一定的 CPU 开销，在纯 CPU 密集型空闲检测场景中需要权衡
4. **负载判断粗糙**：`G1PeriodicGCSystemLoadThreshold` 基于系统级负载平均值，无法精确反映单个 JVM 的活跃度
5. **内存归还不即时**：从应用变空闲到内存实际归还需要至少一个 `G1PeriodicGCInterval` 周期
