# JEP 345: NUMA-Aware Memory Allocation for G1

> **Status**: Final | **Type**: Feature | **Component**: HotSpot / GC
>
> **Lead**: Sangheon Kim (Oracle)
>
> **JDK Version**: JDK 14 (2020-03)
>
> **Organization**: [Oracle](../../contributors/orgs/oracle.md)

---

## 概览

JEP 345 为 G1 垃圾收集器添加了 NUMA（Non-Uniform Memory Access）感知的内存分配能力。在多插槽（multi-socket）服务器上，G1 的堆区域（heap region）会优先分配到当前线程所在 NUMA 节点的本地内存中，从而减少跨节点内存访问，显著提升大型服务器上的应用性能。

---

## 背景与动机

### 什么是 NUMA

NUMA（Non-Uniform Memory Access，非一致性内存访问）是多插槽服务器的内存架构。在这种架构中，每个 CPU 插槽都有自己直接连接的本地内存。CPU 访问本地内存的速度远快于访问其他插槽的远程内存：

```
+-------------------+         互联总线         +-------------------+
|   Socket 0        |<========================>|   Socket 1        |
|                   |                           |                   |
|  CPU 核心 0-7     |                           |  CPU 核心 8-15    |
|  本地内存 (快速)  |                           |  本地内存 (快速)  |
+-------------------+                           +-------------------+
     ↑                                               ↑
     | 本地访问: ~80ns                               | 本地访问: ~80ns
     | 远程访问: ~140ns                              | 远程访问: ~140ns
```

在典型的双插槽服务器上，远程内存访问的延迟可能是本地访问的 1.5 到 2 倍。对于内存密集型应用，NUMA 感知的内存分配能带来可观的性能提升。

### G1 之前缺乏 NUMA 支持

在 JDK 14 之前，JVM 的 NUMA 感知内存分配仅在 Parallel GC 中实现（通过 `-XX:+UseNUMA`）。G1 垃圾收集器虽然自 JDK 9 起成为默认 GC，却不支持 NUMA 优化。这意味着在多插槽服务器上，G1 的内存分配对 NUMA 拓扑无感知，可能导致大量跨节点内存访问，影响性能。

随着 G1 成为默认 GC 并在生产环境中广泛使用，为其添加 NUMA 支持变得越来越重要。

## 设计原理

### 核心机制

G1 将堆划分为大小相等的区域（region）。JEP 345 的核心思想是：在分配新区域时，优先选择位于当前线程所在 NUMA 节点本地内存中的区域。

```
NUMA 节点 0                          NUMA 节点 1
+----------------------------------+ +----------------------------------+
| Region 0 | Region 1 | Region 2  | | Region 3 | Region 4 | Region 5  |
| (Eden)   | (Eden)   | (Surv.)   | | (Eden)   | (Eden)   | (Surv.)   |
+----------------------------------+ +----------------------------------+
    ↑                                     ↑
    |                                     |
  线程 A (运行在节点 0)               线程 B (运行在节点 1)
  → 优先分配节点 0 的区域              → 优先分配节点 1 的区域
```

### 年轻代分配策略

NUMA 优化对年轻代（Young Generation）的效果最为显著，因为年轻代的分配频率最高：

1. **Eden 区域分配**：当线程需要新的 TLAB（Thread-Local Allocation Buffer）时，G1 优先从线程所在 NUMA 节点分配 Eden 区域
2. **Survivor 区域分配**：在年轻代回收过程中，存活对象复制到 Survivor 区域时，同样遵循 NUMA 亲和性
3. **区域选择逻辑**：G1 维护每个 NUMA 节点的空闲区域列表，分配时直接从对应节点的列表中获取

### 老年代与巨型对象

老年代的 NUMA 优化效果相对有限——老年代对象存活时间长，可能被不同节点的线程访问。混合回收中的对象晋升不严格遵循 NUMA 亲和性，巨型对象（Humongous Object）的分配也不考虑 NUMA 节点。

## 使用方式

### 启用 NUMA 感知分配

```bash
# 在 G1 GC 中启用 NUMA 支持
java -XX:+UseG1GC -XX:+UseNUMA -Xmx32g MyApplication
```

由于 G1 是 JDK 14+ 的默认 GC，可以简写为：

```bash
java -XX:+UseNUMA -Xmx32g MyApplication
```

### 推荐配置

```bash
# 多插槽服务器上的典型配置（含 NUMA 调试日志）
java -XX:+UseG1GC -XX:+UseNUMA -Xmx64g -Xlog:gc*,gc+heap+numa=debug MyApplication
```

## 实现细节

### 空闲区域管理

G1 的空闲区域列表被按 NUMA 节点分组管理：

| 组件 | NUMA 感知行为 |
|------|--------------|
| **空闲区域列表** | 按 NUMA 节点分组，每个节点维护独立的空闲列表 |
| **Eden 分配** | 从线程所在节点的空闲列表中优先分配 |
| **Survivor 分配** | GC 工作线程根据自身节点分配 Survivor 区域 |
| **老年代晋升** | 尽量遵循 NUMA 亲和性，但不严格保证 |
| **巨型对象** | 不考虑 NUMA 节点（需要连续区域） |
| **TLAB 分配** | 与 Eden 区域的 NUMA 节点一致 |

### 操作系统交互

JVM 通过操作系统的 NUMA API 实现内存亲和性。在 Linux 上，使用 `mbind()` 系统调用将堆页面绑定到指定 NUMA 节点。G1 在启动时根据堆地址范围和 NUMA 拓扑，建立区域到节点的映射关系。

## 性能影响

在典型的多插槽服务器上，启用 NUMA 感知分配可带来显著的性能提升：

| 场景 | 预期改善 |
|------|---------|
| **内存密集型应用** | 吞吐量提升 10%-30% |
| **大堆应用（>32GB）** | 分配延迟降低，GC 暂停时间缩短 |
| **高线程并发应用** | 减少跨节点内存争用 |
| **SPECjbb 等基准测试** | 得分显著提升 |

性能提升幅度取决于 NUMA 节点数量、远程/本地延迟比、应用的内存访问局部性以及堆大小。在单插槽系统上，`-XX:+UseNUMA` 不会产生负面影响——JVM 会检测到只有一个 NUMA 节点并跳过优化逻辑。

---

## 局限性

1. **仅限 Linux**：JDK 14 中的实现主要针对 Linux 平台。其他操作系统的支持有限或不可用
2. **年轻代优化为主**：老年代和巨型对象的 NUMA 优化效果有限
3. **对象迁移**：对象从年轻代晋升到老年代后，可能不再位于最优的 NUMA 节点
4. **线程迁移**：如果操作系统将线程迁移到不同的 NUMA 节点，之前分配的本地内存变为远程内存
5. **内存不均衡**：如果各 NUMA 节点上的线程负载不均，可能导致某些节点的内存压力更大
6. **容器环境**：在容器中运行时，NUMA 拓扑可能受到 cgroup 限制，影响优化效果
