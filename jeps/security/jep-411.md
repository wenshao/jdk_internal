## 1. 概述

JEP 411 在 JDK 17 中将 Java Security Manager 及其关联 API 标记为
`@Deprecated(forRemoval=true)`——正式宣布该机制已过时，将在未来版本中被移除。

JEP 411 并未立即禁用 Security Manager，而是采取渐进策略：保留完整功能的同时发出
运行时废弃警告，给予生态系统迁移时间。最终 JDK 24 的 [JEP 486](jep-486.md) 完成永久禁用。

### 1.1 核心变更

- **标记废弃**：`SecurityManager`、`Policy`、`AccessController` 等核心类标注 `@Deprecated(forRemoval=true)`
- **运行时警告**：在启动时使用 Security Manager 相关选项会输出警告信息
- **功能保留**：Security Manager 仍可被启用和使用，但不建议继续依赖
- **信号明确**：正式向开发者和框架作者传达迁移信号

## 2. 背景：Applet 时代的安全设计

### 2.1 为不可信代码而生（JDK 1.0, 1996）

Security Manager 诞生于 Java Applet 时代。1996 年，Java 的核心愿景之一是在浏览器中
安全地执行从网络下载的不可信代码。Security Manager 提供了沙箱模型——限制 Applet
对文件系统、网络、系统属性等资源的访问，与 `ClassLoader` 配合构成 Java 安全架构的基石。

### 2.2 权限模型的膨胀

随着 Java 平台的发展，Security Manager 的职责不断扩展。JDK 1.1（1997）引入签名代码，
JDK 1.2（1998）引入 `Policy`、`CodeSource`，从二元沙箱演变为基于策略文件的细粒度权限
模型。此后累积了数十种 `Permission` 子类，安全检查遍布 JDK 核心类库。到 JEP 411
提出时，JDK 内部约有 **1,100 处** `SecurityManager` 检查调用，对维护和优化构成沉重负担。

### 2.3 Applet 的消亡

Security Manager 的主要服务对象——Applet——已先行退场。JDK 9（2017）废弃了
Applet API，JDK 11（2018）标记为 `forRemoval`，JDK 17（2021）完成移除。主流浏览器
早在 2015-2017 年间就已停止对 NPAPI 插件的支持。

## 3. JDK 17 中的变更

### 3.1 API 废弃标注

以下核心类和方法被标注 `@Deprecated(since="17", forRemoval=true)`：

```
java.lang.SecurityManager                # 核心类
java.lang.System.getSecurityManager()    # 获取当前 SM
java.lang.System.setSecurityManager()    # 安装 SM
java.security.AccessController           # 权限检查与 doPrivileged
java.security.AccessControlContext       # 权限上下文
java.security.Policy                     # 策略配置
```

### 3.2 运行时警告

JDK 17 在以下场景发出标准错误流警告：

| 场景 | 警告行为 |
|------|---------|
| `-Djava.security.manager` 启动 | 启动时输出警告到 stderr |
| `-Djava.security.manager=<class>` 启动 | 启动时输出警告到 stderr |
| 代码调用 `System.setSecurityManager()` | 首次调用时输出警告到 stderr |
| `-Djava.security.manager=allow` 启动 | **不输出警告**（显式声明允许） |

`-Djava.security.manager=allow` 是 JDK 17 新增的选项值，表示应用程序明确知晓废弃状态
但仍需使用 Security Manager，为库和框架提供过渡机制。

### 3.3 编译时警告

使用被废弃的 Security Manager API 时，`javac` 会产生 `removal` 级别的废弃警告，
提示开发者尽早迁移。

## 4. 为何废弃

### 4.1 维护负担巨大

约 1,100 处 `checkPermission()` 调用遍布文件 I/O、网络、反射、类加载等核心模块，
增加了 API 变更的审查成本，阻碍了性能优化（如虚拟线程需要绕开栈遍历模型），
导致 JDK 内部复杂度居高不下。

### 4.2 极少有应用正确使用

Oracle 调研发现绝大多数 Java 应用从未启用 Security Manager。少数启用的场景中，
大部分配置了 `AllPermission`，等同于未启用。真正利用细粒度权限策略的应用极为罕见。

### 4.3 无法应对现代安全威胁

Security Manager 的 90 年代威胁模型无法覆盖供应链攻击（恶意依赖库共享权限域）、
反序列化漏洞、容器逃逸等现代安全挑战。JVM 内沙箱无法替代操作系统级隔离。

### 4.4 更好的替代方案已成熟

| 需求 | 现代替代方案 |
|------|------------|
| 代码隔离 | 容器（Docker、Kubernetes） |
| 封装控制 | 模块系统（JPMS, JDK 9+） |
| 进程隔离 | OS 级沙箱（seccomp、SELinux、AppArmor） |
| 网络限制 | 防火墙、网络策略 |
| 文件访问控制 | OS 文件权限、容器卷挂载 |

## 5. 通往 JEP 486 的时间线

JEP 411 是 Security Manager 退役的起点。从废弃到永久禁用，Java 平台提供了
**7 个版本、约 3.5 年** 的过渡期：

| 时间 | 版本 | 事件 |
|------|------|------|
| 1996 | JDK 1.0 | Security Manager 引入，用于 Applet 沙箱 |
| 2017 | JDK 9 | JPMS 引入，Applet API 废弃 |
| **2021** | **JDK 17** | **JEP 411：标记为 Deprecated for Removal** |
| 2021-2024 | JDK 18-23 | 过渡期，生态系统逐步迁移 |
| **2025** | **JDK 24** | **[JEP 486](jep-486.md)：永久禁用** |

JDK 17 作为 LTS 版本，确保了大量用户在废弃初期就接收到迁移信号。JDK 21（下一个 LTS）
进一步巩固了过渡，使 JDK 24 的永久禁用水到渠成。

## 6. 迁移建议

### 6.1 常见迁移路径

| 当前用法 | 迁移方案 |
|---------|---------|
| 限制文件访问 | 使用 OS 文件权限、容器卷挂载控制 |
| 限制网络连接 | 使用防火墙规则、网络策略 |
| 隔离不可信代码 | 使用容器或独立进程运行 |
| 拦截 `System.exit()` | JUnit 5 等测试框架的专用机制 |
| `AccessController.doPrivileged()` | 直接调用目标代码，移除权限包装 |
| 自定义 `Permission` 检查 | 迁移至应用层授权框架（如 Spring Security） |

### 6.2 代码清理示例

```java
// 迁移前：条件性安全检查 → 迁移后：直接移除
// sm.checkPermission() 调用可直接删除；如需封装控制，改用 JPMS module-info.java

// 迁移前：doPrivileged 包装 → 迁移后：直接调用
// AccessController.doPrivileged(() -> operation()) → operation()
```

### 6.3 分阶段迁移策略

1. **JDK 17**：使用 `-Djava.security.manager=allow` 消除启动警告，开始规划迁移
2. **JDK 21**：完成主要代码的 Security Manager 依赖清理
3. **JDK 24**：确认应用在无 Security Manager 环境下正常运行（参见 [JEP 486](jep-486.md)）
