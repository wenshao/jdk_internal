## 1. 概述

JEP 497 在 JDK 24 中引入了 ML-DSA（Module-Lattice-Based Digital Signature Algorithm）的实现，
这是一种抗量子计算攻击的数字签名算法，基于 NIST FIPS 204 标准。ML-DSA 为代码签名、证书验证、
文档签名等场景提供了量子安全的数字签名能力。

### 1.1 核心价值

- **抗量子攻击**：基于格（lattice）数学难题，抵御 Shor 算法威胁
- **标准化**：实现 NIST FIPS 204 标准，全球互操作
- **无缝集成**：使用现有 `java.security.Signature` 和 `KeyPairGenerator` API
- **保护完整性**：确保签名数据在量子计算时代不可伪造

---

## 2. 背景：量子计算威胁与数字签名

### 2.1 数字签名面临的量子威胁

数字签名广泛用于软件发布验证、TLS 证书链、文档真实性证明等场景。当前主流签名算法
（RSA、ECDSA、EdDSA）的安全性依赖于大整数分解和离散对数问题，均可被量子计算机上的
Shor 算法在多项式时间内破解。

与密钥交换不同，数字签名的威胁更为紧迫：

- **长期有效的签名**：软件包签名、法律文档签名需要在未来数十年内保持有效
- **证书链信任**：根 CA 证书有效期长达 20-30 年
- **不可否认性**：签名被伪造意味着信任体系崩溃

### 2.2 NIST 后量子签名标准化

- **2022 年**：NIST 选定 CRYSTALS-Dilithium 作为主要数字签名标准候选
- **2024 年 8 月**：正式发布 FIPS 204（ML-DSA），基于 Dilithium 的标准化版本

## 3. ML-DSA 算法

### 3.1 算法原理

ML-DSA 基于 Module Learning With Errors（Module-LWE）和 Module Short Integer Solution
（Module-SIS）问题。其安全性来源于在模格（module lattice）中寻找短向量的计算困难性。

**签名流程**：

```
1. KeyGen()        → 生成公钥 (pk) 和私钥 (sk)
2. Sign(sk, msg)   → 使用私钥对消息生成签名 (σ)
3. Verify(pk,msg,σ) → 使用公钥验证签名的正确性
```

### 3.2 与 CRYSTALS-Dilithium 的关系

ML-DSA 是 CRYSTALS-Dilithium 的 NIST 标准化版本。标准化过程中对 Dilithium 的部分参数编码
和内部处理进行了调整，最终以"ML-DSA"名称发布于 FIPS 204。

### 3.3 安全性保证

- 基于格问题的计算困难性，目前无已知高效经典或量子攻击算法
- EUF-CMA 安全性（选择消息攻击下的存在性不可伪造性）
- 确定性签名生成（使用内部随机化），避免随机数生成器弱点导致私钥泄露

## 4. 参数集

ML-DSA 定义了三个安全级别的参数集：

| 参数集 | NIST 安全级别 | 公钥大小 | 私钥大小 | 签名大小 | 对应经典安全性 |
|--------|-------------|---------|---------|---------|--------------|
| ML-DSA-44 | 2 | 1,312 bytes | 2,560 bytes | 2,420 bytes | ~AES-128 |
| ML-DSA-65 | 3 | 1,952 bytes | 4,032 bytes | 3,309 bytes | ~AES-192 |
| ML-DSA-87 | 5 | 2,592 bytes | 4,896 bytes | 4,627 bytes | ~AES-256 |

### 4.1 参数命名说明

参数名中的数字对（如 44、65、87）表示矩阵维度 (k, l)：
- **ML-DSA-44**：k=4, l=4
- **ML-DSA-65**：k=6, l=5
- **ML-DSA-87**：k=8, l=7

### 4.2 参数选择建议

- **ML-DSA-44**：适用于一般应用场景，提供足够的抗量子安全性
- **ML-DSA-65**：推荐的平衡选择，适用于大多数企业和政府应用
- **ML-DSA-87**：最高安全级别，适用于需要长期安全保证的场景

## 5. Java API

### 5.1 API 集成

JEP 497 使用 Java 现有的标准安全 API，开发者无需学习新接口：

```java
// 密钥对生成
KeyPairGenerator kpg = KeyPairGenerator.getInstance("ML-DSA");
NamedParameterSpec paramSpec = new NamedParameterSpec("ML-DSA-65");
kpg.initialize(paramSpec);
KeyPair kp = kpg.generateKeyPair();

// 签名
Signature signer = Signature.getInstance("ML-DSA");
signer.initSign(kp.getPrivate());
signer.update(message);
byte[] signature = signer.sign();

// 验签
Signature verifier = Signature.getInstance("ML-DSA");
verifier.initVerify(kp.getPublic());
verifier.update(message);
boolean valid = verifier.verify(signature);
```

### 5.2 核心 API 组件

```
java.security.KeyPairGenerator       # 密钥对生成
├── getInstance("ML-DSA")            # 获取 ML-DSA 生成器
├── initialize(NamedParameterSpec)   # 设置参数集
└── generateKeyPair()                # 生成密钥对

java.security.Signature              # 签名与验签
├── getInstance("ML-DSA")            # 获取 ML-DSA 签名实例
├── initSign(PrivateKey)             # 初始化签名
├── initVerify(PublicKey)            # 初始化验签
├── update(byte[])                   # 提供待签名/验签数据
├── sign()                           # 生成签名
└── verify(byte[])                   # 验证签名

java.security.spec.NamedParameterSpec # 参数规范
└── new NamedParameterSpec("ML-DSA-65")
```

### 5.3 Provider 支持

ML-DSA 由 SUN Provider 实现，无需引入第三方库。

## 6. 签名尺寸对比

ML-DSA 的签名和公钥尺寸显著大于传统签名算法，这是后量子算法的固有特性：

| 算法 | 安全级别 | 公钥大小 | 签名大小 | 总传输开销 |
|------|---------|---------|---------|-----------|
| ECDSA P-256 | ~128-bit | 64 bytes | 64 bytes | 128 bytes |
| RSA-2048 | ~112-bit | 256 bytes | 256 bytes | 512 bytes |
| RSA-3072 | ~128-bit | 384 bytes | 384 bytes | 768 bytes |
| EdDSA (Ed25519) | ~128-bit | 32 bytes | 64 bytes | 96 bytes |
| **ML-DSA-44** | ~128-bit | 1,312 bytes | 2,420 bytes | **3,732 bytes** |
| **ML-DSA-65** | ~192-bit | 1,952 bytes | 3,309 bytes | **5,261 bytes** |
| **ML-DSA-87** | ~256-bit | 2,592 bytes | 4,627 bytes | **7,219 bytes** |

### 6.1 尺寸影响分析

- **TLS 握手**：证书链中包含多个签名，ML-DSA 会显著增加握手数据量
- **嵌入式/IoT**：存储和带宽受限的设备需要评估可行性
- **区块链/分布式账本**：每笔交易的签名增大会影响吞吐量
- **软件签名**：一次性验证场景，尺寸增大影响较小

## 7. 性能特性

### 7.1 操作速度对比

| 操作 | ECDSA P-256 | RSA-2048 | ML-DSA-65 |
|------|------------|----------|-----------|
| 密钥生成 | 快 | 慢 | 快 |
| 签名 | 快 | 中等 | 快 |
| 验签 | 中等 | 快 | 快 |

### 7.2 性能特点

- **密钥生成**：ML-DSA 的密钥生成速度通常快于 RSA
- **签名操作**：ML-DSA 的签名速度在特定工作负载下与 ECDSA 相当
- **验签操作**：ML-DSA 的验签速度通常较快
- **注意**：具体性能数据取决于硬件平台、JVM 版本和工作负载特征，建议在目标环境中进行基准测试

### 7.3 内存开销

由于密钥和签名尺寸增大，ML-DSA 在处理大量密钥或签名时会消耗更多内存。
在高并发签名验证场景中需要评估内存影响。

## 8. 迁移路径

### 8.1 迁移策略

**短期**：在新系统中开始使用 ML-DSA，同时保持对传统算法的支持。对于需要长期有效的签名
（如代码签名、CA 证书），优先考虑迁移。

**中长期**：随着行业标准和协议（TLS 1.3、X.509 v3 扩展）的更新，逐步淘汰传统签名算法。

### 8.2 迁移步骤

1. **盘点**：识别系统中所有使用数字签名的位置（TLS 证书、代码签名、API 认证等）
2. **测试**：在测试环境中使用 ML-DSA，评估签名尺寸和性能影响
3. **双签名**：在过渡期对关键内容同时使用传统算法和 ML-DSA 签名
4. **协议更新**：关注 TLS、X.509、CMS 等协议对后量子签名的支持进展
5. **全面迁移**：在条件成熟后切换到纯 ML-DSA

### 8.3 代码迁移示例

```java
// 迁移前：ECDSA 签名
KeyPairGenerator kpg = KeyPairGenerator.getInstance("EC");
kpg.initialize(new ECGenParameterSpec("secp256r1"));
KeyPair kp = kpg.generateKeyPair();

Signature sig = Signature.getInstance("SHA256withECDSA");
sig.initSign(kp.getPrivate());
sig.update(data);
byte[] signature = sig.sign();

// 迁移后：ML-DSA 签名
KeyPairGenerator kpg = KeyPairGenerator.getInstance("ML-DSA");
kpg.initialize(new NamedParameterSpec("ML-DSA-65"));
KeyPair kp = kpg.generateKeyPair();

Signature sig = Signature.getInstance("ML-DSA");
sig.initSign(kp.getPrivate());
sig.update(data);
byte[] signature = sig.sign();
```

### 8.4 兼容性注意事项

- ML-DSA 签名无法被不支持该算法的旧系统验证
- 证书链中混合使用传统和后量子算法时需要确保所有验证方支持
- JDK 24 之前的 Java 版本不支持 ML-DSA
