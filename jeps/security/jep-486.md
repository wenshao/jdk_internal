## 1. 概述

JEP 486 在 JDK 24 中永久禁用了 Java Security Manager。这标志着这一有近 30 年历史的安全机制
正式退出 Java 平台。Security Manager 无法再被启用——无论通过编程方式还是命令行参数，
任何试图激活它的操作都将被拒绝或忽略。

### 1.1 核心变更

- **永久禁用**：Security Manager 不可被启用，相关 API 调用抛出异常
- **类保留**：`SecurityManager`、`Policy`、`Permission` 等类仍然存在，以确保编译兼容性
- **功能移除**：所有安全检查均为空操作（no-op）或抛出异常
- **警告提示**：使用已废弃的命令行选项时会发出警告信息

## 2. 背景：Security Manager 的完整生命周期

### 2.1 起源：Applet 沙箱时代（JDK 1.0, 1996）

Security Manager 于 1996 年随 JDK 1.0 一同引入，最初的设计目标是为 Java Applet 提供沙箱
安全模型。在那个时代，Applet 从网络下载并在浏览器中执行不可信代码，Security Manager 负责
限制这些代码的文件系统访问、网络连接和系统资源操作。

### 2.2 演进与复杂化

随着 Java 平台的发展，Security Manager 的权限模型不断扩展。JDK 1.1 引入签名代码，
JDK 1.2 引入 `Policy` 和基于 `CodeSource` 的权限系统，使模型从简单的二元沙箱演变为
复杂的策略文件配置。此后数十种 `Permission` 子类被加入，权限检查遍布核心类库。
JDK 9 引入模块系统（JPMS）后，Security Manager 的角色被进一步边缘化。

### 2.3 废弃与移除

2021 年，[JEP 411](https://openjdk.org/jeps/411) 在 JDK 17 中正式将 Security Manager 标记为
"Deprecated for Removal"。原因包括：维护成本高昂、极少有应用正确使用、无法应对现代安全
威胁、更好的替代方案已成熟。从 JDK 24 到 JDK 24，经过 7 个版本的过渡期后，JEP 486
完成了 Security Manager 退役的最终步骤。

## 3. JDK 24 中的变更

### 3.1 API 行为变更

```java
// System.setSecurityManager() — JDK 24 中抛出 UnsupportedOperationException
System.setSecurityManager(new SecurityManager());
// → throws UnsupportedOperationException

// System.getSecurityManager() — JDK 24 中始终返回 null
SecurityManager sm = System.getSecurityManager(); // 永远为 null
```

### 3.2 命令行选项

| 选项 | JDK 24 行为 |
|------|------------|
| `-Djava.security.manager` | 被忽略，启动时发出警告 |
| `-Djava.security.manager=allow` | 被忽略，启动时发出警告 |
| `-Djava.security.manager=<class>` | 被忽略，启动时发出警告 |
| `-Djava.security.policy=<file>` | 被忽略 |

### 3.3 保留的类与接口

以下类和接口在 JDK 24 中**仍然存在**，以确保依赖这些类型的代码仍能编译通过，
但它们已不具备实际功能：

```
java.lang.SecurityManager          # 保留，但无法被安装
java.security.Policy               # 保留，空实现
java.security.Permission           # 保留，用于类型兼容
java.security.AccessController     # 保留，doPrivileged() 直接执行 action
java.security.AccessControlContext # 保留，非功能性
```

`AccessController.doPrivileged()` 在 JDK 24 中直接执行传入的 action，不做任何权限检查，
等同于直接调用内部逻辑。

## 4. Security Manager 为何失败

### 4.1 为 Applet 而生，随 Applet 而亡

Security Manager 的核心设计场景——在浏览器中运行不可信 Applet 代码——早已不复存在。
Applet 在 JDK 9 中废弃（JEP 289）、JDK 17 中被标记为 Deprecated for Removal（JEP 398），Security Manager 失去了主要使用场景。

### 4.2 过于复杂的权限模型

Security Manager 的权限模型要求开发者理解并正确配置数十种 `Permission` 类型
（`FilePermission`、`SocketPermission`、`RuntimePermission` 等）。策略文件的配置复杂且
容易出错，即便是经验丰富的开发者也难以编写出既安全又不过度限制的策略。

### 4.3 AllPermission：名存实亡的安全

在实际生产环境中，绝大多数应用程序直接配置 `grant { permission java.security.AllPermission; }`，
等同于完全关闭安全检查。应用服务器和框架通常默认授予所有权限，因为正确配置细粒度权限的
成本远超收益。

### 4.4 性能开销

每次安全检查都需要遍历调用栈（stack walking）来确定代码来源和权限域，在高频调用路径上
（如文件 I/O、网络操作）引入不可忽视的性能开销，影响了 JDK 内部的优化空间。

### 4.5 现代替代方案

| 需求 | Security Manager 方案 | 现代替代方案 |
|------|---------------------|------------|
| 代码隔离 | Applet 沙箱 | 容器（Docker, Podman） |
| 封装控制 | 权限检查 | 模块系统（JPMS） |
| 进程隔离 | SecurityManager 策略 | 操作系统级进程隔离 |
| 网络限制 | SocketPermission | 防火墙、网络策略 |
| 文件访问控制 | FilePermission | OS 文件权限、SELinux |
| 最小权限 | Policy 文件 | 容器安全上下文、seccomp |

## 5. 迁移指南

### 5.1 使用操作系统级沙箱

对于需要进程隔离的应用，应采用操作系统提供的沙箱机制：

- **Linux**：seccomp、AppArmor、SELinux
- **容器**：Docker、Podman，使用受限的安全配置文件（`no-new-privileges`、`cap_drop: ALL`）
- **虚拟化**：轻量级虚拟机（如 Firecracker）

### 5.2 使用 JPMS 实现封装

Java 模块系统（JPMS, [JEP 261](https://openjdk.org/jeps/261)）提供编译时和运行时的
强封装机制，通过 `module-info.java` 控制包的导出和访问，替代 Security Manager 的
访问控制功能。

### 5.3 代码迁移要点

1. **移除 `System.setSecurityManager()` 调用**：直接删除，JDK 24 中会抛出异常
2. **移除策略文件配置**：`-Djava.security.policy` 不再生效
3. **替换 `AccessController.doPrivileged()`**：直接调用内部逻辑，无需包装
4. **更新自定义 Permission 类**：如果仅用于 Security Manager 检查，可以移除

```java
// 迁移前
AccessController.doPrivileged((PrivilegedAction<Void>) () -> {
    sensitiveOperation();
    return null;
});

// 迁移后
sensitiveOperation();
```

## 6. 对生态系统的影响

### 6.1 受影响的框架和库

- **Apache Tomcat**：曾使用 Security Manager 隔离 Web 应用，自 10.1.x 起已移除依赖
- **Elasticsearch**：曾使用 Security Manager 限制插件行为，已迁移至 OS 级安全模型
- **Gradle / Maven**：构建工具已在新版本中清理 Security Manager 相关配置
- **JUnit**：`System.exit()` 拦截等测试工具需替换为其他机制

### 6.2 应用服务器

传统 Java EE 应用服务器（WildFly、GlassFish）曾在规范中要求支持 Security Manager。
Jakarta EE 10+ 已移除此要求，与 JDK 24 的方向一致。

### 6.3 对库作者的建议

- 移除 `System.getSecurityManager()` 的空值检查和 `AccessController.doPrivileged()` 包装
- 更新文档，移除 Security Manager 相关配置说明
- 使用多版本 JAR（Multi-Release JAR）在不同 JDK 版本间保持兼容
