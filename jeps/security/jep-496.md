## 1. 概述

JEP 496 在 JDK 24 中引入了 ML-KEM（Module-Lattice-Based Key Encapsulation Mechanism）的实现，
这是一种抗量子计算攻击的密钥封装机制，基于 NIST FIPS 203 标准。ML-KEM 提供了在量子计算时代
保护密钥交换安全性的能力，是 Java 平台向后量子密码学（Post-Quantum Cryptography, PQC）迈进
的重要一步。

### 1.1 核心价值

- **抗量子攻击**：基于格（lattice）数学难题，抵御 Shor 算法对传统公钥密码的威胁
- **标准化**：实现 NIST FIPS 203 标准，确保互操作性
- **无缝集成**：复用 JDK 21 引入的 `javax.crypto.KEM` API（[JEP 452](jep-452.md)）
- **向前兼容**：为"先收集、后解密"（harvest now, decrypt later）攻击提供防护

---

## 2. 背景：量子计算威胁

### 2.1 传统密码学面临的挑战

当前广泛使用的公钥密码算法（RSA、ECDH、ECDSA）的安全性依赖于大整数分解和离散对数等数学难题。
量子计算机上的 Shor 算法能够在多项式时间内解决这些问题，从根本上威胁这些算法的安全性。

| 算法 | 安全基础 | 量子威胁 |
|------|---------|---------|
| RSA | 大整数分解 | Shor 算法可破解 |
| ECDH/ECDSA | 椭圆曲线离散对数 | Shor 算法可破解 |
| AES | 对称加密 | Grover 算法削弱一半（仍可用） |
| ML-KEM | 模格问题（Module-LWE） | 目前无已知高效量子攻击 |

### 2.2 NIST 后量子密码标准化

NIST 于 2016 年启动后量子密码标准化项目，经过多轮评选：

- **2022 年**：选定 CRYSTALS-Kyber 作为 KEM 标准候选
- **2024 年 8 月**：正式发布 FIPS 203（ML-KEM），基于 CRYSTALS-Kyber 的最终标准化版本

### 2.3 "先收集、后解密"威胁

攻击者可以现在收集加密通信数据，等待量子计算机成熟后再进行解密。这意味着即使大规模量子计算机
尚未出现，今天传输的敏感数据仍面临未来的解密风险。因此，尽早迁移到抗量子算法至关重要。

## 3. ML-KEM 算法

### 3.1 算法原理

ML-KEM 基于 Module Learning With Errors（Module-LWE）问题，这是格密码学中的核心困难问题。
其安全性源于在带有噪声的线性方程组中求解未知向量的计算困难性。

**密钥封装流程**：

```
1. KeyGen()     → 生成公钥 (ek) 和私钥 (dk)
2. Encaps(ek)   → 生成共享密钥 (K) 和密文 (c)
3. Decaps(dk,c) → 从密文恢复共享密钥 (K)
```

### 3.2 与 CRYSTALS-Kyber 的关系

ML-KEM 是 CRYSTALS-Kyber 的标准化版本。NIST 在标准化过程中对 Kyber 进行了微调，
最终以"ML-KEM"名称发布于 FIPS 203。两者在核心算法上一致，但在参数编码和某些细节上存在差异。

### 3.3 安全性保证

- 基于格问题的计算困难性，目前无已知高效经典或量子算法可破解
- IND-CCA2 安全性（选择密文攻击下的不可区分性）
- 经过 NIST 多轮公开评审和密码学社区广泛分析

## 4. 参数集

ML-KEM 定义了三个安全级别的参数集：

| 参数集 | NIST 安全级别 | 公钥大小 | 密文大小 | 共享密钥 | 对应经典安全性 |
|--------|-------------|---------|---------|---------|--------------|
| ML-KEM-512 | 1 | 800 bytes | 768 bytes | 32 bytes | ~AES-128 |
| ML-KEM-768 | 3 | 1,184 bytes | 1,088 bytes | 32 bytes | ~AES-192 |
| ML-KEM-1024 | 5 | 1,568 bytes | 1,568 bytes | 32 bytes | ~AES-256 |

### 4.1 参数选择建议

- **ML-KEM-768**：推荐的默认选择，在安全性和性能之间取得良好平衡
- **ML-KEM-512**：适用于资源受限场景，提供基本的抗量子安全性
- **ML-KEM-1024**：适用于需要最高安全级别的场景（政府、军事等）

## 5. Java API

### 5.1 API 集成

JEP 496 复用了 JDK 24 中 [JEP 452](jep-452.md) 引入的 `javax.crypto.KEM` API。
开发者使用相同的接口，只需更换算法名称即可从传统 KEM 切换到后量子 KEM。

```java
// 密钥对生成
KeyPairGenerator kpg = KeyPairGenerator.getInstance("ML-KEM");
NamedParameterSpec paramSpec = new NamedParameterSpec("ML-KEM-768");
kpg.initialize(paramSpec);
KeyPair kp = kpg.generateKeyPair();

// KEM 操作
KEM kem = KEM.getInstance("ML-KEM");
```

### 5.2 核心 API 组件

```
java.security.KeyPairGenerator     # 密钥对生成
├── getInstance("ML-KEM")          # 获取 ML-KEM 生成器
├── initialize(NamedParameterSpec) # 设置参数集
└── generateKeyPair()              # 生成密钥对

javax.crypto.KEM                   # 密钥封装机制
├── getInstance("ML-KEM")          # 获取 ML-KEM 实例
├── newEncapsulator(PublicKey)      # 创建封装器
└── newDecapsulator(PrivateKey)     # 创建解封装器

javax.crypto.KEM.Encapsulator      # 封装操作
└── encapsulate()                  # 返回 KEM.Encapsulated

javax.crypto.KEM.Decapsulator      # 解封装操作
└── decapsulate(byte[])            # 恢复共享密钥
```

### 5.3 Provider 支持

ML-KEM 由 SunJCE Provider 实现，无需引入第三方库。

## 6. 使用示例

### 6.1 基本密钥封装

```java
import javax.crypto.KEM;
import java.security.KeyPair;
import java.security.KeyPairGenerator;
import java.security.spec.NamedParameterSpec;
import javax.crypto.SecretKey;

// 1. 接收方生成密钥对
KeyPairGenerator kpg = KeyPairGenerator.getInstance("ML-KEM");
kpg.initialize(new NamedParameterSpec("ML-KEM-768"));
KeyPair receiverKeyPair = kpg.generateKeyPair();

// 2. 发送方使用接收方公钥封装
KEM kemSender = KEM.getInstance("ML-KEM");
KEM.Encapsulator enc = kemSender.newEncapsulator(receiverKeyPair.getPublic());
KEM.Encapsulated encapsulated = enc.encapsulate();
SecretKey senderSharedKey = encapsulated.key();          // 发送方的共享密钥
byte[] ciphertext = encapsulated.encapsulation();        // 发送给接收方的密文

// 3. 接收方解封装获取共享密钥
KEM kemReceiver = KEM.getInstance("ML-KEM");
KEM.Decapsulator dec = kemReceiver.newDecapsulator(receiverKeyPair.getPrivate());
SecretKey receiverSharedKey = dec.decapsulate(ciphertext); // 与发送方相同的密钥
```

### 6.2 指定派生密钥算法

```java
// 封装时指定生成 AES 密钥
KEM.Encapsulator enc = kemSender.newEncapsulator(publicKey);
KEM.Encapsulated encapsulated = enc.encapsulate(0, 16, "AES");
// 生成 128-bit AES 密钥
```

## 7. 性能特性

### 7.1 与 RSA/ECDH 的对比

ML-KEM 在密钥生成和封装/解封装操作上通常比 RSA 更快，但密钥和密文尺寸更大：

| 指标 | RSA-2048 | ECDH P-256 | ML-KEM-768 |
|------|----------|------------|------------|
| 公钥大小 | 256 bytes | 65 bytes | 1,184 bytes |
| 密文/交换数据 | 256 bytes | 65 bytes | 1,088 bytes |
| 密钥生成速度 | 较慢 | 快 | 快 |
| 封装/协商速度 | 中等 | 快 | 快 |

### 7.2 注意事项

- 密钥和密文尺寸显著大于传统算法，在网络带宽受限场景需要评估影响
- ML-KEM 的计算性能在特定工作负载下通常优于 RSA，但具体表现取决于硬件平台和 JVM 实现
- 建议在目标环境中进行实际基准测试

## 8. 迁移路径

### 8.1 迁移策略

**短期（推荐）**：采用混合模式（Hybrid KEM），同时使用传统 ECDH 和 ML-KEM，
确保即使其中一个算法被攻破，通信仍然安全。

**中长期**：在 NIST 标准成熟和行业实践充分验证后，逐步过渡到纯后量子算法。

### 8.2 迁移步骤

1. **评估**：识别系统中使用 RSA/ECDH 进行密钥交换的位置
2. **测试**：在测试环境中使用 ML-KEM，评估密钥尺寸和性能影响
3. **混合部署**：在 TLS 等协议中启用混合密钥交换
4. **监控**：关注 NIST 标准更新和密码学社区的安全评估
5. **全面迁移**：在条件成熟后切换到纯 ML-KEM

### 8.3 代码迁移示例

```java
// 迁移前：ECDH 密钥协商
KeyPairGenerator kpg = KeyPairGenerator.getInstance("EC");
kpg.initialize(new ECGenParameterSpec("secp256r1"));
KeyPair kp = kpg.generateKeyPair();

// 迁移后：ML-KEM 密钥封装
KeyPairGenerator kpg = KeyPairGenerator.getInstance("ML-KEM");
kpg.initialize(new NamedParameterSpec("ML-KEM-768"));
KeyPair kp = kpg.generateKeyPair();
```
