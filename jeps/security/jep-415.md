## 1. 概述

JEP 415 在 JDK 17 中引入了可配置的、上下文相关的反序列化过滤器。该特性扩展了
JEP 290（JDK 9）的序列化过滤框架，允许应用程序通过 **Filter Factory** 为每个
`ObjectInputStream` 实例动态选择反序列化过滤器，而非依赖单一的全局过滤器。

- **Filter Factory**：通过工厂函数为每个反序列化流配置上下文相关的过滤器
- **全局与流级别过滤器组合**：支持将应用级默认过滤器与单个流的过滤器合并
- **基于模式的过滤器**：支持类名匹配、数组大小限制、对象图深度限制等
- **向后兼容**：完全兼容 JEP 290 引入的现有过滤器 API

JEP 290 要求每个 `ObjectInputStream` 单独设置过滤器，第三方库创建的反序列化流
无法被应用程序控制。JEP 415 通过全局 Filter Factory 使应用程序能够为所有反序列化
操作（包括第三方库内部创建的流）设置统一的安全策略。

---

## 2. 背景：反序列化漏洞与 JEP 290 的局限性

Java 反序列化攻击利用 **gadget chain** 实现远程代码执行（RCE）。攻击者构造
序列化数据，利用 classpath 中的危险类组合（如 `InvocationHandler` ->
`ChainedTransformer` -> `Runtime.exec()`）触发任意代码执行。

### 2.1 JEP 290 的局限

JEP 290（JDK 9）引入了 `ObjectInputFilter` 和 `jdk.serialFilter` 系统属性，但存在
关键局限：无法设置应用级默认过滤器工厂；第三方库内部创建的 `ObjectInputStream` 无法
拦截；`jdk.serialFilter` 只能设置一个静态全局过滤器，无法按上下文区分。

## 3. 核心机制：Filter Factory

### 3.1 setSerialFilterFactory API

```java
ObjectInputFilter.Config.setSerialFilterFactory(
    (currentFilter, nextFilter) -> {
        // currentFilter: 流当前已设置的过滤器（可能为 null）
        // nextFilter: 通过 setObjectInputFilter() 设置的过滤器
        return ObjectInputFilter.merge(nextFilter, currentFilter);
    }
);
```

Filter Factory 是一个 `BinaryOperator<ObjectInputFilter>`，在两个时机被调用：

1. **`ObjectInputStream` 构造时**：传入 `(全局过滤器, null)`，返回流的初始过滤器
2. **`setObjectInputFilter()` 调用时**：传入 `(当前过滤器, 新过滤器)`，返回合并后的过滤器

`ObjectInputFilter.merge()` 将两个过滤器组合：先检查第一个过滤器，若返回
`UNDECIDED` 则检查第二个过滤器。

### 3.2 基于模式的过滤器

```java
// 类名匹配 + 默认拒绝
ObjectInputFilter patternFilter = ObjectInputFilter.Config.createFilter(
    "com.myapp.model.*;java.util.ArrayList;!*"
);

// 资源限制过滤器
ObjectInputFilter limitFilter = ObjectInputFilter.Config.createFilter(
    "maxdepth=5;maxrefs=100;maxbytes=4096;maxarray=256"
);
```

## 4. 过滤策略：拒绝列表与允许列表

**拒绝列表（Reject-list）**：阻止已知危险类，允许其他所有类通过。易于实施，
但无法防御未知 gadget chain。

```java
ObjectInputFilter rejectList = ObjectInputFilter.Config.createFilter(
    "!org.apache.commons.collections.functors.**;" +
    "!org.apache.commons.collections4.functors.**;" +
    "!javassist.**;!com.mchange.v2.c3p0.**"
);
```

**允许列表（Allow-list）**：仅放行已知安全的类，拒绝所有其他类。更安全，
但需精确了解反序列化需求。

```java
ObjectInputFilter allowList = ObjectInputFilter.Config.createFilter(
    "java.lang.String;java.lang.Number;java.lang.Integer;" +
    "java.util.ArrayList;java.util.HashMap;com.myapp.dto.**;!*"
);
```

| 特性 | 拒绝列表 | 允许列表 |
|------|----------|----------|
| 默认行为 | 允许所有，阻止已知危险 | 拒绝所有，放行已知安全 |
| 防御未知攻击 | 无法防御 | 有效防御 |
| 适用场景 | 快速部署、遗留系统 | 新系统、高安全要求 |

## 5. 配置方式

### 5.1 系统属性

```bash
# 全局序列化过滤器（JEP 290 引入，JEP 415 继续支持）
java -Djdk.serialFilter="com.myapp.**;java.util.*;!*" MyApp

# Filter Factory 类名（JEP 415 新增）
java -Djdk.serialFilterFactory=com.myapp.security.MyFilterFactory MyApp
```

### 5.2 安全属性文件

在 `$JAVA_HOME/conf/security/java.security` 中配置 `jdk.serialFilter` 和
`jdk.serialFilterFactory` 属性。

### 5.3 编程式 API

```java
// 设置 Filter Factory（推荐）
ObjectInputFilter.Config.setSerialFilterFactory((current, next) -> {
    ObjectInputFilter baseFilter = ObjectInputFilter.Config.createFilter(
        "com.myapp.dto.**;java.lang.*;java.util.*;!*"
    );
    if (next == null) {
        return ObjectInputFilter.merge(baseFilter, current);
    }
    return ObjectInputFilter.merge(next,
               ObjectInputFilter.merge(baseFilter, current));
});
```

**注意事项**：`setSerialFilterFactory()` 只能调用一次，重复调用抛出
`IllegalStateException`。必须在任何 `ObjectInputStream` 创建之前设置。

## 6. 安全背景：真实攻击场景

Apache Commons Collections 3.x/4.x 中的 `InvokerTransformer`、`ChainedTransformer`
等类是最著名的反序列化 gadget chain 来源。2015 年 FoxGlove Security 披露此漏洞后，
Java 反序列化安全成为行业关注焦点，直接推动了 JEP 290 和 JEP 415 的开发。

JEP 415 随 JDK 17 于 2021 年 9 月发布，恰在 Log4Shell（CVE-2021-44228）曝光前夕。
虽然 Log4Shell 是 JNDI 注入而非反序列化漏洞，但它极大提升了 Java 生态系统对
序列化/反序列化安全的重视。

### 常见 Gadget Chain 来源

| 库 | 危险类/包 | 攻击向量 |
|----|----------|----------|
| Commons Collections 3.x/4.x | `o.a.c.collections[4].functors` | 任意方法调用 |
| Commons BeanUtils | `o.a.c.beanutils` | 属性注入 |
| Spring Framework | `o.s.beans.factory` | Bean 实例化 |
| C3P0 | `com.mchange.v2.c3p0` | JNDI 查找 |

## 7. 最佳实践

**优先使用允许列表。** 拒绝列表只能阻止已知危险类，新的 gadget chain 可能随时被
发现。允许列表从根本上限制了反序列化的攻击面。

- **尽早配置**：在 `main()` 方法开头设置 Filter Factory
- **减少序列化使用**：优先使用 JSON、Protocol Buffers 等替代方案
- **监控与审计**：利用 Filter Factory 记录反序列化活动

```java
// 审计示例：利用 Filter Factory 记录反序列化活动
ObjectInputFilter.Config.setSerialFilterFactory((current, next) -> {
    ObjectInputFilter auditFilter = info -> {
        if (info.serialClass() != null)
            logger.debug("反序列化类: {}", info.serialClass().getName());
        return ObjectInputFilter.Status.UNDECIDED;
    };
    ObjectInputFilter base = (next == null) ? current : ObjectInputFilter.merge(next, current);
    return ObjectInputFilter.merge(auditFilter, base);
});
```
