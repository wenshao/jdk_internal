## 1. 概述

JEP 498 在 JDK 24 中对 `sun.misc.Unsafe` 的内存访问方法发出运行时警告。当应用程序
或库调用这些方法时，JVM 向 stderr 输出警告，提醒开发者迁移至标准 API。

这是 Unsafe 退役路径的第二步：JEP 471（JDK 24）完成编译期废弃标注，JEP 498（JDK 24）
增加运行时警告，后续版本将逐步抛出异常直至彻底移除。

### 1.1 核心变更

- **运行时警告**：首次调用 Unsafe 内存访问方法时，JVM 输出一次性警告到 stderr
- **覆盖范围**：堆上内存访问（get/put 系列）与堆外内存操作（allocateMemory、freeMemory 等）
- **无功能变更**：方法仍正常执行，不影响现有行为，仅增加警告输出
- **可配置**：通过 `--sun-misc-unsafe-memory-access` 命令行选项控制警告行为

## 2. 背景：sun.misc.Unsafe 的历史角色

`sun.misc.Unsafe` 自 JDK 1.4 起作为 JDK 内部类存在，从未被纳入 Java 标准 API。
它提供了直接内存访问、CAS 操作、对象字段偏移量计算等底层能力，被高性能库广泛依赖：
序列化框架用它绕过构造器，并发库用它执行原子操作，网络框架用它操作堆外内存。

Java 平台经过多年努力，已为 Unsafe 的各项能力提供了标准替代：

| Unsafe 能力 | 标准替代 API | 引入版本 |
|-------------|-------------|---------|
| 堆上字段的 get/put | `java.lang.invoke.VarHandle` | JDK 9（JEP 193） |
| 堆外内存分配与访问 | Foreign Memory API（`java.lang.foreign`） | JDK 22（JEP 454） |
| CAS 操作 | `VarHandle.compareAndSet()` | JDK 9 |
| 内存屏障 | `VarHandle` 访问模式 | JDK 9 |

JEP 471（JDK 23，2024-09）将这些内存访问方法标注为 `@Deprecated(forRemoval=true)`，
明确传达移除信号。JEP 498 是其自然延续，从编译期警告升级到运行时警告。

---

## 3. 触发警告的方法

以下 `sun.misc.Unsafe` 方法在 JDK 24 中首次被调用时将触发运行时警告：

| 分类 | 代表方法 |
|------|---------|
| 堆上 get/put | `getInt(Object,long)`、`putLong(Object,long,long)`、`getObject` 等全部 9 种类型的 get/put |
| volatile 语义 | `getIntVolatile`、`putLongVolatile`、`putOrderedObject` 等 |
| CAS 操作 | `compareAndSwapInt`、`compareAndSwapLong`、`compareAndSwapObject` |
| 堆外内存 | `allocateMemory`、`reallocateMemory`、`freeMemory`、`setMemory`、`copyMemory` 及单地址 get/put |
| 字段偏移量 | `objectFieldOffset`、`staticFieldOffset`、`arrayBaseOffset`、`arrayIndexScale` |

---

## 4. 警告机制与行为

JDK 24 在首次调用上述方法时向 stderr 输出警告，每个调用者类只触发一次，避免日志洪泛：

```
WARNING: A terminally deprecated method in sun.misc.Unsafe has been called
WARNING: sun.misc.Unsafe::allocateMemory has been called by com.example.MyClass
WARNING: sun.misc.Unsafe::allocateMemory will be removed in a future release
```

通过 `--sun-misc-unsafe-memory-access` 选项控制行为：

| 选项值 | 行为 |
|--------|------|
| `allow` | 允许调用，不输出警告 |
| `warn`（**JDK 24 默认**） | 允许调用，首次调用输出警告 |
| `debug` | 允许调用，每次调用输出警告并附带堆栈跟踪 |
| `deny` | 抛出 `UnsupportedOperationException` |

`deny` 模式允许开发者提前验证应用是否已完成迁移，为未来版本的默认行为做准备。

## 5. 迁移路径

### 5.1 堆上访问：迁移至 VarHandle

```java
// 迁移前                                      // 迁移后
long off = unsafe.objectFieldOffset(           VarHandle vh = MethodHandles.lookup()
    MyClass.class.getDeclaredField("v"));          .findVarHandle(MyClass.class, "v", int.class);
int v = unsafe.getInt(obj, off);               int v = (int) vh.get(obj);
unsafe.compareAndSwapInt(obj, off, v, 99);     vh.compareAndSet(obj, v, 99);
```

VarHandle 提供完整的内存访问模式（plain、opaque、acquire/release、volatile），
且受模块系统访问控制保护。

### 5.2 堆外访问：迁移至 Foreign Memory API

```java
// 迁移前                                      // 迁移后（JEP 454）
long addr = unsafe.allocateMemory(1024);       try (Arena arena = Arena.ofConfined()) {
unsafe.putInt(addr, 42);                           MemorySegment seg = arena.allocate(1024);
unsafe.freeMemory(addr);                           seg.set(ValueLayout.JAVA_INT, 0, 42);
                                               }  // 自动释放
```

Foreign Memory API 提供确定性释放、边界检查和线程安全保障。

## 6. 从废弃到移除的时间线

| 时间 | 版本 | 事件 |
|------|------|------|
| 2004 | JDK 5 | `sun.misc.Unsafe` 被广泛使用（非标准、无承诺） |
| 2017 | JDK 9 | VarHandle 引入（JEP 193），堆上访问有了标准替代 |
| 2024-03 | JDK 22 | Foreign Memory API 正式发布（JEP 454），堆外访问有了标准替代 |
| **2024-09** | **JDK 23** | **JEP 471：内存访问方法标记 `@Deprecated(forRemoval=true)`** |
| **2025-03** | **JDK 24** | **JEP 498：运行时警告（默认 `warn` 模式）** |
| 未来 | JDK 25+ | 预计默认模式升级为 `deny`（抛出异常） |
| 未来 | 后续版本 | 最终移除方法实现 |

这种渐进策略——废弃 → 警告 → 异常 → 移除——与 Security Manager 的退役路径
（JEP 411 → JEP 486）一脉相承，给予生态系统充分的迁移时间。

## 7. 对主流库的影响

| 库/框架 | Unsafe 用途 | 迁移状态与建议 |
|---------|------------|---------------|
| **Netty** | 堆外内存分配、零拷贝缓冲区（`PlatformDependent`） | 4.1.x 会触发警告；4.2+/5.x 迁移至 Foreign Memory API |
| **Guava** | `UnsignedBytes` 高性能数组比较 | 较新版本已有不依赖 Unsafe 的回退路径 |
| **Kryo** | 字段直接读写（`getObject`/`putObject`） | 迁移至 VarHandle（`allocateInstance` 不受影响） |
| Apache Lucene | 堆外索引数据访问 | 迁移至 Foreign Memory API |
| LMAX Disruptor | 序列号原子操作 | 迁移至 VarHandle |
| Jackson | 高性能字段访问 | 迁移至 VarHandle |
| Hadoop/Spark | 堆外内存排序 | 迁移至 Foreign Memory API |

对于尚未完成迁移的库，可通过 `--sun-misc-unsafe-memory-access=allow` 临时抑制警告。
