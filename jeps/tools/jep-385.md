## 1. 概述

JEP 385 在 JDK 15 中将 RMI Activation 机制标记为
`@Deprecated(forRemoval=true)`——正式宣告这一远程对象激活框架将在未来版本中被移除。
后续由 JEP 407 在 JDK 17 中完成实际移除。

### 1.1 核心变更

- **`java.rmi.activation` 包**：整个包标注 `@Deprecated(forRemoval=true)`
- **`rmid` 守护进程**：RMI Activation 守护进程标记为废弃
- **编译与运行时警告**：使用相关 API 时触发 removal 级别警告
- **RMI 核心不受影响**：`java.rmi` 包的核心远程调用功能完全保留

---

## 2. 背景：RMI Activation 的历史

RMI Activation 在 JDK 1.2（1998）中引入，解决了一个特定问题：让远程对象在不被
使用时休眠，在客户端调用时由 `rmid` 守护进程自动启动 JVM 并重新实例化。

2000 年代，企业级分布式计算转向 EJB 和 Web 服务，随后微服务架构和容器编排
（Kubernetes）进一步取代了传统远程对象模型，RMI Activation 失去了使用场景。

---

## 3. JDK 15 中的变更

### 3.1 废弃的 API 清单

```
java.rmi.activation.Activatable           # 可激活远程对象基类
java.rmi.activation.ActivationDesc        # 激活描述符
java.rmi.activation.ActivationGroup       # 激活组
java.rmi.activation.ActivationSystem      # 激活系统接口
java.rmi.activation.Activator             # 激活器接口
rmid                                      # RMI Activation 守护进程
```

### 3.2 不受影响的组件

| 组件 | 状态 |
|------|------|
| `java.rmi.Remote` | **保留** — RMI 核心接口不受影响 |
| `java.rmi.server.UnicastRemoteObject` | **保留** — 非激活远程对象正常使用 |
| `rmiregistry` | **保留** — RMI 注册表工具不受影响 |
| 序列化（`java.io.Serializable`） | **保留** — RMI 底层序列化不受影响 |

---

## 4. 为何废弃

RMI Activation 的实际使用率接近于零。即使在 RMI 仍被使用的场景中，开发者也几乎
总是使用 `UnicastRemoteObject`（非激活模式）。`rmid` 守护进程和激活框架需持续
维护安全补丁和兼容性测试，为无人使用的功能投入资源不合理。此外 `rmid` 可被指示
启动任意 JVM 并加载任意类，存在安全隐患。

| 需求 | 现代替代方案 |
|------|------------|
| 远程对象激活 | 容器编排（Kubernetes Pod 自动扩缩） |
| 服务生命周期管理 | EJB 容器、Spring 容器 |
| 分布式通信 | gRPC、REST API、消息队列 |
| 对象持久化与恢复 | 数据库 + ORM 框架 |

---

## 5. 通往 JEP 407 的时间线

| 时间 | 版本 | 事件 |
|------|------|------|
| 1998 | JDK 1.2 | RMI Activation 引入 |
| 2000s | — | EJB、Web 服务逐步取代 RMI Activation 场景 |
| **2020** | **JDK 15** | **JEP 385：标记为 Deprecated(forRemoval=true)** |
| **2021** | **JDK 17** | **JEP 407：从 JDK 中移除** |

从废弃到移除仅间隔两个版本（约一年），反映了社区对移除该功能的高度共识。

---

## 6. 迁移建议

| 当前用法 | 迁移方案 |
|---------|---------|
| `Activatable` 远程对象 | 改为 `UnicastRemoteObject`，由外部容器管理生命周期 |
| `rmid` 守护进程管理 | 使用 systemd、Kubernetes 等管理 JVM 进程 |
| RMI Activation 注册 | 使用 `rmiregistry` 直接注册非激活远程对象 |
| 整体 RMI 架构 | 考虑迁移至 gRPC 或 REST 等现代通信框架 |
