## 1. 概述

JEP 372 在 JDK 15 中移除了 Nashorn JavaScript 引擎及其相关的 `jjs` 命令行工具。
这是 JDK 11 中 [JEP 335](https://openjdk.org/jeps/335) 废弃操作的后续步骤。

Nashorn 最初通过 JDK 8 的 [JEP 174](https://openjdk.org/jeps/174) 引入，
作为 Rhino JavaScript 引擎的替代品。然而 ECMAScript 语言规范的快速演进使得
Nashorn 的维护变得不可持续，最终导致其从 JDK 中移除。

### 1.1 移除的组件

- **`jdk.scripting.nashorn`** 模块——Nashorn 引擎实现
- **`jdk.scripting.nashorn.shell`** 模块——`jjs` 交互式 Shell 工具
- **`jjs`** 命令行工具

---

## 2. 背景：Nashorn 的生命周期

### 2.1 引入与定位（JDK 8, 2014）

Nashorn 通过 JEP 174 在 JDK 8 中引入，实现了 ECMAScript 5.1 规范。它基于
`invokedynamic` 指令构建，将 JavaScript 编译为 JVM 字节码执行，相比前代
Rhino 引擎有显著的性能提升，并支持 `javax.script` (JSR 223) 脚本 API。

### 2.2 ECMAScript 演进的挑战

2015 年发布的 ECMAScript 6 引入了大量新语法和语义特性，此后 TC39 采用
年度发布节奏。Nashorn 团队发现跟上这一演进速度需要的工程资源远超预期。

### 2.3 JEP 335 废弃（JDK 11, 2018）

JDK 11 通过 JEP 335 将 Nashorn 标记为 `Deprecated for Removal`，给予
社区两年的迁移时间。

---

## 3. JDK 15 中的变更

### 3.1 模块与工具移除

`jdk.scripting.nashorn` 和 `jdk.scripting.nashorn.shell` 两个模块从 JDK 中
完全移除，`jjs` 命令行工具不再包含在发行版中。

### 3.2 API 影响

通过 `javax.script.ScriptEngineManager` 请求 `"nashorn"` 或 `"js"` 引擎
将返回 `null`。依赖该 API 的应用需要迁移至替代引擎。

---

## 4. 为何移除

### 4.1 ECMAScript 演进速度过快

从 ES6 开始，ECMAScript 每年发布新版本。Nashorn 仅实现了 ES5.1，完整实现
现代 ECMAScript（`async/await`、模块系统、可选链等）需要的工程投入远超
JDK 团队在该领域可分配的资源。

### 4.2 维护负担与回报不对称

Nashorn 的用户基础较小，但引擎需要与 HotSpot JIT 编译器协调工作，
`invokedynamic` 调用链的调优和 bug 修复持续消耗工程资源。

### 4.3 更优秀的替代品已出现

GraalJS 提供了完整的 ECMAScript 2020+ 支持，性能优于 Nashorn，并兼容
`javax.script` API。其独立的发布节奏可以更快跟进 ECMAScript 标准演进。

---

## 5. 迁移方案

### 5.1 独立 Nashorn（Maven Central）

Nashorn 引擎以独立项目 `org.openjdk.nashorn:nashorn-core` 发布在 Maven
Central 上，支持 JDK 15+。该版本由社区维护，但不保证长期支持。

### 5.2 GraalJS（推荐）

GraalJS（`org.graalvm.js:js`）是推荐的长期迁移方案，提供完整的现代
ECMAScript 支持，兼容 `javax.script` API，大多数情况下可直接替换 Nashorn。

### 5.3 迁移注意事项

| 场景 | 建议 |
|------|------|
| 简单脚本执行 | 迁移至 GraalJS，API 基本兼容 |
| Nashorn 扩展 API | 检查 GraalJS 兼容模式或重写 |
| `jjs` 命令行使用 | 使用 GraalVM 的 `js` 命令替代 |
