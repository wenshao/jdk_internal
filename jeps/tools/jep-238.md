## 1. 概述

JEP 238 扩展了 JAR 文件格式，允许一个 JAR 中包含针对不同 JDK 版本的类实现。这种 **Multi-Release JAR（MRJAR）** 使库作者能够在单个发布制品中同时提供面向多个 Java 版本的代码，JVM 在运行时根据自身版本自动选择最合适的类文件。该特性从根本上解决了"一个库如何同时利用新版 JDK 特性又保持对旧版 JDK 兼容"的长期问题。

---

## 2. 背景

Java 平台以向后兼容性著称，但库作者面临两难：希望利用新版 JDK 的 API 和性能改进（如 JDK 9 的 `VarHandle`、JDK 11 的 `HttpClient`），同时需要支持旧版 JDK 用户。在 JEP 238 之前，可选策略都不理想：

| 策略 | 缺点 |
|------|------|
| **只使用最低版本 API** | 无法利用新版 JDK 的性能和功能改进 |
| **发布多版本 JAR** | 增加发布和依赖管理复杂性 |
| **反射调用新 API** | 代码复杂、不安全、难以维护 |
| **维护代码分支** | 维护成本高，版本发布矩阵膨胀 |

许多开源库不得不维护多个源代码分支或使用复杂构建配置，导致重复代码和用户选择正确 JAR 版本的认知负担。JEP 238 通过在 JAR 格式层面提供原生支持来解决这些问题。

---

## 3. JAR 结构

MRJAR 通过 `META-INF/versions/` 目录放置版本特定的类文件：

```
jar-root/
├── META-INF/
│   ├── MANIFEST.MF              (包含 Multi-Release: true)
│   └── versions/
│       ├── 9/
│       │   └── com/example/Util.class    (JDK 9+ 专用实现)
│       └── 11/
│           └── com/example/Util.class    (JDK 11+ 专用实现)
└── com/example/Util.class                (基础版本，JDK 8 及以下)
```

- 基础目录包含面向最低支持版本的类；`META-INF/versions/N/` 包含 JDK N+ 的替代实现
- 版本目录中的类与基础目录使用完全相同的包路径和类名
- `MANIFEST.MF` 必须包含 `Multi-Release: true` 属性

---

## 4. 工作原理

当 JVM 加载 MRJAR 中的类时，首先检查 `MANIFEST.MF` 是否声明 `Multi-Release: true`，然后查找 `META-INF/versions/` 下不大于当前 JDK 版本号的最高版本目录中的类文件。如果没有找到匹配，回退到基础目录。

以包含基础版本、JDK 9 和 JDK 11 覆盖的 MRJAR 为例：

| 运行时 JDK | 加载的 Util.class |
|-----------|-------------------|
| JDK 8 | `com/example/Util.class`（基础版本） |
| JDK 9 / 10 | `META-INF/versions/9/com/example/Util.class` |
| JDK 11+ | `META-INF/versions/11/com/example/Util.class` |

对于不识别 Multi-Release 属性的旧版 JVM（JDK 8 及以下），`META-INF/versions/` 目录被自动忽略，使用基础版本，保证完全向后兼容。

## 5. 构建工具支持

**Maven**：通过 `maven-compiler-plugin` 多次执行编译不同版本源代码，`maven-jar-plugin` 设置 `<Multi-Release>true</Multi-Release>`。

**Gradle**：通过额外的 source set 和 jar 任务配置：

```groovy
jar {
    into('META-INF/versions/9') { from sourceSets.java9.output }
    manifest.attributes('Multi-Release': 'true')
}
```

**jar 命令行工具**（JDK 9+）直接支持 `--release` 选项：

```bash
jar --create --file lib.jar \
    -C classes . \
    --release 9 -C classes-9 . \
    --release 11 -C classes-11 .
```

---

## 6. 使用场景

- **利用新版 API 提升性能**：基础版本用 `sun.misc.Unsafe` 或 `synchronized`，JDK 9 版本改用 `VarHandle` 获得更好的性能和安全性
- **适配模块系统**：JDK 9 封装了部分内部 API，MRJAR 允许库在 JDK 9+ 上使用公开 API（如 `java.util.Base64`），在旧版 JDK 上继续使用内部 API（如 `sun.misc.BASE64Encoder`）
- **运行时特性检测**：在新版 JDK 上用 `StackWalker`（JDK 9）替代 `Thread.getStackTrace()` 实现更高效的调用栈遍历

---

## 7. 限制与注意事项

| 限制 | 说明 |
|------|------|
| **公共 API 一致性** | 版本覆盖的类必须保持与基础版本相同的 public/protected 签名，`jar` 工具会执行此验证 |
| **测试复杂性** | 每个版本覆盖都需要在对应 JDK 版本上测试，CI/CD 需配置多个 JDK |
| **构建复杂性** | 需要配置多个编译执行，IDE 对多版本源代码的支持参差不齐 |
| **不能引入新类型** | 版本覆盖目录中不能引入基础目录中不存在的新公共类型 |
| **JAR 体积** | 过多版本层级会增加 JAR 体积 |

## 8. 实际采用情况

MRJAR 已被多个重要开源项目采用：

| 项目 | 使用方式 |
|------|----------|
| **ASM** | 字节码操作库，适配不同 JDK 版本的类文件格式变化 |
| **Guava** | Google 核心库，在高版本 JDK 上使用 `VarHandle` 等新 API |
| **Jackson** | JSON 处理库，在 JDK 9+ 上适配模块系统并利用新特性 |
| **Log4j 2** | 日志框架，在 JDK 9+ 上利用 `StackWalker` 提升性能 |
| **Byte Buddy** | 运行时代码生成库，适配不同 JDK 版本的内部 API 变化 |
