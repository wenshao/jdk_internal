## 1. 概述

JEP 174 在 JDK 8 中引入了 Nashorn——一个全新的、基于 JVM 的 JavaScript 引擎，用于取代自 JDK 6 以来内置的 Rhino 引擎。Nashorn 将 JavaScript 代码直接编译为 JVM bytecode，而非像 Rhino 那样解释执行，从而在性能上实现了质的飞跃。

Nashorn 这个名字来自德语"犀牛"（Rhinoceros 的前辈引擎 Rhino 同为犀牛之意），项目代号延续了 JavaScript-on-JVM 的传统。

---

## 2. 背景

### 2.1 Rhino 的局限性

Rhino 是 Mozilla 开发的 Java 实现的 JavaScript 引擎，JDK 6 将其内置于 `javax.script` 框架中。但随着 JavaScript 语言和 JVM 技术的演进，Rhino 暴露出多项不足：

- **性能低下**：Rhino 基于解释器执行，缺乏 JIT 编译优化，在计算密集型场景中表现较差
- **ECMAScript 支持不完整**：停留在 ECMAScript 3 水平，对 ES5.1 的支持非常有限
- **代码库老旧**：源码维护困难，难以利用 JDK 7 引入的 `invokedynamic` 指令

### 2.2 JVM 技术演进的契机

JDK 7 引入的 `invokedynamic` (JSR 292) 为在 JVM 上高效运行动态语言提供了关键基础设施。Nashorn 项目正是 `invokedynamic` 最重要的实际应用之一，展示了该指令在动态语言实现中的威力。

### 2.3 javax.script (JSR 223) 集成

Nashorn 完全兼容 JSR 223 (Scripting for the Java Platform) 规范，用户只需通过 `ScriptEngineManager` 获取引擎即可无缝切换：

```java
ScriptEngineManager manager = new ScriptEngineManager();
ScriptEngine engine = manager.getEngineByName("nashorn");
engine.eval("print('Hello from Nashorn')");
```

## 3. 架构设计

### 3.1 编译管线

Nashorn 采用 AOT 编译策略（相对于 Rhino 的解释执行），其核心管线为：

```
JavaScript Source
    ↓  Parser
   AST (Abstract Syntax Tree)
    ↓  Compiler
   IR (Intermediate Representation)
    ↓  Code Generator
  JVM Bytecode (.class)
    ↓  JVM
  HotSpot JIT 优化执行
```

每个 JavaScript 函数被编译为一个 Java 类，由 JVM ClassLoader 加载后交由 HotSpot JIT 进一步优化。

### 3.2 invokedynamic 的运用

Nashorn 广泛使用 `invokedynamic` 处理 JavaScript 中的动态特性：

- **属性访问**：`obj.prop` 编译为 `invokedynamic` 调用，运行时通过 `MethodHandle` 链接到实际属性
- **函数调用**：JavaScript 函数调用通过 `invokedynamic` 分派，支持 inline cache 优化
- **类型转换**：动态类型间的隐式转换由 `invokedynamic` 的 bootstrap method 处理

### 3.3 Optimistic Typing（乐观类型推断）

JDK 8u40 中引入了 optimistic typing 优化（通过 `--optimistic-types` 启用，后来默认开启）：

1. **初始假设**：编译器假设变量为 `int` 类型
2. **执行验证**：运行时检查类型假设是否成立
3. **去优化回退**：若类型假设失败，触发 deoptimization，以更宽泛的类型重新编译
4. **逐步稳定**：经过几次 deopt 后，类型假设趋于稳定

这一策略与 V8 等现代 JavaScript 引擎的思路类似，使 Nashorn 在数值计算场景中性能显著提升。

## 4. 核心特性

### 4.1 ECMAScript 5.1 完整支持

Nashorn 通过了 ECMAScript 5.1 conformance test suite 的绝大部分测试用例，支持包括：

- Strict mode
- `Object.create()` / `Object.defineProperty()`
- `JSON.parse()` / `JSON.stringify()`
- `Array.prototype.forEach/map/filter/reduce`

### 4.2 jjs 命令行工具

JDK 8 附带了 `jjs` 命令行工具，提供交互式 JavaScript REPL 和脚本执行能力：

```bash
# 交互模式
$ jjs
jjs> var x = 10
jjs> print(x * 2)
20

# 执行脚本文件
$ jjs script.js

# 传递参数
$ jjs -- arg1 arg2 script.js
```

### 4.3 Java-JavaScript 双向互操作

Nashorn 的一大亮点是与 Java 的深度互操作：

```javascript
// 在 JavaScript 中使用 Java 类
var ArrayList = Java.type('java.util.ArrayList');
var list = new ArrayList();
list.add('hello');

// 实现 Java 接口
var Runnable = Java.type('java.lang.Runnable');
var task = new Runnable(function() {
    print('Running in a thread');
});
new java.lang.Thread(task).start();

// Lambda 风格：自动将 JavaScript 函数适配为 SAM 接口
var thread = new java.lang.Thread(function() {
    print('Simplified thread creation');
});
```

### 4.4 ScriptEngine API 集成

通过标准 `javax.script` API 在 Java 中嵌入 JavaScript：

```java
ScriptEngine engine = new ScriptEngineManager()
    .getEngineByName("nashorn");

// 绑定变量
engine.put("name", "World");
engine.eval("print('Hello, ' + name)");

// 调用 JavaScript 函数
engine.eval("function greet(n) { return 'Hi, ' + n; }");
Invocable inv = (Invocable) engine;
String result = (String) inv.invokeFunction("greet", "Java");
```

## 5. 性能

Nashorn 相比 Rhino 在特定工作负载下实现了显著的性能提升：

| 场景 | Rhino | Nashorn | 提升幅度 |
|------|-------|---------|---------|
| 数值计算 | 基线 | 2-10x 更快 | 受益于 bytecode 编译 + JIT |
| 字符串操作 | 基线 | 2-5x 更快 | 受益于 `invokedynamic` |
| 启动时间 | 较快 | 较慢（首次编译开销） | 编译成本换取运行速度 |

> **注意**：以上数据来自 Nashorn 团队在 JavaOne 2013-2014 演讲中的参考值，实际性能因工作负载而异。

性能优势的核心来源：

- JavaScript 被直接编译为 JVM bytecode，享受 HotSpot C1/C2 JIT 编译器的全部优化
- `invokedynamic` 实现了高效的 inline cache 和方法分派
- Optimistic typing 减少了不必要的装箱/拆箱操作

## 6. 生命周期

### 6.1 时间线

| 版本 | 年份 | JEP | 状态 |
|------|------|-----|------|
| JDK 8 | 2014-03 | JEP 174 | 引入 Nashorn |
| JDK 8u40 | 2015-03 | — | 引入 optimistic typing |
| JDK 9 | 2017-09 | — | 部分 ECMAScript 6 支持 |
| JDK 11 | 2018-09 | [JEP 335](https://openjdk.org/jeps/335) | 标记为 deprecated |
| JDK 15 | 2020-09 | [JEP 372](https://openjdk.org/jeps/372) | 正式移除 |

### 6.2 废弃与移除原因

Nashorn 从引入到移除仅经历了约 6 年，主要原因包括：

1. **ECMAScript 演进速度过快**：ES6 (2015) 之后 ECMAScript 进入年度发布节奏，Nashorn 团队难以跟上 ES2016-ES2020 的语言特性
2. **维护负担过重**：全面实现现代 JavaScript 特性（如 `Promise`、`async/await`、modules）需要大量工程投入
3. **GraalJS 作为替代方案**：GraalVM 项目提供了 GraalJS，基于 Truffle 框架实现，支持最新 ECMAScript 标准，且性能更优

### 6.3 迁移路径

对于依赖 Nashorn 的用户，有以下迁移选项：

- **GraalJS**（GraalVM 项目）：最直接的替代方案，支持 `javax.script` API，兼容性较好
- **独立 Nashorn**：Nashorn 被移除后以 [standalone 项目](https://github.com/openjdk/nashorn) 继续维护
- **其他方案**：如嵌入 V8（J2V8）等

## 7. 遗产与经验教训

### 7.1 技术遗产

Nashorn 虽然已被移除，但其技术影响深远：

- **验证了 `invokedynamic` 的实用价值**：Nashorn 是 `invokedynamic` 最大规模的实际应用之一，推动了 JVM 对动态语言支持的优化
- **推动了 JVM 语言生态**：为后续的 Truffle/GraalVM 多语言框架提供了经验基础
- **影响了 Java 语言设计**：Lambda 表达式与 Nashorn 的 SAM 类型自动转换相辅相成

### 7.2 经验教训

1. **内置 vs 独立**：将快速演进的外部语言引擎内置于 JDK 中，长期来看难以维护。GraalVM 采用独立发布模型是更好的选择
2. **标准追赶困境**：当外部标准的演进速度超过平台的发布节奏时，紧耦合必然导致落后
3. **`invokedynamic` 的成功**：虽然 Nashorn 本身被移除，但它推动的 `invokedynamic` 优化惠及了 JVM 上的所有动态语言（JRuby、Groovy、Kotlin 等）
