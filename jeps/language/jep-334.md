## 1. 概述

JEP 334 引入了 `java.lang.constant` 包，提供一组用于描述类文件常量池中可加载常量（loadable constants）的名义描述符（nominal descriptors）。这些描述符以纯符号形式表示常量，无需实际加载类或解析引用，为编译器、字节码工具和框架提供了标准化的常量建模能力。

---

## 2. 背景

### 2.1 常量池与可加载常量

JVM 类文件的常量池（constant pool）存储了类运行所需的各种常量——字符串、整数、浮点数、类引用、方法引用等。自 JDK 11 的 JEP 309 引入 `CONSTANT_Dynamic` 后，常量池可表达的常量类型进一步扩展，支持通过引导方法（bootstrap method）在运行时惰性解析常量。

可加载常量是指可以通过 `ldc` 指令或作为 `invokedynamic`/`CONSTANT_Dynamic` 引导方法的静态参数出现在常量池中的值。JDK 12 之前，Java 语言层面缺乏统一的方式来描述这些常量。

### 2.2 invokedynamic 与引导方法

`invokedynamic` 指令及其引导方法（bootstrap method）机制允许将调用点的链接逻辑推迟到运行时。引导方法接收的参数包括常量池中的静态参数——这些参数本身也是可加载常量。然而，操作这些常量的工具（如编译器、字节码库）需要在不加载类的前提下描述和操作常量。直接使用 `Class`、`MethodType` 等运行时对象需要真正的类加载，这在编译期或跨 ClassLoader 场景中往往不可行。

## 3. API 设计

### 3.1 核心接口：Constable 与 ConstantDesc

`java.lang.constant` 包围绕两个核心接口构建。`Constable` 表示可以用名义描述符描述自身的类型；`ConstantDesc` 是所有可加载常量描述符的根接口，知道如何将自身还原为运行时值：

```java
public interface Constable {
    Optional<? extends ConstantDesc> describeConstable();
}

public interface ConstantDesc {
    Object resolveConstantDesc(MethodHandles.Lookup lookup)
        throws ReflectiveOperationException;
}
```

JDK 12 中，`String`、`Integer`、`Long`、`Float`、`Double`、`Class`、`MethodType`、`MethodHandle` 及枚举常量均实现了 `Constable` 接口。

### 3.2 ClassDesc

`ClassDesc` 以描述符字符串的形式表示类或接口，无需加载对应的 `Class` 对象：

```java
ClassDesc listDesc = ClassDesc.of("java.util.List");
ClassDesc intArrayDesc = ClassDesc.ofDescriptor("[I");
ClassDesc stringArrayDesc = ClassDesc.of("java.lang.String").arrayType();
```

### 3.3 MethodTypeDesc 与 MethodHandleDesc

`MethodTypeDesc` 用名义方式描述方法签名（参数类型列表和返回类型）。`MethodHandleDesc` 及其子接口 `DirectMethodHandleDesc` 描述方法句柄的行为方式（虚调用、静态调用、构造器等）：

```java
// 描述 (String, int) -> boolean 的方法类型
MethodTypeDesc mtd = MethodTypeDesc.of(
    ClassDesc.ofDescriptor("Z"),
    ClassDesc.of("java.lang.String"), ClassDesc.ofDescriptor("I"));

// 描述 String.length() 的虚方法句柄
DirectMethodHandleDesc mhd = MethodHandleDesc.ofMethod(
    DirectMethodHandleDesc.Kind.VIRTUAL,
    ClassDesc.of("java.lang.String"), "length",
    MethodTypeDesc.of(ClassDesc.ofDescriptor("I")));
```

### 3.4 DynamicConstantDesc

`DynamicConstantDesc` 对应 JEP 309 引入的 `CONSTANT_Dynamic` 常量池条目，通过引用引导方法及其静态参数来描述一个惰性解析的动态常量。它还预定义了常用描述符（如 `NULL`、基本类型默认值）：

```java
// 描述枚举常量 TimeUnit.HOURS（无需加载 TimeUnit 类）
DynamicConstantDesc<Enum<?>> enumDesc = DynamicConstantDesc.ofNamed(
    ConstantDescs.BSM_ENUM_CONSTANT, "HOURS",
    ClassDesc.of("java.util.concurrent.TimeUnit"));
```

## 4. 使用场景

### 4.1 编译器与字节码工具

JVM Constants API 的首要用户是编译器和字节码生成/分析工具，它们需要在不加载目标类的前提下操作常量池条目：

| 工具类型 | 应用方式 |
|---------|---------|
| **javac 编译器** | 在编译期使用名义描述符表示常量池条目，避免触发类加载 |
| **字节码库（ASM、ByteBuddy）** | 用类型安全的 API 构造常量池条目，替代手工拼接描述符字符串 |
| **静态分析工具** | 以符号形式分析类文件中的常量引用，无需解析类依赖 |

### 4.2 框架运行时

框架在运行时生成字节码时，可以使用名义描述符安全地构造引导方法参数，无需强制加载目标类：

```java
ClassDesc targetClass = ClassDesc.of("com.app.MyService");
MethodTypeDesc factoryType = MethodTypeDesc.of(targetClass);
// 传递给字节码生成器，无需加载 MyService 类
```

## 5. 与 JEP 309 的关系

JEP 334 与 JEP 309（Dynamic Class-File Constants）是互补关系：

| 维度 | JEP 309 | JEP 334 |
|------|---------|---------|
| **层级** | JVM 层面 | Java API 层面 |
| **引入版本** | JDK 11 | JDK 12 |
| **职责** | 定义 `CONSTANT_Dynamic` 常量池条目格式与 JVM 解析语义 | 提供 Java 语言层面的名义描述符来表示和操作这些常量 |
| **关键产物** | `CONSTANT_Dynamic` 字节码结构 | `DynamicConstantDesc` 等描述符类 |

JEP 309 让 JVM 能够在常量池中表达动态计算的常量，而 JEP 334 让 Java 程序和工具能够以类型安全、符号化的方式构造和操作这些常量。`DynamicConstantDesc` 正是连接两者的桥梁——它用名义描述符表示一个 `CONSTANT_Dynamic` 条目。
