## 1. 概述

JEP 104 扩展了 Java 注解的适用范围，使注解不仅可以出现在声明（类、方法、字段等）上，还可以出现在任何使用类型的位置。JDK 8 之后，注解可以标注泛型类型参数、数组元素类型、类型转换、`implements` 子句以及 `throws` 子句中的异常类型等。这一变更为静态分析和可插拔类型系统奠定了语言层面的基础。

---

## 2. 背景

JEP 104 的设计源自 JSR 308（Annotations on Java Types），由 Michael Ernst 于 2006 年提出，目标是让注解能够出现在所有类型使用位置而非仅限于声明。该规范经过多年发展，最终被纳入 JDK 8。

在 JDK 8 之前，Checker Framework、FindBugs 等静态分析工具已展示了在类型上添加注解的价值（例如用 `@NonNull` 防止空指针异常），但语言本身不支持类型使用位置的注解，工具只能通过声明注解间接表达约束，无法精确标注泛型参数或嵌套类型。例如，无法区分"容器本身可为空"与"容器中的元素可为空"。

---

## 3. 变更内容

### ElementType.TYPE_USE 和 TYPE_PARAMETER

JDK 8 在 `java.lang.annotation.ElementType` 枚举中新增了两个常量：

| 常量 | 含义 |
|------|------|
| `TYPE_USE` | 注解可以出现在任何类型使用的位置 |
| `TYPE_PARAMETER` | 注解可以出现在类型参数声明上（如 `<@Immutable T>`） |

定义注解时通过 `@Target(ElementType.TYPE_USE)` 指定即可。`TYPE_USE` 注解可出现在以下位置：泛型类型参数（`List<@NonNull String>`）、数组元素类型（`@NonNull String[]`）、类型转换（`(@NonNull String) obj`）、`extends`/`implements` 子句、`throws` 子句、构造器调用（`new @Interned String("hello")`）以及通配符边界（`List<@Positive ? extends Number>`）。

---

## 4. 使用示例

```java
// 标注变量的类型为非空
@NonNull String name = "Alice";

// 标注泛型参数：列表中的每个 Integer 必须为正数
List<@Positive Integer> scores = new ArrayList<>();

// 标注数组：数组本身与元素均非空
@NonNull String @NonNull [] items;

// Map 的键非空，值可为空
Map<@NonNull String, @Nullable Object> config;

// 标注类型参数声明
class Cache<@Immutable K, V> { ... }

// 在 throws 子句中标注异常类型
void process() throws @Recoverable IOException { ... }
```

## 5. 可插拔类型系统与 Checker Framework

JEP 104 为 Java 提供了可插拔类型系统（pluggable type system）的语言基础。类型注解本身不改变编译器的类型检查行为——编译器只负责将注解记录在 class 文件中，真正的检查逻辑由外部注解处理器或编译器插件实现。开发者可以根据项目需要选择不同的类型系统插件，在编译期捕获空指针、并发错误、单位不匹配等问题。

Checker Framework 是 JEP 104 最重要的实际应用，也是推动 JSR 308 设计的核心动力。该框架提供了一系列内置的类型检查器：

| 检查器 | 功能 | 示例注解 |
|--------|------|----------|
| Nullness Checker | 检测空指针异常 | `@NonNull`, `@Nullable` |
| Interning Checker | 检测对象 intern 使用 | `@Interned` |
| Lock Checker | 检测并发锁使用错误 | `@GuardedBy` |
| Units Checker | 检测物理单位不匹配 | `@m`, `@kg`, `@s` |

使用时在编译命令中指定处理器即可启用检查：

```bash
javac -processor org.checkerframework.checker.nullness.NullnessChecker MyApp.java
```

若代码中存在类型注解违规（例如将 `@Nullable` 值赋给 `@NonNull` 变量），编译器将报告错误。

## 6. 限制与约束

- **默认无运行时效果**：类型注解本身不改变程序的运行时行为。即使标注了 `@NonNull`，JVM 也不会自动进行空值检查。
- **需要注解处理器配合**：类型注解的价值完全取决于是否有对应的注解处理器在编译期执行检查。不使用处理器时，注解仅作为文档存在。
- **注解位置的歧义**：在某些位置（如 `@A int[] arr`），注解可能修饰声明也可能修饰类型，具体取决于 `@A` 的 `@Target` 定义。
- **标准注解缺失**：JDK 8 引入了类型注解机制，但未提供 `@NonNull` 等标准类型注解。开发者需依赖 Checker Framework、JetBrains Annotations、JSR 305 等第三方库，导致生态碎片化。

---

## 7. 对开发者的影响

不使用类型注解的项目不受任何影响。对于注重静态分析的团队，类型注解提供了在编译期发现更多错误的能力，尤其在空安全方面具有显著价值。许多现代 Java 项目已将 `@NonNull` / `@Nullable` 注解作为编码规范的一部分。
