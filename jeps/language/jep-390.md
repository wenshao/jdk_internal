# JEP 390: Warnings for Value-Based Classes

> **Status**: Final | **Type**: Feature | **Component**: core-libs
>
> **Lead**: Chris Hegarty (Oracle)
>
> **JDK Version**: JDK 16 (2021-03)
>
> **Organization**: [Oracle](../../contributors/orgs/oracle.md)

---

## 1. 概述

JEP 390 将原始类型的包装类（`Integer`、`Long`、`Double` 等）正式标记为**值基类（value-based classes）**，并在编译期和运行期对不当使用发出警告。具体措施包括：将包装类构造器标记为 `forRemoval` 弃用、对 `synchronized` 锁定值基类实例时发出编译器警告、以及通过 JVM 参数启用运行时同步诊断。这些变更是 Project Valhalla 将包装类迁移为值类型的准备工作。

---

## 2. 背景

Project Valhalla 目标是引入**值类型（value types）**——没有对象标识（identity）的轻量级类型，实例可被 JVM 扁平化存储，无需对象头开销。`Integer`、`Long` 等包装类在语义上是值的容器，是迁移为值类型的天然候选。

然而当前包装类是普通引用类型，具有对象标识——可以对其使用 `synchronized`、通过 `==` 比较标识、调用 `System.identityHashCode()`。当 Valhalla 迁移完成后，这些依赖标识的操作将不再有效。JEP 390 在迁移之前提前发出警告，给开发者时间调整代码。

---

## 3. 什么是值基类

JDK 16 中被标记为值基类的类型包括：

| 分类 | 类型 |
|------|------|
| **原始包装类** | `Byte`、`Short`、`Integer`、`Long`、`Float`、`Double`、`Boolean`、`Character` |
| **java.time** | `LocalDate`、`LocalTime`、`LocalDateTime`、`Instant`、`Duration`、`Period` 等 |
| **java.util** | `Optional`、`OptionalInt`、`OptionalLong`、`OptionalDouble` |

值基类的共同特征：声明为 `final`；`equals()`/`hashCode()`/`toString()` 仅依赖实例状态；通过工厂方法而非构造器创建；不应用作 `synchronized` 锁或通过 `==` 比较。

---

## 4. @ValueBased 注解

JEP 390 引入了 `jdk.internal.ValueBased` 注解用于标记值基类。该注解为 `javac` 提供编译时检测信号，为 HotSpot 提供运行时诊断依据。注意此注解是 JDK 内部 API，不属于公共 API。

---

## 5. 弃用包装类构造器

JEP 390 将所有原始包装类的构造器标记为 `@Deprecated(forRemoval=true)`：

```java
// JDK 16 时标记为 @Deprecated，JEP 390 升级为 forRemoval
@Deprecated(since="9", forRemoval=true)
public Integer(int value) { ... }

@Deprecated(since="9", forRemoval=true)
public Integer(String s) { ... }
```

编译器将对任何直接调用包装类构造器的代码发出更强烈的移除警告。

---

## 6. 同步警告

当开发者在值基类实例上使用 `synchronized` 时，`javac` 发出编译器警告：

```java
Integer count = Integer.valueOf(0);
synchronized (count) {  // warning: [synchronization] attempt to synchronize
                        // on an instance of a value-based class
    count = count + 1;
}
```

该警告属于 `synchronization` 类别，通过 `-Xlint:synchronization` 开启。

对值基类实例同步的风险在于两个层面：**当前**——`Integer.valueOf()` 可能返回缓存实例，多个不相关代码路径可能锁定同一对象，导致意外竞争和死锁；**未来**——Valhalla 迁移后值类型不再有对象标识，`synchronized` 将直接抛出异常。

---

## 7. 运行时警告

JEP 390 引入了 HotSpot 层面的运行时诊断选项：

| 参数 | 说明 |
|------|------|
| `-XX:DiagnoseSyncOnValueBasedClasses=1` | 遇到值基类同步时发出警告日志 |
| `-XX:DiagnoseSyncOnValueBasedClasses=2` | 遇到值基类同步时直接抛出致命错误 |

运行时检查可以捕获编译器无法检测的场景——例如通过 `Object` 引用间接锁定值基类实例：

```java
Object lock = Integer.valueOf(1);  // 静态类型为 Object
synchronized (lock) {              // javac 无法警告，但运行时可以检测
}
```

---

## 8. 迁移指南

### 8.1 从构造器到工厂方法

| 弃用用法 | 推荐替代 |
|---------|---------|
| `new Integer(42)` | `Integer.valueOf(42)` |
| `new Long(100L)` | `Long.valueOf(100L)` |
| `new Double(3.14)` | `Double.valueOf(3.14)` |
| `new Boolean(true)` | `Boolean.TRUE` |

`valueOf()` 允许 JVM 返回缓存实例（`Integer` 对 -128 到 127 进行缓存），减少内存分配，并与未来值类型迁移兼容。

### 8.2 替换同步对象

使用包装类实例作为锁对象的代码应替换为 `new Object()` 或 `java.util.concurrent.locks.ReentrantLock`。

---

## 9. 通向 JEP 401 之路

JEP 390 是 Project Valhalla 路线图的关键里程碑：

| 阶段 | JEP | 说明 |
|------|-----|------|
| **准备** | JEP 390（JDK 16） | 标记值基类，发出警告，给开发者迁移时间 |
| **演进** | JEP 401（Primitive Classes） | 引入值类型语法，将包装类迁移为原始类 |

JEP 390 的警告机制确保当 JEP 401 最终落地时，生态系统中依赖包装类对象标识的代码已大幅减少。这种"先警告、后变更"的渐进策略是 Java 平台兼容性哲学的典型体现。
