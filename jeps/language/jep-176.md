# JEP 176: Mechanical Checking of Caller-Sensitive Methods

> **Status**: Final | **Type**: Feature | **Component**: core-libs/java.lang
>
> **Lead**: John Rose (Oracle)
>
> **JDK Version**: JDK 8 (2014-03)
>
> **Organization**: [Oracle](../../contributors/orgs/oracle.md)

---

## 1. 概述

JEP 176 引入 `@CallerSensitive` 注解，替代 JVM 内部维护的硬编码调用者敏感方法列表。该注解标记那些行为依赖于直接调用者的方法（如 `Class.forName()`、`Method.invoke()`），并通过编译器和 JVM 的机械化检查确保 `sun.reflect.Reflection.getCallerClass()` 仅在正确标注的方法中使用。

---

## 2. 背景

JDK 中有一类特殊方法，其行为取决于调用它们的代码的身份（类加载器、模块、权限等）。在 JDK 7 及之前，这些方法的列表在 JVM 中硬编码维护，存在以下问题：

- **手动维护易出错** — 新增调用者敏感方法时必须同步更新 JVM 内部列表
- **缺乏编译期验证** — 如果方法未在列表中注册，`getCallerClass()` 会返回错误的调用者
- **安全隐患** — 遗漏注册可能导致权限检查被绕过

---

## 3. 调用者敏感方法示例

以下是 JDK 中典型的调用者敏感方法：

| 方法 | 调用者敏感行为 |
|------|---------------|
| `Class.forName(String)` | 使用调用者的类加载器加载类 |
| `Class.newInstance()` | 对调用者执行访问控制检查 |
| `Method.invoke()` | 对调用者执行访问控制检查 |
| `Constructor.newInstance()` | 对调用者执行访问控制检查 |
| `ResourceBundle.getBundle()` | 使用调用者的类加载器和模块 |
| `DriverManager.getConnection()` | 使用调用者的类加载器过滤驱动 |

---

## 4. @CallerSensitive 注解

JDK 8 在 `sun.reflect` 包（后迁移至 `jdk.internal.reflect`）中定义了 `@CallerSensitive` 注解。所有调用者敏感方法必须标注此注解：

```java
@CallerSensitive
public static Class<?> forName(String className) throws ClassNotFoundException {
    Class<?> caller = Reflection.getCallerClass();
    return forName0(className, true, ClassLoader.getClassLoader(caller));
}
```

---

## 5. 机械化检查机制

JEP 176 建立了两层检查保障：

**编译期检查** — 编译 JDK 源码时，编译器插件验证所有调用 `Reflection.getCallerClass()` 的方法都已标注 `@CallerSensitive`。未标注的方法将导致编译错误。

**运行时检查** — JVM 在执行 `getCallerClass()` 时验证调用方法是否带有 `@CallerSensitive` 注解。如果未标注，JVM 抛出 `InternalError` 而非返回错误的调用者信息。

---

## 6. getCallerClass() 的工作原理

`Reflection.getCallerClass()` 遍历调用栈，跳过被 `@CallerSensitive` 标注的方法本身及其框架内部的委托链，返回真正的外部调用者。这确保即使调用者敏感方法经过内部包装或委托，仍然能获取正确的原始调用者：

```
外部调用者代码           ← getCallerClass() 返回此帧
  └─ Class.forName()    ← @CallerSensitive 方法
       └─ forName0()    ← 内部实现
```

---

## 7. 与 Java 模块系统的演进

JDK 9 引入模块系统后，`@CallerSensitive` 从 `sun.reflect` 包迁移至 `jdk.internal.reflect` 包。模块系统进一步加强了对调用者敏感方法的保护——`jdk.internal.reflect` 包不对外导出，第三方代码无法直接使用此注解，从而防止滥用。

---

## 8. 安全性影响

机械化检查消除了一类潜在的安全漏洞。在旧机制下，如果一个调用者敏感方法未在 JVM 列表中注册，`getCallerClass()` 可能返回中间框架而非真正的外部调用者，导致：

- 权限提升：低权限代码可能被误认为高权限调用者
- 类加载器混淆：使用错误的类加载器加载类
- 访问控制绕过：反射调用跳过预期的安全检查
