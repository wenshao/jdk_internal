## 1. 概述

JEP 201 将 JDK 源码仓库从沿用多年的单体式目录布局重新组织为以模块为中心的目录结构。这是 Project Jigsaw（模块化系统）的基础性前置工作——在引入 `module-info.java` 和运行时模块边界之前，先在源码层面确立模块的物理边界。JEP 201 本身不改变任何 API 或运行时行为，仅改变源码文件在仓库中的存放位置。

---

## 2. 背景：旧的单体源码布局

JDK 9 之前，源码分散在多个顶层仓库（`jdk`、`hotspot`、`langtools`、`corba`、`jaxp`、`jaxws`、`nashorn`），每个仓库按技术类型（而非模块边界）组织：

```
jdk/src/
├── share/
│   ├── classes/           # 所有 Java 源码混在一起，无模块边界
│   │   ├── java/lang/
│   │   ├── java/util/
│   │   ├── javax/swing/
│   │   └── ...            # 数千个包
│   └── native/            # 共享本地代码
├── solaris/               # Solaris/Linux 特有代码
│   ├── classes/
│   └── native/
├── windows/
└── macosx/
```

这种布局存在严重问题：

- **模块边界不可见**：`java.base`、`java.desktop`、`java.sql` 等模块的源文件混杂在同一个 `classes/` 目录树中，无法判断某个包属于哪个模块。
- **跨模块依赖难以追踪**：物理上不分离，开发者容易引入不合理的模块间依赖。
- **平台特有代码分散**：同一个类的不同平台实现分布在 `share/`、`solaris/`、`windows/` 等目录下。
- **构建系统复杂**：Makefile 需要复杂逻辑将分散在多个目录中的文件拼装成正确的编译单元。

## 3. 新的模块化目录结构

### 3.1 顶层模块目录

重组后，`src/` 直接包含以模块名命名的子目录，每个模块物理隔离：

```
src/
├── java.base/
├── java.compiler/
├── java.desktop/
├── java.logging/
├── java.sql/
├── java.xml/
├── jdk.compiler/
├── jdk.javadoc/
├── jdk.jshell/
└── ...                    # 共计约 70 个模块
```

### 3.2 模块内部结构

每个模块目录内部按固定约定组织（以 `java.base` 为例）：

```
src/java.base/
├── share/
│   ├── classes/           # 平台无关 Java 源码 + module-info.java
│   ├── native/            # 平台无关本地代码 (libjava, libzip 等)
│   └── conf/              # 配置文件
├── unix/                  # Unix 特有的 classes/ 和 native/
├── windows/
└── linux/
```

平台分层结构（`share/`、`unix/`、`windows/`）保留，但作用域从"整个 JDK"缩小到"单个模块"。`module-info.java` 放在 `share/classes/` 根目录下。

### 3.3 主要模块

| 模块 | 内容 |
|------|------|
| `java.base` | 核心语言类（`java.lang`、`java.util`、`java.io`、`java.nio`） |
| `java.desktop` | AWT、Swing、Java2D |
| `java.sql` | JDBC API |
| `java.xml` | JAXP（DOM、SAX、StAX） |
| `jdk.compiler` | `javac` 编译器 |
| `jdk.javadoc` | Javadoc 工具 |
| `jdk.jshell` | JShell REPL 工具 |

## 4. 构建系统变更

JDK 使用基于 GNU Make 的构建框架，主要变更包括：

- **模块感知的编译规则**：新系统按模块逐一编译，每个模块使用 `SetupJavaCompilation` 宏指定源码根目录列表。
- **源码路径动态生成**：构建系统根据目标平台自动拼接 `share/classes`、`unix/classes` 等路径，形成每个模块的完整源码路径集。
- **本地库构建隔离**：每个模块的本地代码独立编译为共享库，使用 `SetupNativeCompilation` 宏按模块组织。
- **依赖追踪简化**：构建依赖图与模块依赖图对齐，增量构建更精确。

构建命令本身不变（`configure` + `make`），但新增了模块粒度的构建目标：

```bash
make java.base-java        # 只编译 java.base 的 Java 源码
make java.desktop-libs      # 只构建 java.desktop 的本地库
```

模块粒度的目标大幅缩短了迭代周期——修改 `java.base` 后无需重新编译 `java.desktop`。

## 5. 对 JDK 开发的影响

**代码导航性提升**：开发者可以立即定位某个包属于哪个模块。`javax.swing` 的源码全部在 `src/java.desktop/` 下，不再散落在 `jdk/src/share/classes/` 深处。

**模块边界强制执行**：源码物理隔离后，跨模块的不当依赖在编译期即可被 `module-info.java` 阻止。

**为仓库合并铺路**：JEP 201 为 JEP 296（Consolidate the JDK Forest into a Single Repository, JDK 9）奠定基础。旧的 `jdk`、`hotspot`、`langtools` 等独立仓库合并为单一仓库后，模块化目录结构使其仍然可导航。

**降低贡献者门槛**：模块名即目录名的约定直观易懂，新开发者无需理解 `solaris/` 实际也用于 Linux 等历史约定。

**增量构建效率**：模块粒度的构建目标使局部修改后的重编译时间从分钟级降至秒级。
