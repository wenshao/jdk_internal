## 1. 概述

JEP 320 从 Java SE 平台和 JDK 中移除了 Java EE 和 CORBA 相关的模块。这些模块在 JDK 11 中通过 JEP 261 被标记为 `@Deprecated(forRemoval=true)`，在 JDK 11 中被正式移除。移除范围涵盖 JAX-WS、JAXB、JAF、Common Annotations、CORBA 以及 JTA 共六大技术栈对应的七个模块，以及相关的工具（`wsgen`、`wsimport`、`schemagen`、`xjc`、`idlj`、`orbd`、`servertool`、`tnameserv`）。

---

## 2. 背景

### 2.1 Java EE 技术进入 Java SE 的历史

Java EE 的部分技术（如 JAX-WS、JAXB、JAF）在 JDK 6 时期被纳入 Java SE，方便开发者在不依赖外部库的情况下使用 Web Services 和 XML 绑定功能。然而随着时间推移，JDK 中捆绑的版本落后于 Java EE 独立发布的版本，JDK 团队需要维护不属于 SE 范畴的技术，且独立 JAR 与 JDK 内置模块在模块路径上产生 split package 冲突。

### 2.2 JDK 9 中的废弃

JDK 9 引入模块系统（JPMS）后，通过 JEP 261 将这些模块标记为 `@Deprecated(forRemoval=true)`，并在默认模块图中排除它们。开发者需要显式使用 `--add-modules java.xml.bind` 等选项才能在 JDK 9/10 中继续使用，这一过渡期为正式移除提供了缓冲。

---

## 3. 移除的模块

### 3.1 模块清单

| 模块名 | 技术 | 主要功能 |
|--------|------|----------|
| `java.xml.ws` | JAX-WS (Java API for XML Web Services) | SOAP Web 服务客户端和端点 |
| `java.xml.bind` | JAXB (Java Architecture for XML Binding) | Java 对象与 XML 之间的双向绑定 |
| `java.activation` | JAF (JavaBeans Activation Framework) | MIME 数据类型处理，被 JAX-WS/JAXB 依赖 |
| `java.xml.ws.annotation` | Common Annotations | `@Resource`、`@PostConstruct`、`@PreDestroy` 等通用注解 |
| `java.corba` | CORBA (Common Object Request Broker Architecture) | 分布式对象通信协议，包括 IDL 映射和 ORB 运行时 |
| `java.transaction` | JTA (Java Transaction API) | `@Transactional` 和 `TransactionManager` 接口 |
| `java.se.ee` | 聚合模块 | 引用上述所有 Java EE 模块的聚合模块 |

### 3.2 同时移除的工具

JAX-WS 工具 `wsgen`、`wsimport`，JAXB 工具 `schemagen`、`xjc`，以及 CORBA 工具 `idlj`、`orbd`、`servertool`、`tnameserv` 均一并从 JDK 中移除。

---

## 4. 为什么移除

### 4.1 Java EE 移交 Eclipse 基金会

2017 年 Oracle 宣布将 Java EE 移交给 Eclipse 基金会，后者将其重新命名为 **Jakarta EE**。这意味着 JAX-WS、JAXB 等技术的规范制定和参考实现的维护主体不再是 JDK 项目，继续在 JDK 中捆绑这些技术既不合理也不可持续。Jakarta EE 社区以独立项目的形式持续维护和演进这些 API。

### 4.2 CORBA 已过时

CORBA 是 1990 年代的分布式对象技术，在现代 Java 开发中几乎没有新的使用场景。OMG 虽然仍在维护 CORBA 规范，但 Java 语言映射已长期没有更新，也没有第三方愿意接手维护其 Java 实现。

### 4.3 减轻 JDK 维护负担

移除这些模块使 JDK 团队能够集中精力在核心平台演进上，同时减少了 JDK 的体积和安全攻击面。需要这些技术的开发者可通过独立的 Maven 依赖获得更好的版本管理。

---

## 5. 迁移指南

### 5.1 JAXB 迁移

JAXB 是受影响面最广的技术。迁移到 Jakarta 版本只需添加 Maven 依赖：

```xml
<dependency>
    <groupId>jakarta.xml.bind</groupId>
    <artifactId>jakarta.xml.bind-api</artifactId>
    <version>4.0.0</version>
</dependency>
<dependency>
    <groupId>org.glassfish.jaxb</groupId>
    <artifactId>jaxb-runtime</artifactId>
    <version>4.0.3</version>
</dependency>
```

### 5.2 JAX-WS 迁移

```xml
<dependency>
    <groupId>jakarta.xml.ws</groupId>
    <artifactId>jakarta.xml.ws-api</artifactId>
    <version>4.0.0</version>
</dependency>
<dependency>
    <groupId>com.sun.xml.ws</groupId>
    <artifactId>jaxws-rt</artifactId>
    <version>4.0.1</version>
</dependency>
```

### 5.3 其他模块的替代依赖

| 原模块 | 替代 Maven 依赖 |
|--------|-----------------|
| `java.activation` (JAF) | `jakarta.activation:jakarta.activation-api` |
| `java.xml.ws.annotation` | `jakarta.annotation:jakarta.annotation-api` |
| `java.transaction` (JTA) | `jakarta.transaction:jakarta.transaction-api` |
| `java.corba` | 无官方替代，如仍需 CORBA 可使用 GlassFish CORBA ORB (`org.glassfish.corba:glassfish-corba-orb`) |

## 6. 对开发者的影响

### 6.1 企业应用的影响

大量企业级 Java 应用依赖 JAXB 进行 XML 序列化/反序列化，或使用 JAX-WS 调用 SOAP 服务。从 JDK 8 直接升级到 JDK 11+ 时，如果未添加相应的外部依赖，应用会在运行时抛出 `ClassNotFoundException` 或 `NoClassDefFoundError`。受影响场景包括：使用 `JAXBContext` 进行 XML 绑定、使用 `@WebService` 调用 SOAP 服务、依赖 `@PostConstruct`/`@PreDestroy` 注解的 Spring/CDI 组件，以及使用 `xjc` 从 XSD 生成 Java 类的构建脚本。

### 6.2 构建系统与包名变更

使用 `xjc`、`wsimport` 等工具的项目需切换到对应的 Maven/Gradle 插件（如 `org.glassfish.jaxb:jaxb-maven-plugin`、`com.sun.xml.ws:jaxws-maven-plugin`）。此外，Jakarta EE 9+ 将命名空间从 `javax.*` 迁移到 `jakarta.*`——使用 4.x 版本依赖时需将代码中的 `javax.xml.bind.*` 改为 `jakarta.xml.bind.*`；若希望保持 `javax` 包名兼容，可使用 2.x/3.x 版本的依赖。
