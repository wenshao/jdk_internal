## 1. 概述

JEP 269 在 `List`、`Set`、`Map` 接口上定义了静态工厂方法（`of()`），用于简洁地创建不可变集合实例，消除了长期以来创建小型不可变集合时的冗长写法：

```java
List<String> names = List.of("Alice", "Bob", "Charlie");
Set<Integer> primes = Set.of(2, 3, 5, 7, 11);
Map<String, Integer> scores = Map.of("Alice", 90, "Bob", 85);
```

---

## 2. 背景

### 2.1 冗长的集合创建方式

在 JDK 9 之前，创建不可变集合需要多行样板代码：

```java
List<String> list = Collections.unmodifiableList(Arrays.asList("Alice", "Bob"));

Map<String, Integer> map = new HashMap<>();
map.put("Alice", 90);
map.put("Bob", 85);
Map<String, Integer> unmodMap = Collections.unmodifiableMap(map);
```

代码冗长、容易遗漏 `unmodifiable` 包装、且 `Collections.unmodifiable*` 只是可变集合的只读视图。

### 2.2 第三方库与其他语言

Google Guava 早就提供了 `ImmutableList.of("a", "b", "c")` 等简洁 API，但返回的是 Guava 特有类型而非标准 `java.util` 接口，给 API 边界传递带来摩擦。大多数现代语言也都支持简洁的集合字面量（Python 的 `[1, 2, 3]`、Kotlin 的 `listOf(1, 2, 3)`）。JEP 269 的目标是将类似的便利性引入 JDK 标准库，直接返回标准接口类型。

## 3. API 设计

### 3.1 List.of()

`List.of()` 提供了从 0 到 10 个元素的固定参数重载（fixed-arity overloads），以及一个接受可变参数的 varargs 重载：

```java
static <E> List<E> of()                          // 空列表
static <E> List<E> of(E e1)                      // 1 个元素
static <E> List<E> of(E e1, E e2)                // 2 个元素
static <E> List<E> of(E e1, E e2, E e3)          // 3 个元素
// ... 依次类推到 10 个元素的固定参数重载
@SafeVarargs
static <E> List<E> of(E... elements)             // 任意数量（varargs）
```

**为什么同时提供固定参数重载和 varargs？** 固定参数重载避免了 varargs 调用中的数组分配开销。对于最常见的小集合场景（1-10 个元素），这意味着零额外分配。

### 3.2 Set.of()

`Set.of()` 的重载模式与 `List.of()` 完全一致（0-10 个元素的固定参数重载 + varargs）。**关键区别**：传入重复元素会抛出 `IllegalArgumentException`：

```java
Set<String> ok = Set.of("a", "b", "c");       // 正常
Set<String> bad = Set.of("a", "b", "a");       // IllegalArgumentException!
```

这是有意为之的设计——调用者在写 `Set.of("a", "a")` 时几乎一定是编程错误，静默去重反而会掩盖 bug。

### 3.3 Map.of() 与 Map.ofEntries()

`Map.of()` 接受交替排列的键值对参数，提供 0 到 10 个键值对的重载（最多 20 个参数）：

```java
static <K, V> Map<K, V> of()                         // 空 Map
static <K, V> Map<K, V> of(K k1, V v1)               // 1 个条目
static <K, V> Map<K, V> of(K k1, V v1, K k2, V v2)   // 2 个条目
// ... 到 10 个条目
```

对于超过 10 个条目的 Map，使用 `Map.ofEntries()` 配合 `Map.entry()`：

```java
import static java.util.Map.entry;
Map<String, Integer> map = Map.ofEntries(
    entry("Alice", 90), entry("Bob", 85), entry("Charlie", 78));
```

`Map.of()` 不提供 varargs 重载，因为交替的 K/V 参数无法用单一类型的 varargs 表达。`Map.ofEntries()` 通过 `Map.Entry` 对象解决此问题。与 `Set.of()` 类似，重复键会抛出 `IllegalArgumentException`。

## 4. 集合属性

通过工厂方法创建的集合实例具有以下特征：

| 属性 | 说明 |
|------|------|
| **不可变 (Immutable)** | 调用 `add()`、`set()`、`put()` 等修改方法会抛出 `UnsupportedOperationException` |
| **禁止 null** | 元素（或键/值）不能为 `null`，否则抛出 `NullPointerException` |
| **可序列化** | 实现 `Serializable`，反序列化后仍保持不可变性 |
| **基于值 (Value-based)** | 不应依赖实例的 identity（不应使用 `==` 比较，不应同步） |
| **迭代顺序** | `List.of()` 保持插入顺序；`Set.of()` 和 `Map.of()` 的迭代顺序不保证 |

JDK 文档区分 "unmodifiable"（只读视图，底层可变）和 "immutable"（真正不可变）。`List.of()` 等返回真正的 immutable 集合。禁止 `null` 是因为不可变集合中 `null` 几乎无合理用途，且导致 `Map.get()` 返回值歧义。

## 5. 实现细节

### 5.1 紧凑的内部实现类

JDK 内部使用了一组专门优化的实现类（位于 `java.util` 包内部）：

| 内部类 | 用途 | 存储方式 |
|--------|------|----------|
| `List12` | 1-2 个元素的 List | 直接字段（`e0`, `e0`+`e1`） |
| `ListN` | 3+ 个元素的 List | `Object[]` 数组 |
| `Set12` | 1-2 个元素的 Set | 直接字段 |
| `SetN` | 3+ 个元素的 Set | 基于探测的哈希表 |
| `Map1` | 1 个条目的 Map | 直接字段（`k0`, `v0`） |
| `MapN` | 2+ 个条目的 Map | 基于探测的哈希表 |

### 5.2 基于字段的小尺寸优化

对于 1-2 个元素的场景，`List12`、`Set12` 直接使用实例字段（`e0`、`e1`）而非数组存储元素。这消除了数组对象头和长度字段的开销（每个实例节省约 16-20 字节），同时允许 JVM 进行更激进的内联和逃逸分析优化。

### 5.3 Set 和 Map 的哈希策略

`SetN` 和 `MapN` 使用线性探测的开放地址哈希表（而非链表桶），大小在构造时一次性确定。为抵御哈希碰撞攻击，存储位置会用随机 salt 值混淆。

## 6. 与已有方式的对比

| 方式 | 可变性 | null | 代码量 | 依赖 | 返回类型 |
|------|--------|------|--------|------|----------|
| `Arrays.asList(...)` | 固定大小（可 set） | 是 | 1 行 | 无 | `List` |
| `Collections.unmodifiableList(...)` | 不可修改视图 | 是 | 2+ 行 | 无 | `List` |
| Guava `ImmutableList.of(...)` | 不可变 | 否 | 1 行 | Guava | `ImmutableList` |
| **`List.of(...)` (JDK 9+)** | **不可变** | **否** | **1 行** | **无** | **`List`** |

核心区别：`Arrays.asList()` 是底层数组的视图；`Collections.unmodifiableList()` 是可变列表的只读包装，原始引用仍可修改内容；`List.of()` 返回真正不可变的集合，无底层数据泄漏。
