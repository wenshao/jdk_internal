## 1. 概述

Hidden Classes 是不可通过名称发现的类，专为框架在运行时动态生成类而设计。Hidden class 无法通过 `Class.forName()` 查找，生命周期由使用方控制，可独立于类加载器被卸载。JEP 371 提供了标准 API `Lookup.defineHiddenClass()` 来替代非标准的 `sun.misc.Unsafe.defineAnonymousClass()`。

---

## 2. 背景

### 2.1 运行时生成类的需求

许多 JVM 框架需要在运行时动态生成类——`LambdaMetafactory` 生成 functional interface 实现类、`Proxy` 生成代理类、ASM/ByteBuddy/CGLIB 生成辅助类、Nashorn 将脚本编译为类。这些类是纯粹的实现细节，不应对外暴露，且数量可能非常大。

### 2.2 Unsafe.defineAnonymousClass 的问题

在 JDK 15 之前，框架通常使用 `sun.misc.Unsafe.defineAnonymousClass()` 来定义运行时类。该方法存在多项问题：

| 问题 | 说明 |
|------|------|
| **非标准 API** | 位于 `sun.misc` 包中，随时可能移除 |
| **安全隐患** | `Unsafe` 绕过所有访问检查 |
| **模块化冲突** | JPMS 下需 `--add-opens`，破坏强封装 |
| **语义不清** | "host class" 与 nest/access control 交互规则不明确 |

## 3. API 设计

### 3.1 Lookup.defineHiddenClass()

核心 API 位于 `java.lang.invoke.MethodHandles.Lookup` 类中。该方法接收类字节码、是否立即初始化的标志和可选的 `ClassOption`，使用调用者的 `Lookup` 上下文确定访问权限和所属包：

```java
public Lookup defineHiddenClass(byte[] bytes, boolean initialize, ClassOption... options)
    throws IllegalAccessException
```

### 3.2 ClassOption

`ClassOption` 枚举定义了 hidden class 的附加属性：

| 选项 | 说明 |
|------|------|
| `NESTMATE` | 将 hidden class 添加为 lookup class 的 nest member，允许访问 nest 中其他类的 private 成员 |
| `STRONG` | 使 hidden class 与其类加载器之间建立强引用关系，阻止类在类加载器存活期间被卸载 |

默认不指定选项时，hidden class 独立于任何 nest，与类加载器之间为弱引用关系。`NESTMATE` 对 lambda 至关重要——lambda body 被编译为宿主类的私有方法，hidden class 需要通过 nest 访问权限调用该方法。`STRONG` 适用于需要长期存在的类（如动态代理），防止过早被卸载。

### 3.3 使用示例

```java
byte[] classBytes = generateClassBytes();  // ASM 等工具生成字节码
Lookup hiddenLookup = MethodHandles.lookup()
    .defineHiddenClass(classBytes, true, ClassOption.NESTMATE);
Class<?> hc = hiddenLookup.lookupClass();
// hc.getName() => "com.example.Generated/0x0000000800c04400"
```

## 4. Hidden Classes 的属性

| 属性 | 普通类 | Hidden Class |
|------|--------|-------------|
| **名称发现** | `Class.forName()` 可找到 | 无法通过名称发现 |
| **Binary name** | 合法 binary name | 含 `/` 字符，非合法名称 |
| **字节码引用** | 可被 `CONSTANT_Class_info` 引用 | 不能被字节码直接引用 |
| **类加载器** | 强引用，由类加载器管理 | 默认弱引用，可独立卸载 |
| **Nest 归属** | 编译时确定 | 可通过 `NESTMATE` 动态加入 |
| **序列化** | 可序列化 | 不可序列化 |

**不可发现性**是核心特性——名称含 `/`（如 `com.example.Foo/0x0000000800c04400`），任何通过名称加载的操作都会失败。**独立卸载**意味着当所有实例被回收后，GC 可卸载该类并释放元数据空间，即使定义类加载器仍存活。

## 5. 使用场景

### 5.1 Lambda 表达式

Lambda 是 hidden classes 最重要的应用场景。`LambdaMetafactory` 在 JDK 15 中已从 `Unsafe.defineAnonymousClass()` 迁移到 `Lookup.defineHiddenClass()`，使用 `NESTMATE` 选项访问宿主类的 private desugared lambda 方法：

```java
// LambdaMetafactory 内部（简化），JDK 15+
byte[] classBytes = generateLambdaClass(interfaceClass, implMethod);
Lookup hiddenLookup = caller.defineHiddenClass(classBytes, true, ClassOption.NESTMATE);
```

### 5.2 动态代理

`java.lang.reflect.Proxy` 生成的代理类可以作为 hidden class 定义，利用独立卸载能力降低内存压力。

### 5.3 字节码框架

主流字节码框架都受益于 hidden classes：

| 框架 | 迁移状态 |
|------|---------|
| **ASM** | JDK 内部（`LambdaMetafactory`）已迁移 |
| **ByteBuddy** | 已支持 `defineHiddenClass()` |
| **CGLIB** | 通过 ByteBuddy 间接支持 |

## 6. 与 Unsafe.defineAnonymousClass 的对比

| 维度 | `Unsafe.defineAnonymousClass()` | `Lookup.defineHiddenClass()` |
|------|-------------------------------|------------------------------|
| **API 层级** | 非标准内部 API（`sun.misc`） | 标准 API（`java.lang.invoke`） |
| **访问控制** | 无，绕过所有检查 | 基于 `Lookup` 上下文，遵循模块/包访问规则 |
| **Nest 支持** | "host class" 概念，语义模糊 | 显式 `NESTMATE` 选项，基于 JEP 181 |
| **卸载控制** | 总是弱引用，不可配置 | 默认弱引用，可通过 `STRONG` 配置 |
| **常量池修补** | 支持 `cpPatches` | 不支持，需改用 `condy`（JEP 309） |
| **模块兼容** | 需 `--add-opens` | 完全兼容 JPMS |
| **维护状态** | `@Deprecated(forRemoval=true)` | 长期维护的标准 API |

---

## 7. 安全性与强封装

Hidden classes 带来了三方面安全改进：

- **访问控制**：`defineHiddenClass()` 遵循 `Lookup` 的访问权限，只有 `PRIVATE` 级别的 `Lookup` 才能定义 hidden class，且包和模块归属由 lookup class 决定，框架不能将 hidden class 注入任意包或模块
- **强封装**：作为标准 API，不需要 `--add-opens` 暴露内部模块，完全兼容 JPMS
- **不可伪造性**：hidden class 没有合法 binary name，无法通过 `Class.forName()` 或 `CONSTANT_Class_info` 获取引用，唯一途径是 `defineHiddenClass()` 返回的 `Lookup`

```java
Class<?> hc = hiddenLookup.lookupClass();
Class.forName(hc.getName());  // 抛出 ClassNotFoundException
```
