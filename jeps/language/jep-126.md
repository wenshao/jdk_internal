## 1. 概述

Lambda Expressions 为 Java 引入了轻量级的匿名函数语法，使得将行为作为参数传递成为可能；Virtual Extension Methods（即 default methods）允许在接口中添加带有默认实现的方法，从而在不破坏已有实现类的前提下演进接口。

---

## 2. 背景

### 2.1 Java 的困境

在 JDK 8 之前，Java 是少数主流语言中缺乏 lambda/closure 支持的语言之一。将一段行为传递给方法（如排序比较器、事件回调、线程任务）需要使用 anonymous inner class：

```java
// JDK 7 及之前：匿名内部类
Collections.sort(list, new Comparator<String>() {
    @Override
    public int compare(String a, String b) {
        return a.length() - b.length();
    }
});

// JDK 8+：Lambda 表达式
Collections.sort(list, (a, b) -> a.length() - b.length());
```

### 2.2 匿名内部类的问题

| 问题 | 说明 |
|------|------|
| **冗长** | 大量样板代码（class 声明、方法签名、return 语句） |
| **性能** | 每个匿名类生成独立的 `.class` 文件，增加加载和元数据开销 |
| **语义混乱** | `this` 指向内部类实例而非外围类，容易出错 |
| **可读性** | 嵌套层级深时难以阅读，"vertical problem" |

### 2.3 Project Lambda

JEP 126 是 Project Lambda 的核心成果。该项目从 2009 年开始（由 Mark Reinhold 发起），经历了多轮语法设计讨论（BGGA proposal、CICE proposal 等），最终由 Brian Goetz 主导确定了当前语法。

## 3. 关键设计决策

### 3.1 语法设计

Lambda 表达式采用 `(parameters) -> expression/body` 语法：

```java
// 无参数
Runnable r = () -> System.out.println("Hello");

// 单参数（可省略括号）
Consumer<String> c = s -> System.out.println(s);

// 多参数
BinaryOperator<Integer> add = (a, b) -> a + b;

// 带类型声明
Comparator<String> cmp = (String a, String b) -> a.compareTo(b);

// 多行语句块
Function<String, Integer> parser = s -> {
    int result = Integer.parseInt(s);
    return result * 2;
};
```

**设计权衡**：Java 团队选择了 `->` 而非 `=>` (C#) 或 `fun` 关键字 (Kotlin)，以避免与已有语法冲突。参数类型在大多数情况下可以通过 target type inference 推断。

### 3.2 Functional Interface

Lambda 表达式的目标类型必须是 functional interface——仅包含一个抽象方法的接口：

```java
@FunctionalInterface
public interface Predicate<T> {
    boolean test(T t);
}

// @FunctionalInterface 注解是可选的，但推荐使用
// 它让编译器验证接口确实只有一个抽象方法
@FunctionalInterface
public interface Converter<F, T> {
    T convert(F from);
    // 如果添加第二个抽象方法，编译器会报错
}
```

**关键决策**：Java 没有引入新的 function type（如 `(int) -> String`），而是复用已有的接口机制。这保证了与现有 API 的向后兼容——任何接受单方法接口的旧 API 都自动支持 lambda。

### 3.3 Method Reference

Method reference 是 lambda 的简写形式，使用 `::` 操作符：

```java
// 静态方法引用
Function<String, Integer> parser = Integer::parseInt;

// 实例方法引用（绑定实例）
String str = "Hello";
Supplier<Integer> len = str::length;

// 实例方法引用（未绑定）
Function<String, Integer> f = String::length;

// 构造函数引用
Supplier<ArrayList<String>> factory = ArrayList::new;
```

| 类型 | 语法 | Lambda 等价形式 |
|------|------|----------------|
| 静态方法 | `ClassName::staticMethod` | `x -> ClassName.staticMethod(x)` |
| 绑定实例 | `instance::method` | `() -> instance.method()` |
| 未绑定实例 | `ClassName::method` | `x -> x.method()` |
| 构造函数 | `ClassName::new` | `() -> new ClassName()` |

### 3.4 变量捕获

Lambda 可以捕获外围作用域的变量，但被捕获的变量必须是 **effectively final**（事实上不可变）：

```java
String prefix = "Hello";  // effectively final
Function<String, String> greeter = name -> prefix + " " + name;

// ❌ 编译错误：被捕获的变量不能被修改
int count = 0;
Runnable r = () -> count++;  // 编译错误
```

**设计理由**：
- 避免并发环境下的 race condition
- 与 anonymous inner class 的 `final` 要求一致（JDK 8 放松为 effectively final）
- 简化实现，不需要像 closure 那样维护可变状态的 heap allocation

### 3.5 类型推断

Lambda 表达式依赖 target typing（目标类型推断）确定其类型：

```java
// 同一个 lambda 可以匹配不同的 functional interface
Predicate<String> p = s -> s.isEmpty();
Function<String, Boolean> f = s -> s.isEmpty();

// 编译器根据上下文推断参数类型
Comparator<String> cmp = (a, b) -> a.compareTo(b);
// 编译器知道 a, b 都是 String

// 泛型方法中的推断
List<String> names = Arrays.asList("Alice", "Bob", "Charlie");
names.sort(Comparator.comparing(String::length));
```

JDK 8 显著增强了 Java 的类型推断能力（JLS 第 18 章），支持 poly expressions 和 target typing。

## 4. 实现细节

### 4.1 invokedynamic 指令

Lambda 表达式在字节码层面使用 `invokedynamic` 指令（JSR 292，JDK 7 引入），而非生成匿名内部类：

```
// 字节码示例（简化）
invokedynamic #0:test()Ljava/util/function/Predicate;
  BootstrapMethod: LambdaMetafactory.metafactory(...)
```

首次执行时，JVM 调用 bootstrap method 生成实现类；后续调用直接使用已链接的 CallSite，无需重复创建。

### 4.2 LambdaMetafactory

`java.lang.invoke.LambdaMetafactory` 是 lambda 翻译的核心 bootstrap method：

```java
// LambdaMetafactory.metafactory() 签名（简化）
public static CallSite metafactory(
    MethodHandles.Lookup caller,       // 调用方的查找上下文
    String interfaceMethodName,        // 目标接口方法名（如 "test"）
    MethodType factoryType,            // 工厂方法类型
    MethodType interfaceMethodType,    // 接口方法的擦除类型
    MethodHandle implementation,       // 实际实现方法
    MethodType dynamicMethodType       // 运行时方法类型
)
```

运行时生成策略由 JVM 实现决定，HotSpot 默认使用 `InnerClassLambdaMetafactory` 在运行时通过 ASM 生成轻量级类。

### 4.3 为什么不用内部类

Java 团队明确拒绝了将 lambda 直接编译为匿名内部类的方案：

| 维度 | 内部类方案 | invokedynamic 方案 |
|------|-----------|-------------------|
| **类文件数量** | 每个 lambda 生成一个 `.class` | 无额外类文件 |
| **类加载开销** | 每个 lambda 占用 ClassLoader 和元数据空间 | 运行时按需生成，可优化 |
| **未来优化空间** | 固化在字节码中，无法优化 | JVM 可自由选择最优策略 |
| **启动性能** | 慢（加载大量小类） | 快（延迟生成） |
| **序列化** | 天然支持 | 需特殊处理 |

**核心原则**："不要将翻译策略固化在字节码中"（Brian Goetz，Translation of Lambda Expressions）。使用 `invokedynamic` 给予 JVM 最大的优化自由度。

### 4.4 翻译策略

编译器将 lambda body 提取为宿主类的私有方法（desugared method），然后通过 `invokedynamic` 在运行时将其与 functional interface 绑定：

```java
// 源码
public class Example {
    void run() {
        Predicate<String> p = s -> s.isEmpty();
    }
}

// 编译器生成（简化）
public class Example {
    // lambda body 被提取为私有静态方法
    private static boolean lambda$run$0(String s) {
        return s.isEmpty();
    }

    void run() {
        // invokedynamic 调用 LambdaMetafactory
        // 将 lambda$run$0 包装为 Predicate 实例
        Predicate<String> p = /* invokedynamic */;
    }
}
```

捕获变量的 lambda 会生成实例方法而非静态方法，捕获的值作为参数传入。

## 5. Virtual Extension Methods (Default Methods)

JEP 126 的另一部分是 virtual extension methods，即接口中的 `default` 方法。这是为了在不破坏已有实现的情况下向接口添加新方法——特别是为了支持 Stream API 向 `Collection` 接口添加 `stream()` 等方法：

```java
public interface Iterable<T> {
    Iterator<T> iterator();

    // JDK 8 新增的 default method
    default void forEach(Consumer<? super T> action) {
        for (T t : this) {
            action.accept(t);
        }
    }
}

public interface Collection<E> extends Iterable<E> {
    // JDK 8 新增
    default Stream<E> stream() {
        return StreamSupport.stream(spliterator(), false);
    }
}
```

**设计要点**：
- Default methods 允许接口演进而不破坏二进制兼容性
- 如果一个类实现的多个接口有冲突的 default methods，编译器会要求显式 override
- 接口中也可以包含 `static` 方法（JDK 8 新增）

## 6. 影响

### 6.1 Stream API

Lambda 的直接受益者是 [JEP 107: Bulk Data Operations for Collections](jep-107.md)，即 Stream API：

```java
// 没有 lambda 的话，Stream API 几乎不可用
List<String> result = names.stream()
    .filter(s -> s.length() > 3)       // Predicate<String>
    .map(String::toUpperCase)           // Function<String, String>
    .sorted(Comparator.naturalOrder())  // Comparator<String>
    .collect(Collectors.toList());      // Collector
```

Stream API 改变了 Java 处理集合数据的方式，从命令式循环转向声明式管道操作。

### 6.2 函数式编程范式

Lambda 使 Java 能够采用函数式编程的核心模式：

| 模式 | JDK 8 实现 | 示例 |
|------|-----------|------|
| **高阶函数** | 接受/返回 lambda 的方法 | `list.sort(comparing(...))` |
| **惰性求值** | Stream 的惰性管道 | `stream.filter().map()` |
| **组合** | `Function.compose()`, `Predicate.and()` | `f.andThen(g)` |
| **Optional** | `Optional.map()`, `flatMap()` | `opt.map(String::length)` |

### 6.3 java.util.function 包

JDK 8 新增了 `java.util.function` 包，提供 43 个标准 functional interface：

| 接口 | 方法 | 用途 |
|------|------|------|
| `Function<T,R>` | `R apply(T t)` | 转换 |
| `Predicate<T>` | `boolean test(T t)` | 过滤 |
| `Consumer<T>` | `void accept(T t)` | 消费 |
| `Supplier<T>` | `T get()` | 生成 |
| `BiFunction<T,U,R>` | `R apply(T t, U u)` | 双参数转换 |
| `UnaryOperator<T>` | `T apply(T t)` | 一元操作 |
| `BinaryOperator<T>` | `T apply(T t1, T t2)` | 二元操作 |

此外还有针对基本类型的特化版本（`IntFunction`、`LongPredicate` 等），避免 autoboxing 开销。
