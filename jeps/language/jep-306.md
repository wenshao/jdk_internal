## 1. 概述

JEP 306 将所有浮点运算恢复为始终严格（strict）语义，使 `strictfp` 修饰符不再具有实际效果。在 JDK 17 之前，Java 浮点运算默认使用"默认"（default）语义，允许中间结果使用扩展精度；`strictfp` 则强制要求严格遵守 IEEE 754 标准。JEP 306 消除了这一区别——所有浮点表达式的计算结果都必须严格符合 IEEE 754，无论是否标注 `strictfp`。

---

## 2. 背景

### 2.1 JDK 1.2 引入 strictfp 的历史原因

在 Java 最初发布时（JDK 1.0），所有浮点运算都是严格的，要求中间结果精确舍入到 `float`（32 位）或 `double`（64 位）对应的 IEEE 754 格式。然而 x87 浮点处理单元（FPU）使用 80 位扩展精度寄存器，在该硬件上实现严格语义需要额外的指令来截断中间结果，带来显著的性能损耗。

为了解决这一性能问题，JDK 1.2（1998 年）将浮点语义分为两种模式：

| 模式 | 说明 |
|------|------|
| **默认模式（default）** | 允许中间结果使用扩展精度（x87 的 80 位），最终结果再舍入到 `float`/`double` |
| **严格模式（strict）** | 要求所有中间结果都精确舍入到 IEEE 754 格式，通过 `strictfp` 关键字启用 |

### 2.2 SSE2 使严格模式免费

2001 年 Intel Pentium 4 引入了 SSE2 指令集，使用 128 位 XMM 寄存器原生支持 IEEE 754 的 32/64 位浮点运算，不存在扩展精度问题。在 SSE2 硬件上，严格模式与默认模式产生完全相同的结果，且没有任何性能差异。自 2001 年以来，所有 x86/x64 处理器均支持 SSE2，ARM/AArch64/RISC-V 等架构也原生使用标准精度。

## 3. 变更内容

### 3.1 语义统一

JDK 17 起，Java 语言规范不再区分"默认"和"严格"两种浮点模式。所有浮点运算统一使用严格语义：

| 方面 | JDK 16 及之前 | JDK 17 及之后 |
|------|-------------|-------------|
| **默认模式** | 允许扩展精度 | 不再存在，统一为严格 |
| **严格模式** | 需 `strictfp` 显式启用 | 始终启用 |
| **`strictfp` 关键字** | 有实际语义效果 | 保留但无效果 |

### 3.2 strictfp 修饰符的处理

`strictfp` 作为保留关键字仍然被接受，但编译器会发出警告提示该修饰符已无实际意义。在字节码层面，`ACC_STRICT` 标志（`0x0800`）不再由 `javac` 生成，JVM 在加载类时忽略该标志——旧版本编译的 class 文件包含 `ACC_STRICT` 也不会产生任何差异。

```java
// JDK 17+ 编译器警告示例
strictfp class Example {  // warning: [strictfp] as of release 17, all floating-point
                          // expressions are evaluated strictly and 'strictfp' is not required
}
```

## 4. 为什么是现在

### 4.1 硬件已全面支持

所有现代处理器都原生支持 IEEE 754 浮点运算，不再需要扩展精度的"逃生通道"：x86/x64 自 2001 年 Pentium 4 起支持 SSE2，ARM/AArch64 的 VFP/NEON 使用标准精度，RISC-V 的 F/D 扩展同样如此。

### 4.2 JVM 早已默认使用 SSE2

HotSpot JVM 在 x86 平台上早已默认使用 SSE2 指令而非 x87 指令。JDK 17 要求的最低硬件远超 SSE2 时代，区分两种模式的意义在实践中早已不存在，JEP 306 只是让语言规范与现实对齐。

### 4.3 减少认知负担

`strictfp` 是 Java 中最令人困惑的关键字之一。大多数开发者不理解其含义，也不知道何时应该使用它。消除这种区别简化了语言规范，降低了学习成本。

---

## 5. 对开发者的影响

### 5.1 行为变化

对于运行在现代硬件上的代码，JEP 306 **不会导致任何行为变化**。所有主流 JVM 实现已经使用 SSE2 或等效指令集，浮点运算的实际结果在 JDK 17 和 JDK 17 之间完全一致。

### 5.2 源码迁移与二进制兼容性

现有使用 `strictfp` 的代码无需修改——编译器仍然接受该关键字，仅产生警告。开发者可选择移除 `strictfp` 以消除警告。旧版本编译的 class 文件在 JDK 17+ 上完全兼容，`ACC_STRICT` 标志被忽略。

## 6. 技术细节

### 6.1 IEEE 754 与扩展精度

IEEE 754 定义了 `float`（binary32，23 位尾数）和 `double`（binary64，52 位尾数）。x87 FPU 使用 80 位扩展精度（64 位尾数），中间结果保留更多有效位数，导致相同代码在不同平台或优化级别下产生不同结果。SSE2 消除了这一问题。

### 6.2 JLS 变更

Java 语言规范的以下部分被简化：**JLS 15.4** 移除了 FP-strict 与非 FP-strict 表达式的区别；**JLS 8.1.1.3, 8.4.3.5, 9.1.1.2** 中 `strictfp` 标注为不再具有语义效果；**JLS 4.2.3** 浮点类型的值集不再区分 default 和 strict 模式。
