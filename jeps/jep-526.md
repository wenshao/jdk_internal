## 1. 背景与动机

### 问题

Java 的 `final` 字段在类加载时初始化：

```java
public class Config {
    // 类加载时立即初始化
    private static final ExpensiveObject CACHE = new ExpensiveObject();
    
    // 可能不需要，但已初始化
}
```

问题：
- 启动时间增加
- 内存浪费
- 初始化顺序问题

### 解决方案

JEP 526 引入延迟常量：
- 首次访问时初始化
- 线程安全
- JVM 优化支持

---

## 2. 使用示例

### 延迟常量声明

```java
public class Config {
    // 延迟初始化的常量
    private static lazy ExpensiveObject CACHE = new ExpensiveObject();
    
    // 首次访问时初始化
    public static ExpensiveObject getCache() {
        return CACHE;  // 此时才初始化
    }
}
```

### 与 final 对比

```java
// final: 类加载时初始化
private static final Logger LOGGER = LoggerFactory.getLogger();

// lazy: 首次访问时初始化
private static lazy Logger LOGGER = LoggerFactory.getLogger();
```

### 使用场景

```java
public class Services {
    // 可能不需要的服务
    private static lazy ExpensiveService service = new ExpensiveService();
    
    public static ExpensiveService getService() {
        return service;  // 仅在调用时初始化
    }
}
```

## 3. 实现分析

### 语义

```java
// 源码
private static lazy Object VALUE = computeValue();

// 编译后等价于
private static volatile Object VALUE;
private static final Object $VALUE_LOCK = new Object();

public static Object getValue() {
    if (VALUE == null) {
        synchronized ($VALUE_LOCK) {
            if (VALUE == null) {
                VALUE = computeValue();
            }
        }
    }
    return VALUE;
}
```

### JVM 优化

```java
// JVM 可以优化延迟常量访问
// 一旦初始化，访问无同步开销

// HotSpot 优化
// - 内联访问
// - 消除 null 检查
// - 常量折叠
```

## 4. 开发者影响

### 与其他方案对比

| 方案 | 初始化时机 | 线程安全 | 性能 |
|------|-----------|----------|------|
| final | 类加载 | ✅ | 最快 |
| lazy | 首次访问 | ✅ | 快 |
| 手动双重检查 | 首次访问 | 需小心 | 中 |
| Supplier | 每次调用 | 取决于实现 | 慢 |

### 最佳实践

```java
// ✅ 推荐：昂贵资源延迟初始化
private static lazy DatabaseConnection DB = createConnection();

// ✅ 推荐：可能不需要的资源
private static lazy DebugLogger DEBUG = new DebugLogger();

// ⚠️ 注意：循环依赖
// 延迟常量可以帮助解决循环依赖问题

// ❌ 避免：简单值
private static lazy int MAX_SIZE = 100;  // 不必要
```
