# G1 GC: 通过减少同步提高吞吐量

> JEP 522 实现分析 | Ivan Walulya & Thomas Schatzl | JDK 26

## 概述

本文档分析 **G1 GC: Improve Throughput by Reducing Synchronization（G1 GC：通过减少同步提高吞吐量）** 的技术细节，这是 JEP 522 的核心内容。通过引入双卡表（Dual Card Table）机制，减少应用线程与 GC 优化线程之间的同步开销，从而显著提升 G1 收集器的吞吐量。

## 背景与问题

### G1 卡表机制

G1 使用**卡表（Card Table）**来追踪跨 Region 的对象引用。每当应用线程修改对象的引用字段时，JVM 通过注入的**写屏障（Write Barrier）**代码来更新卡表。后台的优化器线程（Optimizer Threads）需要并发地扫描和处理这张卡表，以维护 G1 的 Remembered Set 数据结构。

```
传统 G1 写屏障流程:

应用线程                              GC 优化器线程
   │                                      │
   │  obj.field = newRef;                  │
   │       │                               │
   │       ▼                               │
   │  ┌──────────────────┐                 │
   │  │ 写屏障代码 (~50条  │                 │
   │  │ x64 指令)        │                 │
   │  │ 1. 过滤自引用     │                 │
   │  │ 2. 检查卡表状态   │   ◄── 同步竞争 ──►│ 扫描卡表
   │  │ 3. 更新卡表       │                 │ 处理脏卡
   │  │ 4. 入队脏卡       │                 │ 更新 RSet
   │  └──────────────────┘                 │
   │                                      │
   ▼                                      ▼

问题:
1. 写屏障代码量大 (~50 条 x64 指令)，阻碍编译器优化 (循环展开、内联)
2. 应用线程与优化器线程共享同一张卡表，产生同步竞争
3. G1 写屏障可导致 10-20% 的吞吐量下降
```

### 同步开销的根源

在传统实现中，应用线程和 GC 优化器线程需要同时访问同一张卡表，带来以下问题：

- **缓存行竞争**：多个线程频繁更新同一内存区域，导致 CPU 缓存失效
- **屏障指令复杂**：写屏障需要检查卡表状态、进行条件过滤和队列入队等操作
- **编译器优化受阻**：庞大的写屏障代码（约 50 条 x64 指令）使 JIT 编译器难以执行循环展开和方法内联等优化

---

## 解决方案：双卡表机制

### 核心设计

JEP 522 引入第二张卡表，使应用线程与优化器线程实现无锁化（lock-free）操作。

```
双卡表机制:

              卡表 A (活跃表)              卡表 B (处理表)
              ┌──────────────┐            ┌──────────────┐
              │ □ ■ □ □ ■ □  │            │ □ □ □ □ □ □  │
              │ □ □ ■ □ □ ■  │            │ □ □ □ □ □ □  │
              │ ■ □ □ ■ □ □  │            │ □ □ □ □ □ □  │
              └──────────────┘            └──────────────┘
                    ▲                           ▲
                    │                           │
              应用线程更新                  优化器线程处理
              (无需同步)                   (无需同步)


GC 暂停时交换:

阶段 1: 正常运行
  应用线程 ──► 卡表 A (写入脏卡)
  优化器   ──► 卡表 B (处理脏卡)

阶段 2: GC 暂停检测到卡表 A 扫描可能超出暂停时间目标
  ┌────────────────────────────────────┐
  │  原子交换 (Atomic Swap)             │
  │  卡表 A ◄──────────────► 卡表 B    │
  └────────────────────────────────────┘

阶段 3: 交换后
  应用线程 ──► 卡表 B (空表, 继续写入)
  优化器   ──► 卡表 A (满表, 独立处理)
```

### 工作流程

1. **应用线程**更新活跃卡表（第一张），无需任何同步操作，写屏障逻辑大幅简化
2. **优化器线程**独立处理第二张（最初为空的）卡表，无需与应用线程协调
3. 当 G1 在 GC 暂停期间检测到扫描活跃卡表可能超出暂停时间目标时，**原子交换**两张卡表
4. 交换后，应用线程在原来的第二张（现已清空的）卡表上继续更新，优化器线程处理原来的第一张（已满的）卡表
5. 交换机制利用**通用线程局部握手（Generic Thread-Local Handshakes, JEP 312）**实现，避免依赖操作系统特定的同步原语

### 写屏障优化

由于无需同步，写屏障代码量大幅缩减：

```
写屏障指令数对比 (x64 架构):

传统 G1 写屏障:    ~50 条指令
JEP 522 写屏障:    ~12 条指令
                    ────────────
减少幅度:           ~76%

优化后的写屏障 (伪代码):
┌─────────────────────────────┐
│ 1. 计算卡表偏移              │
│ 2. 标记卡为脏               │
│ (无需检查当前状态)            │
│ (无需条件入队)               │
│ (无需同步操作)               │
└─────────────────────────────┘
```

写屏障的精简带来额外的编译器优化机会：
- **循环展开**：更短的屏障代码使 JIT 更愿意展开包含引用写操作的循环
- **方法内联**：更小的代码体积降低了内联预算压力
- **指令缓存友好**：更少的指令意味着更高的 I-Cache 命中率

## 性能影响

### 吞吐量提升

```
引用修改密集型工作负载 (如大量对象图修改、频繁引用更新):

                    传统 G1        JEP 522 G1       变化
────────────────────────────────────────────────────────
吞吐量              基准值          +5% ~ +15%       显著提升
写屏障开销          10-20%         3-5%             大幅降低
────────────────────────────────────────────────────────

提升来源:
- 写屏障指令减少 ~76%
- 消除应用线程与优化器线程之间的同步竞争
- JIT 编译器获得更多优化机会


轻量级引用修改工作负载 (如计算密集型、少量引用操作):

                    传统 G1        JEP 522 G1       变化
────────────────────────────────────────────────────────
吞吐量              基准值          ~ +5%            适度提升
────────────────────────────────────────────────────────

提升来源:
- 写屏障代码缩短提高 I-Cache 效率
- 即使引用修改较少，更简洁的屏障仍有正面影响
```

### GC 暂停时间

GC 暂停时间略有改善。新的卡表结构相比此前的辅助数据结构更加高效，扫描和处理速度更快。

### 目标与非目标

**目标：**
- 减少 G1 的同步开销
- 减少写屏障注入代码大小
- 保持 G1 现有架构不变，对用户透明

**非目标：**
- 并非追求与其他收集器（如 Parallel GC、ZGC）的吞吐量持平

## 内存开销

### 卡表内存消耗

```
每张卡表占用 Java 堆容量的 ~0.2%:

堆大小          单张卡表大小      双卡表总增量
──────────────────────────────────────────
1 GB           ~2 MB            +2 MB
4 GB           ~8 MB            +8 MB
8 GB           ~16 MB           +16 MB
32 GB          ~64 MB           +64 MB
──────────────────────────────────────────

计算: ~2 MB 原生内存 / 1 GB 堆
```

### 内存开销评估

- 第二张卡表需要额外 ~0.2% 堆容量的原生内存
- 但它**替换了此前的辅助数据结构**，因此在某些场景下实际内存使用反而更少
- JDK 20 和 JDK 21 已经移除了更大的数据结构，使这一开销在历史上下文中是可接受的
- 对于绝大多数生产环境（堆 4-32 GB），额外的 8-64 MB 原生内存开销可以忽略不计

### JVM 参数

JEP 522 的优化在 JDK 26 中使用 G1 收集器时默认启用，无需额外配置。G1 仍然是 JDK 的默认 GC 收集器。

```bash
# G1 为默认收集器，JEP 522 优化自动生效
java -Xms4g -Xmx4g -jar app.jar

# 显式指定 G1 (与默认行为相同)
java -XX:+UseG1GC -Xms4g -Xmx4g -jar app.jar
```
