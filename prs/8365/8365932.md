## Summary

实现 JEP 516: Ahead-of-Time Object Caching with Any GC。当前 AOT 缓存使用 mmap 将文件中的字节直接放置在 GC 管理的堆中来缓存堆对象，这要求所有 GC 具有逐位相同的对象和引用格式。这意味着使用 ZGC 时无法使用需要堆对象的 AOT 缓存优化。本实现确保所有 GC（包括 ZGC）都能使用更高级的 AOT 缓存功能。

新机制通过"在线布局"逐个分配对象并使用 Access API 链接它们，使归档对象对 GC 来说与普通对象无异，从而将 GC 实现与归档机制解耦。

---

## Motivation

- 原有的堆归档机制使用 mmap 映射，与 ZGC 不兼容
- 必须在 GC 导致的尾延迟和启动导致的尾延迟之间二选一
- 需要一种 GC 无关的对象缓存机制

---

## Technical Approach

### 核心机制

1. **对象索引**: 使用对象索引（如 1, 2, 3）代替原始指针来表示对象间引用
2. **惰性物化**: 使用惰性物化和并发。当线程请求根对象时，后台 AOTThread 执行批量工作
3. **DFS 预序布局**: 对象按 DFS 预序排列，使后台线程的 DFS 遍历顺序与线性物化顺序一致
4. **批次处理**: 后台物化按批次执行，每个批次包含 N 个根及其可传递到达的对象

### 与 CDS 的集成

AOT 缓存与 CDS (Class Data Sharing) 协同工作。JDK 包含两个基线 AOT 缓存：一个包含 GC 无关的缓存对象，另一个包含 GC 特定的缓存对象。

### 字符串处理

不再转储整个字符串表。归档中的 interned 字符串被标记，加载器在解包时知道需要对字符串进行 intern。

## Configuration
### 启用 AOT 缓存
```bash
java -XX:AotCache=example.jar -Aot:cache=/path/to/cache.aot
```
### 缓存位置
默认: `$JAVA_HOME/lib/aot.cache`
可配置: `-XX:AotCachePath=/custom/path`

### 生成缓存
```bash
# 使用 jlink 生成 AOT 缓存
jlink --class-data-sharing:dump-class-list \
     --aot-cache=/path/to/cache.aot

```

## Compatibility

- **JDK 版本**: 26+
- **平台**: Linux, macOS, Windows
- **架构**: x86_64, AArch64
- **GC**: 支持所有 GC，包括 G1, ZGC, Shenandoah, Serial GC
- **COOPs**: 压缩指针关闭时使用新的流式机制，开启时使用旧的映射机制

---

## Limitations
1. **缓存大小**: 默认 1GB，可通过 `-XX:AotCacheSize` 配置
2. **缓存失效**: 当类定义变化时需要重新生成缓存
3. **安全**: 缓存文件需要保护，防止篡改
4. **动态类加载**: 不支持运行时类重定义
5. **反射修改**: 已缓存的类的反射修改可能导致不一致
