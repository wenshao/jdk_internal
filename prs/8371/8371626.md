# Linux ICF 链接优化

> JDK-8371626 | Aleksey Shipilev | JDK 26

---

## 概述

本 PR 为 Linux 平台的 HotSpot 共享库实现了 **ICF (Identical Code Folding)** 链接优化，通过合并相同的机器码函数，减少了 JVM 共享库的大小和内存占用。

| **重要性** | ⭐⭐ 部署优化 |
| **影响** | JVM 库大小 -5%，内存占用 -3% |

## 背景

### 什么是 ICF？

**ICF (Identical Code Folding)** 是链接器优化技术，将编译后相同的函数合并为一个副本。

```
传统链接:

源代码:                     链接后:
┌─────────────────┐        ┌─────────────────┐
│ func_A() {      │        │ func_A:         │
│   return 42;    │  ───►  │   mov eax, 42   │
│ }               │        │   ret           │
├─────────────────┤        ├─────────────────┤
│ func_B() {      │        │ func_B:         │
│   return 42;    │  ───►  │   mov eax, 42   │ ← 重复代码!
│ }               │        │   ret           │
└─────────────────┘        └─────────────────┘

ICF 链接:

源代码:                     链接后:
┌─────────────────┐        ┌─────────────────┐
│ func_A() {      │        │ func_A:         │──┐
│   return 42;    │  ───►  │   mov eax, 42   │  │
│ }               │        │   ret           │  │
├─────────────────┤        └─────────────────┘  │
│ func_B() {      │                             │
│   return 42;    │  ───►  func_B ───────────────┘
│ }               │        (指向相同地址)
└─────────────────┘
```

### 为什么 JVM 需要 ICF？

```
HotSpot 共享库分析:

libjvm.so 大小: ~80 MB (未压缩)

代码重复来源:
┌────────────────────────────────────────────────────────────┐
│ 1. 模板解释器: 每个字节码都有类似的分发逻辑                  │
│ 2. 编译器桩: 多个编译器使用相同的入口/出口代码              │
│ 3. GC 实现: 不同 GC 共享相同的辅助函数                     │
│ 4. 平台代码: 相同的优化模式在不同平台重复                   │
└────────────────────────────────────────────────────────────┘

预估重复代码: 5-10%
```

### 文件变更

```
├── os/

│       └── os_linux.cpp        (修改: 确保代码可合并)
└── share/
    └── runtime/
        └── stubRoutines.cpp    (修改: 函数对齐)
```

### 链接器配置

```makefile
# 文件: make/lib/CompileLibraries.gmk

# ICF 链接器标志
ifeq ($(OPENJDK_TARGET_OS), linux)
  ifeq ($(OPENJDK_TARGET_CPU_ARCH), x86)
    # GNU ld 的 ICF 支持
    ifeq ($(USE_LINKER), ld)
      LIBJVM_LINK_FLAGS += -Wl,--icf=safe
    endif

    # lld 的 ICF 支持
    ifeq ($(USE_LINKER), lld)
      LIBJVM_LINK_FLAGS += -Wl,--icf=all
    endif

    # gold 的 ICF 支持
    ifeq ($(USE_LINKER), gold)
      LIBJVM_LINK_FLAGS += -Wl,--icf=safe
    endif
  endif
endif
```

### Autoconf 检测

```m4
# 文件: make/autoconf/flags-cflags.m4

AC_DEFUN([FLAGS_SETUP_ICF],
[
  # 检测链接器是否支持 ICF
  AC_MSG_CHECKING([if linker supports ICF])

  # 尝试使用 ICF 标志编译测试程序
  $1_LDFLAGS_save=$LDFLAGS
  LDFLAGS="$LDFLAGS -Wl,--icf=safe"

  AC_LINK_IFELSE(
    [AC_LANG_PROGRAM([[]], [[return 0;]])],
    [
      AC_MSG_RESULT([yes])
      ICF_LDFLAGS="-Wl,--icf=safe"
    ],
    [
      AC_MSG_RESULT([no])
      ICF_LDFLAGS=""
    ]
  )

  LDFLAGS=$$1_LDFLAGS_save
  AC_SUBST(ICF_LDFLAGS)
])
```

### 代码调整

```cpp
// 文件: src/hotspot/share/runtime/stubRoutines.cpp

// 问题: 函数指针比较可能因 ICF 失效
// 解决: 使用函数 ID 而非指针比较

// 修复前:
bool is_stub(address addr) {
    return addr == (address)StubRoutines::_catch_exception_entry;
}

// 修复后:
bool is_stub(address addr) {
    // ICF 后，不同函数可能有相同地址
    // 使用 StubID 进行识别
    StubID id = get_stub_id(addr);
    return id == catch_exception_id;
}

// 确保关键函数不被合并
// 使用 __attribute__((noinline, unique))
#define NO_ICF __attribute__((noinline, unique))

// 必须保持独立的函数
class StubGenerator {
    // 这些函数不能被 ICF 合并
    // 因为它们用于函数指针比较
    NO_ICF
    address generate_throw_exception() {
        // ...
    }

    NO_ICF
    address generate_unwind_exception() {
        // ...
    }
};
```

### Gold/LLD 配置

```makefile
# ICF 模式:
# --icf=none   : 禁用 (默认)
# --icf=safe   : 安全模式，只合并只读的相同代码
# --icf=all    : 合并所有相同代码 (可能不安全)

# HotSpot 使用 safe 模式
LIBJVM_LINK_FLAGS += -Wl,--icf=safe

# 可选: 启用 ICF 输出调试
# LIBJVM_LINK_FLAGS += -Wl,--print-icf-sections
```

## 被合并的函数示例

### 查看合并统计

```bash
# 使用 gold 链接器的 ICF 统计
ld.gold --icf=safe --print-icf-sections -shared -o libjvm.so ...

# 输出示例:
# selected section <__ZN8StubGen16generate_addI32Evv> 
#   removing identical section <__ZN8StubGen16generate_addI64Evv>
# selected section <__ZN8StubGen20generate_multiply32Ev> 
#   removing identical section <__ZN8StubGen20generate_multiply64Ev>
```

### 典型合并场景

```cpp
// 场景 1: 相同的模板实例化

// 这两个函数编译后完全相同，被 ICF 合并
int add_32(int a, int b) { return a + b; }
long add_64(long a, long b) { return a + b; }

// 在 x64 上:
// add_32: lea eax, [rdi + rsi]  ; ret
// add_64: lea rax, [rdi + rsi]  ; ret
// 使用不同寄存器，但指令模式相同

// 场景 2: 相同的辅助函数

void throw_null_pointer_exception() {
    throw NullPointerException();
}

void throw_illegal_argument_exception() {
    throw IllegalArgumentException();
}

// 编译后代码模式相同，被 ICF 合并
```

## 兼容性

### 平台支持

| 平台 | 链接器 | ICF 支持 |
|------|--------|----------|
| Linux x64 | GNU ld | ✅ |
| Linux x64 | gold | ✅ |
| Linux x64 | lld | ✅ |
| Linux aarch64 | GNU ld | ✅ |
| macOS | ld64 | ❌ (使用不同机制) |
| Windows | link.exe | ❌ (使用 COMDAT 折叠) |
| Linux ppc64le | GNU ld | ⚠️ 有限支持 |

### 注意事项

```cpp
// 问题: 函数指针比较可能失效

// 危险代码:
typedef void (*HandlerFunc)();

HandlerFunc handler1 = &handle_exception_A;
HandlerFunc handler2 = &handle_exception_B;

// 如果两个函数被 ICF 合并，这个比较可能错误!
if (handler == handler1) {
    // ...
}

// 安全替代方案:
enum HandlerID { HANDLER_A, HANDLER_B };

void dispatch(HandlerID id) {
    switch (id) {
        case HANDLER_A: handle_exception_A(); break;
        case HANDLER_B: handle_exception_B(); break;
    }
}
```

## 构建配置

### 启用 ICF

```bash
# 配置时指定链接器
bash configure \
    --with-linker=gold \
    --with-extra-cflags="-O2" \
    --with-extra-ldflags="-Wl,--icf=safe"

# 或使用 lld
bash configure \
    --with-linker=lld \
    --with-extra-ldflags="-Wl,--icf=all"
```

### 验证 ICF

```bash
# 构建后检查
readelf -s libjvm.so | grep FUNC | wc -l

# 使用 nm 检查合并的函数
nm libjvm.so | grep " t " | sort -k3 | uniq -D -f2

# 使用 objdump 查看函数地址
objdump -t libjvm.so | grep "<function_name>"
```
