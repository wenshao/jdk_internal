## 背景

`StringBuilder` 是 Java 中最常用的字符串构建工具，`append(char[])` 是其核心方法之一。在日志格式化、JSON序列化、模板渲染等场景中被频繁调用。

**为什么重要？**
- `StringBuilder` 在 `java.base` 模块，是JDK最核心的类之一
- 每次 `String +` 操作编译后都会调用此方法
- 高频调用场景（如日志、JSON）中，此方法是热点路径

**改动位置**
```
src/java.base/share/classes/java/lang/AbstractStringBuilder.java
    └── append(char[] str)  ← 唯一改动的方法
```

## 问题分析

### 原始实现

```java
// 文件: java/lang/AbstractStringBuilder.java
// 行数: ~500-550

/**
 * Appends the string representation of the {@code char} array
 * argument to this sequence.
 */
public AbstractStringBuilder append(char[] str) {
    int len = str.length;                    // 1. 读取长度
    ensureCapacityInternal(count + len);      // 2. 确保容量足够
    System.arraycopy(str, 0, value, count, len);  // 3. 数组复制 ← 瓶颈
    count += len;                            // 4. 更新计数
    return this;
}
```

### 性能瓶颈分析

| 瓶颈点 | 原因 | 影响 |
|--------|------|------|
| **方法调用开销** | `System.arraycopy` 是 native 方法，调用需要跨越JNI边界 | 每次调用 ~5-10ns |
| **边界检查** | `arraycopy` 需要检查源/目标数组边界 | CPU分支预测失败 |
| **类型检查** | 需要验证数组类型兼容性 | 额外的类型比较 |
| **长度转换** | char[] 到 byte[] 的长度计算 (x2) | 简单但重复的计算 |

### 为什么 `System.arraycopy` 这么"重"？

```java
// System.arraycopy 的简化伪代码
public static void arraycopy(Object src, int srcPos,
                            Object dest, int destPos, int length) {
    // 1. 空指针检查
    if (src == null || dest == null) throw NPE;

    // 2. 类型检查
    if (!src.getClass().isArray() || !dest.getClass().isArray())
        throw ArrayStoreException;

    // 3. 边界检查 (4个)
    if (srcPos < 0 || destPos < 0 || length < 0 ||
        srcPos + length > Array.getLength(src) ||
        destPos + length > Array.getLength(dest))
        throw IndexOutOfBoundsException;

    // 4. 类型兼容性检查
    if (!compatibleTypes(src, dest)) throw ArrayStoreException;

    // 5. 处理重叠情况
    if (overlapping(src, srcPos, dest, destPos, length)) {
        // ... 特殊处理
    }

    // 6. 最后才执行复制
    unsafe_copy_memory(...);
}
```

对于 `StringBuilder.append(char[])`：
- 容量已通过 `ensureCapacityInternal` 确保 → 边界检查多余
- 调用者保证类型正确 → 类型检查多余
- 源和目标不会重叠 → 重叠检查多余

## 解决方案

### 核心改动

```java
// 变更后: 直接使用 Unsafe.copyMemory

public AbstractStringBuilder append(char[] str) {
    int len = str.length;
    ensureCapacityInternal(count + len);

    // 直接内存复制，绕过所有检查
    UNSAFE.copyMemory(
        str,                                    // 源数组
        CHAR_ARRAY_BASE_OFFSET,                 // 源起始位置 (常量)
        value,                                  // 目标数组
        CHAR_ARRAY_BASE_OFFSET + (count << 1),  // 目标位置 (char = 2 bytes)
        len << 1                                // 复制字节数
    );

    count += len;
    return this;
}
```

### 关键优化点

| 优化点 | 说明 |
|--------|------|
| **消除JNI边界** | `Unsafe.copyMemory` 是 `@IntrinsicCandidate`，JIT会内联 |
| **消除边界检查** | 容量已确保，无需检查 |
| **消除类型检查** | 内部调用，类型已知 |
| **常量折叠** | `CHAR_ARRAY_BASE_OFFSET` 是编译时常量 |

### 为什么安全？

| 潜在风险 | 缓解措施 |
|----------|----------|
| 数组越界 | `ensureCapacityInternal` 已确保容量 |
| 类型错误 | 内部私有方法，调用链可控 |
| 内存错误 | 只在受控环境中使用 |

## 设计决策

### Q1: 为什么不用 `Arrays.copyOf`？

> **Reviewer**: `Arrays.copyOf(char[])` 也可以，为什么选 `Unsafe`？

**A**: `Arrays.copyOf` 最终还是调用 `System.arraycopy`，没有绕过检查。而且它还需要分配新数组，这里我们只需要复制。

### Q2: 性能提升主要来自哪里？

> **Reviewer**: 是减少了边界检查，还是减少了方法调用？

**A**: 主要是 JIT 优化效果：
- `Unsafe.copyMemory` 是 `@IntrinsicCandidate`
- JIT 会将其内联为单一指令 (`rep movsq` 或类似)
- `System.arraycopy` 虽然也可能内联，但保留更多检查

### Q3: 为什么只优化 `char[]`，不优化 `String`？

> **Reviewer**: `append(String)` 调用更频繁，为什么不一起优化？

**A**:
1. `append(String)` 内部调用 `getChars()` 再调用 `append(char[])`，会间接受益
2. `String` 需要解压 `coder` 检查，逻辑更复杂
3. 遵循 "一次一个PR" 原则，后续可单独优化

public class StringBuilderAppendBenchmark {

    @Param({"10", "100", "1000"})

    @Setup

        data = new char[arraySize];
        Arrays.fill(data, 'a');

    @Benchmark
    public StringBuilder appendCharArray() {
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < 100; i++) {
            sb.append(data);
        }
        return sb;
    }
}

## 风险评估

| 风险类型 | 等级 | 缓解措施 |
|----------|------|----------|
| **内存安全** | 🟢 低 | 容量检查已确保 |
| **兼容性** | 🟢 无影响 | 纯内部实现 |
| **平台差异** | 🟡 中 | Unsafe在不同JVM实现可能不同 |
| **回归风险** | 🟢 低 | 变更范围小，测试覆盖全 |

### 平台差异说明

`Unsafe.copyMemory` 是 HotSpot 特有，但：
- AbstractStringBuilder 本身就是 HotSpot 实现
- 其他JVM实现可以保留原有 `System.arraycopy`
- 性能提升只在 HotSpot 有效，但行为一致
